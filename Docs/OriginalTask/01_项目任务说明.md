# 东软智慧云脑诊疗平台——实践任务说明

> 文档说明：本文件是「智慧云脑诊疗平台」实践项目的任务总览，面向高校师生与全栈开发学习者，涵盖项目背景、目标用户、技术栈、核心任务、拓展任务、成果要求与非功能要求。本文件是后续编码与联调工作的基线参考。

> 项目定位：本项目属于基于 Spring Boot 3 + Vue 3 前后端分离架构，融合 AI 大模型、知识图谱与大数据分析技术的全流程智能诊疗综合实践。
>
> 终端约定：
>
> 1. 患者端建议技术栈为 Vue 3，模拟手机窗口即可，使用其它技术亦可。
> 2. 医生端、管理端是 PC 版的。

---

## 项目概述

随着人工智能、大数据和医疗信息化的发展，诊疗服务正从传统的“医生经验驱动、人工操作为主”逐步走向“数据驱动、AI 辅助、全流程协同”。与此同时，企业级医疗应用开发已全面转向**前后端分离架构**，前端聚焦用户体验与交互逻辑，后端专注业务逻辑与数据服务，AI 能力作为独立的服务层被灵活集成。

传统的医疗信息化课程或项目通常聚焦于单一环节，例如电子病历录入、挂号系统设计或药品信息管理，学生虽然能够完成某一个功能模块的开发，但对一个完整的智能诊疗系统如何串联诊前、诊中、诊后各环节，如何**将传统业务功能（挂号、病历管理、处方管理）与 AI 能力（智能分诊、辅助审核、自动生成）有机融合**，以及如何基于 **Spring Boot 3 + Vue 3** 构建企业级前后端分离应用，还缺乏深入的理解。

本平台围绕“AI+诊疗”展开，采用**前后端分离架构**。后端基于 Spring Boot 3 框架，提供 RESTful API，整合 AI 模型接口、知识图谱、数据持久化与业务逻辑。前端基于 Vue 3 配合 TypeScript 与 Element Plus 组件库，实现患者端与医生端交互界面，通过 HTTP 客户端调用后端 API 完成数据展示与操作。AI 能力层则封装为独立服务或 API 接口，供后端业务层调用，实现传统业务与 AI 能力的无缝融合。

课程以**智能分诊、AI 处方审核、AI 病历生成**三大 AI 模块与传统业务模块（挂号、患者管理、处方管理、病历归档）的融合开发为主要实践对象。让学生在理解前后端分离架构的基础上，完成从**后端 API 开发、前端页面实现、AI 能力集成、前后端联调**的完整全栈训练。

平台最终希望学生理解智慧医疗系统中“患者需求感知—智能分诊匹配—在线挂号—诊疗决策支持—处方审核—病历自动生成—诊疗质量反馈”的完整闭环，同时掌握前后端分离架构下的 Vue 3 交互界面构建、Spring Boot 3 标准化 API 开发，以及传统 CRUD 操作如何与 AI 能力有机结合、AI 如何为传统业务赋能。

## 目标用户

本课程面向高校师生、全栈开发学习者，以及人工智能、大数据、医疗信息化、Java 后端、Vue 前端方向的学习者。

---

## 技术栈说明

本课程涉及的前端技术主要包括 Vue 3 框架、Vite 构建工具、Element Plus 或 Ant Design Vue UI 组件库、Axios HTTP 客户端以及 Vue Router 和 Pinia 状态管理。

后端技术栈以 Spring Boot 3 为核心框架，配合 Spring MVC 处理 Web 请求、Spring Data JPA 或 SQL-Toy 进行数据持久化操作，数据库采用 MySQL 或 PostgreSQL。AI 集成方面通过 HTTP API 调用大模型服务与知识图谱服务。API 文档使用 Knife4j（Swagger3）自动生成。前后端通过 RESTful API 加 JWT 认证的方式进行协作。

---

## 学生参与说明

根据课程设计，学生实际参与的工作围绕**后端开发、前端开发、AI 能力集成、前后端联调**四个维度展开。

- **后端维度**：学生需要基于 Spring Boot 3 开发 RESTful API，实现患者管理、挂号、处方、病历等传统业务模块，并设计调用 AI 能力的 Service 层组件。
- **前端维度**：学生需要基于 Vue 3 配合 TypeScript 开发患者端与医生端页面，实现表单交互、数据展示、路由管理，并通过 Axios 调用后端 API。
- **AI 融合维度**：学生需要理解 AI 能力如何被封装为可调用的服务接口，以及如何在传统业务流程中嵌入 AI 调用，例如挂号前调用分诊 AI、开方时调用审核 AI、就诊后调用病历生成 AI。
- **联调维度**：学生需要完成前后端联调，处理跨域、异步状态、错误边界等实际问题。

平台本身已具备基础 AI 能力接口与业务脚手架，但学生需要完成从零到一的完整功能开发，包括后端接口实现、前端页面编写、AI 集成逻辑编写，最终交付一个可运行的全栈应用。

## 核心任务

### 任务一：项目初始化与环境搭建

学生需要完成前后端开发环境的准备。

- **后端方面**：使用 Spring Initializr 创建 Spring Boot 3 项目，选择 Spring Web、Spring Data JPA、MySQL Driver、Lombok、Validation 等依赖。随后配置 application.yml 文件连接 MySQL 数据库，集成 Knife4j（Swagger3）生成 API 文档（可通过 doc.html 路径访问），配置统一跨域处理允许前端 Vue 应用调用，并编写统一的响应结构 Result 泛型类与全局异常处理器。
- **前端方面**：使用 Vite 创建 Vue 3 加 TypeScript 项目，安装 Element Plus、Axios、Vue Router、Pinia 等依赖。随后配置 Axios 封装，包括请求拦截器、响应拦截器和错误处理逻辑，并在 vite.config.ts 中配置开发环境代理以解决跨域问题。

该任务要求学生能够独立搭建企业级前后端分离项目脚手架，理解项目结构、依赖配置与基础通信机制。

### 任务二：传统业务模块开发（前后端完整实现）

学生需要完成患者管理与在线挂号两个传统业务模块的前后端开发，在不引入 AI 的情况下先跑通完整业务流程。

- **后端接口开发**：
  - **患者管理**：患者注册接口（`api/patient/register`）、患者登录接口（`api/patient/login`，登录后返回 JWT 令牌）、患者信息获取接口（`api/patient/info`）。
  - **医生管理**：医生列表接口（`api/doctor/list`）、医生详情接口（`api/doctor/detail`）。
  - **挂号管理**：创建挂号接口（`api/registration/create`）、挂号记录列表接口（`api/registration/list`）、取消挂号接口（`api/registration/cancel`）。
  - **数据持久化**：使用 JPA 或 SQL-Toy 设计患者表、医生表和挂号记录表，完成 Repository 或 Mapper 层的开发。
- **前端页面开发**：患者端需要实现注册页面、登录页面和个人中心页面；挂号页面需要实现科室筛选功能、医生列表展示、时间段选择、提交挂号操作以及我的挂号记录展示。状态管理方面，使用 Pinia 存储用户登录状态与基本信息。

该任务让学生理解前后端分离模式下传统 CRUD 业务的标准开发流程，即从数据库设计到后端接口编写，再经过 Swagger 调试，最后到前端页面实现、Axios 调用和数据渲染的完整链路。

### 任务三：AI 融合开发——诊前智能分诊模块

在传统挂号模块基础上，学生需要引入 AI 分诊能力，实现智能推荐科室与医生的功能融合。

- **后端开发**：开发智能分诊接口（路径为 `api/triage/consult`），该接口接收患者主诉文本参数；同时创建 AI 分诊服务组件（`AITriageService`），封装对 AI 分诊 API 的调用，包括请求 AI 模型和解析返回结果。在挂号流程中嵌入 AI 分诊逻辑：用户在挂号前输入症状描述，后端调用 AI 分诊 API，返回推荐的科室与医生列表供前端展示和用户选择。此外，需要将分诊记录持久化到数据库的分诊记录表中，存储主诉、推荐科室、推荐医生、患者 ID 和时间戳等信息。
- **前端开发**：开发智能分诊页面或组件，包含症状描述输入框（支持多行文本输入）和调用分诊 API 的按钮。前端需要展示 AI 推荐的科室与医生列表，并支持用户一键跳转至该医生的挂号页面。同时需要处理异步加载状态（显示 loading 效果）和 AI 调用失败时的降级提示。

该任务让学生理解如何将 AI 能力作为后端 Service 组件嵌入传统业务流程，以及前端如何优雅地展示 AI 推荐结果。AI 不再是孤立的能力，而是挂号流程的智能入口。

### 任务四：AI 融合开发——AI 处方辅助审核模块

在传统处方管理基础上，学生需要引入 AI 处方审核能力，实现用药安全的智能辅助。

- **后端开发**：开发传统处方管理接口，包括医生开方接口（`api/prescription/create`）和处方列表接口（`api/prescription/list`）。同时开发处方审核接口（`api/prescription/check`），该接口调用 AI 处方审核 API，传入处方药品列表与患者个体信息，返回用药建议、药物相互作用检测结果与风险提示。融合逻辑的设计上，医生在开方时可以点击“AI 审核”按钮调用审核接口预览审核结果，确认无误后再提交保存处方；或者在保存处方时自动触发 AI 审核并将审核结果存入审核记录表。可选地，还可以集成知识图谱查询来补充本地规则校验。
- **前端开发**：开发医生端处方开具页面，包含药品选择或录入控件、剂量输入、用法说明以及患者信息展示区域。添加“AI 辅助审核”按钮，点击后调用审核接口，在弹窗或侧边栏中展示用药建议和风险提示。医生可以根据 AI 建议调整处方内容，确认后提交保存。同时需要展示历史处方列表及对应的 AI 审核结果，如风险等级和检测项。

该任务让学生理解 AI 作为医生决策的“副驾驶”角色，以及前端如何通过异步调用在不阻塞主流程的情况下提供辅助信息。

### 任务五：AI 融合开发——AI 病历自动生成模块

在传统病历管理基础上，学生需要引入 AI 病历生成能力，实现从医患对话到结构化病历的自动生成。

- **后端开发**：开发病历生成接口（`api/medical-record/generate`），该接口接收医患对话文本或结构化问诊记录参数。后端调用大语言模型 API，将非结构化对话转换为结构化病历字段，包括主诉、现病史、既往史、体格检查、初步诊断和治疗意见。同时需要开发传统病历管理接口，包括病历保存接口（`api/medical-record/save`）、病历列表接口（`api/medical-record/list`）和病历详情接口（`api/medical-record/detail`）。融合逻辑的设计上，医生可以通过“AI 生成病历”按钮将本次就诊的对话记录发送到 AI，生成草稿病历，经医生确认或编辑后再保存。
- **前端开发**：开发医生端问诊对话记录录入页面，可以设计为模拟对话文本输入区域。添加“AI 生成病历”按钮，点击后调用生成接口，返回的结构化病历自动回填到表单的相应字段中，包括主诉、现病史、既往史等。医生可以对 AI 生成的内容进行修改，确认后提交保存。同时需要展示病历列表与详情，支持按患者姓名或 ID 查询。

该任务让学生理解大语言模型如何与前端表单协同工作，实现“AI 生成加人工确认”的人机协作模式。

### 任务六：前后端联调与综合场景演示

学生最终需要完成前后端的完整联调，并演示一个覆盖传统业务与 AI 融合的完整诊疗场景。

联调需要关注的要点包括：

- **跨域问题**：通过后端 CORS 配置或前端代理配置解决。
- **异步状态**：请求 loading 展示、错误提示、超时重试。
- **JWT 认证**：token 的存储（放在 localStorage 或 Pinia 中）与自动携带（通过 Axios 拦截器实现）。
- **降级逻辑**：AI 服务不可用时提示用户并回退到手动操作模式。

典型演示场景如下：患者注册登录，前端调用后端注册和登录接口；患者进入智能分诊页面，输入“胸痛伴气短”，调用 AI 分诊接口，系统推荐心内科医生；患者根据推荐结果完成在线挂号；医生登录医生端，查看挂号患者列表，进入问诊页面并模拟录入对话记录；医生调用 AI 生成病历，系统自动填充结构化病历表单，医生确认后保存；随后医生开具处方，点击 AI 审核按钮，系统返回用药建议与风险提示，医生确认后提交处方；最后患者端可以查看自己的挂号记录、处方记录和电子病历。

## 拓展任务

能力较强的小组可以进一步完成以下拓展任务：

1. **WebSocket 实时通知**：当 AI 审核发现高风险用药时，后端通过 WebSocket 向医生端推送实时告警。
2. **AI 流式响应**：病历生成接口支持 SSE 或 WebSocket 流式输出，前端逐字展示 AI 生成内容，提升用户体验。
3. **前端状态机优化**：使用 Pinia 模块化管理患者信息、医生信息、挂号记录、处方记录等状态，实现复杂交互流程的状态流转。
4. **AI 提示词工程优化**：在后端 Service 中设计可配置的 Prompt 模板，针对不同科室（如儿科、心内科）优化病历生成质量。
5. **双端分离部署**：前端打包为 Nginx 静态资源，后端打包为可独立运行的 Jar 包，理解生产环境的部署流程。
6. **微服务拆分（进阶任务）**：将 AI 能力层抽取为独立的微服务，使用 Spring Cloud Feign 实现服务间调用。

## 成果要求

每组学生最终需要提交一个基于 Spring Boot 3 加 Vue 3 的完整前后端分离项目。

- **后端成果**：Spring Boot 3 项目源代码、完整的 RESTful API（通过 Swagger 或 Knife4j 可以查看所有接口）、数据库设计文档（包含 ER 图和表结构说明）。
- **前端成果**：Vue 3 加 TypeScript 项目源代码、患者端与医生端主要页面的截图或录屏。
- **项目文档**：项目汇报 PPT（内容包含架构图、数据库设计、模块划分、AI 融合逻辑和核心代码讲解）；典型场景演示视频（要求完整跑通一个诊疗案例）；前后端联调测试记录（包含 Postman 测试截图和前端操作截图）。

答辩时学生需要说明：前后端分离架构的设计思路，Spring Boot 3 和 Vue 3 分别承担什么职责；传统业务模块（挂号、处方、病历）与 AI 能力如何融合，哪些环节调用了 AI；前端如何处理异步 AI 请求，如何处理 loading 与错误状态；后端如何封装 AI 调用，AI 服务不可用时有什么降级策略；以及数据库如何设计以支撑传统业务与 AI 记录。

## 非功能要求

1. **可运行性**：项目能够通过前端 `npm run dev` 命令和后端 `mvn spring-boot:run` 命令正常启动，完整业务流程可以走通。
2. **可演示性**：能够通过浏览器演示从患者注册到医生开方的完整流程，清晰展示传统业务与 AI 融合的效果。
3. **代码规范性**：后端遵循 Spring Boot 分层架构（Controller-Service-Repository 三层分离），前端遵循 Vue 3 组合式 API 规范。
4. **体验友好性**：AI 调用时前端需要展示加载状态，调用失败时需要给出明确提示，不允许出现页面卡死或无响应的情况。
5. **扩展性**：AI 能力应当封装为独立的 Service 组件，便于后续替换为不同的 AI 服务提供商或升级为本地部署模型。

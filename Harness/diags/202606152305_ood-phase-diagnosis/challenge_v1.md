## 质询要点

### 问题 1：P1-4（业务模块缺少 API/impl 分层）— 误诊

- **问题**：诊断将 patient/doctor/admin 定位为需要 api/impl 拆分的模块，并将此列为 P1 阻断性问题。但原文中这些模块是 REST 对外暴露的**终端模块**（将来直接成为微服务边界），与 common-module 和 ai（为业务模块提供 Java 门面的**共享服务模块**）架构定位根本不同。终端模块通过 Controller（REST API）对外暴露，不需要 Java 接口门面子模块。
- **原因**：诊断未区分"共享服务模块"和"终端模块"两种架构角色，将统一的结构模式强加于所有模块，且未引用 §2.2 第 283 行"三者之间不允许互相依赖"的架构约束。这会误导团队为 patient/doctor/admin 引入不必要的 api/impl 拆分。
- **建议方向**：承认 patient/doctor/admin 的扁平结构是合理设计。若需进一步验证，可补充说明跨终端模块调用在 Phase 1+ 通过 REST API 而非 Java 接口门面实现的演进路径。

### 问题 2：P2-2（Maven 聚合结构两套不可兼得）— 误判

- **问题**：诊断称根 POM 的扁平叶子模块列表与中间聚合 POM "互斥"且"中间聚合 POM 在 reactor 中永远不会被执行"。实际上这是 Maven 多模块项目的标准"部分聚合"模式，两种结构完全可共存。中间聚合 POM 为子模块提供 parent POM 引用点，既不是死代码，也不是互斥方案。
- **原因**：对 Maven 聚合(parent)与反应堆(reactor)机制的认知偏差导致将此标准实践误判为冲突，浪费团队排查时间。
- **建议方向**：接受两种结构并存的合理性，仅需在文档中注释中间聚合 POM 的用途（为子模块提供共享 parent POM，不做独立构建时无需在根 POM 的 modules 中注册）。

# 任务：评估任务二/三/四的可行性与并行方案

## 用户指令原文

> 🟩 任务二：挂号中心 + 医嘱记账后端（Phase 2 包 A + Phase 3 包 G）
> 定位：交付诊疗主流程的后端双引擎——挂号中心驱动患者侧主流程，医嘱记账驱动医生侧诊疗闭环。两者通过挂号单 ID 串联，形成"挂号→接诊→开方→记账"后端骨架。
>
> 交付清单：
>
> Phase 2 包 A 挂号中心后端：双分支挂号（门诊/检查）、取消规则引擎（时间窗内可取消/窗外引导线下）、分诊记录表、挂号状态机、REST API
> Phase 3 包 G 医嘱开立+记账：医嘱主表/明细表（药品/检查/检验三类）、收费前置单生成（不接真实收费）、与处方中心的 MedicationOrder DTO 契约预留
> 挂号单 → 医嘱的关联链路：医生接诊时基于挂号单开立医嘱
> 单元测试覆盖取消规则时间窗边界、医嘱三类开立、记账前置单生成
>
> 🟥 任务三：AI 能力底座 + 三件套首次落地（Phase 2 包 C + Phase 3 包 D-AI1/D-AI2/E + Phase 5 包 G）
> 定位：交付核心三件套（3.4.1 分诊/3.4.2 处方审核/3.4.3 病历生成）+ 辅助开方（3.4.10）的 Mock 实现，并同步搭建 AI 进阶底座骨架作为统一承载层。这是 AI 主线最重的一条，合并多个包可避免底座与业务 AI 接入的反复重构。
>
> 交付清单：
>
> Phase 5 包 G AI 进阶底座骨架：五大基础能力接口冻结 + Mock 实现（LlmProvider/ModelRouter/PromptTemplate/Experiment/AiMetricCollector）
> Phase 2 包 C 智能分诊（3.4.1）：单轮/多轮双对话、规则可配置、Mock 兜底回退科室列表；直接落地在底座上（而非独立接入，规避 Phase 5 迁移成本）
> Phase 3 包 D-AI1 处方审核（3.4.2）：风险等级差异化阻断、AI 超时回退本地规则校验打标；同样落地在底座
> Phase 3 包 D-AI2 病历生成（3.4.3）：对话转结构化病历、按科室配置规则、关键字段缺失提示补全；落地在底座
> Phase 3 包 E 辅助开方（3.4.10）：剂量阈值告警，与处方审核强耦合同步落地
>
> 🟨 任务四：医生端诊疗闭环 UI + 病历/处方中心后端（Phase 3 包 A + 包 B + 包 C）
> 交付清单：
>
> Phase 3 包 A 医生端诊疗 UI：挂号管理（叫号队列）、病人信息卡、病情录入页、病历保存/列表/详情页
> Phase 3 包 B 医生端 AI 相关 UI：占位诊断入口、开立检查入口、辅助开方页、处方审核结果展示、病历生成结果展示，全部带降级标识 UI（AI 不可用时显式标注降级路径）
> Phase 3 包 C 处方中心 + 病历中心后端：
> 处方实体/表结构、处方状态机（草稿/待审/已审/已驳回）、处方明细
> 病历实体/表结构、按科室的病历模板配置、病历版本管理（草稿/正式）
> REST API：POST /api/doctor/prescriptions、GET /api/doctor/prescriptions/{id}、POST /api/doctor/medical-records、GET /api/doctor/medical-records?patientId=。
>
> 这几个任务对于现在项目的现状（roadmap phase1ABD完成）是否可以继续？是否可以并行开发？

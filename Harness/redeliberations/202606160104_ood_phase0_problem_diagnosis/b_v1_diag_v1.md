## 0. 整体评价

诊断报告在 §8.2 AI DTO 字段缺失的识别（2.1）、Phase 0 范围越界识别（2.3）、Vite workspace 术语误用识别（2.5）、JSON 序列化约定未声明识别（2.7）上具有较高质量，逐项均给出文档位置、字段差异与影响分析，可操作性强。但在**关键能力遗漏（3.4.3、3.4.11 整段缺失）+ 输入契约多字段遗漏（3.4.5/3.4.6/3.4.7 输入仅有 1-2 项缺失被标记，其余必填字段未列出）+ 优先级排序部分倒置 + 修复方案副作用未提示**等方面存在显著质量问题，整体诊断深度不足以让执行者"直接采取行动"。建议按本报告问题清单回填后再次提交。

## 1. 关键遗漏（高严重度，直接影响产出可用性）

### 1.1 【高】3.4.3 AI 病历生成 整段遗漏

**问题描述**：诊断报告 §2.1 的"逐项不符清单"涵盖 3.4.1 / 3.4.2 / 3.4.4 / 3.4.5 / 3.4.6 / 3.4.7 / 3.4.8 / 3.4.9 / 3.4.12 共 9 项，但 **3.4.3 AI 病历生成完全未列入**。用户需求 task 1/2/3（定义矛盾、事实错误、逻辑错误）的诊断基线是全部 13 项能力，3.4.3 是"核心三件套"之一（路线图 §0 阅读指引明确指出"核心三件套（3.4.1/3.4.2/3.4.3）→ 拓展 AI"），遗漏属于基线缺失。

**所在位置**：`a_v1_diag_v1.md` §2.1 不符清单（行 33-53）；§5"本诊断范围内"声明（行 339-345）声称"§8.2 AI 13 项 DTO 字段定义与需求 3.4.x 的对齐状态"在范围内，但实际未覆盖 3.4.3。

**事实核对**（对照需求 §3.4.3 行 857-883 与 OOD §8.2 行 954-958）：

| 需求字段 | OOD §8.2 `MedicalRecordGenRequest` / `MedicalRecordGenResponse` | 差异 |
|---------|---------------------------------------------------------------|------|
| 输入 `dialogue_text`（必填，50-10000 字符） | `dialogueText: String` | 命名风格偏差，无字符数约束 |
| 输入 `patient_id`（必填） | **缺失** | 必填字段缺失 |
| 输入 `encounter_id`（可选） | **缺失** | 可选字段缺失 |
| 输入 `stream`（bool，可选，默认 false） | **缺失** | 流式/非流式分流能力未预留 |

**严重程度**：高。遗漏核心三件套之一，整体诊断覆盖率不达标。

**改进建议**：
- 在 §2.1 不符清单新增 3.4.3 行，逐字段列出上述 4 项输入缺失 + 8 项输出缺失；
- 在 §2.4"§8.2 行 907 对齐范围声明与实际不符"补充：3.4.3 的实际 OOD 定义与"对齐"承诺的偏差；
- 在 §4 修复指引方向中显式纳入 3.4.3，并指出"流式/非流式架构选型未在 OOD §8.2 中预留"是 Phase 2+ 流式输出的阻塞点。

### 1.3 【高】3.4.5 / 3.4.6 / 3.4.7 输入契约字段遗漏不完整

**问题描述**：诊断报告 §2.1 对 3.4.5 / 3.4.6 / 3.4.7 的输入字段不符清单仅各列出 1 项，但每项能力的需求输入契约均含多个必填字段被 OOD 整体遗漏。

### 1.4 【高】3.4.10 输出 PrescriptionDraft 字段语义偏差未标记

**问题描述**：OOD §8.2 `PrescriptionAssistResponse.prescriptionDraft.drugs` 复用 3.4.2 的 `PrescriptionDrug` 类型，而 `PrescriptionDrug` 字段定义（drugName/dosage/frequency/days）已在 §2.1 行 38 被诊断为"字段语义偏差（dosage→dose、days→duration）"。这一偏差同样影响 3.4.10 输出契约，但诊断报告未在 3.4.10 行标记，形成"上游已诊断 / 下游未关联"的不一致。

## 2. 优先级排序问题（中等严重度，影响执行效率）

### 2.1 【中】Mock 占位规则语义风险（§2.6）优先级应上调

**问题描述**：诊断报告将 §3.4 Mock 占位规则的"枚举字段 values()[0]"风险列为"中"优先级。但该规则应用于全部 13 项 AI 能力的 Mock 输出（OOD §3.4 行 624-629），一旦实施者按规则实现 `MockAiService`，前端 UI 开发将基于错误数据（全部 PASS / LOW / MILD 等），且**Mock 实现是 Phase 0 前端独立开发的依据**（路线图 §0.2 推荐补齐"AI 能力模块 Mock 占位"），影响范围与 2.1 的字段缺失同样广泛。

**所在位置**：`a_v1_diag_v1.md` §2.6 行 162-187；§4 修复指引（行 326-334）将其列为第 6 优先级。

**严重程度**：中。语义风险而非字段缺失，Phase 0 不一定立即暴露（前端开发初期可能未基于 Mock 分支开发），但 Phase 1+ 联调时会造成返工。

**改进建议**：
- 优先级上调为"中偏高"，与 2.1 并列；
- 在 §3.4 行 626 的"枚举字段：填充目标枚举类型的第一个枚举值"补充"或显式声明每个枚举字段的目标占位值，避免依赖 `values()[0]`"；
- 在 OOD 中补充对 `audit_result` / `risk_level` / `severity` / `urgency_hint` 等强语义枚举的目标占位值约定（如"Mock 阶段返回枚举的中间值或业务中性值"）。

### 2.2 【中】DTO 字段命名风格（§2.7）优先级应上调

**问题描述**：诊断报告将"camelCase vs snake_case 未声明 JSON 序列化约定"列为中优先级。该问题一旦进入 Phase 1+ 编码阶段，可能导致后端实际 JSON 字段为 camelCase，前端按 snake_case 调用，全字段映射错误——属于"全链路契约错位"风险。

**所在位置**：`a_v1_diag_v1.md` §2.7 行 190-208。

**严重程度**：中偏高。影响范围与 2.1 DTO 字段缺失同等广泛（前端 `packages/shared/types/` 全部类型均受影响），但修复成本极低（单一 `@JsonNaming` 配置或全局 Jackson 配置即可）。

**改进建议**：
- 优先级上调为"中偏高"；
- 修复方向补充：在 `application.yml` 增加 `spring.jackson.property-naming-strategy: SNAKE_CASE` 全局配置，无需逐 DTO 加 `@JsonProperty`；
- 与 §8.3 行 1130 的"Phase 1+ 引入 openapi-generator"协同：若 Phase 0 即统一 snake_case，Phase 1+ 自动生成的 TypeScript 类型也保持一致。

### 2.3 【中】§3.3 权限模型"完整 JPA 关系映射"越界判断需复核

**问题描述**：诊断报告 §2.3（行 87-109）将"§3.3 完整 JPA 关系映射"列为越界 Phase 0 范围。但路线图 §0.2（行 38-42）骨架必备明确包含"**数据与权限模型骨架：数据实体基类与权限模型就位**"，路线图 §0.3（行 64）也明确"搭建任务 E：数据库基础骨架搭建（Schema 基类定义 + 初始迁移脚本）"。

**判断分歧**：诊断报告的"完整 JPA 关系映射 = 模块级接口契约冻结"推断与路线图 §0.2 的"权限模型骨架就位"之间存在张力。Phase 0 需"骨架"，但骨架是否包含"完整 JPA 关系映射约定"（含 `@JoinTable` 名、`fetch = LAZY`、`cascade` 策略等），需对照路线图 Phase 0 的语义进一步澄清。

**所在位置**：`a_v1_diag_v1.md` §2.3 行 92-93、§4 行 329。

**严重程度**：中。可能造成过度回收——若按诊断建议"回收完整 JPA 关系映射"，可能影响 Phase 1 权限模型落地（Phase 1 §1.2 行 86 明确"权限矩阵：角色—岗位—功能三级权限模型落地"）。

**改进建议**：
- 在诊断时引入路线图 §0.2 "骨架必备" vs §0.4 "明确不包含"的对照表，明确"骨架必备"包含的实体定义粒度（如"实体类 + 主键 + 基本字段 vs 含完整关联映射"）；
- 若 §3.3 完整 JPA 映射超出 Phase 0 范围，建议将 §3.3 拆分：实体类骨架保留，完整 JPA 映射约定移至 Phase 1 启动前的"模块级接口契约冻结"阶段（路线图 §0.4 行 69）；
- §4 修复指引方向第 2 项"回收 OOD 中超出骨架必备粒度的内容"需精确化，避免误删 Phase 0 必须保留的实体骨架。

## 3. 修复方案潜在副作用提示缺失（中等严重度）

### 3.1 【中】§2.1 修复副作用：DTO 字段对齐会触发连锁变更

**问题描述**：诊断报告 §2.1 给出 9 项能力的字段缺失清单与 §2.2 `PatientInfo` 字段语义问题，但**未提示修复会触发的连锁变更**。

**所在位置**：`a_v1_diag_v1.md` §2.1 影响范围（行 60-63）、§2.2 影响范围（行 80-81）。

**潜在副作用清单**（执行者需知晓）：
1. **Mock 实现层 `MockAiService` 重写**：13 个方法中至少 9 个的输入/输出 DTO 需重新生成 Mock 数据，§3.4 行 624-629 的占位规则应用面扩大；
2. **TypeScript 类型同步**：`packages/shared/types/` 全部 AI 相关 interface 需重新生成（§8.3 行 1130），若 Phase 0 已生成 TS 类型，diff 工作量大；
3. **前端 UI 分支**：已基于 OOD 当前 DTO 编写的前端占位 UI 需调整字段引用（如 `TriageResponse.recommendedDept` → `recommended_departments` 数组结构）；
4. **OpenAPI 规范追溯**：§8.3 行 1130 的 springdoc-openapi 配置生成的 OpenAPI schema 与 snake_case 命名约定（见 2.7）需协同调整；
5. **需求文档 vs OOD 优先级**：若诊断建议全部回退到需求契约，修复者需面对"13 项 AI 能力全部按需求字段集实现"的工作量，与 Phase 0 "骨架" 定位可能冲突——是否仅对齐"骨架必备"的能力（3.4.1 / 3.4.2 / 3.4.3 / 3.4.12）而其他能力保留 Mock 占位，需明确。

**严重程度**：中。诊断报告给出"问题清单"但未给出"实施影响面"，执行者可能低估工作量。

**改进建议**：
- 在 §2.1 / §2.2 影响范围章节补充上述 5 项副作用；
- 在 §4 修复指引方向中区分"Phase 0 必须对齐的能力"（建议对齐 3.4.1 / 3.4.2 / 3.4.3 / 3.4.12 的核心必填字段）与"Phase 1+ 启动前冻结的能力"（其他能力保留 Mock 占位即可），避免执行者做"13 项全对齐"的过度工作。

## 4. 事实/逻辑准确性复核（低严重度，但需修正）

### 4.1 【低】§2.6 枚举风险分析可更精确

**问题描述**：诊断报告 §2.6（行 162-187）假设"`audit_result.values()[0] = PASS`"、"`risk_level.values()[0] = LOW`"等，但枚举值的顺序由定义顺序决定，并非一定为 PASS/LOW/MILD/LOW。

### 4.2 【低】§2.10 FallbackAiService 隐含假设判断准确但表述模糊

**严重程度**：低。问题真实但表述需澄清。

**改进建议**：§2.10 行 248 改为"OOD §3.4 行 634 声明 FallbackAiService'作为装饰器始终注册为 Bean'，但未显式给出其 `@Service` 注解、`implements AiService` 实现关系及 `@Primary` 装饰器标注，导致 Spring 类型注入 `List<AiService>` 时是否包含自身依赖 OOD 隐含行为推断"。

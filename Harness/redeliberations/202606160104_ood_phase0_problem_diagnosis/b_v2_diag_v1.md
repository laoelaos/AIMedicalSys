## 1. 事实错误（必须修正）

### 1.1 【高】3.4.9 `ExaminationRecommendResponse.disclaimerRequired` 实际已存在却被标记为缺失

**问题描述**：诊断报告 §2.1 不符清单针对 "3.4.9 开立检查/检验输出" 行声称 "缺 `disclaimer_required`（固定 true）"，§2.4 第 3 项再次声称 "`ExaminationRecommendResponse`（对应 3.4.9）缺 `disclaimer_required`（需求固定 true，见需求 §3.4.9 行 979）"。但经核对 OOD §8.2 行 1022-1025 原文：

```
ExaminationRecommendResponse
  - List<RecommendedItem> recommendedItems // 推荐检查/检验项目列表
  - Map<String, String> categorySummary    // 分类推荐理由摘要
  - Boolean disclaimerRequired      // 是否必须显示免责声明
```

`Boolean disclaimerRequired` 字段实际已在 OOD 中定义（仅命名为 camelCase，与 §2.7 命名风格问题相关）。诊断报告将其归入 "字段缺失" 范畴属事实判断错误。

**所在位置**：
- `a_v2_diag_v1.md` §2.1 不符清单 "3.4.9 开立检查/检验输出" 行（约第 74 行）；
- `a_v2_diag_v1.md` §2.4 第 3 项（约第 165 行）；
- `a_v2_diag_v1.md` §1 概述列举 3.4.9 在 "11 项存在字段缺失或语义偏差" 中也间接受影响（约第 15 行）。

**严重程度**：高。该错误位于 §2.1 高优先级章节的核心证据表中，会误导执行者新增冗余字段；且重复出现在 §2.4，证明并非笔误而是事实核对疏漏。

**改进建议**：
- §2.1 "3.4.9 开立检查/检验输出" 行：删除 "缺 `disclaimer_required`" 表述，可改为 "`disclaimerRequired` 命名为 camelCase，与需求 snake_case 契约不一致（详见 §2.7）"，或将该行整体移除（如认为已被 §2.7 全局命名问题覆盖）；

- §1 概述的 "11 项存在不符点" 计数若已包含 3.4.9 的本错误，应核校；
- §1.1 覆盖率核查表行 "3.4.9 AI 开立检查/检验 | 已覆盖（含 `PatientInfo` 偏差）"：若移除上述 disclaimer 错误，3.4.9 仅剩 `PatientInfo` 偏差，此标注本身可保留但 "关键问题位置" 应仅指向 §2.2。

## 2. 内部一致性问题

### 2.1 【中】§1.1 覆盖率 100% 声明与 §2.1 实际未为 3.4.13 列出独立字段偏差行不一致

**问题描述**：§1.1 覆盖率核查表 "3.4.13 AI 综合讨论结论" 行标注为 "已覆盖（含 `PatientInfo` 偏差）"，关键问题位置指向 "§2.2 / §2.4"。但 §2.2 仅讨论 `PatientInfo` 在 5 项 AI 能力间共享的过敏字段语义偏差，§2.4 讨论 OOD §8.2 行 907 对齐声明范围，二者均未真正核查 3.4.13 自身输入/输出契约的具体字段偏差。覆盖率 "100%（13 项全部覆盖）" 的声明实际上对 3.4.13 的独立字段偏差未做核查。

**具体未核查的 3.4.13 字段偏差**（举例，需要诊断报告作者复核）：
- 需求 §3.4.13 行 1060 要求 `discussion_text` "最低质量门槛 ≥ 50 字符"，OOD §8.2 行 1107 仅声明 `String discussionText` 无字符数约束；
- 需求 §3.4.13 行 1061 要求 `summary` 项数为 "0–5 项，每项 ≤ 100 字符"，OOD 行 1108 `List<String> summary` 未声明项数与项长度约束；
- 需求 §3.4.13 行 1055 要求 `encounter_id` 为 **必填**，但需求 §3.4.13 line 1057 又对 "三者中**任一**缺失即返回 `DISCUSS_AI_INPUT_INVALID`" 给出特殊校验规则；OOD §8.2 行 1104 仅声明 `String encounterId` 无必填/校验声明。

**所在位置**：
- `a_v2_diag_v1.md` §1.1 覆盖率核查表 "3.4.13" 行（约第 39 行）；
- `a_v2_diag_v1.md` §1.1 覆盖率修正值声明（约第 41 行）。

**严重程度**：中。覆盖率声明的精确性是诊断报告的关键质量指标，若 "100%" 实为对部分能力 "通过共享字段间接涉及" 的口径，与读者预期的 "逐项核查" 不一致，可能误导执行者忽略 3.4.13 自身的字段偏差。

**改进建议**：
- 选项 A：在 §2.1 不符清单新增 3.4.13 独立行，逐项核查 `discussion_text` 字符约束、`summary` 项数约束、`encounter_id` 必填校验等具体偏差；
- 选项 B：保留当前覆盖表，但将 3.4.13 标注修正为 "PatientInfo 共享偏差已覆盖；自身字段约束（字符数、项数、必填校验）未做独立核查，纳入 §5 范围外" 并相应调整覆盖率声明为 "字段集核查完整覆盖率 12/13"。

## 3. 关键遗漏

### 3.1 【中】§2.2 PatientInfo 分析未纳入 OOD `allergyDetails` 字段，影响修复者判断

**问题描述**：诊断报告 §2.2 仅分析 OOD `PatientInfo` 的 `allergies: List<String>` 与需求 `allergy_history: string` 偏差，但未提及 OOD §8.2 行 941 实际还定义了 `List<AllergyDetail> allergyDetails // 结构化过敏详情（可选，缺省时回退到 allergies）`，且 `AllergyDetail` 包含 `allergen` / `reactionType` / `severity` / `occurredAt` 四子字段（OOD §8.2 行 943-947）。

OOD 实际通过 `allergyDetails` 提供了比需求更结构化的过敏信息，并显式声明 "缺省时回退到 allergies"。这与需求 `allergy_history` 的 "中文逗号拼接 allergen 字符串" 之间存在映射机制（结构 → 字符串）。修复者若按诊断报告建议简单地将 `allergies` 改为 `allergy_history` 字符串，可能误删 `allergyDetails` 字段或忽略映射逻辑。

**所在位置**：`a_v2_diag_v1.md` §2.2 现象段（约第 104 行）；§2.1 不符清单 "3.4.4 智能诊断`PatientInfo`" 行（约第 64 行）。

**严重程度**：中。直接影响 §2.2 修复方案的准确性。

**改进建议**：
- §2.2 现象段补充 "OOD §8.2 行 941 同时定义了 `List<AllergyDetail> allergyDetails`（可选，缺省时回退到 allergies），与需求 `allergy_history: string` 之间需明确映射规则（结构化对象列表 → 中文逗号拼接字符串）"；
- §2.2 影响范围或修复建议补充 "修复时需决定：(a) 移除 `allergyDetails` 仅保留 `allergy_history` 字符串；(b) 同时保留两者并明确序列化时如何映射；(c) 重命名 `allergies` → `allergy_history` 且类型改为 string，删除 `allergyDetails`"，避免修复者面对 3 种合理选项时无依据。

### 3.2 【中】OOD `PatientInfo.name` 字段超出需求未提及

**问题描述**：OOD §8.2 行 936 `PatientInfo` 包含 `String name // 患者姓名` 字段。但需求文档 §3.4.2 / §3.4.4 / §3.4.9 / §3.4.10 / §3.4.13 的 `patient_info` 契约均未包含 `name` 字段（仅 `patient_id` / `age` / `gender` / `allergy_history` / `comorbidities`）。诊断报告 §2.2 仅讨论字段缺失/语义偏差，未提及该 "OOD 超出需求的字段" 问题。

**所在位置**：`a_v2_diag_v1.md` §2.2（约第 102-115 行）。

**严重程度**：中。修复方向若仅 "对齐需求字段集"，可能错过 OOD 已超出需求的 `name` 字段；若简单保留，则前端可能依赖 `name` 渲染但需求未要求后端 AI 契约中传递；若简单删除，则需要复核是否有 OOD 内部其他模块依赖。

**改进建议**：§2.2 不符清单或现象段补充 "OOD `PatientInfo` 包含 `String name` 字段（行 936），但需求 3.4.x 各能力的 `patient_info` 契约均未定义 `name`。修复时需决定保留/删除，并核查其他模块依赖"。

## 4. 可操作性细节问题

### 4.1 【低】§2.3 "回收对象 vs 保留对象" 清单中 JPA 关联映射的具体边界不明确

**问题描述**：§2.3 末尾 "回收对象 vs 保留对象清单" 中 "保留：实体类骨架（含主键 + 基本字段，不含 JPA 关联映射）" 与 "回收：完整 JPA 关联映射（`@JoinTable` / `@JoinColumn` / `fetch` / `cascade` / `orphanRemoval` 显式声明）"。但执行者面对 OOD §3.3 中 User-Role-Post-Function 的多对多 / 一对多关系时，会面临以下不明确：

1. 若不保留 `@ManyToMany` 等关联注解，User 与 Role 之间如何在 Phase 0 表示关联？仅声明 `Long roleId` 外键？还是完全省略？
2. 关联表 `user_role` / `user_post` / `post_function` 的 schema 是否需要在 Phase 0 初始化迁移脚本中创建？还是延迟到 Phase 1？
3. 路线图 §0.3 搭建任务 E "数据库基础骨架搭建（Schema 基类定义 + 初始迁移脚本）" 是否包含关联表 schema？

执行者按报告建议 "回收完整 JPA 关联映射" 时，可能误删关联表本身，导致 Phase 1 启动时需要追加迁移脚本（带来数据迁移风险）。

**所在位置**：`a_v2_diag_v1.md` §2.3 清单（约第 150-153 行）。

**严重程度**：低。属可操作性细化建议，不影响诊断的核心结论。

**改进建议**：§2.3 清单细化为三档：
- **保留**：实体类 + 主键 + 基本字段 + 关联表 schema（在迁移脚本中创建 user_role 等关联表）+ 外键字段；
- **延后冻结**：`@JoinTable` 命名、`fetch` 策略、`cascade` 策略、`orphanRemoval` 等行为约定，Phase 1 启动前明确；
- **回收**：当前 OOD 中过早冻结的 `FetchType.LAZY` 显式声明、`orphanRemoval = false` 等具体策略约束。

### 4.2 【低】§2.10 第 3 项判断与 OOD 设计意图存在偏差

**问题描述**：§2.10 第 3 项 "OOD 未明示 FallbackAiService 自身的 `@ConditionalOnProperty` 配置（仅 §3.4 行 632-633 描述 MockAiService 的条件）"。但 OOD §3.4 行 634 已明确声明 "FallbackAiService：作为装饰器**始终注册为 Bean**"。"始终注册" 意味着无任何条件注解（无 `@ConditionalOnProperty`），是设计意图而非遗漏。

报告将其列为 "隐含假设未明示" 可能造成执行者误增 `@ConditionalOnProperty` 条件，反而违背 OOD 设计意图。

**所在位置**：`a_v2_diag_v1.md` §2.10 第 3 项（约第 298 行）。

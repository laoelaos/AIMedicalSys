## 发现问题

### 问题 1：[中] T6 的"独立于修复批"断言与 T2 行为变更存在矛盾

**所在位置**：`a_v3_diag_v2.md` 第 553 行（任务依赖图），修复批描述第 492 行

**问题描述**：
依赖图将 T5/T6/T7 一并标注为"与被测代码修复批独立，可先行或并行"。但 T6（缺失异常刷新检测单元测试）的被测对象正是 `AuthServiceImpl.refresh()` 第 270-283 行的异常刷新检测逻辑；而 T2（位于修复批 1）将**更改这一逻辑的行为**——从 log-only 改为拒绝请求并强制清除本地 token。若 T6 测试在 T2 之前编写，将验证旧行为（仅告警不阻断），T2 实施后需大量返工。

T5（UserFacadeImplTest mock 覆盖问题）和 T7（LoggingSecurityAuditLogger 降级测试）确实独立，但 T6 与 T2 之间存在实质性行为级依赖，不应笼统归入"独立"行列。

**严重程度**：中

**改进建议**：
在依赖图中将 T6 从"独立"组中分离，标注 `T6 ──→ T2（测试对象行为变更依赖）`；或至少补充说明："T6 的检测条件部分可先行编写，但响应行为验证需在 T2 实施后更新"。

### 问题 2：[中] T11→T17 依赖箭头缺乏代码级依据

**所在位置**：`a_v3_diag_v2.md` 第 549 行

**问题描述**：
依赖图标注 `T11 ──→ T17`，解释为"同属 Filter/Handler 层"。但经核实：
- T11 修改对象：`AuthServiceImpl.getCurrentUser()`（Service 层），与 Filter/Handler 无调用关系
- T17 修改对象：`RestAuthenticationEntryPoint`、`RestAccessDeniedHandler`，属 Spring Security 的认证/授权异常出口
- `AuthServiceImpl.java` 中不存在对 `RestAuthenticationEntryPoint` 或 `RestAccessDeniedHandler` 的任何引用，反之亦然

"同属 Filter/Handler 层"描述的是架构层次上的归属相似性，而非代码层面的依赖或执行时序依赖。此箭头可能导致实施者错误地串行化两个完全独立的修复任务。

**严重程度**：中

**改进建议**：
删除 T11→T17 箭头（两者完全独立，可并行实施）。如需保留层次关联信息，改用无箭头的虚线或分组标注。

### 问题 3：[低] T18 内存估算术语"堆外"使用错误

**所在位置**：`a_v3_diag_v2.md` 第 276 行

**问题描述**：
原文："约 10MB 堆外"。`refreshTimestamps` 声明为 `ConcurrentHashMap<Long, Deque<Long>>`（`AuthServiceImpl.java:68`），所有数据存储在 JVM 堆内内存中，而非堆外（off-heap）。"堆外"特指通过 `DirectByteBuffer`、`Unsafe.allocateMemory` 等方式分配的本机内存，与 `ConcurrentHashMap` 的分配方式不符。

**严重程度**：低（术语不准确，但不影响诊断结论）

**改进建议**：
将"堆外"改为"堆内"或"内存"。

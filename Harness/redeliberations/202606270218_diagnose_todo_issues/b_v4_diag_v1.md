### 问题 2：[一般] T17 编码修复方案缺少技术路线指引

**所在位置**：第 497 行（修复批 2）

**问题描述**：
> "编码同步更新 Result.fail 调用"

此描述过于笼统。`RestAuthenticationEntryPoint` 和 `RestAccessDeniedHandler` 是 Spring Security 配置层面的处理器，它们的代码路径与 `GlobalExceptionHandler`（`@ControllerAdvice`）处于不同层次。要使这两个处理器经过消息插值管线，需要明确给出技术方案选择：

- 方案 A：为 `GlobalExceptionHandler.formatMessage()` 抽取公共静态工具方法，在 Security handler 中直接调用
- 方案 B：在 Security handler 中独立维护插值逻辑（复制代码，不推荐）
- 方案 C：将插值逻辑封装为 `@Component` bean，Security handler 注入使用

当前描述让执行者自行承担设计决策风险，不同开发者可能选择不同方案，导致实现风格不一致或引入代码重复。

**改进建议**：
给出推荐方案（如方案 C：抽取 `MessageInterpolator` 组件，`GlobalExceptionHandler` 和 Security handler 共同注入使用），并说明理由（避免代码重复、统一维护入口、可独立单元测试）。

### 问题 3：[一般] T13 根因分析未辨识 OOD 文本内在模糊性

**所在位置**：第 198-206 行（T13 根因分析）

**问题描述**：
报告将 T13（全局锁 vs 每 IP 独立锁）完全归类为"编码与 OOD 不一致"，但 OOD 4.1 节本身存在语义张力：

- 第 433 行："确保**每个 IP 的窗口对象独立加锁**"——暗示每 key 独立锁定
- 第 444 行："**ReentrantLock** 保护跨窗口操作的原子性"——暗示一个全局锁

OOD 文本未明确说明 ReentrantLock 的作用域是"每次操作创建临时锁"还是"类级别的全局唯一锁"，也未解释在 `ConcurrentHashMap.compute` 已提供 per-key 原子性的前提下，额外的 ReentrantLock 究竟保护什么"跨窗口操作"。

当前报告将此问题完全归责于编码实现，但 OOD 文本本身的模糊性是编码偏差的诱因之一。如果 OOD 文本足够清晰（例如"SlidingWindowCounter 内部不设额外锁，依赖 compute 的原子性"），编码者可能不会引入全局锁。

**改进建议**：
补充 OOD 层面的局限性分析：OOD 4.1 节对锁粒度的描述存在歧义，应建议在修复 T13 时同步澄清 OOD 文本，明确 SlidingWindowCounter 的锁策略（是否完全依赖 `ConcurrentHashMap.compute` 的原子性，还是每 key 独立 ReentrantLock）。

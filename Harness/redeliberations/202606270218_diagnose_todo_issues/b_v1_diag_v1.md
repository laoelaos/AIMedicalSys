### 问题 5：[中] T12 修复建议遗漏配套的 Base64 解码器修改

- **位置**：T12 第 185-193 行
- **问题描述**：报告指出现有代码（JwtTokenProvider.java:37-38）使用标准 Base64 字符集正则校验但 OOD 要求 URL-safe 字符集，建议"改为 URL-safe 字符集正则"。然而第 42 行 `Base64.getDecoder().decode(secret)` 使用的是标准 Base64 解码器。如果仅改正则而不改解码器，URL-safe 编码的密钥（含 `-` 和 `_`）会通过正则校验但被 `Base64.getDecoder()` 拒绝（URL-safe Base64 中的 `-` 和 `_` 是标准 Base64 的非法字符）。
- **严重程度**：中。修复建议不完整，执行者若仅照做正则部分，会遇到运行时异常。
- **改进建议**：在修复建议中补充说明需同时将第 42 行从 `Base64.getDecoder()` 改为 `Base64.getUrlDecoder()`，或者确认实际使用的密钥格式并统一采用一种编码方案。

### 问题 6：[中] T13 修复建议未讨论 cleanup 方法的同步依赖

- **位置**：T13 第 195-203 行，修复批 4 第 495-500 行
- **问题描述**：建议将全局 `ReentrantLock` 改为"`ConcurrentHashMap.compute` 内原子操作 + 每个 key 独立锁"。但 `SlidingWindowCounter` 中有一个 `cleanup()` 方法（第 59-66 行）在定时任务中遍历并移除空窗口，该方法也受全局锁保护。移除全局锁后，cleanup 中的 `windows.entrySet().removeIf(...)` 与业务路径的 `windows.compute(key, ...)` 之间将失去同步保证，可能产生竞争（如 cleanup 与活跃的 compute 同时操作同一 entry）。
- **严重程度**：中。修复建议遗漏关键的并发安全边界，可能引入新的 bug。
- **改进建议**：在修复建议中补充说明 cleanup 方法的同步策略：可以采用 `ConcurrentHashMap` 的 `values().removeIf(...)`（基于迭代器的弱一致性）替代全局锁下的 `entrySet().removeIf(...)`，或者保持专用清理锁。

## 问题清单

### 问题一：T13 根因分类与汇总表之间存在事实矛盾

**问题描述**：T13 正文分析（第 198-211 行）明确指出：

> "根因归属：**OOD 文本模糊性导致实现方向不明确（主因）**，编码实现跟随了其中一种解读但未质疑矛盾（次因）"

即 OOD 文档内在矛盾是**主要**根因。但在汇总表中：
- "OOD 文档缺陷"汇总表（第 447-451 行）**未包含** T13
- "编码缺陷导致的问题"汇总表（第 454-468 行）**将 T13 列为编码缺陷**，根因简述为"全局锁而非每 IP 独立锁"，完全未反映 OOD 为主因的事实

这是报告内部的事实性矛盾——正文判定 OOD 是主因，但汇总表将其归为纯编码缺陷。

**所在位置**：第 198-211 行（T13 正文） vs 第 447-451 行（OOD 缺陷汇总） vs 第 454-468 行（编码缺陷汇总）

**严重程度**：一般

**改进建议**：统一分类方案。可选方案：a) 在"OOD 文档缺陷"汇总表中新增 T13 条目，同时在"编码缺陷"汇总表保留 T13 但根因简述改为"OOD 文本模糊 -> 编码跟随"，并注明双归属；b) 仅纳入 OOD 汇总表，编码汇总表移除 T13，因为主因是 OOD。

### 问题二：修复批次遗漏"存量测试同步更新"指引，可能导致执行后测试中断

**问题描述**：修复批 1-5（第 500-533 行）详细描述了代码和 OOD 文档的修改方案，但完全未提及**因行为变更导致现有测试失效**的风险。经验证的代码事实：

- **T1 修复**（Access Token 补 type claim）：`JwtTokenProviderTest.java` 中生成 Access Token 的测试用例将产生与当前不同的 token 内容，现有断言需要同步更新。
- **T9 修复**（过期 token 记录审计日志）：`AuthServiceTest.java:535` 的 `logout_shouldNotAuditWhenTokenInvalid()` 专门验证"无效 token 不记日志"——而 T9 修复恰好要改变此行为，该测试**必须重写或删除**。
- **T11 修复**（getCurrentUser 签名变更）：`AuthServiceTest.java:565` 的 `getCurrentUser_shouldSucceed()` 使用 `getCurrentUser("token")` 签名，若方法签名变更则编译失败。
- **T15 修复**（record* 添加窗口过期防御）：`LoginAttemptTrackerTest` 中的 record* 方法调用预期需要调整。

报告将测试工作全部归入"修复批 6：测试增强与文档对齐"，但该批次仅关注新增测试，未覆盖存量测试适配需求。

**所在位置**：第 500-533 行（修复批 1-5），第 535-556 行（修复批 6）

**严重程度**：一般

**改进建议**：为每个涉及行为变更的修复批次（1-5）增加"受影响存量测试"子项，明确列出需要同步更新的测试文件及变更性质（更新断言/移除旧测试/新增场景）。避免执行者完成代码修改后才发现测试失败。

### 问题三：修复批 6 分组方案缺乏可操作的具体映射

**问题描述**：修复批 6 涵盖 T3-T7、T20-T34（T22 除外）共 18 项任务。报告第 556 行仅给出"建议切分为 5-6 个子任务，每个子任务覆盖 3-5 个测试文件"的泛泛建议，但：

- 未提供具体的 task-to-group 分配映射（哪些任务归入同一子任务）

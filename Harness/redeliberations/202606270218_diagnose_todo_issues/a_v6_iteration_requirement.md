T13 正文分析（第 198-211 行）明确指出根因归属为 **"OOD 文本模糊性导致实现方向不明确（主因）"**，编码实现跟随了其中一种解读但未质疑矛盾（次因）。但在汇总表中：

**严重程度：一般**

修复批 1-5（第 500-533 行）详细描述了代码和 OOD 文档的修改方案，但完全未提及因行为变更导致现有测试失效的风险。经验证的代码事实：

- **T9 修复**（过期 token 记录审计日志）：`AuthServiceTest.java:535` 的 `logout_shouldNotAuditWhenTokenInvalid()` 专门验证"无效 token 不记日志"——而 T9 修复恰好要改变此行为，该测试必须重写或删除。
- **T11 修复**（getCurrentUser 签名变更）：`AuthServiceTest.java:565` 的 `getCurrentUser_shouldSucceed()` 使用 `getCurrentUser("token")` 签名，若方法签名变更则编译失败。
- **T15 修复**（record* 添加窗口过期防御）：`LoginAttemptTrackerTest` 中的 record* 方法调用预期需要调整。

**严重程度：一般**
**改进建议**：为每个涉及行为变更的修复批次（1-5）增加"受影响存量测试"子项，明确列出需要同步更新的测试文件及变更性质（更新断言/移除旧测试/新增场景）。避免执行者完成代码修改后才发现测试失败。

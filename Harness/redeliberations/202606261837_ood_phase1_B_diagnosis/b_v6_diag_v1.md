## 问题 1（严重）：根因分析对需求第 2 项「OOD 文档维度」的响应不充分

**所在位置：** 概述 — OOD 设计覆盖度评价段落；逐项诊断 — T1~T27 各节的根因分类

**问题描述：** 需求第 2 项定位目标要求根因分析覆盖 4 类 OOD 文档问题：**矛盾、偏差、不完善、错误**。但产出仅对 3 项（T3/T8/T9）提及了 OOD「粒度不足」（属于"不完善"一类），其余 21 项纯实现偏差项未做任何 OOD 维度的反事实分析。读者无法判断：这 21 项中有哪些项如果 OOD 文档更完整/更精确，实现者本可避免偏差；又有哪些项确实与 OOD 文档质量无关。

例如：
- T6（路径 `/me` vs `/profile`）— OOD 4.4 和 6.1 均明确定义路径为 `/profile`，但实现侧为何仍使用了 `/me`？是否存在 OOD 文档变更后未同步更新的问题？或 OOD 与原型代码交接时存在契约丢失？
- T19（PUT vs PATCH）— OOD 定义 PATCH 但实现用了 PUT，这与 T6 同为 API 契约偏差但 OOD 是否有足够醒目的规范标注？
- T22（`component` 字段缺失）— `sys_function` 表有此列但实体未映射，如果 OOD 在 2.1 目录结构或 5.2 节中明确标注了 PermissionFunction 实体应包含的所有字段映射，是否可避免遗漏？

**严重程度：** 严重 — 直接偏离第 2 项定位目标的核心要求

**改进建议：** 新增一个「OOD 文档质量评估」独立章节（或在模式识别章节中扩展），对全部 26 项做 OOD 维度的反事实分析：若 OOD 文档对某项的具体要求更加精确/完整，实现偏差是否仍会发生？以此区分"OOD 可预防的偏差"和"纯实现侧失误"。对 T6/T19/T22 等 OOD 已有明确规定的偏差项，至少应在正文中确认 OOD 定义无误（即实现侧完全未遵循已有规定）。

## 问题 2（中）：T25 前端契约风险评估存在事实偏差

**所在位置：** T25 — 风险与注意事项

**问题描述：** 产出称「若前端已有 429 处理逻辑（如显示'请稍后重试'）则可正常适配」，暗示前端存在 429 专用处理路径。事实是（验证自 `frontend/packages/shared/src/api/index.ts:42-47`），前端 API 响应拦截器仅对 HTTP 401 和 403 有专用处理逻辑，429 状态码会走通用分支（`index.ts:48-53`），用户将看到类似"请求失败（429）"的通用错误信息而非业务错误消息。

**严重程度：** 中 — 事实偏差可能导致执行者低估或错误判断前端适配的改造成本

**改进建议：** 将风险评估修改为：「前端响应拦截器当前仅处理 401/403 状态码（见 `frontend/packages/shared/src/api/index.ts:42-47`），无 429 专用处理。后端切换 429 后，前端通用分支（第 48-53 行）将提取 `response.data.code` 和 `response.data.message` 作为业务错误处理，因此业务错误消息仍可送达用户；但 HTTP 状态码变化会影响前端的分支路由逻辑（如 token 过期的 401 自动跳转登录），需确认限流/锁定场景是否需要特殊 UI 处理。」

## 问题 3（中）：T9 修改建议存在逻辑矛盾

**所在位置：** T9 — 修改建议

**问题描述：** 产出建议将 `UserFacadeImpl` 委托给 `UserConverter`，但同时明确列出了 `UserConverter` 的三项缺陷（无 Role::getEnabled 过滤、sort 字段 NPE 风险、无 PermissionFunction::getEnabled 过滤），而 `UserFacadeImpl` 的对应实现在这三处均正确。若直接委托而不先修复 UserConverter，将**丢失已正确的 null-safe 行为**——这与修改建议的目标（消除重复**且**保证行为正确）自相矛盾。

**严重程度：** 中 — 若执行者按字面意思直接委托，将引入新缺陷

**改进建议：** 在修改建议中明确修复顺序：（1）先以 `UserFacadeImpl` 的实现为准，修正 `UserConverter` 的三处缺陷；（2）再将 `UserFacadeImpl.toUserInfoResponse()` 委托给 `UserConverter`。或者在"风险与注意事项"中增加警告：「不可在未修复 UserConverter 的前提下直接委托，否则 null 安全行为将退化」。

## 问题 4（中）：T22 被归入批次 2「编码规范对齐」的风险被低估

**所在位置：** 修复执行策略 — 批次 2

**问题描述：** 批次 2 被描述为"纯文本修改/枚举值替换/字段补充，无运行时行为变更风险，可并行修复"。但 T22（`PermissionFunction` 实体新增 `component` 字段）涉及 JPA 实体类的 schema 变更，可能产生以下非纯文本风险：
- 若 `spring.jpa.hibernate.ddl-auto` 为 `validate`，新增字段会导致启动失败（需手动执行 DDL）
- 若 `ddl-auto` 为 `update`，Hibernate 自动修改表结构可能在生产环境产生锁表风险
- 新增字段后需确认 `data.sql` 或 `data-dev.sql` 中的初始数据是否需要补充该字段
- 若其他查询（如 Specification/QueryDSL）引用了 `PermissionFunction` 的全部字段，新增字段可能影响查询结果

与 T4/T27（纯消息文本修改）和 T20（枚举值替换）的技术风险不在同一量级。

**严重程度：** 中 — 批次划分不当可能导致执行者的工作量估算和并行策略偏差

**改进建议：** 将 T22 移至批次 6（基础设施修复），或在批次 2 说明中将 T22 标注为"含实体变更，需确认 DDL 策略"，与 T4/T27/T20/T14 保持风险分组区分。

## 问题 5（低）：T11 备选方案忽略 Spring Security 异常包装行为的验证成本

**所在位置：** T11 — 修改建议备选方案 B

**问题描述：** 产出推荐方案 B（新增 `AccountDisabledAuthenticationException`，在 `RestAuthenticationEntryPoint` 通过 `instanceof` 判断），但未评估在 Spring Security Filter Chain 中抛出自定义 AuthenticationException 的集成测试成本。`JwtAuthenticationFilter` 在 Filter Chain 内部抛出异常后，Spring Security 的 `ExceptionTranslationFilter` 和 `FilterChainProxy` 会实施特定的异常包装逻辑，模拟该行为的单元测试往往需要 `SecurityMockServerRequest` 或 `@WithMockUser` 等较重的测试基础设施，而非简单的 Mockito stub。产出在"验证方式"中仅写了"编写 RestAuthenticationEntryPointTest"，低估了实际测试复杂度。

**严重程度：** 低 — 不影响方案正确性，但执行者可能低估测试工作量

**改进建议：** 在验证方式中补充测试路径提示：「因 JwtAuthenticationFilter 运行在 Spring Security Filter Chain 内部，集成测试建议使用 `@WebMvcTest` + `@Import` + MockMvc 模拟请求从 Filter Chain 入口到 AuthenticationEntryPoint 的完整链路，而非仅对 RestAuthenticationEntryPoint 做单元测试。」

## 问题 6（低）：T23 并发场景分析缺失

**所在位置：** T23 — 分析及优先级判定

**问题描述：** 产出基于单请求场景（3-7 次 SQL 查询）将 T23 从 P0 降为 P1，但未考虑并发场景下的复合影响。在管理后台典型场景下（如 50 用户并发加载菜单），单请求 3-7 次查询放大为 150-350 次查询对数据库连接池的瞬时压力。虽然 HikariCP 默认连接池可应对此负载，但若同时存在其他 N+1 查询，连接池可能成为瓶颈。

**严重程度：** 低 — 不影响 P1 优先级判定的合理性，仅为分析完整性的补充

**改进建议：** 在分析段落末尾补充：「并发场景下（如 50 并发），3-7 次/请求放大为 150-350 次/秒的数据库查询。当前数据量级下连接池压力可控，但在菜单-权限体系扩展后（岗位数>10 或功能组数>50）应重新评估。」

## 问题 7（低）：修复执行策略批次 6 中存在未声明的隐含依赖

**所在位置：** 修复执行策略 — 批次 6 说明

**问题描述：** 批次 6（基础设施修复）同时包含 T8（安全审计日志）和 T23（N+1 查询），两者表面无交叉依赖。但若 T23 的 `MenuServiceImpl.getUserMenuTree()` 修复需要访问用户-岗位-功能数据，而 T8 引入的 `SecurityAuditLogger` 可能在日志基础设施层面与 `UserFacade` 或 `MenuServiceImpl` 的数据访问路径产生间接交互（例如审计日志需要记录"菜单数据访问"事件）。此外，T8 新建的 `SecurityAuditLogger` 与 T9 要合并的 `UserConverter`/`UserFacadeImpl` 是否存在日志记录逻辑的交叉，也未在批次 6 内分析。批次 2/4/5/6 被标为"可并行"，但 T8 和 T23 的平行修改可能导致后续需要适配接口。

**严重程度：** 低 — 实际冲突概率低，但显式标注可减少执行者的协调成本

**改进建议：** 在批次 6 说明追加：「T8（SecurityAuditLogger）引入的日志接口可能被 T23 的 MenuServiceImpl 或 T9 的 UserFacadeImpl 引用，两者虽非硬性依赖，但建议 T8 先行确认接口契约，T23/T9 在集成阶段检查是否需要接入审计日志。」

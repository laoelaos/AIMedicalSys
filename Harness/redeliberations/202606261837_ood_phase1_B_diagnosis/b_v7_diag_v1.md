## 发现的问题

### 问题 1：T21 独立修复声明与批次策略矛盾

- **位置**：T21 节「风险与注意事项」末行（"与 T19 独立，可各自单独修复"）vs 批次策略「批次 5」描述（"T19 和 T21 均修改 MenuController.java，同理需顺序修改"）
- **严重程度**：中
- **问题描述**：T21 的诊断备注称 T19 和 T21"可各自单独修复"，但批次策略明确要求两者"顺序修改"（均修改 MenuController.java 的同一方法 `update()`）。两个相邻页面存在矛盾的修复指导：若执行者按 T21 注释独立修复，会导致批次 5 的 `MenuController.java` 修改冲突；若按批次策略修复则无法理解为何备注说"可各自单独修复"。
- **改进建议**：统一两处表述。T21 的「风险与注意事项」应改为与批次策略一致的表述，例如："与 T19 均修改 MenuController.update() 方法，建议一次提交中顺序修改以避免合并冲突，参见批次策略批次 5。"

### 问题 2：T10 修改建议与自身职责划分描述自相矛盾

- **位置**：T10 节「修改建议」第 2 段（`secret.length() < 32` → `Base64.getDecoder().decode(secret).length < 32`）vs 「风险与注意事项」（"JwtConfig.validate() 负责配置层校验……限定在无需 SecretKey 解码的浅层检查"）
- **严重程度**：中
- **问题描述**：该节「修改建议」推荐 T10 将 JwtConfig 的校验改为检查解码后字节长度（即执行 `Base64.getDecoder().decode()`），但紧随其后的「风险与注意事项」明确要求"JwtConfig.validate() 的校验应限定在无需 SecretKey 解码的浅层检查"。两条指导直接矛盾——前者让 JwtConfig 做解码检查，后者禁止 JwtConfig 做解码检查。执行者无法判断应遵循哪条。此问题在 v7 修订中也未纠正（Q 质询未覆盖此矛盾）。
- **改进建议**：选择一个方向并删除另一方向：
  - 若采纳浅层检查方案（与风险描述一致）：T10 的修改建议应改为"保留 `secret.length() < 32` 但将阈值修正为 Base64 编码后的合理最小值（如 44 字符对应 32 字节），同时注明深层的解码后字节长度检查由 T1 的 JwtTokenProvider.init() 完成"。
  - 若采纳解码检查方案（与修改建议一致）：删除风险描述中限制 JwtConfig 做浅层检查的段落，并同步协调 T1 中的重叠校验逻辑。

### 问题 3：T3 与 T16 的修复-测试依赖描述存在模糊地带

- **位置**：T3「修改建议」（多方案并列）vs T16「修改建议」（测试策略与 T3 的推荐方案不一致）
- **严重程度**：中
- **问题描述**：T3 的修改建议提供了三种插值方案（GlobalExceptionHandler / Result / BusinessException）并推荐"在 GlobalExceptionHandler 中完成插值"。但 T16 的测试策略描述为"AuthServiceTest 中增加锁场景的消息内容断言"——若 T3 按推荐方案在 GlobalExceptionHandler 中插值，BusinessException 中仍保留模板原文，AuthServiceTest 断言 `e.getMessage()` 将获取到模板原文而非插值结果，导致 T16 断言与 T3 行为不一致。实际插值结果只能在 HTTP Response 层面验证（集成测试），而非 AuthServiceTest 的单元测试层面。
- **改进建议**：
  1. T3 应明确锁定推荐方案为唯一修改路径，删除或降级其他备选方案。
  2. T16 的测试策略应调整为：AuthServiceTest 验证 args 参数正确传递（`e.getArgs()` 含预期插值参数），GlobalExceptionHandlerTest（集成测试）验证 HTTP Response message 为插值后文本。

### 问题 4：T14 缺陷分类与功能正确性结论不一致

- **位置**：T14 节「判定」（"真实缺陷"）vs「分析」（"功能正确，不存在数据竞争或并发问题……锁缺失不影响功能正确性"）
- **严重程度**：低
- **问题描述**：T14 同时声明"真实缺陷"和"功能正确"。ConcurrentHashMap.compute() 已提供原子性保证，锁的缺失不影响正确性且不引入数据竞争。严格来说这是设计约定（OOD 要求 ReentrantLock）与实现不一致，而非功能缺陷。当前分类可能导致执行者优先修复 T14（P3 中优先级较高），而实际价值是纯设计合规性调整。
- **改进建议**：在「判定」中补充说明"功能正确，属设计约定不一致"的定性，或修改为"设计约定偏差（功能正确）"。修改建议中明确推荐一个方案（建议删除 lock 字段而非添加锁），避免执行者做无谓的加锁修改引入无意义的锁竞争开销。

### 问题 5：批次执行流程图示易引起误解

- **位置**：「修复执行策略」第 105 行：
  ```
  批次 1 → 批次 3 → 批次 2, 4-6（可并行，但与批次 3 无依赖关系，批次 3 修复后其他批次可继续）→ 批次 7
  ```
- **严重程度**：低
- **问题描述**：箭头序列图示 `批次 1 → 批次 3 → 批次 2, 4-6` 暗示批次 2/4-6 在批次 3 之后执行，但文字说明"可并行"且"与批次 3 无依赖关系"表明实际执行顺序并非如此。执行者若按图示理解，会不必要地等待批次 3 完成才开始批次 2/4-6。这种图示/文字的不一致在执行层面上会造成排期决策失误。
- **改进建议**：将序列图示改为更清晰的并行表示，如：
  ```
  批次 1 → 批次 3 ─┐
                    ├→ 批次 2, 4-6（并行）→ 批次 7
          其他批次 ─┘
  ```
  或明确标注"批次 2/4-6 可在批次 1 完成后立即启动（与批次 3 并行）"。

### 问题 6：T8 备选方案未进行实际项目验证

- **位置**：T8「修改建议」备选方案段落
- **严重程度**：低
- **问题描述**：备选方案说"若项目已有 AuditLogger……可检查复用"，但实际项目中不存在任何安全审计日志框架（已确认：无 `AuditLogger`、`AuditService`、`SecurityAudit`、`AuditAspect` 等组件，仅有 JPA 的 `@EnableJpaAuditing` 用于实体时间戳审计）。此备选方案不仅多余，而且会引导执行者花费时间搜索不存在的框架。此外，JPA 的 `@EnableJpaAuditing` 也可用于记录实体变更的 `@CreatedDate`/`@LastModifiedDate`，但与安全审计日志（操作人、操作行为、结果、时间戳的不可篡改记录）是完全不同的关注点，不应混淆。
- **改进建议**：删除备选方案段落。直接将默认行动路径提升为唯一路径，并确认`@EnableJpaAuditing` 与安全审计日志的职责边界——前者可用于自动填充 `created_at`/`updated_at`，后者需独立实现。

### 问题 7：T22 批次分类双重冗余表述

- **位置**：「修复执行策略」批次 2 和批次 6 对 T22 的说明
- **严重程度**：提示
- **问题描述**：批次 2 说"T22 从本批次移出——因其涉及 JPA 实体 schema 变更，归入批次 6"；批次 6 又说"T22 从批次 2 移入本批次"。两条表述本质相同但分别出现在两个批次中，读起来像是重复或矛盾。为避免混淆，同一决策应只在与其直接相关的批次条目中声明一次。
- **改进建议**：仅在批次 6 中写明"T22：JPA 实体 schema 变更（由批次 2 移入）"，批次 2 中无需再说明 T22 的排除。

### 问题 8：产出未对项目整体审计能力进行前置排查

- **位置**：T8 修改建议全篇（未确认审计基础设施现状即给出默认路径）
- **严重程度**：低
- **问题描述**：产出对 T8（安全审计日志）的修改建议以"默认行动路径"先行创建 SecurityAuditLogger，但并未在诊断阶段确认项目是否已有审计基础设施。对于一份定位为"可执行"的诊断报告，应在给出修复建议前完成这类事实性确认，避免执行者在修复时重新做一遍排查。其他依赖外部基础设施的修复项（如 T23 的 N+1 依赖 UserRepository 方法存在性、T9 的循环依赖排查）均做了到位的前置确认，只有 T8 存在此 gap。建议其他同类项保持一致水平。
- **改进建议**：在 T8 分析中明确说明"已确认项目中不存在安全审计日志框架"，基于此给出确定的修复路径。

### 问题 9：T25 前端适配策略存在相互矛盾的指令

- **位置**：T25「风险与注意事项」中"版本发布协调"（"需确保前端已适配 429"）vs"灰度/兼容期处理"（"可临时将 RATE_LIMITED/ACCOUNT_LOCKED 保留 400"）
- **严重程度**：低
- **问题描述**：T25 给出了两种矛盾的适配过渡策略——一个要求前端适配后再部署，另一个建议临时保留旧状态码。执行者拿到两个矛盾选项，需要自行判断选择。考虑到 T25 的代码修改本身仅涉及 GlobalExceptionHandler 中的状态码映射（纯后端变更，不涉及 API 契约的格式/语义变化，仅状态码从 400 变为 429），且前端 axios 拦截器的通用分支已能正确提取业务消息，推荐策略应是优先直接部署（无需等待前端），而非要求前端先适配。
- **改进建议**：明确推荐策略："直接部署，不需等待前端适配。前端 400 和 429 均走通用错误码提取路径，用户看到的业务消息不变。若需严谨过渡，可在部署后监控前端是否因状态码变化出现预期外的错误处理。"

## 发现问题列表

### 问题1：汇总表 T4 的「相关项/修复依赖」标注与正文分析相矛盾

- **位置**：汇总表 T4 行（第13行），标注内容"T11（消息变更影响字符串匹配）"
- **严重程度**：中
- **问题描述**：
  - 汇总表标注 T4 与 T11 存在修复依赖关系——"消息变更影响字符串匹配"。
  - 但正文 T4 第131-132行明确说明："与 T11 联动：...但不受 UNAUTHORIZED 消息变更影响（它检查的是 ACCOUNT_DISABLED 消息）。仍需关注 T11 本身的修复。"
  - 即：T4 修复（UNAUTHORIZED 消息文本变更）不会影响 T11 的字符串匹配逻辑（T11 检查的是 ACCOUNT_DISABLED 消息），两者**无修复依赖关系**。汇总表的标注会使执行者错误地认为必须先修 T11 才能修 T4。
- **改进建议**：
  - 删除 T4 汇总表行中的"T11（消息变更影响字符串匹配）"依赖标记。
  - 若需保留此信息，应移至 T4 正文的"风险与注意事项"中并明确说明"T4 修复不影响 T11，但 T11 本身的字符串匹配方式存在设计级脆弱性，建议关注"。

### 问题2：汇总表 T11 的「相关项/修复依赖」标注方向倒置

- **位置**：汇总表 T11 行（第27行），标注内容"T4/T27（消息变更使匹配静默失效）"
- **严重程度**：中
- **问题描述**：
  - 该标注暗示"T4/T27 的变更会使 T11 的匹配逻辑静默失效"，读作 T11 依赖 T4/T27 修复必须先处理。
  - 但正文 T11 第261行明确说明："T4/T27 的消息文本变更不影响此处的 instanceof 判断逻辑"——前提是 T11 按推荐方案（选项 B）修复为 instanceof 判断。**原文真正想表达的是"在 T11 修复之前，任意消息文本变更都会使字符串匹配静默失效"，而非 T4/T27 与 T11 存在强依赖。** 方向倒置导致执行者可能误解修复顺序。
  - 此外，`instanceof` 是 Java 关键字而非实例名——原文"instanceof 判断"是正确的，此处无拼写问题。
- **改进建议**：
  - 将标注改为"T4/T27（T11 修复前，文本变更会造成匹配静默失效）"，明确限定前提条件。
  - 或删除此依赖标记，将说明移至 T4/T27 的风险段落中。

### 问题3：T23 P0 优先级与 T26（启动阻塞）同级别缺乏可比性论证

- **位置**：汇总表 T23 行（第14行），优先级标注为 P0
- **严重程度**：中
- **问题描述**：
  - T26（重复 Bean 定义）为 P0 合理——直接导致启动失败，是阻塞级缺陷。
  - T23（N+1 查询性能）同为 P0，但诊断未提供实际数据量级验证（用户数、岗位数、每个岗位的功能数）。在没有数据支撑的情况下，无法判断 N+1 是否达到"阻塞性"级别。在典型的后台管理系统中，用户总数通常在数百以内，每个用户最多 1-3 个岗位，每个岗位 10-30 个功能——此量级下 N+1 的额外查询数为 M（岗位数），增加的延迟在毫秒级，远未达到"阻塞"标准。
  - 两个 P0 之间缺乏区分度会误导执行者的修复优先级——先修 T26 还是先修 T23？
- **改进建议**：
  - 补充数据量级估算（参考 data.sql 或业务调研数据）来支撑 P0 评级，若无大量级数据则应将 T23 降为 P1 或 P2。
  - 或调整 P0 定义为"启动阻塞/数据安全性"类，P1 为"核心业务流程缺陷"，T23 归入 P1。

### 问题4：T22 `@Column` 修改建议存在内部矛盾

- **位置**：第471-472行
- **严重程度**：低
- **问题描述**：
  - 前半句："若列名为 component 则无需显式指定"——建议不添加 @Column。
  - 后半句："建议显式指定 @Column(name = 'component') 以避免歧义"——建议添加 @Column。
  - 两条意见自相矛盾，执行者无法确定是加还是不加。
  - 实际上，`component` 在 Hibernate 中是 `@Embeddable`/`@Embedded` 相关概念，作为普通 `@Column(name = "component")` 不会产生歧义，Spring Data JPA 能正确处理。
- **改进建议**：
  - 删除前半句"若列名为 component 则无需显式指定"，统一建议"推荐显式添加 `@Column(name = 'component')` 以提高代码可读性"。

### 问题7：T13 虚拟哈希常量未给出具体值或生成指引

- **位置**：第290-291行
- **严重程度**：低
- **问题描述**：
  - 修改建议使用 `passwordEncoder.matches("dummy", "$2a$10$dummyHash...")`，但 `$2a$10$dummyHash...` 仅为一个占位符，未提供具体可用的 BCrypt 哈希值或生成方法。
  - 执行者需要额外步骤来生成一个 BCrypt 哈希（启动应用、调用 PasswordEncoder.encode()、复制输出），增加了实现摩擦。且不同 BCrypt 实现的格式略有差异（`$2a$` vs `$2b$` vs `$2y$`）。
  - 在测试环境或本地调试时，执行者可能使用一个非法格式的哈希导致 `matches()` 抛出异常。
- **改进建议**：
  - 提供一个已验证可用的 BCrypt 哈希常量，例如常见的 `$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy`（对应明文 "dummy"），并注明该哈希仅用于 dummy 比对，不存储密码。
  - 或在修改建议中直接给出生成命令：`System.out.println(passwordEncoder.encode("dummy"))` 作为生成指引。

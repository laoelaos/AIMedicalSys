# 质量质询报告（v1）

## 质询结果

CHALLENGED

## 逐维度审查

### 1. 证据充分性

**[问题-严重]** 问题2（chiefComplaint/additionalResponses 互斥语义）的证据存在对需求文档的误读。审查报告称需求文档中 `chief_complaint` 始终为必填、`additional_responses` 为增量补充而非替代，因此设计文档的"互斥"定义与需求文档矛盾。但实际需求文档 §3.4.1 输入契约原文为"`additional_responses`……与单字段 `chief_complaint` 二选一使用，不同时提供"——这正是需求文档显式定义的互斥语义。审查报告将互斥等同于"后续轮不传 chiefComplaint"，又引用 §3.1.3.1 多轮组装约定作为反证，但 §3.1.3.1 描述的是"首次调用仅传主诉"+"后续追问通过同一 session_id 增量传入 additional_responses"——后续轮确实仅传 additionalResponses + sessionId，chiefComplaint 不传，这恰好是需求文档互斥约束的体现，而非矛盾。审查报告将此标注为"严重"并断言与需求文档语义矛盾，属关键判断未经实际内容确认，证据不足以支撑"矛盾"结论。

**[建议]** 重新审视需求文档 §3.4.1 输入契约原文中"二选一使用，不同时提供"的显式互斥约束。该互斥约束与 §3.1.3.1 多轮组装约定完全一致——首轮传 chiefComplaint、后续轮传 additionalResponses + sessionId。设计文档的互斥语义与需求文档对齐，不应标注为矛盾。若审查报告认为互斥语义在编码实现中存在歧义（如服务端如何校验互斥），应重新定位问题性质并调整严重程度。

**[通过]** 问题1（DoctorFacade 包语义）、问题3（MatchedRule 字段缺失）、问题5（DosageAlert 缺 errorCode）、问题7（处方版本比较语义）、问题9（allergyHistory 数据来源）、问题10（AuditAlert.severity 枚举值域）、问题12（RegistrationEvent 事件契约）等问题的证据均经过设计文本实际内容确认，证据充分。

**[通过]** 问题6（AllergyCheckRule 遗漏合并症检查）的证据参照了需求文档 §3.4.2 最小检查项集 #2 的明确要求（"是否与患者的合并症冲突"），与设计文本中 AllergyCheckRule 仅覆盖过敏的事实直接对照，证据充分。

### 2. 逻辑完整性

**[问题-严重]** 问题4（RecordGenerateResponse 缺少 missingFields 原始字段）的改进建议与设计决策逻辑存在内在矛盾。审查报告建议"保留 missingFields（List\<String\>）字段以对齐需求文档原始输出契约，同时保留 missingFieldHints 作为结构化增强版本"。但 RecordGenerateResponse 已有 missingFieldHints（List\<FieldMissingHint\>），每个 FieldMissingHint 含 missingField 标识，前端可通过 `missingFieldHints.stream().map(h -> h.missingField).toList()` 一行代码还原出等价于 missingFields 的简单字符串列表——两字段完全冗余。审查报告将此标注为"一般"问题，但改进建议引入的冗余字段在编码实践中是不必要的重复，与其自身宣称的"结构化升级版本"定位矛盾（既然是升级版，原版即被替代而非并存）。

**[建议]** 将问题4的改进建议调整为：在 §7 设计决策中明确 missingFieldHints 是 missing_fields 的超集替代（每个 FieldMissingHint.missingField 即为 missing_fields 的元素），前端可通过 missingFieldHints 直接还原 missingFields 列表，无需两字段并存。或将严重程度降低为"轻微"——此问题为字段冗余选择，不影响编码实现。

**[通过]** 其余问题之间无逻辑矛盾。问题6（合并症检查遗漏）与问题15（辅助开方与处方审核过敏检查重复）分别关注两个不同维度（需求覆盖完整性 vs 设计冗余清晰度），不矛盾。

### 3. 覆盖完备性

**[问题-严重]** 审查报告遗漏了一个关键的需求响应充分度问题：需求文档 §3.4.1 输出契约中 `matched_rules` 字段无子结构定义——原文仅写"`matched_rules`"后以句号结束，未列出每项应含的子字段。这与 §3.4.2 alerts（显式列出 alert_code / alert_message / severity）的详略程度明显不同。设计文本 §1.3 MatchedRule DTO 定义为 ruleId / ruleName / score，审查报告问题3声称需求文档规定每项含"rule_id / rule_name / confidence / reason"——但需求文档 §3.4.1 输出契约原文中 `matched_rules` 后并无子结构字段列表，审查报告对象（需求文档 3.4.1）所引用的子字段（confidence / reason）缺乏来源。审查报告自身在此处存在未验证的假设，但更关键的是：需求文档中 matched_rules 的子结构定义本身缺失，这一需求侧缺口应被识别为设计文档需补充定义的设计决策（而非简单对齐需求字段），审查报告未覆盖此维度。

**[建议]** 补充问题或调整问题3的定位：需求文档 §3.4.1 matched_rules 子结构定义缺失是需求侧缺口，设计文档需主动定义子字段而非被动对齐。建议审查报告补充此需求响应充分度问题——设计文档应在 §7 设计决策中说明 matched_rules 子字段的设计选择及理由，而非仅标注与需求文档对齐。

**[问题-严重]** 审查报告未覆盖"底层落地约束"维度的充分审查。再审议需求明确要求"所有包均落地在底座上（非独立接入），规避 Phase 5 迁移成本"和"需参考已有 Phase0/Phase1ABD 的设计风格和架构模式保持一致性"。审查报告未评估设计文档是否充分论证了"底座直接落地"与规避迁移成本的实现方式——例如：设计文档中三个新模块均直接依赖 ai-api 并通过 AiService 接口调用 AI，Phase 5 迁移时 AiService 接口是否需要变更、迁移成本是否真正规避？设计文档仅在 §6.1 提及内存存储迁移，但 AI 调用接口迁移路径未讨论。此维度在内部审议（技术可行性）中可能被部分覆盖，但从需求响应充分度视角（迁移成本是否被真正规避vs仅声明性对齐），审查未充分评估。

**[建议]** 补充对"底座直接落地约束"的需求响应充分度评估：设计文档是否说明了 Phase 5 迁移时 AiService 接口的兼容策略？各模块 Converter 类的迁移影响范围是否有预估？若内部审议已充分验证此维度，审查报告应显式声明"底座落地约束已由内部审议技术可行性维度覆盖"而非完全忽略。

**[问题-一般]** 审查报告未充分评估设计文档对"异常场景和边界条件"的覆盖完整度（任务描述要求）。以下边界场景在审查报告中未提及：(1) DialogueSession 超过 TTL 30 分钟后恰好有请求到达的竞态处理（请求到达瞬间 ScheduledExecutorService 清理该 session）；(2) AuditRecord 原始处方快照（originalPrescription JSON TEXT）与对话级并发更新（如医生在两个浏览器标签页同时修改处方）的竞态场景；(3) 多轮分诊场景下规则版本在对话中途发生了热更新（ruleVersion 快照 vs 当前版本冲突），设计文档虽提及了快照机制，但未说明快照版本对应的规则集因管理员操作被删除/禁用后的行为。这些边界场景从"可直接指导编码实现"视角应被识别。

**[建议]** 在覆盖完备性维度补充对上述边界场景的审查意见，或评估设计文档已有定义对编码实现者是否足够明确。

### 4. 报告必要性

**[通过]** 审查报告的问题均与任务描述（需求响应充分度、整体深度和完整性、可直接指导编码实现、接口定义是否支持下游消费者、异常场景和边界条件）直接相关，无无关紧要的细节问题。问题1（DoctorFacade 包语义）虽为包路径问题但也属于接口定义清晰性维度，未超出审查范围。

**[问题-一般]** 问题14（缺少需求契约对齐矩阵）建议增加集中化的映射表——此改进建议对编码实现者确有便利性，但将此标注为"一般"问题略显高估：文档已有散点式对齐描述，若每个 OOD 文档都要求集中映射矩阵则文档膨胀不可避免。此问题更接近"改进建议"而非"质量问题"。

**[建议]** 将问题14严重程度降低为"轻微"，或在改进建议中标注为"可选增强"。

## 质询要点

### 要点1

- **问题**：问题2（chiefComplaint/additionalResponses 互斥语义与需求文档矛盾）的核心证据与需求文档原文不符——需求文档 §3.4.1 显式声明"二选一使用，不同时提供"，设计文档的互斥定义与此完全一致，不存在矛盾
- **原因**：审查报告对"互斥"的理解为"chiefComplaint 始终必填、additionalResponses 为增量补充"，但需求文档的互斥语义是"首轮仅传 chiefComplaint、后续轮仅传 additionalResponses + sessionId"，二者均为调用级互斥而非字段级互斥。将此标注为"严重"问题且判定为需求文档矛盾，导致审查结论在该问题上的可信度不足
- **建议方向**：重新核实需求文档 §3.4.1 输入契约原文和 §3.1.3.1 组装约定，确认设计文档互斥语义与需求文档一致后，撤回该问题或重新定位为"互斥语义编码实现细节澄清"类的一般问题

### 要点2

- **问题**：覆盖完备性维度遗漏——需求文档 §3.4.1 matched_rules 子结构定义本身缺失（需求侧缺口），审查报告问题3错误声称需求文档定义了 confidence / reason 子字段
- **原因**：审查报告问题3引用的需求文档子字段（confidence / reason）实际上在需求文档 §3.4.1 输出契约中不存在（matched_rules 后无子字段列举），此为审查报告自身的未验证假设。更关键的是，需求侧子结构定义缺失意味着设计文档需主动定义而非被动对齐，审查报告未从需求响应充分度视角识别此缺口
- **建议方向**：核实需求文档 §3.4.1 matched_rules 原文，纠正问题3中关于需求文档定义子字段的错误引用；补充对 matched_rules 子结构需求侧缺口的设计决策评估

### 要点3

- **问题**：覆盖完备性维度未评估"底座直接落地约束"的需求响应充分度
- **原因**：再审议需求明确列出"所有包均落地在底座上（非独立接入），规避 Phase 5 迁移成本"为核心约束，但审查报告未评估设计文档对此约束的实现是否真正规避了迁移成本（vs声明性对齐）。AI 调用接口迁移路径未讨论
- **建议方向**：补充对底座落地约束的需求响应充分度评估，或显式声明该维度已由内部审议技术可行性维度覆盖

### 要点4

- **问题**：问题4（RecordGenerateResponse 缺少 missingFields）的改进建议引入冗余字段，与"结构化升级版本"定位矛盾
- **原因**：missingFieldHints 已包含 missingField 标识，前端可一行代码还原 missingFields 列表，两字段并存为纯冗余。审查报告将此标注为"一般"但改进建议引入设计劣化
- **建议方向**：调整改进建议为明确 missingFieldHints 为 missing_fields 的超集替代，或降低问题严重程度

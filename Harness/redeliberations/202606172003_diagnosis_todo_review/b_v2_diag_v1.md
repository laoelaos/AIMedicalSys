## 发现的问题

### 问题 1（严重）：T10 分类修正说明与 T10 详细分析自相矛盾

- **所在位置**：第 23 行（分类修正说明）vs 第 183–185 行（T10 Spring DI 可行性分析）
- **问题描述**：分类修正说明声称"OOD 未考虑 Spring 构造器依赖注入的工程约束"，但 T10 详细分析明确写道"在构造器中直接检查 `delegates.isEmpty()` 并输出 ERROR 日志是**技术可行的**，OOD 的构造器阶段要求并非不切实际"。两处陈述直接矛盾——若 OOD 要求在 Spring DI 环境下技术可行，则"未考虑工程约束"的指控不成立。经代码核实，`FallbackAiService.java:52-58` 的构造器已通过构造函数注入接收 `List<AiService>` 和 `List<DegradationStrategy>`，在该构造器末尾添加空检测完全可行。OOD 真正欠缺的是对"惰性检测 vs 构造器检测"工程权衡的讨论，而非"未考虑可行性约束"。
- **严重程度**：严重 — 同一文档内部控制逻辑矛盾，且误导对 OOD 文档问题的定性判断
- **改进建议**：将分类修正说明中"OOD 未考虑 Spring 构造器依赖注入的工程约束"修正为"OOD 未讨论惰性首次检测与构造器阶段检测之间的工程权衡（如运维感知延迟 vs 启动失败快速反馈）"

### 问题 2（一般）：T10 双分类标注在结论表中未体现

- **所在位置**：第 16 行（总体结论表 T10 行）vs 第 23 行（分类修正说明）
- **问题描述**：分类修正说明声明 T10 应"同时标注真实存在与 OOD 文档问题"，但结论表 T10 行仅在"真实存在"列有 ✓，"OOD 文档问题"列为空。表格呈现与文字说明不一致。
- **严重程度**：一般 — 呈现不一致，读者需自行对照两处才能理解完整结论
- **改进建议**：方案 A：在 T10 行的"OOD 文档问题"列添加 ✓。方案 B：如果结论表只展示主分类，在表注中说明"T10 为交叉分类，详见正文分析"

### 问题 3（一般）：T6 代码文件路径引用不精确，影响可操作性

- **所在位置**：第 112 行、第 120 行、第 125 行（T6 条目引用）
- **问题描述**：报告引用响应拦截器代码时使用 `index.ts:10-26` 和 `index.ts:11-12`，但前端代码库中存在多个 `index.ts`（`packages/shared/src/api/index.ts`、`packages/ui-core/src/index.ts`、`packages/shared/src/index.ts` 等）。实际代码位于 `packages/shared/src/api/index.ts`。笼统的"index.ts"在构建产物的 IDE 中可能指向错误文件。
- **严重程度**：一般 — 路径不精确可能使修复者打开错误的文件查看代码
- **改进建议**：将 T6 条目中所有"index.ts"前缀为 `packages/shared/src/api/` 或 `shared/src/api/`，消除在多个同名文件间的歧义

### 问题 4（一般）：T5 缺乏 OOD 文档修复的具体行动指引

- **所在位置**：第 94–105 行（T5 条目）
- **问题描述**：T5 被正确诊断为 OOD 文档问题（OOD §10.1 设计规范正文与阶段性声明自相矛盾），但整个条目缺少"修复者行动指引"子节。执行者读完知道"代码没问题"，但不知道应当如何修复 OOD 文档中的矛盾——应删除 §10.1 正文中的 MeterRegistryCustomizer 描述？还是保留正文但增加醒目标注？还是将该项整体移出到"推荐补齐"附录？相比之下，T1/T2/T6/T9/T10/T11 均有明确的行动指引，T5 的缺失破坏了全文的一致性。

- **改进建议**：增加"修复者行动指引"子节，建议具体方案（例如："从 §10.1 设计规范正文中删除 MeterRegistryCustomizer 的具体实现描述，仅在路线图 0.2 '推荐补齐' 部分保留；或在 §10.1 段首增加醒目标记（如 `> **可选**`）以区分规范正文中的可选条目"）

## 发现问题

### 问题1：优先级方法论与T6实际定级之间存在张力

- **问题描述**：优先级排序方法论声明为"影响范围 × 修复风险"（§优先级排序说明）。T6被评为**高（功能）**，理由为"契约行为偏差直接改变 API 消费者得到的响应结构"。然而T6分析中明确标注"零冲击面——经代码库实测，前端除测试文件外不存在任何对上述四个包装函数的现有调用"。当前实际影响范围为0，与"高"定级之间存在张力。若T6是凭借"趁无消费者时低成本修复"的窗口期逻辑定级，则方法论需补充该维度，否则仅凭"影响范围"无法支撑"高"评级。

- **所在位置**：
  - 优先级排序表 T6 行（a_v8_diag_v1.md:330）
  - T6分析"零冲击面"声明（a_v8_diag_v1.md:158）
  - 优先级排序说明（a_v8_diag_v1.md:325）

- **严重程度**：一般

- **改进建议**：二选一：
  1. 在优先级排序说明中补充"修复窗口期"维度，说明为何当前零影响的问题仍为高优先级（趁无消费者时一次性修复，避免后续产生技术债务）
  2. 若严格按字面"影响范围×修复风险"框架，T6应降为"中"级，其当前影响为0，且修复方案已有明确指引，修复风险可控

---

### 问题2：T11修复指引缺少测试基础设施前提讨论

- **问题描述**：T11指引建议"添加 @SpringBootTest（或 @DataJpaTest）测试类或测试方法，在 JPA 上下文中持久化实体并验证审计字段"。但 @DataJpaTest 需要测试类路径中存在嵌入式数据库配置（如 H2 in-memory）、JPA 实体扫描配置等前提。报告未讨论这些基础设施在测试模块中是否已就绪。若测试模块未配置嵌入式数据库，执行者将无法直接实施该指引。

- **所在位置**：T11条目"修复者行动指引"子节（a_v8_diag_v1.md:314）

- **严重程度**：轻微

- **改进建议**：在修复指引中补充前提检查描述，如"确认 test scope 下已配置嵌入式数据库（H2），或提示执行者需先添加测试数据库依赖及 @DataJpaTest 的自动配置扫描范围"。

---

### 问题3：T5修复方案缺少执行责任归属说明

- **问题描述**：T5的修复方案涉及修改OOD文档正文（§10.1），而非代码修改。但报告未说明此修复的执行者角色——OOD文档修改通常由架构师/技术负责人负责，而非开发工程师。执行者在阅读该指引时无法判断"这是否属于我的职责范围"，可能导致修复被延迟或无人认领。

- **所在位置**：T5条目"修复者行动指引"子节（a_v8_diag_v1.md:106-108）

- **严重程度**：轻微

- **改进建议**：在修复指引前增加执行者角色说明，如"此修复需由OOD文档维护者（架构师/技术负责人）执行"。

---

### 问题4：T3/T4合并PR建议缺少修改后验证步骤

- **问题描述**：T3和T4建议"合并为一个PR提交"。虽报告注明"修复为纯删除无副作用"，但 dependencyManagement 的修改（删除5个Spring Boot Starter的显式版本号、删除h2的scope声明）仍需编译和依赖解析验证才可确认修改正确。缺少此验证步骤，执行者无法确认修改后父POM传递性依赖解析行为是否符合预期。

- **所在位置**：
  - T3"修复关联"（a_v8_diag_v1.md:75）
  - T4"修复关联"（a_v8_diag_v1.md:90）

- **严重程度**：轻微

- **改进建议**：在T3或T4的修复指引中补充验证步骤，如"修改后运行 `mvn compile -pl .` 确认父POM解析成功，再运行全量编译 `mvn compile` 确认无间接影响"。

## 发现的问题

### 问题 1：T10 偏重构造器方案作为主推荐，未充分论证"启动期"不同实现路径的等价性

- **所在位置**：T10 条目"根因"段落（"将 ERROR 触发时机从'构造器阶段'延迟到了'首次调用阶段'"）、"修复者行动指引"方案 B 第 1 步、"推荐理由"段落
- **严重程度**：轻微
- **问题描述**：OOD §3.4（`04_ood_phase0.md:699` 及 `:706`）的原文表述为"**启动期**输出 ERROR 日志、运行期输出 WARN 日志"，并未指定"构造器阶段"。报告在根因分析中称代码应将 ERROR 时机放在"构造器阶段"，此诊断源于第 1 轮的分析视角并延续至今，但该隐含等价关系在 OOD 原文中不成立。"启动期"同样可以经由 `@PostConstruct` 或其他 `ApplicationListener` 机制实现。报告虽在"备选启动期方案——@PostConstruct"子节中讨论了 `@PostConstruct` 路径（第 316-317 行），但主推荐（方案 B）仍以"构造器加 3 行"为唯一推荐方案，未明确说明构造器方案相对于 `@PostConstruct` 的优先性依据——"改动量小"不足以解释为何构造器路径优于语义更精确的 `@PostConstruct` 路径。
- **改进建议**：
  1. 在根因分析中区分"启动期"（OOD 原文）与"构造器阶段"（实现选择），明确 OOD 原文对实现方式未做约束
  2. 在"Spring DI 环境可行性分析"中补充 `@PostConstruct` 作为替代方案的对比：构造器方案线程安全风险低（Phase 0 单线程启动）但耦合 Bean 创建时序；`@PostConstruct` 方案语义更准确（"启动期"而非"构造期"）但需额外方法声明
  3. 推荐方案的论证从"构造器可行"转向"两种启动期方案的工程权衡"，而非仅推荐一方

### 问题 2：T6 返回类型 `Promise<T | BusinessError>` 未统一约束 error 拦截器的返回形状

- **所在位置**：T6 条目"实现路径分析"方案 B "关键要点"（"返回类型变更：`apiGet`/`apiPost`/`apiPut`/`apiDelete` 从 `Promise<ApiResponse<T>>` 调整为 `Promise<T | BusinessError>`"）及"关键要点"第 5 项
- **严重程度**：一般
- **问题描述**：报告规范了业务错误分支返回 `BusinessError`（`{ code, message }`），但 error 拦截器（`index.ts:14-25`）当前返回的形状为 `{ code: 'NETWORK_ERROR', message: '...' }`、`{ code: 'HTTP_ERROR', message: '...' }` 等，这些形状与 `BusinessError` 结构吻合。然而报告未明确要求 error 拦截器的各类返回也统一纳入 `BusinessError` 类型约束。执行者可能为 success 拦截器分支定义 `BusinessError` 类型，但 error 拦截器返回字面对象而非类型化实例，造成调用方在使用 `Promise<T | BusinessError>` 时，从 error 分支收到的对象类型无法被 TypeScript 类型守卫收窄。此问题在方案 A 的判断链伪代码（第 149-163 行）中也存在——`handleBusinessError()` 返回的类型与 error 拦截器其他分支返回的字面对象类型不一致。
- **改进建议**：
  1. 明确将所有 error 拦截器返回路径统一为 `BusinessError` 类型实例，包括 network error、HTTP error、unauthorized 等
  2. 在"关键要点"中补充说明：error 拦截器的每个分支返回都应为 `BusinessError` 类型，而非零散的字面对象
  3. 考虑使用 TypeScript discriminated union 的 `kind` 字段（如 `kind: 'BUSINESS' | 'NETWORK' | 'HTTP'`）帮助调用方在消费侧做类型守卫

### 问题 3：T5 方案 A 的设计可追溯性损失副作用未讨论

- **所在位置**：T5 条目"修复者行动指引"方案 A（"从 §10.1 设计规范正文中删除 MeterRegistryCustomizer 的具体实现描述"）
- **严重程度**：轻微
- **问题描述**：方案 A 建议从 OOD §10.1 正文中删除 MeterRegistryCustomizer 描述，但未评估此举的副作用：§10.1 作为 Phase 0 可观测性的唯一设计规范，删除 MeterRegistryCustomizer 的具体描述后，该配置类的设计意图（通用标签 `application=aimedical-sys`、Micrometer 依赖声明方式等）将在 OOD 文档中丢失。后续 Phase 1+ 开发人员需要跨文档交叉引用路线图（`Docs/03_roadmap.md` 0.2 节）才能重建设计上下文，增加了设计可追溯性的断裂风险。相对而言，方案 B（保留正文但增加醒目可选性标记）不损失设计信息，应明确推荐优先。

  2. 明确推荐方案 B 为优先方案（对文档结构和设计可追溯性的影响最小），方案 A 仅在团队约定允许删除设计规范中的可选条目时选用
  3. 若保留方案 A，应在删除的位置添加指向路线图的显式交叉引用备注

### 问题 4：T11 修复指引缺少 `@DataJpaTest` 下的 `JpaConfig` 导入前提

- **所在位置**：T11 条目"前置条件检查"子节
- **严重程度**：轻微
- **问题描述**：报告讨论了 `@DataJpaTest` 和 `@SpringBootTest` 两种测试路径及其 H2 依赖前提。但 `@DataJpaTest` 默认仅扫描 JPA 相关 Bean，不会自动加载 `@EnableJpaAuditing` 配置（位于 `JpaConfig.java:7`）。尽管该注解在当前代码库中存在，`@DataJpaTest` 不会自动导入它。报告未提示执行者需要使用 `@Import(JpaConfig.class)` 配合 `@DataJpaTest`，或使用 `@SpringBootTest` 让全量上下文自动加载。缺少此提示可能导致执行者选择 `@DataJpaTest` 路径后仍遭遇审计字段不自动填充的失败。
- **改进建议**：
  1. 在前置条件检查中补充：若选择 `@DataJpaTest`，需通过 `@Import(JpaConfig.class)` 显式导入审计配置
  2. 或直接建议使用 `@SpringBootTest`（无此问题）作为首选路径，标注 `@DataJpaTest` 为需要额外导入步骤的备选

### 问题 6：T10 的"中"优先级与"Phase 0 实际影响为零"的定级逻辑存在未解释的张力

- **所在位置**：优先级排序表 T10 行（"中（日志行为）"）与 T10 分析中"Phase 0 实际影响为零"段落
- **严重程度**：轻微
- **问题描述**：报告的三维优先级框架（影响范围 × 修复风险 × 修复窗口期）引入了"修复窗口期"维度，用于解释 T6 的"零冲击面 + 高优先级"——窗口即将关闭所以优先级高。但 T10 的分析明确指出"Phase 0 实际影响为零，可推迟至 Phase 2+"，即修复窗口期非常宽裕。按框架的对称逻辑，T10 的优先级应低于同为代码缺陷且有实际 Phase 0 影响的其他"中"级条目（如 T7 有 `mvn dependency:analyze` 干扰、T11 有测试覆盖缺口）。报告未解释为何 T10 在"影响范围为零、窗口期极其宽裕"的条件下仍维持"中"级（仅低于"高"两档）。这种定级方法的不对称性降低了优先级框架的可信度。
- **改进建议**：二选一：
  1. 在优先级方法论中补充"非功能维度权重"规则，说明代码规范的偏离程度（如"与 OOD 规范的不一致"）作为独立于影响范围/窗口期的定级因子
  2. 或将 T10 降为"低"级（与 T5 并列，但注明"代码问题 vs OOD 问题"的区别），使框架逻辑自洽

### 问题 7：T6 方案 A 的 `isBusinessError` 标记设计脆弱性未被显式评估

- **所在位置**：T6 条目"实现路径分析"方案 A 判断链伪代码（`if (error.isBusinessError)`）、推荐优先级段落（方案 A 降为备选的理由）
- **严重程度**：轻微
- **问题描述**：方案 A 依赖在 throw 对象上附加 `isBusinessError: true` 标记来区分业务错误与网络错误。此方案存在以下未被讨论的维护风险：
  1. **属性名拼写风险**：`isBusinessError` 为运行时字符串属性，无类型系统保护。如果 success 拦截器与 error 拦截器中属性名拼写不一致，判断链会静默失效，业务错误被错误映射为 `NETWORK_ERROR`
  2. **路径顺序耦合**：error 拦截器判断链中 `isBusinessError` 检查必须位于 `error.response === undefined` 之前（伪代码已正确排列），但任何后续修改者都可能打乱此顺序，导致同样的静默失效
  3. **与第三方 throw 冲突风险**：如果未来有其他代码路径也向 error 拦截器 throw 含 `isBusinessError` 属性的对象，鉴别逻辑将被污染
- **改进建议**：在方案 A 的"推荐优先级重新评估"中明确列出此设计脆弱性作为降级理由之一，而非仅说"实施复杂度较高，且存在 NETWORK_ERROR 误映射风险"。备选方案说明中建议使用 Error 子类（`class BusinessError extends Error { code: string }`）而非属性标记，以利用 `instanceof` 类型守卫替代字符串属性检查。

## 质量审查报告：a_v10_diag_v1.md

### 1. T10 — Logback 级别阈值事实错误

**所在位置**：T10 条目"修复后验证闭环"第 5 项，logback 配置检查段落。

**问题描述**：报告称"若存在全局根级别限制（如 `<root level="WARN">`），则构造器输出的 ERROR 在生产环境中将不可见"。此断言违反 logback 级别层次规则：优先级顺序为 TRACE < DEBUG < INFO < WARN < ERROR，ERROR 的级别值（40000）高于 WARN（30000），因此在 `<root level="WARN">` 配置下 ERROR 日志**必然可见**。只有 `<root level="ERROR">` 才会遮蔽 ERROR 或 `<root level="OFF">` 才能完全禁止日志输出。此外，报告中提及的 `spring-default.xml` 并非 Spring Boot 自动检测的标准 logback 配置文件名（标准为 `logback-spring.xml` 或 `logback.xml`）。

**严重程度**：严重 — 此错误将导致执行者在修复时投入不必要的时间排查"本不存在"的日志可见性问题，或错误地添加多余的 logger 级别配置。

**改进建议**：删除关于 `<root level="WARN">` 遮蔽 ERROR 的错误描述，将检查内容修正为：确认日志框架未设置针对该类的 `<filter>` 过滤规则，且未设置 `<root level="ERROR">` 或 `<root level="OFF">` 等非标准限制。同时将 `spring-default.xml` 修正为 Spring Boot 标准文件名 `logback-spring.xml`。

### 2. T5 — "OOD 文档矛盾"描述与事实有偏差

**所在位置**：T5 条目"根因分歧说明"段落及"分类修正说明"中 T5 条目。

**问题描述**：报告称 OOD §10.1 "正文与阶段性声明间的矛盾导致审查工具标记为缺失"、"两者自相矛盾"。实际查阅 OOD §10.1：正文以设计规范形式描述 MeterRegistryCustomizer 配置细节（L1180-1182），而段首（L1176）明确标注"该骨架为推荐补齐项，不影响 Phase 0 骨架验收"。这是标准的"设计规范 + 可选性标注"文档模式——设计文档描述构成要件、可选性标注告知实施时机，两者服务于不同目的，不存在逻辑矛盾。审查工具将此标记为缺失的原因是工具未（或无法）理解 OOD 的可选性声明语义，而非 OOD 文档自身存在矛盾。

**严重程度**：一般 — 报告对根因的分析方向正确（问题在 OOD 的呈现方式），但对"矛盾"的定性过度，可能误导后续的 OOD 文档修订方向（执行者可能错误地去"消除矛盾"而非优化可选性标注的可发现性）。

**改进建议**：将根因描述修正为"OOD 文档对 MeterRegistryCustomizer 的设计描述与可选性标注在语义上一致，但审查工具无法理解 OOD 的可选性声明标记，导致误判为缺失。优化方向应是增强可选性标注的机制化可识别性（如统一标记模式），而非断言 OOD 存在自相矛盾。"

### 3. T10 — 未区分 OOD "启动期 ERROR" 规范的生效范围约束

**所在位置**：T10 条目"根因"段落、分类修正说明及整条分析。

**问题描述**：OOD §3.4 对"启动期输出 ERROR" 的描述实际出现在两个上下文中：① Phase 0 说明（L699）明确 Phase 0 禁止将 `ai.mock.enabled` 设为 `false`，该场景受保护不会触发；② 兜底保护描述（L706）描述的是 Phase 2+ 引入真实 AiService 后的行为。在 Phase 0 默认配置（`ai.mock.enabled=true`）下，MockAiService 被注册为 Bean，`delegates` 列表非空，`handleEmptyDelegates()` 在正常流程中**永远不会被调用**。报告将 T10 定性为"真实存在"的代码缺陷时，未明确指出此偏差在 Phase 0 的**实际影响为零**，且 OOD 的规范本身在 Phase 0 语境下属于"不会触发的设计规范"。执行者可能据此在 Phase 0 阶段投入不必要的时间修复一个零影响问题。

**严重程度**：一般 — 报告虽在"当前惰性检测的工程合理性评估"中提及"功能正确性不受影响"，但整体定性（真实存在 + 中优先级）未充分结合 Phase 0 的实际生效范围。

**改进建议**：在根因分析或修复者行动指引中补充说明：Phase 0 默认配置下 delegates 非空，该日志行为差异仅当 `ai.mock.enabled=false` 时暴露（当前受 OOD 约束禁止），可推迟至 Phase 2+ 引入真实 AiService 时一并修复，或在 Phase 0 中标记为"跟踪项"而非"待修复项"。

### 4. T6 方案 B — 消费者侧类型安全负担未充分讨论

**所在位置**：T6 条目"实现路径分析"方案 B 子节及"关键要点"段落。

**问题描述**：方案 B 对业务错误采用 `Promise.resolve(handleBusinessError(...))` 模式，将业务错误与成功数据通过联合类型 `Promise<T | BusinessError>` 返回。这意味着**每个 API 消费者**（前端组件/service）必须显式区分 `T`（业务数据）和 `BusinessError`（业务错误）：每次调用 `apiGet<T>()` 后，消费者需执行类型守卫判断返回的是业务数据还是错误信息。若消费者未处理该分支，`BusinessError` 对象可能被静默当作业务数据使用，导致运行时逻辑错误。前端社区对业务级错误更为常见的实践是通过 `Promise.reject` 抛给 `.catch()` 统一处理，使 `.then()` 中仅处理成功数据。报告未讨论 resolve 模式对消费者心智负担和类型安全的影响，也未与 reject 模式做权衡对比。

**严重程度**：轻微 — 该设计选择本身合理，但不讨论其负面影响使执行者缺少完整的决策信息。

**改进建议**：在方案 B 中补充"消费者影响"子节，讨论 resolve vs reject 两种模式在复杂度、类型安全和错误处理一致性方面的权衡，或引用现有前端架构决策记录（ADR）给出明确推荐。

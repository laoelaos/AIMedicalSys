- **所在位置**：T6 条目"实现路径分析"子节，第 158 行（"此方案的联动影响"段落）及第 164 行
- **严重程度**：一般
- **改进建议**：在"联动影响"分析中补充："经代码库搜索，当前不存在对 `apiGet`/`apiPost`/`apiPut`/`apiDelete` 的任何调用，该返回类型变更为零冲击面。"

## 问题 3（一般）：优先级定级内部分化不足——T3/T4 与 T1/T6 同列"高"级但本质差异未区分

- **问题描述**：优先级排序将 T3（parent POM 中 Starter 冗余版本号）和 T4（h2 scope 误设）定为"高"，与 T1（恶意分页 OOM 安全风险）和 T6（API 契约偏差影响所有前端消费者）同级。T3/T4 的实质是**构建维护性**问题（未来升级风险、构建行为可能异常），修复后无即时功能性影响；而 T1/T6 是**功能性/安全性**问题（当前存在安全漏洞、当前契约行为偏差）。四者并列"高"优先级使执行者难以区分紧急与重要程度，"高"标签的信息量被稀释。
- **所在位置**：优先级排序说明表，第 287-290 行（T3/T4 行）及第 285-286 行（T1/T6 行）
- **严重程度**：一般
- **改进建议**：在优先级排序说明中增加子维度标注，例如在"高"级别内区分：
  - **高-功能/安全**（T1, T6）：存在即时功能或安全风险
  - **高-架构/维护**（T3, T4）：提升长期可维护性，当前无即时风险
  或直接将 T3/T4 调整为"中"级（与 T7/T9/T10 同级），让"高"级更聚焦于有即时影响的条目。

- **所在位置**：T10 条目"推荐理由"段落，第 256 行

- **改进建议**：在"无需 AtomicBoolean 等线程安全机制"后补充限定语，如"（基于当前 Spring 单线程 Bean 创建的实现行为，非 API 契约保证，但实际风险极低）"。

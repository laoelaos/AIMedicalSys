# 质量审查报告

> 审查对象：`a_v7_diag_v1.md`（Phase 0 OOD 诊断报告，v7）
> 审查视角：需求响应充分度、事实/逻辑一致性、深度完整性、可操作性

## 1. §2 诊断项分类体系存在系统性矛盾

**问题描述**：报告§2（偏离需求文档）下全部5个诊断项（§2.1~§2.5）均在其各自的"降级依据"段落中明确声明"不影响 Phase 0 三项验收标准"、"属于 Phase 0→Phase N 衔接准备项"。其中§2.2更显式标注"(Phase 1+ 视角)"。然而这些项却被统一归类在"偏离需求文档"标题下——即同一个章节内，各节自身内容与章节标题构成了实质性的分类矛盾。

此矛盾对修复者的实质影响在于：每个"修复者须知"段落将修改责任指向 OOD 文档的某个具体章节（如§8.2、§3.3），但同节的"降级依据"又说该项不影响 Phase 0。修复者面临"修还是不修"的决策困境——作为 Phase 0 OOD 修复任务，应在当前迭代中处理；作为 Phase 1+ 衔接建议，则应在后续阶段由 Phase 1 开发者处理。当前的分类混合让两类读者（OOD 修复者、Phase 1+ 实现者）都无法从分类层级判断哪些是他们需要立即关注的。

- **所在位置**：`a_v7_diag_v1.md` §2 全部（§2.1 L34-L39, §2.2 L41-L50, §2.3 L52-L62, §2.4 L64-L69, §2.5 L71-L75）
- **严重程度**：严重
- **改进建议**：将§2中明确属于"Phase 1+ 衔接准备"的项（§2.1、§2.3、§2.4、§2.5）移入独立章节（如新增§8 跨阶段衔接建议），或在每项标题后标注"（衔接建议，非Phase 0偏离）"，使诊断分类与项内分析自洽。保留在§2下的应是那些真正阻塞 Phase 0 验收标准或在 Phase 0 范围内构成明显偏离的项。

## 2. §2.2 权限矩阵诊断中"线下接诊医生角色缺失"的有效性不足

**问题描述**：报告将"需求 §2.6 中'线下接诊医生'角色在 OOD 中没有对应的角色定义或实体预留说明"列为 Phase 0 OOD 对需求文档的偏离。但实际 OOD 文档已在§3.3 定义了"角色—岗位—功能"三级权限模型（含 User、Role、Post、Function 四个实体），线下接诊医生作为"医生"角色的子岗位，是 Post 实体在 Phase 1 数据种子阶段的标准实例化场景。OOD §3.3 明确声明"Phase 0 的 common-module-impl 仅提供实体与 Repository 骨架，不提供 PermissionServiceImpl 等门面实现"(L531)，意味着角色枚举与岗位实例化不属于 Phase 0 OOD 的内容边界。

该诊断的实质性缺陷在于：将 Phase 1 的数据播种任务缺失诊断为 Phase 0 OOD 的"偏离需求文档"，混淆了"OOD 文档设计框架是否预留建模能力"与"Phase 1 是否已完成角色数据初始化"两个不同阶段的责任边界。修复者按此诊断修改 OOD，要么补充一条"线下接诊医生将在 Phase 1 作为 Post 实例化"的备注（对修复者在 Phase 0 的实质行动无帮助），要么试图在 Phase 0 就列全七类角色的权限映射（超出 Phase 0 骨架内容边界）。

- **所在位置**：`a_v7_diag_v1.md` §2.2 L44-L48，特别是 b 项（线下接诊医生）
- **严重程度**：一般
- **改进建议**：将该项措辞调整为"OOD 权限模型框架（角色—岗位—功能三级）已预留全部角色/岗位的建模能力，建议在§3.3 末尾补充一句对 Phase 1 数据种子中角色/岗位枚举的指引说明"，或将该项从"偏离需求文档"降级为"Phase 1+ 衔接提示"，移入独立章节。

## 3. §2.4 修复建议存在"伪二选一"困境，修复者无法直接行动

**问题描述**：§2.4 修复者须知为实体阶段映射表的放置提供了两种路径，但路径 A（追加至 OOD 附录）被报告自身分析立即否定——"违背 OOD 最小化修改原则"；路径 B（另建独立文档）附加了前置条件——"需项目管理者确认该独立文档的生命周期和更新责任人"。修复者面对的实际情形是：路径 A 已被报告自身判定为不可接受（修复者不应选择一种被诊断报告判定为"违背原则"的方案），路径 B 需要等待外部决策链。最终修复者无法在当前条件下直接行动。

该问题反映出修复建议未将"修复者可直接执行的路径"与"需外部协调的路径"区分开来。修复者需要的是一个明确的、可立即执行的推荐方案，而非两个皆附带负面条件的选择项。

- **所在位置**：`a_v7_diag_v1.md` §2.4 L69 "修复者须知"
- **严重程度**：一般
- **改进建议**：（1）如果路径 A 确实因违背最小化原则而不可行，应直接推荐路径 B 作为唯一路径，并明确标注"该映射表因超出 OOD 最小化骨架内容边界，不属于 Phase 0 OOD 修改范围。建议由项目管理者在 Phase 1 启动前创建独立文档。"（2）如果两条路径均可接受，则移除对路径 A"违背最小化修改原则"的否定性评价，并给出明确的选择推荐（如"推荐路径 B，理由如下：……"），而非将决策负担转移给修复者。

## 4. §3.2 对 SecurityConfig 超前实现的风险提示与 §4.1 CSRF 修复项之间的优先级倒挂

**问题描述**：§3.2 认可 SecurityConfig 超前实现（含 BCryptPasswordEncoder、AuthenticationEntryPoint、AccessDeniedHandler 等 Phase 1 完整配置）为"合理的前瞻性设计"，仅在风险提示中建议添加 PasswordEncoder 单元测试。§4.1 将 CSRF 未禁用列为 P1 修复项，理由是"Phase 0 骨架就应固化到代码中的标准基线配置"。

从项目风险角度审视这两个问题的权重：SecurityConfig 超前实现中包含了完整的 Phase 1 安全配置代码（PasswordEncoder、AuthenticationEntryPoint、AccessDeniedHandler），但 §4.1 的 CSRF 仅缺一行 `.csrf(csrf -> csrf.disable())`。如果 Phase 1 开发者忘记添加 CSRF，第一个 POST 端点出现 403——这是一个易于发现和修复的问题（几行代码、几分钟调试）。而如果 PasswordEncoder 的 BCrypt 强度参数配置错误（如过高的 `strength` 参数导致认证响应时间从 100ms 升到 5s+），该问题在 Phase 0 完全隐藏，直到 Phase 1 启用认证后才暴露——这是一个难以发现、根因定位耗时更长的风险。

报告将易发现、易修复的 CSRF 问题列为 P1，将对系统性能影响更大、更隐蔽的 PasswordEncoder 配置风险仅作为风险提示而不设优先级，两者不成比例。

- **所在位置**：`a_v7_diag_v1.md` §3.2 L90-L92（风险提示）vs §4.1 L98-L111（P1 修复项）
- **严重程度**：一般
- **改进建议**：（1）在§3.2 风险提示中为 PasswordEncoder 配置错误补充一个优先级标注（至少 P2，建议 P1），使报告整体风险治理策略一致；（2）或在§4.1 中解释为何 CSRF 的 P1 优先级高于 PasswordEncoder 配置错误——例如："CSRF 缺失在 Phase 1 启动时立即暴露且有精确的 Spring 错误消息指引修复方向；PasswordEncoder 配置错误则隐藏至 Phase 1 认证上线且无直观错误提示，但该配置的骨架代码已在 Phase 0 就位，Phase 1 开发者启动前可执行单元测试验证（如§3.2 风险提示所述），因此 PasswordEncoder 风险的缓解措施已在报告中建议。"——让读者明确理解两处优先级差异的权衡依据。

## 5. §3.1 ADR 建议的边际价值不明确

**问题描述**：报告§3.1 建议 OOD"在§8.2 或§7 设计决策中补充一条 ADR，说明'为何在 Phase 0 超前定义全部 13 项 AI 能力的 DTO'，并标注需求变更时的应对策略"。但 OOD §1.1 设计目标已明确"契约先行：统一响应包装、分页、错误码、全局异常处理等接口契约在 Phase 0 冻结，后续业务模块遵循契约即可"，§1.3 核心抽象清单已将 AiService 列为 Phase 0 交付物。超前定义的设计理由在 OOD 已有隐含覆盖，单独补充一条 ADR 的信息增益有限，而新增的维护成本（ADR 需随阶段演进同步更新）可能超过其收益。

- **所在位置**：`a_v7_diag_v1.md` §3.1 L85 "设计权衡记录建议"
- **严重程度**：轻微
- **改进建议**：降级为可选建议，或将建议内容合并到现有 §7 设计决策表"AI 能力接口形态"行的备注字段中（一句话补充），不另立独立 ADR。同时明确标注该建议为"可选的文档补充项，不影响 Phase 0 交付质量"。

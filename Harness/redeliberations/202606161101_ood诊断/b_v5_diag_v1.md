# 质量审查报告 — a_v5_diag_v1.md

> 审查对象：`a_v5_diag_v1.md`（Phase 0 OOD 诊断报告 v5）
> 审查视角：需求响应充分度、事实/逻辑正确性、深度与完整性、可操作性（诊断建议可行动性、修复方案副作用、优先级合理性）

---

## 1. 修复指导自相矛盾——§2.5 超时阈值索引表的位置建议冲突

- **问题描述**：§2.5 主文段（L72）明确建议"应在 §6 中标注引用位置"，而同节的"修复者须知"（L73）则建议"在 OOD §8.2 中增加超时阈值索引表"。§6（并发设计）与 §8.2（DTO 字段定义）是 OOD 中两个完全不同的章节，前者描述非功能设计，后者定义字段级契约。同一修复项给出了两个互不连通的定位建议，修复者无法确定应修改何处。
- **所在位置**：`a_v5_diag_v1.md` §2.5 L72 vs L73
- **严重程度**：中等
- **改进建议**：统一为单一推荐位置。若两处均需修改，应明确表述为"同时在 §6 标注引用位置、在 §8.2 增加索引表"，并说明两处修改各自侧重的信息维度（如 §6 标注超时阈值的契约引用，§8.2 建立能力—阈值索引映射）。

---

## 2. 事实性判断错误——§1.2 JDBC 驱动 `compile` scope 的依赖传播担忧不成立

- **问题描述**：§1.2 以"`compile` 会导致下游模块通过 transitive 依赖引入不必要的 JDBC 驱动包，与 §2.2 模块依赖治理目标不一致"作为建议 JDBC 驱动统一使用 `runtime` scope 的核心理由。但 OOD 中 JDBC 驱动声明在 `application/pom.xml` 中，而 `application` 是最终聚合启动模块，在模块依赖图中属于**叶子节点**——没有任何模块依赖 `application`。因此 `application/pom.xml` 中的 `compile` scope 不会向任何下游模块传播 transitive 依赖，所谓"不必要的 transitive 依赖"问题在此场景下不成立。
- **所在位置**：`a_v5_diag_v1.md` §1.2 L23-L24
- **严重程度**：中等
- **改进建议**：移除关于 transitive 依赖传播的论据，或替换为真实成立的论据（如"JDBC 驱动统一使用 `runtime` 是 Spring Boot 官方推荐实践，便于在 Phase 1+ 通过 profile 切换数据源时保持一致的依赖声明风格"）。修复建议本身（统一使用 `runtime` scope）仍然合理，仅修正其支撑理由即可。

---

## 3. §2.3 缺少"降级依据"段落——结构不完整，修复者无法理解 P1 的判定逻辑

- **问题描述**：§2（偏离需求文档）下 §2.1 和 §2.2 均有独立的"降级依据"段落，逐条说明为何该偏离不阻塞 Phase 0 验收标准、为何定为 P1。而 §2.3（DTO 字段约束对齐检查）直接给出偏离项后跳至"修复者须知"，缺少"降级依据"段落。修复者需要自行推断为什么 DTO 字段约束缺失不阻碍验收标准——对不熟悉路线图内容的修复者可能产生混淆。
- **所在位置**：`a_v5_diag_v1.md` §2.3 L55-L61 之后
- **严重程度**：轻微
- **改进建议**：补充一段"降级依据"，明确表述 DTO 字段级验证约束属于 Phase 1+ 的接口契约细化工作，其缺失不阻塞 Phase 0 的三项验收标准。

---

## 4. §2.2 跨模块授权路径分析超出 Phase 0 OOD 边界——诊断粒度偏细

- **问题描述**：§2.2 指出"OOD 未在权限模型或数据流中描述该跨模块授权路径（处方→就诊→检查/检验报告）"。但 OOD §8.4（L1354）及 §3.3（L531）已多次明确声明"Phase 0 暂不实现任何跨模块调用"、"各业务模块仅有占位 Controller"、"不提供 PermissionServiceImpl 门面实现"。要求 OOD 在 Phase 0 描述一条跨模块数据流路径，与其自身边界声明相违背。虽然该观察在 Phase 1 视角下有参考价值，但作为 Phase 0 诊断输出，此处属于诊断粒度过细——修复者收到后会对"到底该不该改 OOD 以包含该路径"产生困惑。
- **所在位置**：`a_v5_diag_v1.md` §2.2 偏离说明第 3 项（L47）
- **严重程度**：轻微
- **改进建议**：在该项前补充修饰语"该数据流路径属于 Phase 1+ 的数据访问设计范畴，Phase 0 无需在 OOD 中详细描述，但可标注为 Phase 1 需验证的权限映射场景之一"。

---

## 5. §2.4 实体交付阶段映射表的优先级可能偏低（P2 → P1）

- **问题描述**：§2.4 建议"OOD 应补充一份'实体交付阶段映射表'"并标记为 P2。按报告自身定义的优先级标准（P1=影响 Phase 1+ 衔接质量），缺失实体阶段映射表直接导致 Phase 1 开发者在启动时需自行交叉对照需求文档和路线图来确定实体搭建顺序，属于**影响 Phase 1+ 衔接质量**的问题。P2（建议性改善）与此偏离的实际影响不完全匹配。
- **所在位置**：`a_v5_diag_v1.md` §2.4 L67
- **严重程度**：轻微
- **改进建议**：将 §2.4 优先级调整为 P1，并在"降级依据"（当前缺少该段落）中说明其虽不影响 Phase 0 验收标准，但实体阶段归属的缺失会增加 Phase 1 开发者的实体设计决策负担，属于衔接准备项。

---

## 6. §3.2 对 Phase 1 SecurityConfig 超前实现的"无偏离"结论缺乏风险提示

- **问题描述**：§3.2 将 SecurityConfig Phase 1+ profile 切换机制（含 AuthenticationEntryPoint、AccessDeniedHandler、PasswordEncoder 的完整实现）判定为"无偏离"和"合理的前瞻性设计"，结论正确。但该超前实现包含 `PasswordEncoder`（BCryptPasswordEncoder Bean），Phase 0 的 `permitAll` 模式下该 Bean 不会被调用——但若 Phase 1 开发者启用了 `permitAll` + `PasswordEncoder` 的组合且在测试中未显式触发认证流程，该 Bean 的配置错误（如 encoder 强度参数不合适）将被延迟暴露。该风险虽小，但属于"前瞻性设计"的经典权衡点，报告应补充一句风险提示而非仅做正面评价。
- **所在位置**：`a_v5_diag_v1.md` §3.2 L88-L89
- **严重程度**：轻微
- **改进建议**：在"结论"段末尾补充"建议在 Phase 0 验收前对 PasswordEncoder Bean 做一次独立单元测试（无需认证流程，仅验证 encode/matches 基本行为），确保该超前代码在 Phase 1 启动时正常生效"。

## 总体评价

报告 v5 相比 v4 已修复优先级标准引用错误、新增 §7 偏离路线图声明、统一优先级判定逻辑等关键问题。上述发现的 6 项问题中，前 2 项（编号 1、2）属于中等严重度的**可行动性障碍和事实判断偏差**，其余 4 项为结构完整性和粒度方面的轻微瑕疵。修复者可以§1.2 的论据修正和 §2.5 的定位统一作为优先处理项（建议在下一迭代的修订说明中处理），其余项可择机调整。

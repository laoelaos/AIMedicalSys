> 审查视角：需求响应充分度、事实/逻辑正确性、深度与完整性、可操作性（优先级与修复建议的副作用）

## 问题 1：§2.2 权限矩阵缺口被赋予 P0 优先级，与 Phase 0 验收标准不匹配

- **位置**：§2.2，优先级标记 P0
- **问题描述**：诊断报告将权限矩阵对齐缺口标记为 P0（"阻塞骨架验收或运行时错误"），但该发现的实际阻塞程度达不到此标准。验证依据如下：
  - 路线图 §Phase 0"验收标准"（`03_roadmap.md:53`）明确列出三项标准："三端前端可一键启动到占位首页；后端基础骨架可独立启动并响应 ping 健康检查；新人按入门引导文档可在 1 小时内完成本地环境搭建"。权限矩阵的完整角色映射不属于任何一项。
  - 路线图 §Phase 0"骨架必备"项要求的是"数据与权限模型骨架"（`03_roadmap.md:39`）——即 User-Role-Post-Function 的三级抽象模型。OOD §3.3（`04_ood_phase0.md:529-593`）已实现该骨架模型，并明确通过 Post 实体支持"医生端通过岗位区分五个子岗位的可操作菜单"（L551），模型层面已为"线下接诊医生"等子岗位预留了扩展能力。
  - 诊断报告自身描述的缺口是"未列出全部 7 类角色的权限边界映射"和"未显式覆盖线下接诊医生"——这些都是**数据种子和验证清单层面**的缺口，而非模型抽象能力层面的缺失。此类缺口不影响骨架运行和编译，应属 Phase 1 启动前准备项而非 Phase 0 阻塞项。
- **严重程度**：高
- **影响**：P0 标记将引导修复者优先投入时间处理一个不阻塞骨架验收的问题，可能挤占真正阻塞性项目的处理资源（如 §2.1 错误码契约冻结、§1.1 文档矛盾等）。同时 P0 的误用降低了诊断报告优先级体系的可信度。
- **改进建议**：将 §2.2 降级为 P1。该缺口应在 Phase 1 启动前（而非 Phase 0 骨架验收前）完成对齐验证，属于 Phase 0→Phase 1 衔接准备项而非 Phase 0 阻塞项。若坚持 P0，需补充论证该缺口如何阻塞 Phase 0 验收标准中的某一项。

## 问题 2：§2.1（P1）与 §2.2（P0）之间存在可验证的优先级倒挂

- **位置**：§2.1（P1）vs §2.2（P0）
- **问题描述**：诊断报告的优先级分配在 §2.1 和 §2.2 之间自相矛盾：
  - §2.1（错误码命名约定，P1）识别的问题是"Phase 0 的 MockAiService 未要求返回符合 `_AI_` 命名约定的占位错误码"，这属于 Phase 0 接口契约冻结的硬性要求。路线图 §Phase 0 "骨架必备"明确要求"统一响应包装、错误码命名空间"（`03_roadmap.md:38`）。如果 Phase 0 不冻结错误码约定，Phase 1+ 开发者将在错误码命名上各行其是，后续统一成本更高。
  - §2.2（权限矩阵，P0）的问题（见问题 1）并不阻塞 Phase 0 验收。
  - 按"谁更直接影响 Phase 0 契约冻结质量"的标准，§2.1 的实际阻塞程度等于或高于 §2.2，却被赋予低了整整一级的优先级。
- **严重程度**：中
- **影响**：优先级倒挂导致修复者按 P0→P1→P2 顺序处理时，可能先处理权限矩阵角色映射表（Phase 1+ 准备项），而延迟修复错误码约定这个 Phase 0 契约冻结问题。后期若 MockAiService 已经编码完成，修改成本更高。
- **改进建议**：
  - 方案 A（推荐）：将 §2.1 升级为 P0（错误码命名约定是 Phase 0 契约冻结的一部分，MockAiService 若不遵循约定则骨架传递了错误的模式），将 §2.2 降级为 P1。
  - 方案 B（保守）：将两者统一为 P1，同时在 §2.1 和 §2.2 的"修复者须知"中补充一句话明确各自的时间约束——§2.1 应在 Phase 0 MockAiService 编码完成前处理，§2.2 应在 Phase 1 权限模块开发前处理。

## 问题 3：§3.2 推荐补齐项遗漏超出了 OOD 架构设计文档的诊断边界

- **位置**：§3.2（路线图"推荐补齐"项有遗漏）
- **问题描述**：诊断报告将"容器化开发部署脚本"和"本地代码质量检查工具集成"列为 OOD 的偏离问题。但这两项属于：
  - 容器化部署（Docker Compose）——运维/DevOps 工程配置，不属于架构设计文档的职责范围。OOD 作为架构设计文档，职责边界在"模块划分、接口契约、实体设计等技术事项"（v2 诊断报告 §3.1 自身也已确认此边界）。
  - 本地代码质量检查工具集成（lint 工具配置）——属于项目工程工具链配置（推荐在 `.opencode/`、`settings.xml`、`pom.xml` 中配置），也不属于 OOD 架构设计内容。
  - 路线图"推荐补齐"的原文表述（`03_roadmap.md:48-49`）使用的是"可在后续阶段首期补齐"的非强制语气，且归类在运营/工程工具而非架构设计项下。
  - 诊断报告自身在 §3.1 中正确界定了"协作规范"（CONTRIBUTING.md）不属于 OOD 补充范围，但在 §3.2 中对类似性质的 DevOp 工具项采用了相反的判断标准，构成内部不一致。
- **严重程度**：中
- **影响**：将 DevOps 工具配置的缺失列为 OOD 的"偏离"问题，会误导修复者在 OOD 文档中补充 Docker Compose 配置说明和 lint 工具配置说明——这些内容放在架构设计文档中既不合适也不可维护。同时该问题的存在稀释了 §3.1（协作规范缺失）这一真正属于路线图硬性交付项的问题的严肃性。
- **改进建议**：从"偏离"类别移除 §3.2，或将措辞调整为"项目管理提醒"而非"OOD 偏离问题"——明确标注该发现指向项目管理方而非 OOD 作者，建议由项目管理者在 Phase 0 验收前评估是否补齐推荐项（补充在独立的工程工具配置文件或项目 README 中），不属于 OOD 修订范围。

## 问题 4：§5.1 CSRF 配置建议与 Spring Security REST API 标准实践相悖

- **位置**：§5.1（CSRF 未禁用——Phase 1+ 的前瞻性风险）
- **问题描述**：诊断报告在 §5.1 中提出"当前无需在 Phase 0 中添加 `csrf.disable()` 代码（因为 Phase 0 无 POST 端点，增加未验证的配置属于过度防御）"。该建议存在两方面问题：
  - **与技术事实不符**：Spring Security 官方文档明确建议对使用无状态身份验证机制（如 JWT/Bearer Token）的 REST API 禁用 CSRF 保护。OOD 设计的认证架构明确基于 JWT（§4.5 使用 `PasswordEncoder` + `LoginUser` 适配 `UserDetails`，Phase 1 启用 `anyRequest().authenticated()`），这是一个标准的无状态 API 认证架构，CSRF 保护从架构层面不适用。对于无状态 API，`csrf.disable()` 不是"过度防御"而是"标准配置项"——无论 Phase 0 当前是否有 POST 端点。
  - **建议的实际效果适得其反**：诊断报告自身已识别"Phase 1 添加第一个 POST 业务端点时才会暴露"的风险，但推荐的解决方案（"添加注释标记 + Phase 1 Checklist 项"）实际上将风险转化为 Phase 1 开发者的记忆负担。在 Phase 0 添加一行 `csrf.disable()` 是零成本防御，而推迟到 Phase 1 则依赖人的记忆来避免 bug——数据显示这种依赖记忆的风险管理方式在大型项目中失败率高（遗漏调试时间通常 > 1 小时）。
- **严重程度**：低
- **影响**：该建议如果被采纳，OOD 将保留一个与标准实践相反的配置，Phase 1 开发者在添加 POST 端点后有概率遭遇 CSRF 403 错误，需额外花时间排查。
- **改进建议**：将推荐策略从"当前无需添加"调整为"Phase 0 骨架中应添加 `csrf.disable()`——无状态 API 始终需要该配置，Phase 0 添加不会产生副作用，Phase 1 无需记忆 Checklist 项"。

## 问题 5：§4.1 将 Phase 0 的正确实现描述为"风险"，问题定义存在偏差

- **位置**：§4.1（DTO 超前定义带来的需求稳定性风险）
- **问题描述**：诊断报告将 OOD §8.2 为全部 13 项 AI 能力定义完整字段级 DTO 的行为定性为"需求稳定性风险"（P1），但该问题的定义与路线图要求和诊断者自身的分析存在冲突：
  - 路线图 §Phase 0 "骨架必备"明确要求"接口契约框架"和"AI 能力模块接口契约与 Mock 数据占位"（`03_roadmap.md:38`）。OOD §8.2 的 DTO 定义正是"接口契约框架"的核心组成部分——定义字段级输入/输出契约并在 ai-api 中冻结，是 Phase 0 的正确交付物，而非"风险"。
  - 诊断报告自身在 §4.2（SecurityConfig 超前预留的合理性）中正确地判断了类似的"超前实现"属于"合理的前瞻性设计，不构成风险，无需修复"，但在 §4.1 中对同一性质的做法（超前定义 DTO）得出了相反的结论。两处的判断标准不一致。
  - 诊断报告提到的"需求在 Phase 2~5 发生字段变更时需回溯修订"是软件开发中**所有接口契约的常态化维护工作**，并非 Phase 0 特别引入的风险。任何被冻结的接口在需求变更时都需要更新。将这种常态化维护活动标注为"风险"等同于认为任何接口冻结都有风险——这否定了 Phase 0 路线图中"接口契约冻结"这一核心要求的意义。
- **严重程度**：中
- **影响**：该问题定义偏差将引导修复者认为 OOD 做了一个有风险的决定，并着手编写额外的 ADR 文档来解释一个已达成的标准决策。这不仅增加了不必要的文档工作，还可能在团队中引发对 Phase 0 接口契约冻结策略的不必要争论。
- **改进建议**：将 §4.1 从"问题"降级为"补充说明"级别的**设计笔记**，措辞从"需求稳定性风险"调整为"设计决策记录建议——Phase 0 全量 DTO 定义是路线图要求的标准做法，若团队希望记录决策上下文可在 §7 设计决策中增补一行说明"。同时注意与 §4.2 的判断标准保持一致，避免同类情况（超前实现/超前定义）得出相反结论。

## 审查结论

**发现总计**：5 项质量问题（含 1 项高严重度 + 3 项中严重度 + 1 项低严重度）。

**与第 1 轮审查结论的关系**：
- 第 1 轮审查的核心问题（§4 偏离需求文档覆盖深度不足、缺少优先级排序、分类准确性不足）在 v2 诊断报告中已得到全面改善：覆盖深度从 1 项扩展至 5 项，优先级标记全局添加，分类精确度显著提升。
- 第 2 轮审查发现的问题集中在**优先级分配的内部一致性**（P0 误用导致 §2.2 偏高与 §2.1 偏低）、**诊断边界的自我矛盾**（§3.1 正确界定了 OOD 边界但 §3.2 越界）、**判断标准的跨节一致性**（§4.1 vs §4.2 对同类超前行为的判断相反）和**技术建议的准确性**（§5.1 CSRF 建议与标准实践相悖）——这些是 v2 诊断报告的新增问题，未在前一轮中出现。

**整体质量评价**：v2 诊断报告在一轮迭代中实现了显著的改进——修复了第 1 轮的全部 6 项问题，补充了反向边界检查维度，建立了优先级体系，增强了需求对齐的系统验证。当前剩余问题集中于优先级体系的内部分配一致性、诊断边界的自洽性以及个别技术建议的准确性，属于精修层面的改进。建议在第 3 轮迭代中针对性解决上述 5 项问题，诊断报告即可达到较高的交付质量。

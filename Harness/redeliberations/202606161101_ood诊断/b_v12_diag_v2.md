# 质量审查报告 — v12 诊断报告（修订版）

## 问题 1：§3.2——PasswordEncoder Profile 配置假设与 OOD 实际描述不符（严重程度：一般）

- **位置**：`a_v12_diag_v1.md` §3.2 风险提示（L73）
- **问题描述**：诊断报告称"因 `PasswordEncoder` 配置于 `@Profile("!phase0")` 下"，并据此建议测试需激活 `!phase0` Profile。经核实 OOD §4.5 L780，原文明确描述为"骨架中保留真实的 `AuthenticationEntryPoint`、`AccessDeniedHandler` 和 `PasswordEncoder` 配置，**共享给两个 profile 的 SecurityConfig 使用**"。若共享于两个 Profile，则 PasswordEncoder 不应受 `!phase0` Profile 约束，Phase 0 默认 Profile 下 Bean 即应被创建。诊断报告中关于 Profile 归属的断言无 OOD 文本支撑，与 OOD 描述相矛盾。该错误直接影响测试建议的可执行性——若执行者直接按报告中 Profile 激活前提实施测试，将面临不必要的配置复杂度。
- **改进建议**：移除或条件化 Profile 假设。修订为："PasswordEncoder 在 OOD §4.5 中描述为共享配置（非 Profile 约束），Phase 0 默认 Profile 下即可注入测试。若实际实现中 PasswordEncoder 定义于 Phase 1 SecurityConfig 类内部（被 `@Profile("!phase0")` 约束），则需在测试中激活对应 Profile。请确认 OOD 中 PasswordEncoder 的实际配置位置后选择相应的测试策略。"

## 问题 2：§2.3 第三子点——事实断言精度不足（严重程度：轻微）

- **位置**：`a_v12_diag_v1.md` §2.3 第 3 子点（L53-L54）
- **问题描述**：报告声称"OOD 未在对应错误码或校验逻辑中体现该质量门槛的触发条件"。经核实 OOD §8.2 L1274，原文写道："`DiscussionConclusionResponse` 对应补充错误码 `DISCUSS_AI_OUTPUT_INCOMPLETE`，用于 `discussionText` 未达到 50 字符最低质量门槛时的返回语义"。OOD 已在统一约束补充部分将错误码与 50 字符阈值显式关联，报告断言"未体现"不为准确。实际差距在于 OOD 的映射停留在文字说明层而非 DTO 字段级 `@Size` 注解层——应精确表述为"已在文本层面映射但未落实为字段级校验注解"。
- **改进建议**：将措辞修正为"OOD 已在 §8.2 L1274 将 `DISCUSS_AI_OUTPUT_INCOMPLETE` 与 50 字符阈值建立文本关联，但未在 DTO 字段级以 `@Size` 等校验注解体现——Phase 0 Mock 模式下无运行时差异，Phase 2+ 实现者需在 Bean Validation 注解或业务校验中独立实现该映射"。

## 问题 3：§2.2 修复建议可操作性偏弱——执行者无法直接判断交付物边界（严重程度：一般）

- **位置**：`a_v12_diag_v1.md` §2.2 修复者须知（L46）
- **问题描述**：修复建议表述为"建议在 OOD §3.3 中补充权限矩阵与需求 §2.6 的对齐验证结果，明确各角色访问路径的覆盖情况"。但对于一个"最小化骨架"OOD，该建议缺少交付物边界指引：(1) 对齐验证结果的表现形式未定义——是简短的"已逐项对照确认覆盖"声明，还是完整的 7 角色 × 28 功能映射表？(2) 前者不提供实质信息价值，后者超出了 Phase 0 OOD 的典型内容边界。执行者面对该建议需要自行判断范围和粒度，增加了操作决策负担。对比同类建议（如 §2.3 明确要求"补充各字段的必填/长度约束"），§2.2 的粒度不一致。
- **改进建议**：在修复者须知中补充交付物边界指引，如："建议在 §3.3 末尾补充一段简短声明——说明权限模型骨架（User/Role/Post/Function）已覆盖需求 §2.6 矩阵的实体建模需求，完整映射关系属于 Phase 1 数据播种范围并在 §8.3 中记录衔接指引。该声明确认覆盖率但不展开映射细节，避免在 Phase 0 OOD 中引入非本阶段的全量数据。"

## 问题 4：§8.1 外部依赖风险——修复方案依赖项目管理者决策且无后备路径（严重程度：一般）

- **位置**：`a_v12_diag_v1.md` §8.1 修复者须知（L154）
- **问题描述**：修复建议要求创建独立文档 `Docs/entity_stage_mapping.md`，但明确声明"由项目管理者确认生命周期和更新责任人"。这意味着：(1) OOD 作者无法单方面完成此修复——需要外部协调；(2) 若项目管理决策被延迟或搁置，Phase 1 开发者面临同样的实体搭建顺序信息缺口。报告标记为 P1（影响 Phase 1+ 衔接质量），但修复方案的可执行性受外部依赖限制。该风险在报告中被提及（"由项目管理者确认"）但未评估其影响——即该 P1 问题的实际关闭周期不由 OOD 作者控制。
- **改进建议**：在修复者须知末尾补充依赖性评估，如："注意：此建议的实施依赖项目管理者确认文档所有权和责任人。若项目管理决策延迟，§8.1 的 P1 信息缺口将持续存在。建议 OOD 作者在提交诊断报告时同步推动项目管理确认，或在 §8.1 中补充 OOD 作者可独立完成的替代方案（如仅在 §3.2 实体段落中标注'其余 14 个核心实体归属阶段见《实体交付规划》待创建'的占位声明）。"

## 问题 5：§8.3 优先级标记与内容描述不完全一致（严重程度：轻微）

- **位置**：`a_v12_diag_v1.md` §8.3（L163-L169）
- **问题描述**：§8.3 三项子内容均被标记为 P1，但其导语明确表述为"属于 Phase 1 数据种子和权限扩展的**实现参考**，不影响 Phase 0 验收标准"。修复者须知进一步描述本节定位为"**参考备忘录**"。P1 定义是"影响 Phase 1+ 衔接质量"，而"参考备忘录"本质上提供的是一般性指导信息而非缺失的衔接任务。按报告自设标准（P0=阻塞验收/P1=影响衔接/P2=建议性改善），纯参考信息应归为 P2——它们不会因未被处理而阻塞或降低 Phase 1 的衔接质量，只是增加 Phase 1 开发者的查询成本。当前标记与 §8.2 非功能指标（P2）的优先级判定逻辑不一致：§8.2 同样是 Phase 2+ 参考信息但标记为 P2，§8.3 同为参考信息却标记为 P1。
- **改进建议**：将 §8.3 整体优先级从 P1 降级为 P2，或在导语中明确区分——将"线下接诊医生角色数据播种"和"药房医生跨模块授权路径"两项对 Phase 1 启动有直接指引作用的子项保留 P1，将"DataPermissionEvaluator 接口语义覆盖验证"（纯前瞻思考）标记为 P2。

## 问题 6：需求响应充分度与整体深度评估——无显著问题

- **位置**：全文
- **评估结论**：经逐项对照 `requirement.md` 要求的 5 类诊断维度：
  - **定义矛盾**（§1）：已覆盖 2 项，定位精确，分析完整 ✓
  - **事实错误**（§6）：已显式声明"未发现"，附排查范围 ✓
  - **逻辑错误**（§6）：已显式声明"未发现"，附排查范围 ✓
  - **偏离需求文档**（§2）：已覆盖 3 项，每项均有具体需求-OOD 映射分析 ✓
  - **偏离路线图 Phase 0 阶段**（§7）：已单独成节，逐项对照路线图交付物清单和排除清单 ✓
- **补充评估**：§3「反向边界检查」虽非 5 类维度之一，但提供了有价值的 Phase 0 骨架合理性分析（SecurityConfig 超前预留、DTO 超前定义），不影响需求响应充分度的判断。§6 的"事实错误"和"逻辑错误"维度已通过逐条技术断言验证确认，但方法论声明（如何验证 Spring Security 行为、Hibernate 注解语义等）可进一步补充以增强可信度。整体而言，诊断报告在需求响应充分度和深度上已达到较高成熟度，无需因结构性遗漏而追加审查轮次。

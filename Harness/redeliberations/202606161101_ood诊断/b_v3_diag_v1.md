## 问题 1：§2.1 与 §2.2 优先级判定采用双重标准，构成内部逻辑矛盾

- **位置**：`a_v3_diag_v1.md` §2.1「升级依据」 vs §2.2「降级依据」
- **严重程度**：中等
- **问题描述**：

  §2.2（权限矩阵缺口）以降级依据说明"不影响骨架运行和编译，不阻塞 Phase 0 三项验收标准中的任何一项"，因此判定为 P1。§2.1（错误码命名缺口）以"路线图'骨架必备'要求"为理由升级为 P0，未使用与 §2.2 相同的判定标准——即未评估该缺口是否实际阻塞 Phase 0 三项验收标准（前端占位首页、后端 ping 健康检查、1 小时搭建环境）或导致运行时错误。

  事实上，§2.1 缺口（MockAiService 未显式标注应返回符合 `_AI_` 命名约定的占位错误码）与 §2.2 缺口（OOD 未列出全部 7 类角色映射）均不阻塞骨架编译运行，均不影响三项验收标准，均属于 Phase 0→Phase 1 衔接准备的文档完整性缺口。两份缺口的实质性差异不足以支撑 P0 vs P1 的优先级差距。

  该优先级倒挂修正（§2.1↑P0、§2.2↓P1）是对迭代反馈的直接执行，但报告未对修正结果进行独立验证——对 §2.1 使用"路线图骨架必备"作为升级依据，对 §2.2 使用"验收标准不阻塞"作为降级依据，二者判定逻辑无法自洽。

- **改进建议**：统一使用同一判定标准重新评估 §2.1 和 §2.2 的优先级。若以"是否阻塞 Phase 0 验收标准"为准，二者均应定为 P1；若以"路线图任务清单是否完成"为准，则需重新审查 §2.2 是否也应升级。无论选择哪种标准，报告内部应保持逻辑一致。

## 问题 2：缺少"事实错误"和"逻辑错误"维度的显式审查结论

- **位置**：`a_v3_diag_v1.md` 全文
- **严重程度**：轻微
- **问题描述**：

  用户需求要求对 OOD 文档进行 5 维度诊断（定义矛盾、事实错误、逻辑错误、偏离需求文档、偏离路线图 Phase 0）。v3 报告将原 v1/v2 中"事实错误"章节的重分类为"技术风险提示"（§5）、将"逻辑错误"章节的唯一项移至"补充说明"（§6.C），重新归类本身合理，但报告缺少一句显式声明说明这两个维度的检查结果。

  未阅读迭代历史的读者无法确认：报告是否有意忽略了"事实错误"和"逻辑错误"两个维度？这两个维度的结论是"未发现"还是"未检查"？当前的报告结构产生了歧义。

- **改进建议**：在报告末尾（修订说明之后）增加一句显式声明，如："经审查，OOD 文档无事实错误（原归类为事实错误的问题已重新归类为技术风险提示/补充说明），无逻辑错误。"

## 问题 3：§3.1 协作规范缺失项对 OOD 修复者缺乏可操作性

- **位置**：`a_v3_diag_v1.md` §3.1
- **严重程度**：轻微
- **问题描述**：

  §3.1 将协作规范缺失列为"偏离路线图 Phase 0 阶段"诊断项，但在末段明确声明"协作规范缺失的责任方不是 OOD 作者，而是项目管理者"和"不属于 OOD 架构设计文档的补充范围"。这造成了一个矛盾：诊断项以"偏离"问题形式呈现，但报告的修复目标（OOD 作者）无法对该问题采取任何行动。

  报告的前后文指向了不同读者——标题分类告诉 OOD 作者"你的设计偏离了路线图"，但正文告诉项目管理者"这是你的责任"。两份读者均可能忽视该问题。

- **改进建议**：将 §3.1 从"偏离路线图"诊断项移至独立的管理提醒章节（如 §6 补充说明），并明确标注"该问题需由项目管理者在团队仓库中创建独立协作规范文档，不属于 OOD 架构设计文档的修改范围"。

## 问题 4：§1.2 JDBC 驱动 scope 不一致项涉及 Phase 1+ 配置，诊断边界交代不清

- **位置**：`a_v3_diag_v1.md` §1.2
- **严重程度**：轻微
- **问题描述**：

  §1.2 指出 OOD 文档对 H2（Phase 0，runtime scope）和 MySQL/PostgreSQL（Phase 1+，compile scope）使用了不同的 JDBC 驱动 scope 策略，建议统一为 runtime。诊断结论本身正确（JDBC 驱动的最佳实践是 runtime scope），但该问题涉及的是 Phase 1+ 才生效的配置项——MySQL/PostgreSQL 驱动的 scope 设置不在 Phase 0 交付范围内。

  报告未明确说明该问题是否属于 Phase 0 诊断范畴。若属于，应阐述为何 Phase 0 OOD 文档中关于 Phase 1+ 的配置引用也需要被修正；若不属于，则该诊断项超出了"偏离需求文档/路线图 Phase 0"的任务边界。当前描述让读者无法判断该问题的紧急性和归属。

- **改进建议**：在 §1.2 中明确标注"该问题涉及 OOD 对 Phase 1+ 配置的引用，虽不影响 Phase 0 交付，但属于 Phase 0 OOD 文档应修正的技术引用错误"，并相应调整优先级标注逻辑。

## 1. 问题发现

### 1.1 §2.2 降级依据段落未完整覆盖子点内容

- **问题描述**：§2.2 保留了两个独立子点——子点1（角色映射缺口）和子点4（DataPermissionEvaluator 语义覆盖缺口）。但"降级依据"段落仅讨论了"权限矩阵的角色映射缺口属于数据种子和验证清单层面"，未提及 DataPermissionEvaluator 子点。读者无法确认该子点是否也被统一视为"数据种子和验证清单层面"的问题，还是报告遗漏了对它的降级归类说明。

- **所在位置**：`a_v9_diag_v1.md` §2.2，第 47 行（降级依据段落）

- **严重程度**：轻微

- **改进建议**：在降级依据末尾补充对 DataPermissionEvaluator 子点的降级说明，例如"DataPermissionEvaluator 的语义覆盖确认属于 Phase 1+ 扩展点验证范畴，同样不阻塞 Phase 0 验收标准。"

---

### 1.2 §8.2 修复建议超出诊断报告典型边界，表现为实现设计指导

- **问题描述**：§8.2 修复者须知为超时阈值嵌入方式提供了完整的二元选择分析——路径 A（DTO 字段注释形态）和路径 B（独立索引表形态），包含各自的优劣分析、操作复杂度对比、维护成本评估，以及明确的推荐路径。这已从"问题诊断"延伸到了"组件级实现设计建议"。诊断报告的核心产出是"识别问题并指明修复方向"，而非提供细粒度的实现方案设计。这种超纲内容可能导致：
  - 修复者越过 OOD 架构师直接采纳路径建议而无视架构语境
  - OOD 评审会上对非实质的实现细节产生争议
  - 报告篇幅膨胀，稀释核心诊断信息

- **所在位置**：`a_v9_diag_v1.md` §8.2，第 160 行（"**实现区分**"起头的段落）

- **严重程度**：一般

- **改进建议**：将"实现区分"段落简化为一句指向性说明，例如"建议在 OOD §8.2 中补充各 AI 能力的超时阈值引用（从需求 §3.4.x 直接提取），同时在 §6 并发设计中标注超时定义以 §8.2 为统一入口。"移除路径 A/B 的优劣分析、操作复杂度对比和推荐理由，这些属于 OOD 架构师的设计决策范畴。

### 1.3 §2.1 修复建议沿用"或"连接歧义模式，与历史迭代已修正的模式一致

- **问题描述**：§2.1 修复者须知表述为"OOD 应在 §8.2 各 AI 能力 DTO 中补充对应错误码清单，**或**在 §5 ErrorCode 设计部分增加与各能力的映射表"——以"或"连接两个不同修改位置，修复者仍需自行决策。历史迭代 v5 §2.5 中，同样的"或"模式已被质询并修复（改为确定位置）。§2.1 存在相同的歧义模式但未被一致处理。

- **所在位置**：`a_v9_diag_v1.md` §2.1，第 39 行

- **严重程度**：轻微

- **改进建议**：统一为单一推荐路径，例如推荐在 §8.2 补充各能力错误码值列表（具体），同时在 §5 增加映射关系说明（抽象），并标注两处修改的不同侧重。或直接指定一侧（如 §8.2），移除"或"表述。

### 1.4 §8.3 修复建议可操作性不足，缺乏具体行动指引

- **问题描述**：§8.3 修复者须知为"建议 Phase 1 开发者在设计数据种子和权限校验逻辑时参考本节说明"。该表述仅建议"参考"而未给出可执行的具体行动项，修复者（Phase 1 开发者）无法据此直接开始工作。相比之下，同属 §8 的 §8.1 给出了明确的创建独立文档的路径，§8.2 给出了超时阈值索引表的具体位置建议，§8.3 是 §8 中可操作性最弱的一项。

- **所在位置**：`a_v8_diag_v1.md` §8.3，第 167 行（"修复者须知"段落）

- **严重程度**：轻微

- **改进建议**：补充至少一个具体行动示例，例如"Phase 1 启动时应在数据种子初始化脚本中将'线下接诊医生'纳入 Post 枚举的 INSERT 条目中"；或者标注该节仅为 Phase 1 开发者的参考备忘录，不提供具体行动指引，在作者预期使用方式上明确其定位。

### 1.5 §8.1 修复指令内部矛盾——"OOD 应补充"与"不属于 OOD 修改范围"表述冲突

- **问题描述**：§8.1 修复者须知首句要求"OOD 应补充一份实体交付阶段映射表"，但紧接着的放置说明声明"该映射表不属于 Phase 0 OOD 修改范围，建议另建独立文档（如 `Docs/entity_stage_mapping.md`）"。两个相邻语句给出互斥指令：上一句命令 OOD 作者修改 OOD，下一句声明该修改不属于 OOD 范围。修复者无法确定应修改 OOD 文档还是另建独立文档，该矛盾直接影响诊断建议的可操作性。

- **所在位置**：`a_v9_diag_v1.md` §8.1，第 153 行

- **严重程度**：一般

- **改进建议**：统一措辞。若推荐独立文档，应将首句"OOD 应补充一份……"修改为"项目管理者应在独立文档（如 `Docs/entity_stage_mapping.md`）中补充实体交付阶段映射表——列出全部 18+ 核心实体及其归属的阶段"，消除"OOD 应补充"与"不属于 Phase 0 OOD 修改范围"的表述冲突。

### 1.6 修复方案潜在副作用——部分建议缺少副作用评估

- **问题描述**：用户要求从可操作性视角评估"修复方案是否有潜在副作用"。a_v9_diag_v1.md 部分修复建议缺少对潜在副作用的明确评估，存在以下缺口：

  - **§3.2 PasswordEncoder 测试建议**：建议添加 `@ActiveProfiles("!phase0")` 单元测试以验证 PasswordEncoder Bean 创建。潜在副作用未评估——激活 `!phase0` Profile 可能连带创建其他标有 `@Profile("!phase0")` 的 Phase 1 Bean，而这些 Bean 的依赖可能未在 Phase 0 测试上下文中准备完毕，导致测试上下文加载失败或创建了非预期的 Bean 实例。建议在测试建议中补充说明应使用 `@SpringBootTest(classes = {PasswordEncoderConfig.class})` 等精确上下文加载方式，而非全局激活 `!phase0` Profile。

  - **§4.1 CSRF 禁用建议**：建议添加 `csrf.disable()`。报告已在"已有配置上下文"段落中讨论了与既有 CORS 配置的兼容性，该分析基本充分。但仍存在一处未交代的风险——若 Phase 0 骨架中未来添加了非 JWT 认证的端点（如 Phase 0 文档未明确排除但可扩展的 admin 控制台端点），禁用 CSRF 可能降低安全性。该风险场景极低（Phase 0 仅为无状态 REST API 骨架），可作为脚注标注而非硬性限制。

  - **§1.2 JDBC scope 统一建议**：将 MySQL/PostgreSQL 驱动从 `compile` 改为 `runtime`。该变更属于纯声明性修改，无运行时行为变化，无实质副作用，评估充分。

  - **§2.1/§2.2/§2.3/§8.1/§8.2/§8.3**：这些修复建议属于文档补充型修改（补充映射表、添加约定说明等），不涉及代码行为变更，无实质副作用，不需补充评估。

- **所在位置**：`a_v9_diag_v1.md` §3.2（第 75 行）、§4.1（第 92-93 行）、§1.2（第 25-26 行）

- **严重程度**：一般（§3.2 缺口）至轻微（§4.1 缺口）

- **改进建议**：
  1. §3.2 PasswordEncoder 测试建议中补充精确上下文加载指引，替代全局 `!phase0` Profile 激活方式。
  2. §4.1 在脚注中标注"若 Phase 0 后续扩展非 JWT 端点，需重新评估 CSRF 禁用策略"。

### 1.7 优先级标记系统性一致性验证

- **问题描述**：用户要求评估"优先级排序是否合理"。a_v9_diag_v1.md 在 §2 开头声明了统一优先级标准（P0=阻塞验收标准/P1=影响衔接质量/P2=建议性改善），经逐项验证全部 11 处优先级标记与标准的一致性和横向公平性，发现一处横向不一致：

  - **§2.1（错误码命名约定衔接缺口，P1）与 §8.3（权限模型 Phase 1 数据种子，P1）**: 两者均标记为 P1（影响 Phase 1+ 衔接质量），判定逻辑一致。合理。

  - **§3.2（PasswordEncoder 风险，P1）与 §4.1（CSRF 未禁用，P1）**: 两者均为 P1，但风险性质不同——CSRF 是配置完整性缺口（易发现、影响确定），PasswordEncoder 是隐藏配置错误（隐蔽、发现成本高）。a_v9_diag_v1.md 已在 §3.2 中阐明了将 PasswordEncoder 风险定为 P1 的理由（隐蔽性高、影响范围大、Phase 0 排查无额外成本），该解释充分。合理。

  - **§1.1（common 模块 security 依赖矛盾，P1）**: 该问题属于文档内部事实陈述冲突，不影响衔接质量也不阻塞验收标准。按统一标准应归为 P2（建议性改善），但被标为 P1。判定为横向不一致——§1.1 的影响本质是文档正确性（非代码正确性），与 §1.2（JDBC scope，P2）和 §4.2（@SQLRestriction，P2）同类，不应因同属"定义矛盾"维度即默认归 P1。

  - **§1.2（JDBC scope 不一致，P2）**: 文档一致性改善，对 Phase 1+ 衔接无实质影响（不会导致编译或运行错误）。P2 合理。

  - **§4.2（@SQLRestriction，P2）**: Phase 2+ 开发者须知，对当前骨架无影响。P2 合理。

  - **§8.2（非功能指标对齐检查，P2）**: Phase 2+ 实现者信息完整性补充。P2 合理。

- **所在位置**：`a_v9_diag_v1.md` §1.1（第 11 行）、§1.2（第 20 行）、§2.1（第 34 行）、§2.2（第 41 行）、§2.3（第 50 行）、§3.2（第 75 行）、§4.1（第 81 行）、§4.2（第 96 行）、§8.1（第 148 行）、§8.2（第 155 行）、§8.3（第 162 行）

- **严重程度**：一般

- **改进建议**：将 §1.1 的优先级从 P1 降级为 P2，与 §1.2 和 §4.2 保持一致的判定标准——§1.1 是文档内部陈述冲突，不影响 Phase 0 验收标准，对 Phase 1+ 衔接质量的实质影响极低，适合归为"建议性改善"。修订标题从"§1.1 common 模块 security 依赖的陈述理由与实际架构矛盾（P1）"改为"§1.1 common 模块 security 依赖的陈述理由与实际架构矛盾（P2）"。

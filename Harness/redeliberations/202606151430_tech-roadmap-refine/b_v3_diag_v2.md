# 质量审查报告 — 第 3 轮（修订版 v2）

审查范围：需求响应充分度、事实/逻辑正确性、深度与完整性。

## 发现问题

### 1. [P1] Phase 0 未区分"最小可协作骨架"与"推荐补齐配套"
- **问题描述**：需求明确要求"明确区分'最小可协作骨架'和'推荐补齐配套'"，使读者一眼看出哪些是并行协作的硬性前提、哪些是可选的补齐完善内容。当前 Phase 0.2 将 7 项交付物（工程结构、前后端骨架、协作规范、CI 占位、共享组件、本地启动、持续集成）扁平罗列，未做任何优先级或必要性分级标记。其中 CI 占位流水线在 Phase 1 即正式落地，属于"可不在 Phase 0 完成"的推荐补齐项，但当前与骨架级交付物处于同一层级，读者无法区分。对比源文档 `Docs/a_v13_tech_v1.md` 第 35~50 行可见，源文档已通过"硬必需/软预备"层级表实现了这一区分，而本轮产出在简化过程中丢失了此关键信息。
- **所在位置**：Phase 0.2 交付物列表（第 33~41 行）
- **严重程度**：一般（P1）
- **改进建议**：恢复源文档中的"硬必需/软预备"二级分层标记，或将 Phase 0.2 拆分为"骨架必备（阻塞并行）"与"推荐补齐（可在后续阶段首期补齐）"两个子列表。

### 2. [P2] Phase 4 并行任务包数/关键路径数与建议开发人数存在逻辑矛盾
- **问题描述**：Phase 4.1 声明"9 个任务包/7 条关键路径"且"建议至少 5 位开发者并发"，Phase 4.3 拆出 9 个并行包，风险表的缓解策略同样建议"至少配置 5 位开发者"。7 条关键路径至少需要 7 个开发人员并行推进才能避免路径级联等待。5 人无法覆盖 7 条关键路径，建议人数与实际需求存在明显缺口；若部分关键路径可串行，则"7 条关键路径"定义本身需要澄清——否则读者在编排人力时会产生困惑。
- **所在位置**：Phase 4.1（第 171 行）、Phase 4.3（第 188~198 行）、风险表第 3 行（第 299 行）
- **严重程度**：轻微（P2）
- **改进建议**：(a) 将风险表中"至少配置 5 位开发者"修正为"至少 7 位开发者"，或 (b) 明确说明 9 个任务包如何在 7 条关键路径上排列、哪些路径可以串行，或 (c) 精简关键路径数量至 5 条以内使其与建议人数匹配。

### 3. [P2] AI 进阶底座与前期 AI 能力之间的迁移成本未说明
- **问题描述**：Phase 2~Phase 4 分别接入各自的 AI 能力（Phase 2 "Spring AI ChatModel 接入"、Phase 3 "Structured Output"与本地规则校验、Phase 4 多项 AI 推理），而 AI 进阶底座（LLM 凭证集中管理、多模型路由、Prompt 模板版本管理等五组件）直到 Phase 5 才落地。这意味着前期各阶段独立完成的 AI 接入方案需要在 Phase 5 重构迁移至集中底座，存在显式的迁移/重构成本。路线图未提及此成本，也未说明前期实现是否需要预留适配接口。对于读者（尤其是参与 Phase 2 的开发者）来说，这会直接影响前期实现方案的选择。
- **所在位置**：Phase 5.2 "AI 进阶底座（五组件）"（第 224 行）
- **严重程度**：轻微（P2）
- **改进建议**：在 Phase 5 的"做完后新增什么能力"或本阶段"不包含"中增加一行说明文字，标注前期独立 AI 接入方案向集中底座升级的预期工作量和注意事项，或在 Phase 2 入口处增加"AI 基础接入方案将在 Phase 5 统一升级至集中底座，当前设计应考虑后期迁移"的指导性说明。

### 4. [P3] Phase 0 验收条件引用未在交付物中列出的文档
- **问题描述**：Phase 0.2 验收条件写道"新人按 AGENTS.md 可在 1 小时内本地拉起全部工程"，但 Phase 0.2 交付物列表中未包含 AGENTS.md 文档。AGENTS.md 作为新人上手的关键指引，应在交付物中显式列出，否则验收条件无法自洽。
- **所在位置**：Phase 0.2 验收条件（第 42 行）
- **严重程度**：轻微（P3）
- **改进建议**：在 Phase 0.2 交付物列表中补充"入门引导文档（AGENTS.md）"。

## 补充审查：核心需求系统性验证

应上一轮质询要求，对需求中的四项核心要求进行逐项验证。

### 验证项 1：各阶段是否已按"四件事"结构收敛
- **要求**：每个阶段收敛为四件事——①为什么现在做、②做完后新增什么能力、③可以并行分给哪些人、④本阶段明确不包含。
- **结论**：✅ **已满足**
- **证据**：Phase 0~Phase 6 全部 7 个阶段均已采用 0.1/1.1/2.1...（为什么现在做）、0.2/1.2/2.2...（新增能力）、0.3/1.3/2.3...（并行分配）、0.4/1.4/2.4...（不包含）的四段式结构，不存在缺少或顺序错乱的情况。

### 验证项 2：细节下沉是否已落实
- **要求**：详细状态机枚举、接口冻结时点表、具体超时秒数、降级实现细节、依赖占位和工程脚手架细则等设计化内容应从路线图正文移除或弱化。
- **结论**：✅ **已满足**
- **证据**：对比源文档 `Docs/a_v13_tech_v1.md`，原文中的以下设计级内容已从产出中移除：①各类详细状态机枚举（退药状态机、调拨六态、盘库四态等）；②模块级接口冻结时点表（源文档 Phase 0.2 表内嵌的冻结时点列）；③具体超时秒数（如源文档 Phase 2 取消规则中的"2h"等硬编码值）；④降级实现细节（如源文档 Phase 3 中"Mock 占位接口+本地规则表"等描述）；⑤依赖占位细节（如 SQL-Toy 依赖引入说明、Spring Cloud 兼容依赖占位等）；⑥工程脚手架细则（如模块内三层包名约定等）。以上内容已在当前产出中简化为阶段级功能描述。

### 验证项 3：Phase 0 压缩程度与"最小/推荐"区分
- **要求**：(a) Phase 0 再压轻一些；(b) 明确区分"最小可协作骨架"和"推荐补齐配套"。
- **结论**：⚠️ **部分满足**
- **证据**：
  - (a) 压缩程度可接受：Phase 0 从源文档约 120 行（含表格）压缩至当前约 35 行（第 26~61 行），体量已显著减少，符合"再压轻一些"的要求。
  - (b) 未满足：当前 Phase 0.2 仍将 7 项交付物扁平罗列，未做"骨架/补齐"的二级分层对比。源文档第 35~50 行已通过"硬必需/软预备"层级列实现了此区分，但产出在简化过程中丢失了这一关键信息。详见问题 #1。

### 验证项 4：文档是否可读为纯粹路线图而非混合体
- **要求**：文档定位为"指导项目推进的路线图"，而非"路线图和详细设计的混合体"。
- **结论**：✅ **已满足**
- **证据**：文档头部第 3~7 行明确声明定位为"实现路线图"、职责边界（不包含 OOD 类图、方法签名、接口字段表、代码片段等）。正文内容均保持在阶段级功能描述粒度，无代码片段、无类图、无接口字段级细节。各阶段"本阶段明确不包含"子节的交叉引用有助于读者理解文档边界。

## 一、需求响应充分度

### 1.1 已充分响应的方面

- ✅ 保留 Phase 0 ~ Phase 6 主结构未动
- ✅ 各阶段统一采用四要素框架（为什么现在做 / 做完后新增能力 / 并行分配 / 不包含内容）
- ✅ Phase 0 已做"骨架必备"与"推荐补齐"的划分尝试
- ✅ 验收标准已融入"做完后新增能力"段落末尾，不作为独立要素
- ✅ v7~v9 三轮修订已清理大部分实现级术语（如 Structured Output、RAG、SQL-Toy、device-gateway 模块名等）

### 1.2 响应不充分的问题

#### 问题 1：Phase 0.3 并行分配模块命名与"最小化骨架"定位存在张力

**问题描述：** Phase 0.3 列出 7 个并行模块（A~G），模块名称直接使用完整的业务模块名（如"患者端前端 + 患者后端模块""管理员端前端 + 管理员后端模块"）。但 Phase 0.4 声明"不包含任何业务功能"。读者看到模块名时容易误判 Phase 0 的工作范围——以为 Phase 0 就要开始开发这些模块的业务功能，与"最小化骨架"的定位产生矛盾。

**所在位置：** Phase 0.3（第 49~55 行）

**严重程度：** 中

**改进建议：** 在模块命名上明确标注 Phase 0 的工作边界，例如"患者端前端 + 患者后端模块（仅工程骨架搭建，不包含业务功能）"，或在各模块名后追加上标说明，使读者一眼看出这些分配针对的是模块工程结构搭建而非功能开发。也可考虑将 0.3 的模块列表改为以"搭建任务"为粒度而不是以"模块名"为粒度。

#### 问题 2：Phase 0.2 "推荐补齐"仅 1 项，轻重划分的实际指导意义不足

**问题描述：** 用户要求"明确区分'最小可协作骨架'和'推荐补齐配套'"。当前 Phase 0.2 在"骨架必备"下列出 5 项关键交付物，在"推荐补齐"下仅列出"持续集成"1 项。既然用户特地点名要求区分这两类，说明预期中有多项可被识别为"推荐/非阻塞"的内容。仅 1 项的划分结果显得刻意且信息量不足，要么是该划分本身不成立（应合并回骨架），要么是遗漏了应被识别为推荐补齐的其他项。

**所在位置：** Phase 0.2（第 36~45 行）

**严重程度：** 中

**改进建议：** 方案 A：补充更多推荐补齐项，例如"日志聚合框架占位""监控埋点接入点预留""API 文档生成工具集成""Docker 化部署脚本"等，使"推荐补齐"与"骨架必备"在体量上形成有意义的对比。方案 B：如果经过审视确认只有 CI 这一项是可选的，则删除"推荐补齐"这个子分类，将 CI 降级为骨架必备中的一个次要项，或在 0.4 不包含清单中注明"CI 门禁（在 Phase 1 落地，Phase 0 可选）。

#### 问题 3：Phase 0.3 模块 F（基础设施）与 Phase 0.2 推荐补齐的 CI 之间存在隐含矛盾

**问题描述：** Phase 0.2 将 CI 列为"推荐补齐（可在后续阶段首期补齐）"，暗示 CI 不是 Phase 0 的阻塞项。但 Phase 0.3 模块 F 列出"基础设施（DB / CI / 部署脚本）"作为并行分配的一个工作包，且该包与其他骨架模块并列，未标记为可选/推荐。读者会困惑：CI 到底是"不是必须的"还是"有人已经在 Phase 0 里做了"？两处表述口径不一致。

**所在位置：** Phase 0.2（第 43 行）vs Phase 0.3（第 54 行）

**严重程度：** 中

**改进建议：** 统一两处表述：如果 CI 确实是推荐补齐（即可以在 Phase 1 做），则 Phase 0.3 模块 F 应注明"CI 可在 Phase 1 补充，不阻塞 Phase 0 验收"；如果 CI 是 Phase 0 必须完成的基础设施工作，则从 0.2 的"推荐补齐"移入"骨架必备"。

## 二、残留过设计内容

#### 问题 4：验收标准中残留"状态机"实现级术语

**问题描述：** 多处验收标准和交付物描述中直接使用"状态机"一词，属于工程实现概念而非能力描述。路线图级别的文档应描述"系统具备什么能力"，而非"系统用什么工程模式实现"。

具体残留位置：
- Phase 2.2 验收标准："取消规则引擎时间窗与状态机冒烟通过"
- Phase 4.2："缴费记录状态管理"
- Phase 4.3 包 C："药房域（发药/退药/库存/状态机）"
- Phase 4.3 包 E："线下窗口 + 缴费记录状态机"
- Phase 4.1 路径 5："线下挂号/退号 + 收费/退费 + 缴费记录状态机"

**严重程度：** 低（术语粒度可控，但属于未清理干净的"残留"）

**改进建议：** 将"状态机"统一替换为能力级描述，例如：
- "取消规则引擎时间窗与状态机冒烟通过" → "取消规则引擎按时间窗口正确执行"
- "缴费记录状态管理" → "缴费记录全生命周期管理（待支付/已支付/已退款/已对账）"
- "发药/退药/库存/状态机" → "发药/退药/库存全流程管理"

#### 问题 5：Phase 4.1 关键路径表颗粒度过细，超出路线图层级

**问题描述：** Phase 4.1 的 7 条关键路径定义表包含：每条路径的归属任务包、依赖/解耦关系、路径独立性声明、串行化压缩后的并行路数估算、建议开发者人数。这些内容在项目管理计划（Project Plan）层面是合理的，但在路线图（Roadmap）层面属于过度展开。一般读者只需要知道"Phase 4 是最复杂阶段，有若干并行路径，各路径基本解耦"即可，不需要逐条路径地看依赖/解耦关系。

**所在位置：** Phase 4.1 关键路径定义表（第 177~187 行）

**严重程度：** 中

**改进建议：** 将 7 条路径的细节表移出路线图正文，保留一句话概括："Phase 4 是本路线图中工作量最大的阶段，共 7 条可基本独立并行的关键路径（检查域/检验域/药房药库/AI 推理/线下窗口与缴费/硬件接入/数据接入与健康档案）"。详细路径表可作为附录供 PM 参考，或仅在启动 Phase 4 的详细计划中展开。

#### 问题 6：风险表中包含具体数值和供应商名称

**问题描述：** 风险表中出现多个具体数值和名称，属于项目执行中可能随时变化的细节：
- "每个阶段在其工期中预留 15% 缓冲天数"——具体百分比
- "备选供应商评估清单至少 2 家，如 DeepSeek/Qwen/GLM"——具体供应商名
- "外部依赖分类...切换时间窗口已标注"（窗口未给出具体值但暗示存在，且在路线图中无实际用途）

**所在位置：** 风险与依赖表（第 313~323 行）

**严重程度：** 低

**改进建议：** 将"15%"改为"预留缓冲时间"，将具体供应商名改为"至少 2 家备选 LLM 供应商（如国产主流大模型）"。风险表的职责是提示风险方向而非给出精确数字。

## 三、四要素结构一致性

#### 问题 7：AI 能力迁移提醒在正文中重复 4 次

**问题描述：** "注：本阶段 AI 接入方案为独立实现，Phase 5 AI 进阶底座落地后统一迁移"这句话在 Phase 2.2 和 Phase 3.2 中累计出现 4 次（分别针对 3.4.1/3.4.2/3.4.3/3.4.10）。四要素结构的"做完后新增能力"段落应展示差异化的能力交付，而非逐项标注相同的迁移策略。重复 4 次降低了文档的信息密度和可读性。

**所在位置：** Phase 2.2（关于 3.4.1）、Phase 3.2（关于 3.4.2/3.4.3/3.4.10）

**严重程度：** 低

**改进建议：** 删除正文中的重复提醒，改为在"阅读指引"或"编号约定"中统一声明："Phase 2/3/4 中首次落地的 AI 能力均采用独立 AI 接入方案，Phase 5 AI 进阶底座落地后统一迁移"。同时可在附录 C 增加一列"底座迁移"标注哪些能力需要迁移。

#### 问题 8：Phase 0.2 验收标准中包含了超出"最小骨架"范围的验证要求

**问题描述：** Phase 0.2 验收标准是"三端可一键启动到占位首页，新人按入门引导文档可在 1 小时内本地拉起全部工程"。但"三端可一键启动"依赖前端工程已经具备可运行的空壳页面，而用户要求 Phase 0 "再压轻一些"。当前验收标准本质上要求三端前端骨架都已就位且可运行，这与"最小化骨架"的定位是一致的，但验收标准中的"全部工程"一词暗示包含后端。如果后端骨架未完全就位（比如只有空壳结构但未配置启动），验证可能失败。文档未明确"三端一键启动"是否依赖后端服务也在同一时间就位。

**所在位置：** Phase 0.2 验收标准（第 45 行）

**严重程度：** 低

**改进建议：** 明确验收标准中"全部工程"指什么——是否是"三端前端 + 后端基础骨架可同时本地启动"。建议改为："三端前端可一键启动到占位首页；后端基础骨架可独立启动并响应 ping 健康检查；新人按入门引导文档可在 1 小时内完成本地环境搭建。"

## 四、深度与完整性评估

### 4.1 整体评价

产出整体质量较高，经过 v7~v9 三轮修订后，核心过设计问题已得到有效清理。四要素结构在全部 7 个阶段中贯彻一致，Phase 0 的骨架/推荐区分已有初步框架，附录 C 的 AI 能力索引提供了良好的快速参考价值。文档在全阶段覆盖、角色覆盖、业务域覆盖、AI 能力覆盖方面均无显著遗漏。

### 4.2 未发现的问题

- **事实错误：** 未发现编号对照错误、AI 能力计数遗漏、阶段间依赖关系矛盾
- **逻辑矛盾：** 除上述问题 3（CI 角色不统一）外，未发现其他矛盾
- **关键遗漏：** 13 项 AI 能力全覆盖，三大角色全阶段覆盖，六大业务域全阶段覆盖

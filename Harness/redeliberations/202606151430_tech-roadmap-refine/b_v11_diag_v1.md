## 审查发现

### 问题 1（中）：Phase 4.1 正文存在逻辑矛盾——"路径解耦独立"与"串行化压缩"表述冲突

**问题描述**：Phase 4.1 "为什么现在做"段落中，第 182 行声称"7 条关键路径解耦独立、均可并行推进"，但第 184 行紧接着说"经串行化压缩后，有效并行路数约为 5 条"。两条陈述并排放置，读者会立即产生困惑：如果 7 条路径已"解耦独立"（即不存在彼此阻塞的依赖关系），则不存在"串行化压缩"的优化空间；如果存在压缩，则"解耦独立"的表述言过其实。

实际的约束在附录 D 中可以找到——路径 3（药房药库）内部包 D1→D2 存在串行依赖（药库基础管理完成后才能启动调拨管理）；路径 7（数据接入+健康档案增强）标注为"非专用人力"，实质需要复用其他路径完成后的资源。因此"7 条路径均解耦独立"是一个不精确的概括。

**所在位置**：Phase 4.1，第 182 行和第 184 行

**严重程度**：中

**改进建议**：统一表述。建议以下三选一：
- 方案 A（推荐）：删除第 182 行中的"解耦独立"，改为"7 条关键路径均已定义解耦边界，可并行推进"
- 方案 B：删除第 184 行中的"经串行化压缩后"，将结论改为"7 条路径并行推进，考虑内部依赖与资源复用后，建议配置至少 5 位开发者"
- 方案 C：明确解释压缩由来，例如"7 条路径可并行启动；考虑路径 3 内部串行步骤和路径 7 的资源复用约束后，有效并行路数约为 5 条"

### 问题 2（中低）：Phase 0.2 推荐补齐含具体工程工具名，未完全落实"细节下沉"要求

**问题描述**：Phase 0.2 "推荐补齐"子列表中，两项描述仍包含具体产品名和工具名：
- 第 47 行："OpenAPI/**Swagger** 自动生成工具集成"——"Swagger"是具体产品品牌名
- 第 49 行："**ESLint / Checkstyle** 等规范检查工具配置"——"ESLint""Checkstyle"是具体工具名

用户需求明确要求将"工程脚手架细则"从路线图正文中移除或弱化。虽然这两项属于"推荐补齐"（非核心骨架），但具体工具名的存在仍使路线图文风偏向实现细节，与"让读者一眼看出这是路线图而非详细设计"的目标存在差距。

**所在位置**：Phase 0.2 "推荐补齐"段落，第 47 行和第 49 行

**严重程度**：中低

**改进建议**：将具体工具名从描述中移除，只保留能力级描述：
- 第 47 行改为：**"API 文档自动生成**：自动生成工具集成，开发期即可查阅 API 契约"
- 第 49 行改为：**"本地代码质量检查工具集成**：代码规范检查工具配置"

（子标题本身已是抽象描述，正文细节降级为能力级即可）

### 问题 3（低）：Phase 2.2 / 3.2 残留 AI 实现层术语 "Prompt 模板"

**问题描述**：两处使用了"Prompt 模板"这一 AI prompt engineering 领域的实现层术语：
- Phase 2.2 "AI 核心 1 [智能分诊]"（第 117 行）："AI 接入满足单次/多轮双对话模式，**Prompt 模板可配置**"
- Phase 3.2 "AI 核心 3 [AI 病历生成]"（第 150 行）："将对话转结构化病历；**按科室配置 Prompt 模板**"

"Prompt 模板"是具体实现概念（指 prompt engineering 中的 template），属于应下沉至 OOD 或技术设计文档的过度设计内容。在路线图层级，应描述为业务规则的可配置能力，而非 Prompt 技术细节。

**所在位置**：Phase 2.2 第 117 行；Phase 3.2 第 150 行

**严重程度**：低

**改进建议**：
- 第 117 行改为："AI 接入满足单次/多轮双对话模式，**分诊规则可配置**"
- 第 150 行改为："将对话转结构化病历；**按科室配置病历生成规则**"

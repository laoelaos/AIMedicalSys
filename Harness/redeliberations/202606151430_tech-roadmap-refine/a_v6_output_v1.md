# 智慧云脑诊疗平台 —— 项目实现路线图

> 文档定位：实现路线图（Implementation Roadmap）。回答实现路径、阶段划分、阶段内任务清单。
> 文档职责不包含：OOD 类图、方法签名、接口字段表、代码片段、测试用例细节、变更历史。
> 阶段粒度：7 个阶段（Phase 0 ~ Phase 6）。Phase 0 为最小化骨架，是本路线图的硬性起点。
> 编号约定：3.4.x = AI 能力编号；3.1.x = 患者端需求；3.2.x = 医生端需求；3.3.x = 管理员端需求。

## Phase 0 — 最小化骨架（项目初始化阶段）

### 0.1 为什么现在做

多人并行开发的必要条件是需要一个共享的工程"地形"——统一的工程结构、技术基线、接口契约框架和协作规范。Phase 0 不实现任何业务功能，只交付协作所需的最小骨架。

### 0.2 做完后新增什么能力

**骨架必备（阻塞并行）**
- **共享工程结构**：Monorepo 布局，多模块后端工程与多端前端工程就位，支持多人并行贡献
- **接口契约框架**：统一响应包装、分页、错误码命名空间、AI 能力占位类型等规范对齐
- **数据与权限模型骨架**：数据实体基类与权限模型就位，支撑后续业务模块快速搭建
- **协作规范**：分支约定、Commit 格式、PR 模板、Code Review 必查项
- **本地开发体验**：一键启动脚本与入门引导文档，新人在可预期时间内完成本地环境搭建

**推荐补齐（可在后续阶段首期补齐）**
- **持续集成**：CI 占位流水线（配置文件存在、可跑构建+单测占位；完整 CI 门禁在 Phase 1 落地）

验收标准：三端可一键启动到占位首页，新人按入门引导文档可在 1 小时内本地拉起全部工程。

### 0.3 可以并行分给哪些人

- 模块 A：患者端前端 + 患者后端模块
- 模块 B：医生端前端 + 医生后端模块
- 模块 C：检查/检验/药房后端模块
- 模块 D：管理员端前端 + 管理员后端模块
- 模块 E：AI 能力模块的 Mock 实现
- 模块 F：基础设施（DB / CI / 部署脚本）
- 模块 G：（选做）硬件采集模块（接口占位）

### 0.4 本阶段明确不包含

- 任何业务功能（如挂号、分诊、处方）
- 模块级接口契约冻结（在对应阶段启动前冻结）
- CI 门禁清单（构建+单测+OpenAPI diff+覆盖率阈值等，归 Phase 1）
- 真实 LLM/AI 接入
- 微服务化拆分

## Phase 1 — 基础设施 + 统一认证与权限（地基阶段）

### 1.1 为什么现在做

三端所有业务功能都依赖统一的身份认证与权限体系。Phase 0 骨架搭好后，第一件要做的事就是让用户能进入系统。同时，健康档案的基础字段需要尽早录入，为后续 AI 能力提供数据源。

### 1.2 做完后新增什么能力

- **统一认证**：登录（密码+JWT）、刷新 Token、登出、当前用户信息接口
- **权限矩阵**：角色—岗位—功能三级权限模型落地；前端按权限动态渲染菜单与按钮
- **患者端基础**：注册/登录、个人中心（仅查看）、健康档案基础版（基本字段 + 5 类结构化子字段 CRUD；前端 UI 同步提供过敏详情扩展容器的录入能力）
- **医生端基础**：医生登录、岗位标识加载、个人资料维护
- **管理员端基础**：管理员登录、用户/角色/岗位管理、菜单管理、字典管理
- **公共能力**：统一响应包装、全局异常处理、参数校验、审计日志、操作日志

验收标准：三大终端能完成"登录 → 看到与自己角色匹配的菜单"全链路冒烟通过。

### 1.3 可以并行分给哪些人

- 包 A：用户/角色/岗位/菜单/字典数据建模与初始 SQL
- 包 B：Spring Security + JWT 实现
- 包 C：患者端注册/登录/个人中心/健康档案 UI + API
- 包 D：医生端与管理端登录/菜单动态化

### 1.4 本阶段明确不包含

- AI 相关能力
- 患者挂号/分诊流程
- 健康档案的按维度查询/对比/趋势阅读能力（归 Phase 4）
- 过敏详情扩展容器的后端消费（Phase 2 起启用）

## Phase 2 — 患者服务域 + 核心三件套之"智能分诊"（患者端主流程）

### 2.1 为什么现在做

患者端是系统的第一用户入口。在有了基础账号体系后，应首先跑通患者从"出现症状 → AI 分诊 → 挂号 → 查看记录"的主流程，并落地核心三件套中与患者最直接相关的第一件——AI 智能分诊。

### 2.2 做完后新增什么能力

- **患者端主流程**：AI 智能问诊+导诊、智能挂号（门诊/检查双分支）、取消挂号、报告/病历/处方查询（占位可空）、缴费记录查询（占位可空）、首页+便民服务、病情咨询 UI（AI 走 Mock 兜底+免责声明）
- **AI 核心 1 [智能分诊]（3.4.1，首次落地）**：AI 接入满足单次/多轮双对话模式，Prompt 模板可配置；AI 不可用时回退到按科室列表选择。注：本阶段 AI 接入方案为独立实现，Phase 5 AI 进阶底座落地后统一迁移。
- **挂号中心后端**：双分支挂号、取消规则引擎（按距就诊时间窗口执行：窗口内可取消，窗口外引导至线下窗口）、分诊记录表
- **检查预约占位**（占位状态，Phase 4 派单上线后完成真实数据接入）

验收标准：智能分诊（3.4.1）首次落地 + 挂号流程演示链路通畅；AI 不可用时降级路径可演示；取消规则引擎时间窗与状态机冒烟通过。

### 2.3 可以并行分给哪些人

- 包 A：挂号中心后端（含双分支与取消规则引擎）
- 包 B：患者端 UI（智能问诊 + 挂号 + 取消 + 首页 + 便民服务）
- 包 C：AI 智能分诊后端模块 + Prompt 模板 + Mock 兜底
- 包 D：分诊记录表与历史查询 API
- 包 E：病情咨询 UI + Mock 兜底文案 + AI 免责声明标识

### 2.4 本阶段明确不包含

- 真实知识库问答（Phase 5 升级）
- 缴费记录状态机真实数据（Phase 4 接入）
- 检查/检验/药房相关能力
- 3.1.8 药房取药入口接真实数据（Phase 4）

## Phase 3 — 医生端核心 + 核心三件套之"AI 处方审核 / AI 病历生成"（诊疗主战场）

### 3.1 为什么现在做

门诊诊疗是平台最核心的业务闭环。在患者可挂号后，需要让门诊医生能完成"叫号 → 接诊 → 开方 → 审核 → 病历生成"全流程，并在此阶段落地核心三件套的剩余两件——AI 处方审核和 AI 病历生成。3.4.10 辅助开方与 3.4.2 处方审核构成直接上下游耦合，同步落地。

### 3.2 做完后新增什么能力

- **医生端诊疗闭环**：挂号管理、病人信息、病情录入、AI 辅助开方（3.4.10）、AI 处方审核（3.4.2，核心 2）、AI 病历生成（3.4.3，核心 3）、病历保存/列表/详情
- **AI 核心 2 [AI 处方审核]（3.4.2，首次落地）**：基于风险等级对处方进行自动化审核，不同风险档位采取差异化阻断策略；AI 超时回退本地规则校验并打标。注：本阶段 AI 接入方案为独立实现，Phase 5 AI 进阶底座落地后统一迁移。
- **AI 核心 3 [AI 病历生成]（3.4.3，首次落地）**：Structured Output 将对话转结构化病历；按科室配置 Prompt 模板；关键字段缺失时回提示补全。注：本阶段 AI 接入方案为独立实现，Phase 5 AI 进阶底座落地后统一迁移。
- **AI 拓展 [辅助开方]（3.4.10，首次落地）**：剂量阈值告警。注：本阶段 AI 接入方案为独立实现，Phase 5 AI 进阶底座落地后统一迁移。
- **后端**：处方中心、医嘱开立+记账（仅生成收费前置单据）、病历中心、AI 审核记录持久化、本地规则校验（AI 不可用降级）
- **3.4.4/3.4.9 在本阶段为功能入口占位**：AI 智能诊断（3.4.4）和 AI 开立检查/检验（3.4.9）在本阶段提供功能入口与降级引导。真实 AI 推理在 Phase 4 落地。
- **患者端 3.1.5 草稿可见**：病历生成/处方审核落库后，患者端即可查看对应草稿版本

验收标准：核心 2 + 核心 3 首次落地可演示；3.4.10 同步冒烟通过；3.4.4/3.4.9 功能入口占位降级路径冒烟通过；医生端单端诊疗闭环冒烟通过。

### 3.3 可以并行分给哪些人

- 包 A：医生端挂号/病人信息/病情录入/病历 UI
- 包 B：医生端 AI 相关 UI（占位诊断/开立检查/辅助开方/处方审核/病历生成 + 降级标识）
- 包 C：处方中心 + 病历中心后端
- 包 D-AI1：AI 处方审核（3.4.2，核心 2）
- 包 D-AI2：AI 病历生成（3.4.3，核心 3）
- 包 E：AI 辅助开方（3.4.10）
- 包 F：3.4.4/3.4.9 占位实现
- 包 G：医嘱开立 + 记账（收费前置单生成）

### 3.4 本阶段明确不包含

- 智能诊断（3.4.4）和开立检查（3.4.9）的真实 AI 推理（Phase 4）
- 检查/检验/药房/线下窗口相关能力
- 全链路联调（归 Phase 6）
- 流式病历输出（SSE，归 Phase 6）

## Phase 4 — 检查 / 检验 / 药房 / 药库 / 线下窗口 / 硬件接入（业务域拓展）

### 4.1 为什么现在做

医生端核心诊疗闭环跑通后，需要将 5 个医生子岗位全部落地，打通检查、检验、药房、线下窗口四条完整业务流，并承接 Phase 3 占位的 AI 能力的真实推理。本阶段工作量最大（9 个任务包/7 条关键路径），7 条关键路径定义如下：

| 路径编号 | 关键路径名称 | 归属任务包 | 依赖/解耦关系 |
|---------|-------------|-----------|--------------|
| 路径1 | 检查域 | 包 A（检查域全流程 + AI 检查报告 + AI 影像分析 + 执行顺序推荐） | 独立路径，不阻塞其他路径 |
| 路径2 | 检验域 | 包 B（检验域全流程 + AI 检验报告 + 趋势图） | 独立路径，不阻塞其他路径 |
| 路径3 | 药房药库 | 包 C（药房发药/退药/库存）+ 包 D1（药库基础管理）+ 包 D2（调拨管理） | 包 D1/D2 可串行执行（药库基础管理完成后启动调拨管理），与包 C 弱依赖 |
| 路径4 | AI 真实推理 | 包 D（3.4.4/3.4.9 真实 AI 推理升级） | 需要 Phase 3 占位实现已部署，但不阻塞其他路径 |
| 路径5 | 线下窗口+缴费 | 包 E（线下挂号/退号 + 收费/退费 + 缴费记录状态机） | 独立路径，不阻塞其他路径 |
| 路径6 | 硬件接入 | 包 F（device-gateway 模块接口契约 + Mock 数据） | 独立路径，不阻塞其他路径 |
| 路径7 | 数据接入+健康档案增强 | 包 G（患者端报告/缴费/取药真实数据接入 + 健康档案增强查询） | 可复用其他路径人力（优先由包 A/B/C 完成后接力），非专用人力 |

经串行化压缩后，有效并行路数约为 5 条，建议至少配置 5 位开发者并行推进。

### 4.2 做完后新增什么能力

- **检查域**：检查管理全流程、AI 检查报告（3.4.5）、AI 影像分析推理（3.4.7，首次落地）、AI 执行顺序推荐（3.4.11）、AI 开立检查/检验真实推理（3.4.9，从 Phase 3 占位升级）
- **检验域**：检验管理全流程、AI 检验报告（3.4.6，首次落地）、检验趋势图（SQL-Toy 聚合）
- **药房域**：药房管理、发药/退药全流程
- **药库域**：药库管理、调拨与盘库流程
- **线下窗口**：线下挂号/退号、收费/退费、缴费记录状态管理
- **AI 智能诊断真实推理（3.4.4，首次落地）**：从 Phase 3 占位升级为真实 AI 推理
- **硬件接入**：device-gateway 模块接口契约 + Mock 数据（HL7/DICOM/串口/网关四种接入路径）
- **健康档案增强**：档案查询（按时间/机构/类别）、长期健康趋势合并展示
- **患者端真实数据接入**：3.1.5 完整报告接入、3.1.7 缴费记录查询、3.1.8 药房取药

验收标准：检查/检验/药房/线下窗口四条主流程可演示；调拨、盘库、退药、缴费记录各流程冒烟通过；硬件接口契约可联调消费；3.4.5/3.4.6/3.4.7/3.4.11 降级路径冒烟通过。

### 4.3 可以并行分给哪些人

- 包 A：检查域（3.2.3 + 3.4.5 + 3.4.7 分析推理 + 3.4.11）
- 包 B：检验域（3.2.4 + 3.4.6 + 趋势图 SQL-Toy）
- 包 C：药房域（发药/退药/库存/状态机）
- 包 D：3.4.4/3.4.9 真实 AI 推理
- 包 D1：药库基础管理 + 盘库
- 包 D2：调拨管理
- 包 E：线下窗口 + 缴费记录状态机
- 包 F：硬件接入（device-gateway）
- 包 G：真实数据接入 + 健康档案增强

### 4.4 本阶段明确不包含

- 管理员端综合管理（Phase 5）
- 知识库问答（3.4.8，Phase 5）
- AI 排班（3.4.12）、综合讨论结论（3.4.13）
- AI 影像模型管理面（Phase 5）
- 微服务化拆分
- 全链路联调

## Phase 5 — 医院管理域 + 拓展 AI 能力（管理员端与剩余 AI）

### 5.1 为什么现在做

四大业务域落地后，系统的日常运营需要管理员端支撑，剩余 4 项拓展 AI 能力（知识库、排班、综合讨论结论、影像模型管理面）和 AI 进阶底座也需要在此阶段补齐。同时将 Phase 2 的病情咨询从 Mock 升级为真实知识库调用。

### 5.2 做完后新增什么能力

- **管理员端**：科室/医生/检查检验项目/药品字典维护、排班管理（含 AI 排班 3.4.12）、AI 分诊台规则配置（3.3.3）、综合管理
- **AI 拓展能力**：
  - **AI 知识库问答（3.4.8，首次落地）**：RAG 接入医学知识库；Phase 2 病情咨询从 Mock 升级为真实调用
  - **AI 医生排班（3.4.12，首次落地）**：基础规则+AI 推荐双模式
  - **AI 综合讨论结论（3.4.13，首次落地）**
  - **AI 影像模型管理面（3.4.7 扩展）**：模型注册、岗位授权、调用审计
- **AI 进阶底座**：实现 LLM 对接、模型路由、Prompt 管理、A/B 实验、性能观测五大基础能力。注意：Phase 2~Phase 4 各阶段独立接入的 AI 能力（3.4.1/3.4.2/3.4.3/3.4.10 等）将在本阶段统一迁移至本底座，前期各阶段 AI 接入方案存在重构迁移成本。
- **医生端补全**：AI 智能讨论结论（3.2.8）

验收标准：13 项 AI 能力全部完成首次落地；AI 进阶底座五大基础能力均可冒烟；3.4.12/3.4.13 降级路径冒烟通过。

### 5.3 可以并行分给哪些人

## Phase 6 — 全链路联调 + 实时通信 + 部署与运维（收口与上线）

### 6.1 为什么现在做

所有功能已落地，需要将全链路贯通、补齐实时通信与部署能力，并建立 AI 持续优化机制，使系统达到可上线状态。

### 6.2 做完后新增什么能力

- **全链路联调**：注册登录→分诊→挂号→接诊→AI 诊断→开立检查→执行检查→报告→开方→审核→病历→缴费→发药→退药→退费端到端贯通
- **实时通信**：WebSocket 高风险用药告警推送、SSE 病历生成流式输出、前端状态管理同步
- **部署运维**：Nginx 静态资源+Spring Boot Jar、一键启动/停止脚本、生产部署文档、健康检查+就绪探针
- **可观测性**：日志聚合与全链路追踪、性能指标埋点、错误监控、慢调用追踪
- **文档材料**：QUICKSTART.md、DEPLOY.md、ARCHITECTURE.md、API.md、演示 PPT
- **AI 持续优化**：向量库管理（医学知识库分块/索引/回滚）、A/B 实验效果对比页、推理性能基线（P50/P95 时延、成功率、单次成本）

验收标准：六大业务域、三类角色、13 项 AI 能力全部可演示且全部具备降级路径；性能/可观测性/安全性按要求验收。

### 6.3 可以并行分给哪些人

- 包 A：联调（接口契约、跨模块数据流、测试集）
- 包 B：实时通信（WebSocket + SSE）
- 包 C：部署（Nginx + Jar + 脚本 + 部署文档）
- 包 D：可观测性（日志、指标、追踪）
- 包 E：演示材料（架构图、PPT、演示脚本）
- 包 F：AI 持续优化（向量库管理 + A/B 框架 + 性能基线）

### 6.4 本阶段明确不包含

- 新的 AI 能力首次落地（13 项 AI 已在 Phase 2~5 完成首次落地）
- 微服务化拆分（已预留演进路径）

## 风险与依赖（路线图层面战略风险）

| 风险 | 影响阶段 | 缓解策略 |
|------|----------|----------|
| Phase 4 工作量集中度过高（9 个并行任务包/7 条关键路径） | Phase 4 | 经串行化压缩后核心并行路数约为 5 条；建议至少配置 5 位开发者。各路径解耦关系见 Phase 4.1 关键路径定义表。 |
| 骨架阶段"最小化"被误读为省略关键基础设施 | Phase 0 | Phase 0 以"支持多人并行贡献"为最小化判据 |
| 13 项 AI 能力降级路径完整性 | Phase 2~5 | 13 项 AI 能力均已声明降级路径 |
| 单仓多模块→微服务化拆分时复杂度上升 | Phase 5/6 | Phase 0 已预留演进路径与 Spring Cloud 兼容依赖 |
| AI 进阶底座演进影响业务模块兼容性 | Phase 5/6 | ai-ability 抽象层冻结对外接口；五大基础能力模块彼此解耦 |
| **阶段延期传播风险** | Phase 2~6 | 各阶段间依赖关系：Phase 0→Phase 1→Phase 2→Phase 3→Phase 4→Phase 5→Phase 6 为硬串行；Phase 2/3/4 内部并行任务包间已定义解耦边界。每个阶段在其工期中预留 15% 缓冲天数用于应对上游延期传导。驱动依赖图及缓冲分配见阶段关键路径定义。 |
| **关键人员流失风险** | Phase 0~6 | 每阶段的关键任务包均指定 AB 角计划：包负责人（A 角）+ 结对人员（B 角）参与代码 Review 与关键决策同步；核心模块代码必须有文档说明关键设计意图（Architecture Decision Record），确保单点人员流失不影响工程连续性。 |
| **外部依赖失效风险** | Phase 2~5 | 外部依赖分类：LLM 服务（备选供应商评估清单至少 2 家，如 DeepSeek/Qwen/GLM）；硬件接入（HL7/DICOM 为标准协议，Mock 数据优先验证接口契约）；开源组件（锁定版本 + 内部缓存）。供应商不可用时切换备选方案的切换时间窗口已标注。 |
| **监管合规变更风险** | Phase 1~6 | 合规审计接口预留（操作日志全量存储 + 数据结构可回溯）；法规跟踪里程碑在 Phase 0 的 CI 骨架中包含合规检查占位阶段；AI 辅助决策标注义务（AI 生成内容在 UI 层统一标识 + 免责声明）已在 Phase 2/3 UI 需求中声明。 |

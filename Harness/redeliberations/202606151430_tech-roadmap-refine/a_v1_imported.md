## Phase 0 — 最小化骨架（项目初始化阶段）

### 0.1 阶段目标

搭出**让多人可并行继续贡献**的最小工程骨架。骨架阶段不实现任何业务功能，只交付协作所需的"地形"。

> 关于"硬必需 / 软预备"二分层级：Phase 0 的最小化交付需区分"硬必需（直接支撑 Phase 1 起步）"与"软预备（仅固化位置/依赖，可在后续阶段首期补齐）"两类条目。硬必需缺一不可，软预备如未在 Phase 0 启动，可在对应后续阶段首期补齐。本节各表新增"层级"列；Phase 0.7 集中陈列软预备条目并说明触发补齐时机。

### 0.3 关键技术决策

| 决策项 | 决策 | 层级 | 决策理由 |
|--------|------|------|----------|
| 工程组织 | Monorepo（单仓多模块） | **硬必需** | 三端 + 多后端模块 + 共享类型，单仓可原子化重构与跨端接口同步 |
| 工程组织演进路径 | 单仓多模块（Maven 聚合）→ Phase 5/6 视负载热点服务按 `services/` 切分为独立 Jar 部署形态（服务注册发现 Nacos/Eureka + Spring Cloud Gateway + 配置中心 Nacos Config） | **软预备** | 在骨架阶段固化"同进程调用"接口风格，演进时通过配置开关在"同进程"与"远程调用"间切换，避免后期重构；Phase 0 仅声明路径，真实拆分在 Phase 5/6 启动 |
| 后端模块切分 | 按"业务域"垂直切模块，模块间通过内部 API 调用而非共享 DB | **硬必需** | 保证 Phase 1+ 多人可在不同模块内并行开发不相互阻塞 |
| 后端模块命名约定 | `pharmacy` 模块当前以"药房 + 药库子包"形态承载；后续视 OOD 阶段规模评估（药库域实体数量、子模块数）决定是否拆为 `pharmacy` 与 `drug-storage` 两个独立 Maven 模块 | **硬必需**（命名约定本身） | 在 Phase 0 即固化药库与药房在工程结构上的归属位置，避免 Phase 4 业务落地时出现跨模块越界 |
| 前后端契约 | OpenAPI / TypeScript 类型生成（后端导出 → 前端 `@types` 包） | **硬必需** | 解决"前后端改一处要改多处"的协作痛点 |
| 数据库选型 | MySQL（与原始文档一致），编码 utf8mb4 | **硬必需** | 实践路径最成熟、文档支持最全 |
| 持久化方案 | Spring Data JPA（CRUD 主路径）+ SQL-Toy（复杂报表/统计查询） | **硬必需（JPA）/ 软预备（SQL-Toy 仅引入依赖）** | JPA 用于日常 CRUD 主路径，SQL-Toy 用于复杂报表聚合（Phase 4 检验趋势图、Phase 5 排班统计、Phase 6 性能指标），与原始技术栈文档 §2 一致；骨架阶段 JPA 即用、SQL-Toy 仅引入依赖，复杂查询在后续阶段按需启用 |
| 鉴权落地 | Spring Security + JWT，骨架阶段只走通"登录 → 拿 Token → 受保护接口" 三个端点的空实现 | **硬必需** | 后续业务模块只需复用鉴权拦截器即可 |
| AI 接入 | 后端独立 `ai-ability` 模块，对外暴露 AI 服务抽象层内部接口；骨架阶段以"可注入的 Mock 实现"占位 | **硬必需** | 业务模块可在无 LLM 凭证情况下基于 Mock 实现并行开发 |
| 硬件采集预留 | `services/device-gateway` 目录 + README 占位 | **软预备** | 提前固化"硬件采集"作为独立业务模块的位置，避免后期跨模块越界；真实代码在 Phase 4 首期补齐 |

## Phase 1 — 基础设施 + 统一认证与权限（地基阶段）

### 1.1 阶段目标

打通"账号 / 角色 / 岗位 / 菜单 / 接口"五件套，让三大终端能基于统一的身份进入平台骨架。

### 1.2 阶段交付物

| 类别 | 交付项 |

### 4.3 关键技术决策

- 检查预约任务分配：系统自动按最短队列或轮转规则分配至检查医生，无需手动认领。**Phase 2 历史占位申请衔接**：Phase 2 生成的 EXAMINATION 占位申请（派单状态字段=Phase 4 派单前的占位状态）在 Phase 4 派单上线时按**创建时间顺序首次回填入队**；后续新申请按 Phase 4 派单逻辑即时分配
- 趋势图聚合：按 `item_id`（内部编码）匹配；同报告期取最近一次；单位不一致时按 SI 标准单位换算（架构设计阶段固化换算规则），无法换算的不参与绘制并标注；**聚合查询使用 SQL-Toy（与 Phase 0.3 持久化双栈一致）**
- 药房库存扣减/回补：发药扣减药房库存、退药回补药房库存；盘库差额回退药库
- **退药角色矩阵与状态机联动**（颗粒度与 4.2 调拨单对齐；**业务规则边界保留在路线图，事务边界与时间窗下沉 OOD**）：
  - **发起方** = 患者（通过患者端发起）或开单医生（通过医生端处方详情发起）——**保留在路线图**（业务规则边界）
  - **审批方** = 药房医生，按药品分类规则决定是否需双签（具体规则下沉产品配置）
  - **执行方** = 药房医生：执行药房库存回补 + 缴费记录联动变更——**保留在路线图**（业务规则边界）
  - **退药被拒兜底** = 走原渠道（患者端 / 医生端）告知，退药申请状态变更为"已拒绝"，库存与缴费记录均不变更——**保留在路线图**（业务规则边界）
  - **事务边界**（库存回补与缴费记录状态变更的原子性保证）——**下沉到 OOD 阶段**，路线图仅声明"退药库存回补与缴费记录状态变更需保证事务原子性，任一失败触发回滚"
  - **退药时间窗**（已发药后允许退药的时间范围）——**下沉到产品配置层**，路线图仅声明"退药有可配置时间窗（管理员可配置）"
- 缴费记录状态机落地（详见 4.2 通用行）：原缴费记录变更为"待退费"中间态而非新建记录（避免双记录歧义）
- **调拨单状态机**：草稿 → 审批中 → 在途 → 已接收 / 已拒绝 / 已撤销；调拨过账采用"药库出库 + 药房入库"双写对账（出库与入库任一失败触发事务回滚 + 差异告警）；调拨差异按差异类型分级告警（警告/阻塞；具体字段下沉 OOD 阶段）
- **盘库单状态机**：草稿 → 盘点中 → 已复盘 → 已归档；盘盈自动入库，盘亏自动出库，误差超阈值触发管理员复核
- 药房医生查看检查/检验结果：药房医生通过处方详情卡片内嵌入口查看该处方所关联的检查/检验结果（权限：药房医生岗位 + 处方关联校验，不开放独立列表查询能力）
- 硬件接入：`device-gateway` 模块独立于 `exam` / `lab` 模块；通过统一的"设备事件 → 业务事件"映射层将原始设备数据（HL7 消息、DICOM 文件、串口数据、网关推送）转为 `exam` / `lab` 模块可直接消费的业务事件；Phase 4 仅落接口契约与 Mock 数据，真实设备对接在后续阶段
- 3.4.4 / 3.4.9 真实 AI 推理承接：复用 Phase 3 暴露的分诊规则降级提供器 / 检查推荐规则降级提供器接口；新增真实 AI 实现类并通过配置开关切换；本地规则降级实现保留为兜底
- 3.4.5 / 3.4.6 AI 失败回退：调用失败 / 超时时回退到医生手动判读模式（具体 UI 规范下沉 OOD 阶段）
- **3.4.7 / 3.4.11 降级路径**：3.4.7 影像分析推理与 3.4.11 执行顺序推荐在 AI 调用失败 / 超时时，回退到"医生手动操作"路径——3.4.7 退回到检查医生手工录入影像所见与初步结论、3.4.11 退回到检查医生按经验/默认规则手工排序；UI 显著标识"由医生手动判读 / 排序，AI 不可用"，原始申请单数据仍可正常流转，不阻断业务流程

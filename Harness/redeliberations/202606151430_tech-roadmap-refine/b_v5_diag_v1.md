## 问题清单

### P1-1：Phase 0 未充分落实"压轻"要求，仍包含大量工程脚手架细节

- **所在位置**：Phase 0.2「做完后新增什么能力」——骨架必备（阻塞并行）子列表（第 36~41 行）
- **问题描述**：核心需求明确要求"把 Phase 0 再压轻一些"并移除"工程脚手架细则"。当前 Phase 0.2 仍包含：
  - 具体目录名枚举（`apps/`、`services/`、`packages/`、`docs/`、`scripts/`）
  - 具体模块列表（common/gateway/auth/patient/doctor/exam/lab/pharmacy/admin/ai-ability）
  - 具体技术栈/框架名（Vue 手机壳、Vue PC、Spring Boot 多模块）
  - 具体文件名（`scripts/dev-up.sh`、`AGENTS.md`、`QUICKSTART.md`）
  
  这些内容属于"工程脚手架细则"，需求明确要求将其下沉至工程规范文档。
- **严重程度**：高
- **改进建议**：将上述细节抽象为能力级描述，例如：
  - "工程结构" → "标准化 Monorepo 工程结构与接口契约基线"
  - "后端骨架" → "后端多模块工程骨架（集成认证/授权/数据访问基础设施）"
  - "前端骨架" → "三端独立前端工程骨架（含状态管理、请求封装、路由）"
  - 脚本文件名和入门引导文件名只需描述为"一键启动脚本与快速入门文档"即可，无需暴露具体文件名

### P1-2：Phase 3.2 包含需求明确要求移除的详细状态机枚举

- **所在位置**：Phase 3.2，AI 处方审核（3.4.2）描述（第 142~143 行）
- **问题描述**：需求文档明确将"详细状态机枚举"列为应移除的典型内容，但当前 3.4.2 描述中仍完整保留了三档风险等级及其对应的行为处理策略（"高风险阻断、中风险三选（强制提交/修改重审/撤销）、低风险建议不阻断"）。这本质上是审核决策状态机的详细枚举，应下沉至 OOD 或技术设计文档。
- **严重程度**：高
- **改进建议**：将状态枚举抽象为能力级描述，例如："AI 处方审核（3.4.2，首次落地）：支持分级风险阻断——高风险自动拦截、中风险人工确认、低风险仅建议不阻断"。详细的三档规则和交互流程移至 OOD 文档。

### P2-1：Phase 4.1 包含超出路线图粒度的串行化计算过程

- **所在位置**：Phase 4.1，最后一段（第 187 行）
- **问题描述**："经串行化压缩后核心并行路数推断过程"整段（约 120 字）是在展示项目管理层面的计算公式推导（7 − 1 − 1 = 5）。路线图应告知结论和建议，不需要展示推算过程。这与需求要求移除"依赖占位和工程脚手架细则"的精神一致——推算公式属于项目排期规划细节。
- **严重程度**：中
- **改进建议**：删除推算过程，仅保留结论句，例如改写为："7 条关键路径经串行化压缩后有效并行路数约为 5 条，建议至少配置 5 位开发者并行推进（各路径解耦关系见上表）。"

### P2-2：阅读指引中包含应下沉至技术设计文档的技术栈细节

- **严重程度**：中
- **改进建议**：替换为"持久化方案采用关系型数据库+报表查询引擎双栈"，或删除该行（此信息已在 `Docs/02_tech.md` 中详述）。

## 整体质量评价

产出在第 3~4 轮指出的具体问题（Phase 0 分层、Phase 4 关键路径定义、风险表扩充）上已妥善修复，四件事框架和附录 C 索引也较为规范。**但 v5 应重点解决的"抽象提升、细节下沉"核心目标仅部分达成**——产出仍保留了多处路线图不应承载的设计级细节（脚手架明细、状态机枚举、计算公式推导、具体工具/框架名），导致读者难以将本文档与详细设计文档区分开来，与"让读者一眼看出这是指导项目推进的路线图"的要求尚有一定距离。

建议优先修复 P1-1、P1-2（直接违背核心需求），其次处理 P2-1~P2-3，P3-1/P3-2 按余量处理。

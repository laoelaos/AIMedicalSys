根据以下审查结果，迭代上一轮的产出，形成新版的文件，从而更好地满足用户需求。

## 当前审查结果

本轮审查发现 8 个问题，其中 3 个中等严重度、5 个低严重度：

**问题 1（中）：Phase 0.3 模块命名与"最小化骨架"定位存在张力**
Phase 0.3 列出 7 个并行模块（A~G），模块名称直接使用完整的业务模块名（如"患者端前端 + 患者后端模块""管理员端前端 + 管理员后端模块"），但 Phase 0.4 声明"不包含任何业务功能"。读者看到模块名时容易误判 Phase 0 的工作范围，以为 Phase 0 就要开始开发这些模块的业务功能，与"最小化骨架"定位产生矛盾。
改进建议：在模块命名上明确标注 Phase 0 的工作边界，例如在各模块名后追加上标说明"仅工程骨架搭建，不包含业务功能"，或将 0.3 的模块列表改为以"搭建任务"为粒度。

**问题 2（中）：Phase 0.2 "推荐补齐"仅 1 项，轻重划分的实际指导意义不足**
用户要求"明确区分'最小可协作骨架'和'推荐补齐配套'"。当前 Phase 0.2 在"骨架必备"下列出 5 项，在"推荐补齐"下仅列出"持续集成"1 项。仅 1 项的划分结果显得刻意且信息量不足。
改进建议：方案 A：补充更多推荐补齐项（如日志聚合框架占位、监控埋点接入点预留、API 文档生成工具集成、Docker 化部署脚本等）；方案 B：如果确认只有 CI 是可选，则删除"推荐补齐"子分类，将 CI 降级为骨架必备中的次要项。

**问题 3（中）：Phase 0.2 将 CI 列为"推荐补齐"但与 Phase 0.3 模块 F 中 CI 作为并行分配工作包存在隐含矛盾**
Phase 0.2 将 CI 列为"推荐补齐（可在后续阶段首期补齐）"，暗示 CI 不是 Phase 0 的阻塞项。但 Phase 0.3 模块 F 列出"基础设施（DB / CI / 部署脚本）"作为并行分配的一个工作包，且与其他骨架模块并列，未标记为可选/推荐。
改进建议：统一两处表述。如果 CI 是推荐补齐，则 Phase 0.3 模块 F 应注明"CI 可在 Phase 1 补充，不阻塞 Phase 0 验收"；如果 CI 是 Phase 0 必须完成的基础设施，则从 0.2 的"推荐补齐"移入"骨架必备"。

**问题 4（低）：验收标准中残留"状态机"实现级术语**
多处验收标准和交付物描述中使用"状态机"一词，属于工程实现概念而非能力描述。具体残留位置：Phase 2.2 验收标准、Phase 4.2 "缴费记录状态管理"、Phase 4.3 包 C "药房域（发药/退药/库存/状态机）"、Phase 4.3 包 E "线下窗口 + 缴费记录状态机"、Phase 4.1 路径 5 "线下挂号/退号 + 收费/退费 + 缴费记录状态机"。
改进建议：将"状态机"替换为能力级描述，如"缴费记录全生命周期管理（待支付/已支付/已退款/已对账）"、"发药/退药/库存全流程管理"等。

**问题 5（中）：Phase 4.1 关键路径表颗粒度过细，超出路线图层级**
Phase 4.1 的 7 条关键路径定义表包含每条路径的归属任务包、依赖/解耦关系、路径独立性声明、串行化压缩后的并行路数估算、建议开发者人数。这些内容在项目管理计划层面合理，但在路线图层面属于过度展开。
改进建议：将 7 条路径的细节表移出路线图正文，保留一句话概括。详细路径表可作为附录供 PM 参考，或仅在启动 Phase 4 详细计划中展开。

**问题 6（低）：风险表中包含具体数值和供应商名称**
风险表中出现具体百分比（"15% 缓冲天数"）和具体供应商名（"DeepSeek/Qwen/GLM"），属于项目执行中可能随时变化的细节。
改进建议：将"15%"改为"预留缓冲时间"，将具体供应商名改为"至少 2 家备选 LLM 供应商（如国产主流大模型）"。

**问题 7（低）：AI 能力迁移提醒在正文中重复 4 次**
"注：本阶段 AI 接入方案为独立实现，Phase 5 AI 进阶底座落地后统一迁移"这句话在 Phase 2.2 和 Phase 3.2 中累计出现 4 次（分别针对 3.4.1/3.4.2/3.4.3/3.4.10）。
改进建议：删除正文中的重复提醒，改为在"阅读指引"或"编号约定"中统一声明。同时可在附录 C 增加一列"底座迁移"标注哪些能力需要迁移。

**问题 8（低）：Phase 0.2 验收标准中"全部工程"边界不明确**
验收标准"三端可一键启动到占位首页，新人按入门引导文档可在 1 小时内本地拉起全部工程"中的"全部工程"一词暗示包含后端。文档未明确"三端一键启动"是否依赖后端服务也在同一时间就位。
改进建议：明确验收标准中"全部工程"的范围，建议改为"三端前端可一键启动到占位首页；后端基础骨架可独立启动并响应 ping 健康检查；新人按入门引导文档可在 1 小时内完成本地环境搭建。"

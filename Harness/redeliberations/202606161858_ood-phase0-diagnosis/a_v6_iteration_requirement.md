## 当前审查结果

### 问题一：[中] API 文档工具选型偏离技术栈规范未分析

诊断报告声明以 `Docs/02_tech.md`（技术方案）为参考基线，但未检查 OOD 在 API 文档工具选型上是否遵循了技术栈规范。技术栈文档 `02_tech.md` §4.4（第114行）明确指定 API 文档工具为 **Knife4j（Swagger3）**，访问路径为 `doc.html`；§11.1（第259行）同样将 `Knife4j（Swagger3）` 列为核心任务必需技术。然而 OOD §8.3（第936-953行）使用的是 **springdoc-openapi-starter-webmvc-ui**，访问路径为 `swagger-ui.html`。

诊断报告当前仅在 §3.1 中将 springdoc-openapi v2.5.0 作为"版本兼容性风险"标记，但未分析最根本的问题：**库选型本身是否偏离了技术栈规范**。

**改进建议**：
1. 新增一条分析条目，指出 OOD 使用 springdoc-openapi 而非 Knife4j 是否构成对技术栈规范的偏离
2. 若此选择是经团队同意的设计决策，应在 OOD 中显式说明替换理由
3. 评估两种方案的差异：访问路径、配置模式、对 Phase 1+ openapi-generator 自动生成 TypeScript 类型的功能影响

### 问题二：[低] §6.3 方法签名冻结立场与建议 2 的内在张力未说明

§6.3 中同时存在两条相互制衡的分析结论但报告未说明二者之间的关系：

**立场 A**（结构耦合约束分析，第170-171行）："若同时采纳建议 4，则 Phase 0 冻结的 13 个方法签名将成为 Phase 2+ 降级框架设计的**不可变约束**。"

**立场 B**（建议 2，第176-183行）："标注'可调整'的方法签名需在 Javadoc 中显式声明...任何此类调整必须走接口变更评审流程..."

两条立场的逻辑链条是：A 说签名是"不可变约束"，B 说通过评审流程可以改变签名。如果签名可以通过受控流程改变，则"不可变约束"的描述不准确。

**改进建议**：
1. 将"不可变约束"措辞修订为"默认不可变约束（架构评审例外）"以与建议 2 保持一致
2. 在两条建议之间增加一句关联说明，明确：方法签名冻结的强度取决于是否启用弹性变更通道

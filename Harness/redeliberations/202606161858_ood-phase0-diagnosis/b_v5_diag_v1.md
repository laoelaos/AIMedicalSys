## 一、[中] API 文档工具选型偏离技术栈规范未分析

**问题描述**：诊断报告声明以 `Docs/02_tech.md`（技术方案）为参考基线，但未检查 OOD 在 API 文档工具选型上是否遵循了技术栈规范。

技术栈文档 `02_tech.md` §4.4（第114行）明确指定 API 文档工具为 **Knife4j（Swagger3）**，访问路径为 `doc.html`；§11.1（第259行）同样将 `Knife4j（Swagger3）` 列为核心任务必需技术。然而 OOD §8.3（第936-953行）使用的是 **springdoc-openapi-starter-webmvc-ui**，访问路径为 `swagger-ui.html`。

Knife4j 和 springdoc-openapi 虽然同为 Swagger/OpenAPI 规范的实现，但它们是不同的库——Knife4j 是 springdoc-openapi 的增强型 UI 封装层，提供不同的访问路径（`doc.html` vs `swagger-ui.html`）和额外的 UI 功能。OOD 直接选择 springdoc-openapi 而非 Knife4j 构成了对技术栈规范的偏离。

诊断报告当前仅在 §3.1 中将 springdoc-openapi v2.5.0 作为"版本兼容性风险"标记，但未分析最根本的问题：**库选型本身是否偏离了技术栈规范**。这属于任务要求中"事实错误（如版本号、技术规范等）"维度下的技术规范检查遗漏。

**所在位置**：
- 诊断报告：§3.1（仅覆盖版本兼容性，未覆盖工具选型合规性）
- OOD：§8.3（第936-953行），父 POM 第197-202行
- 技术栈规范：`02_tech.md` §4.4（第114-115行）、§11.1（第259行）

**严重程度**：中

**改进建议**：
1. 新增一条分析条目，指出 OOD 使用 springdoc-openapi 而非 Knife4j 是否构成对技术栈规范的偏离
2. 若此选择是经团队同意的设计决策，应在 OOD 中显式说明替换理由（如"Knife4j 与 Vue 3 前端联调存在已知兼容性问题，改用原生 springdoc-openapi"等），而非默认为技术栈规范中指定的选型
3. 评估两种方案的差异：访问路径（`doc.html` vs `swagger-ui.html`）、配置模式、对 Phase 1+ openapi-generator 自动生成 TypeScript 类型的功能影响

## 二、[低] §6.3 方法签名冻结立场与建议 2 的内在张力未说明

**问题描述**：§6.3 中同时存在两条相互制衡的分析结论，但报告未说明二者之间的关系：

**立场 A（结构耦合约束分析，第170-171行）**："若同时采纳建议 4（将降级框架推迟至 Phase 2+），则 Phase 0 冻结的 13 个方法签名将成为 Phase 2+ 降级框架设计的**不可变约束**。"

**立场 B（建议 2，第176-183行）**："保留弹性设计考量...标注'可调整'的方法签名需在 Javadoc 中显式声明...任何此类调整必须走接口变更评审流程...等同于保留了 Phase 0 冻结立场，但为远期能力预留了显式、可控的弹性通道。"

两条立场的逻辑链条是：A 说签名是"不可变约束"，B 说通过评审流程可以改变签名。如果签名可以通过受控流程改变，则"不可变约束"的描述不准确——更精确的说法应是"默认不可变，但可通过架构评审例外变更"。报告未对此一致性做说明，可能误导执行者认为二者是独立并列的建议。

**所在位置**：`a_v5_diag_v1.md` §6.3（第170-183行）

**严重程度**：低

**改进建议**：
1. 将"不可变约束"措辞修订为"默认不可变约束（架构评审例外）"以与建议 2 保持一致
2. 在两条建议之间增加一句关联说明，明确：方法签名冻结的强度取决于是否启用弹性变更通道；如果选用建议 2 的受控调整机制，"不可变约束"仅对未标记可调整的方法签名适用

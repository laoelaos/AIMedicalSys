## 一、分类与严重度不一致

### 1.1 [严重: 中] 4.1 "GlobalExceptionHandler 下沉导致 common 耦合 Web 框架" 分类不当

**问题描述**：报告将 4.1 归类为"逻辑错误"（严重度: 中），但 OOD 已通过 `optional=true`（第 320-322 行）显式声明了依赖传播策略——spring-boot-starter-web 以 `compile + optional` 引入，下游纯契约模块不会被动继承完整 Web 依赖树。将 GlobalExceptionHandler 置于 common 模块并用 optional 控制传播，是 Spring Boot 多模块项目中的常见实践（Spring 官方示例同樣如此处理）。

报告自身在因果链中也引用了 `optional=true` 的事实，却仍然得出"逻辑错误"的结论，与其内部证据不一致。更关键的是，报告 v2 在修订说明中对 5.1（AI 降级框架）做了同类处理——从"逻辑错误"降级为"设计争议"——但对 4.1 未做同样的重新评估。

**所在位置**：`a_v2_diag_v1.md` 第 64-82 行（§4.1）

**改进建议**：将 4.1 从"逻辑错误"（中）降级为"设计争议"（低），或在分析中更明确地说明为何在已有 `optional=true` 的前提下仍认定为逻辑错误。建议与 5.1 的降级逻辑保持一致。

## 二、范围与严重度匹配问题

### 2.1 [严重: 中] 6.1/6.2 将项目级交付物缺失等同于 OOD "偏离路线图"，未讨论 OOD 的设计边界

**问题描述**：报告将协作规范缺失（6.1）和新人引导文档缺失（6.2）均标记为"偏离路线图"（严重度: **高**/阻塞级），但这两项是**项目级交付物**，而非 OOD 架构设计文档的固有内容：
- 路线图 Phase 0.2 的"协作规范"（分支约定、Commit 格式、PR 模板、CR 必查项）通常归入项目管理文档（CONTRIBUTING.md、PR template 等），不属于架构 OOD 设计文档的职责范围
- 入门引导文档（QUICKSTART.md）同样属于项目工程文档，通常由 README 或独立文档承载

OOD 作为架构设计方案，核心产出是模块划分、接口契约、依赖方向、关键抽象等设计决策。报告判定 OOD"偏离路线图"时，未区分"OOD 设计文档未覆盖"与"项目交付物缺失"——OOD 可以引用或规划这些文档的创建，但不一定需要在 OOD 正文中包含其内容。

**所在位置**：`a_v2_diag_v1.md` 第 133-164 行（§6.1、§6.2）

**改进建议**：将这两项的严重度从"高"（阻塞）降为"中"，并将问题描述从"OOD 未覆盖"修订为"OOD 未规划这些项目级交付物的创建"。同时建议在 OOD 新增一节"关联交付物清单"，列出需并行产出的协作规范文档、QUICKSTART.md 等。

### 2.2 [严重: 低] 6.4 未区分"推荐补齐"项的性质差异

**问题描述**：报告将 5 项缺失的推荐补齐项统一标记为"缺失"，并建议部分项为"高"优先级。但路线图明确标注这些为"推荐补齐"且"可跨阶段持续完善，不阻塞 Phase 0 骨架验收"。其中：
- 容器化开发部署脚本（Docker）、本地代码质量检查工具集成、硬件接入接口占位属于**运维/基础设施范畴**，通常在独立文档（DEPLOY.md、lint 配置文件等）中定义，不在 OOD 中展开
- 日志聚合框架和基础监控埋点虽属于设计范畴，但路线图标注为推荐补齐且可顺延

报告未讨论这些项是否**理应**在 OOD 中覆盖，也未参考 OOD §10（CI 占位）已包含 CI 构建阶段配置的事实——这本身就是代码质量/构建基础设施的一部分。

**所在位置**：`a_v2_diag_v1.md` 第 183-200 行（§6.4）

**改进建议**：仍保留"缺失"的标注，但建议在描述中区分：哪些项属于 OOD 应覆盖而遗漏的（如日志格式规范），哪些属于独立文档范畴（如 Docker 配置、lint 工具配置），避免将后者视为 OOD 的设计缺陷。

## 三、建议的副作用分析不完整

### 3.1 [严重: 低] 建议 4 方案 A 遗漏 AiService 注入策略变更分析

**问题描述**：报告建议 4 方案 A（将 FallbackAiService/DegradationStrategy 推迟至 Phase 2+）的副作用分析中，未提及移除 FallbackAiService 后 **AiService Bean 注入策略的变更**：

当前 OOD 的注入策略：
```
application 模块 → @Bean 暴露默认 AiService
  ├── MockAiService（@ConditionalOnProperty）
  └── FallbackAiService（装饰器，从 List<AiService> 排除自身后选委托对象）
业务模块 → private final AiService aiService;（按类型注入，不使用 @Qualifier）
```

如果移除 FallbackAiService，则：
- application 模块的 AiService 注入点需要简化为直接注册 MockAiService（不再需要装饰器）
- 当前 OOD 第 855 行的设计决策"application 模块统一暴露默认 AiService 注入点；FallbackAiService 构造器注入 List<AiService>"需要更新
- 业务模块代码（`private final AiService aiService;`）本身不需要修改，但注入链路的依赖关系发生了变化

**所在位置**：`a_v2_diag_v1.md` 第 238-249 行（建议 4 方案 A）

**改进建议**：在方案 A 的副作用分析中补充 AiService 注入策略的变更说明，并评估对 application 模块配置代码（当前 OOD 第 636-638 行 Bean 装配策略）的影响面积。

## 四、建议内部的逻辑一致性

### 4.1 [严重: 低] 6.3 建议 2 可能与 OOD 的冻结立场矛盾

**问题描述**：报告 6.3 建议 2 "对 Phase 5 才落地的 AI 能力标记'方法签名可调整直至该能力首次落地阶段启动前'"与 OOD §8.2 第 874 行的显式立场矛盾——OOD 明确声明 Phase 0 冻结方法签名是为了"避免越过路线图中'模块级接口契约在对应阶段启动前冻结'的边界"。如果对 Phase 5 能力标注"可调整"，则 Phase 0 的冻结失去意义，Phase 0 后续阶段的 AiService 接口稳定性承诺被削弱。

**所在位置**：`a_v2_diag_v1.md` 第 178-179 行（6.3 建议 2）

**改进建议**：此建议本身是合理的弹性设计考量，但应明确说明：标注"可调整"的方法签名需要在 Javadoc 中显式声明"该接口方法在 Phase 5（AI 医生排班首次落地阶段）启动前可调整"，并且任何此类调整必须走接口变更评审流程（避免被随意修改）。即保留冻结立场但为远期能力预留显式弹性通道。

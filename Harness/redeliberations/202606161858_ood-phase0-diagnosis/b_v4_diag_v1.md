## 一、需求响应充分度

### 1.1 [低] 未检查 OOD 文档编号格式约定错误

**问题描述**：用户需求要求检查"事实错误（如版本号、技术规范等）"。OOD 第 928 行使用了 "3.4.x" 这一编号约定——按 Roadmap 第 6 行约定，`3.4.x` 仅适用于需求文档中的 AI 能力编号体系，不应用于 OOD 设计文档。OOD 在自身文档中使用 "3.4.x" 引用 AI 能力，实质上是将需求文档的编号体系引用到 OOD 中，而 Roadmap 第 6 行的约定并未声明该编号体系可跨文档使用。诊断报告 v2 修订说明曾提及此条目被删除（误判为跨文档编号引用），但此处需要辨析的是：OOD 引用"3.4.x"不是跨文档章节引用错误，而是将需求领域的编号约定带入了设计文档，可能造成读者混淆（OOD 读者会疑惑"3.4.x"是指 OOD 本身的章节编号还是 AI 能力编号）。报告最终完全删除了此条目，但没有对"OOD 中使用需求编号体系是否合适"给出明确结论。

**所在位置**：报告 §二、事实错误（原 v1 条目 2.1，已被删除）；修订说明 v2 第 280 行

**严重程度**：低

**改进建议**：此问题不构成硬性事实错误（按 Roadmap 第 6 行约定，"3.4.x" 在需求语境中确为 AI 能力编号），但属于 OOD 文档风格的观察点。若认为不值得报告，应在诊断报告中明确说明"已评估，不构成问题"而非简单地删除条目。当前报告对此问题无结论性描述。

## 二、深度与完整性

### 2.1 [中] 6.3（方法签名冻结边界）与 6.5/5.1（降级框架超范围）作为独立条目分析，未揭示二者的结构耦合约束

**问题描述**：报告将 6.3（OOD §8.2 冻结 13 项 AiService 方法签名与 Phase 0.4 边界的张力）和 5.1/6.5（降级框架超出 Roadmap 附录 A"Mock 占位"定位）作为两个独立问题分别分析，但二者存在结构耦合：如果依照 6.3 维持 Phase 0 冻结全部 13 个方法签名的立场，同时依照 建议 4（方案 A）将降级框架推迟至 Phase 2+，那么 Phase 2+ 的降级框架实现将受到 Phase 0 已冻结的 AiService 方法签名的约束——`FallbackAiService` 作为 `AiService` 的实现类，其构造器签名和装饰逻辑不能要求 `AiService` 接口新增任何方法。建议 4 方案 A 的副作用分析中虽然提及了"Phase 2+ 不能修改 AiService 结构"，但未将这些分析明确回连到 6.3 的边界分析中，导致执行者拿到报告后无法看到这两个决策的联动影响。

**所在位置**：报告 §6.3（L153-176）、§5.1（L76-96）、§6.5（L197-209）、建议 4 方案 A 副作用分析（L236-243）

**严重程度**：中

**改进建议**：在 6.3 的分析中增加一条联动分析说明，指出"若同时采纳建议 4（将降级框架推迟至 Phase 2+），则 Phase 0 冻结的 13 个方法签名将成为 Phase 2+ 降级框架设计的不可变约束——FallbackAiService 只能以装饰器模式包装已冻结的方法，不能引入新的方法重载或接口变体"。同样地，在建议 4 的副作用分析中增加对 6.3 边界的引用。

### 2.2 [低] 6.4 中"日志格式规范"的优先级评估缺乏依据

**问题描述**：报告在 6.4 中将"日志聚合框架占位（日志输出格式规范与采集配置骨架）"标注为"OOD 应覆盖"且优先级"高——Phase 0 开发期即可受益于统一日志格式"。此判断力度过大但支撑不足：日志输出格式规范通常属于开发约定/编码规范（CONTRIBUTING.md 范畴），不属于 OOD 架构设计文档的固有职责——这一判断标准与报告自身在 6.1/6.2 边界辨析中使用的标准（"项目级交付物，非 OOD 设计文档遗漏"）不一致。报告在 6.1/6.2 中明确主张协作规范和入门引导文档"属于项目级交付物而非 OOD 设计文档的固有职责范围"，却将日志格式规范判定为"OOD 应覆盖"且优先级"高"，两个判据的适用尺度不统一。

**所在位置**：报告 §6.4 对比表第 1 行（L184）

**严重程度**：低

**改进建议**：统一判断标准。如果协作规范和入门引导文档不属于 OOD 职责范围，日志格式规范也应同理对待。建议将日志格式规范的"性质归属"从"OOD 应覆盖"改为"开发规范/项目文档范畴"，优先级下调为"中"（相对于 6.1/6.2 的"高"保持一致）。OOD 仅需在关联交付物清单中标注此项由谁在何时创建。

### 2.3 [低] 4.2（ai-api 依赖注入策略）仅指出问题未提供倾向性方案

**问题描述**：4.2 指出 OOD 未明确 Phase 0 业务模块是否需要在 Controller 中注入 `AiService`，但未给出倾向性建议，仅描述两种可能（注入代码 vs 仅 POM 依赖）。这使得执行者需要自行决策。相比之下，其他条目（如 5.3 GlobalExceptionHandler 位置、5.1 降级框架裁剪）均提供了明确的倾向性分析。

**所在位置**：报告 §4.2（L64-70）

**严重程度**：低

**改进建议**：在 4.2 末尾增加倾向性建议。基于 Phase 0"最小化骨架"定位，建议：Phase 0 业务模块的占位 Controller 不注入 `AiService`，仅声明 ai-api POM 依赖；代码中零 `AiService` 引用。Maven 对未使用的 compile scope 依赖不会报错或警告（相比 provided scope），不存在 CI 误报风险。

## 三、可操作性评估

### 3.1 [中] 建议优先级排序与修复依赖关系不够清晰

**问题描述**：报告§七列出了 5 条修复建议，但未给出推荐执行顺序。其中：
- 建议 4（AI 降级框架裁剪）是幅度最大的改动，方案 A 涉及 route 表重写、Bean 装配策略变更、包结构调整、启动期校验，级联影响面广，应当作为高优先级审慎评估；
- 建议 3（GlobalExceptionHandler 迁移）标记为"设计优化"，非硬性要求，可作为低优先级；
- 建议 2 已标记"验收硬性要求"，优先级明确；
- 建议 5（应用名统一）最简单，可立即执行。

但报告未指明这些建议之间的先后依赖关系（例如：如果决定实施建议 4 方案 A，则 OOD §8.2 中关于降级框架的内容需要先重写，然后才能统一评估对其他章节的影响），执行者无法一眼看出先做哪个。

**所在位置**：报告 §七（L213-252）

**严重程度**：中

**改进建议**：在§七开头增加一张优先级与依赖关系概览表，明确标注每条建议的"建议次序""前置条件"和"阻断关系"。示例：

| 次序 | 建议 | 属性 | 前置条件 | 阻断关系 |
|------|------|------|----------|----------|
| 1 | 建议 2（QUICKSTART.md） | 验收硬性要求 | OOD §9 内容已就绪 | 无 |
| 2 | 建议 5（应用名统一） | 低风险速赢 | 无 | 无 |
| 3 | 建议 1（关联交付物清单） | 关联项目交付 | 项目负责人确认分支策略 | 无 |
| 4 | 建议 4（降级框架裁剪—方案评估） | 高影响变更 | 需架构评审 | 如果实施可简化 OOD §3.4 |
| 5 | 建议 3（GlobalExceptionHandler 迁移） | 设计优化 | 确认 Jackson 版本管理 | 低—可随时执行 |

### 3.2 [低] 建议 1 与建议 4 的前置条件校验不完整

**问题描述**：

- 建议 1 的前置条件为"需项目负责人确认分支策略"。但协作规范不仅包含分支策略，还包含 Commit 格式、PR 模板、CR 必查项——即使分支策略未确认，Commit 格式和 PR 模板完全可以先行制定。将全量协作规范挂起于"分支策略确认"这一前置条件过于保守，可能导致实际执行时其他可并行的工作被延迟。

- 建议 4 方案 B（保守方案：保留降级框架骨架但缩小范围）的副作用分析深度不足方案 A 的 20%。方案 B 同样涉及 `TimeoutDegradationStrategy` 的移除和 route 表的精简，应当与方案 A 有对称的副作用分析。

**所在位置**：报告§七 建议 1（L215-218）、建议 4 方案 B（L246-247）

**严重程度**：低

**改进建议**：
- 建议 1 的前置条件拆分为多条："分支策略——需项目负责人确认；Commit 格式 / PR 模板 / CR 必查项——可由技术负责人先行制定，不依赖分支策略确认"
- 建议 4 方案 B 补充副作用分析：移除 `TimeoutDegradationStrategy` 后 `ai-impl` 的 `degradation/` 包仅保留 `NoOpDegradationStrategy`；route 表精简到仅 `true`/未配置两行后需同步更新 Phase 0 说明中"Phase 0 应始终使用 `ai.mock.enabled=true`"的描述；方案 B 的实施工作量评估（约为方案 A 的 30%~40%）。

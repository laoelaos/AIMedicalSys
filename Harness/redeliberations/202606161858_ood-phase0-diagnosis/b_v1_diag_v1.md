## 一、诊断报告自身的事实错误

### 1.1 [严重: 高] 2.1 "3.4.x 跨文档编号引用问题" — 结论不成立，属于误判

**问题描述**：诊断报告声称 OOD 第 928 行的 "3.4.x" 是"需求文档第 3.4 节的子节编号体系"，且"OOD 读者在本文档内无法定位"。但根据 `Docs/03_roadmap.md` 第 6 行明确声明：

> 编号约定：3.4.x = AI 能力编号

该编号约定是项目级统一规范，在 Roadmap（第 6、24 行）、需求文档（第 814 行）、OOD（第 928 行）中一致使用。OOD 第 928 行的 "3.4.x AI 能力错误码"正是按此约定引用"所有 AI 能力"的错误码命名规则，并非引用某文档的章节编号。熟悉项目约定的读者可以明确理解其含义。

**所在位置**：`a_v1_diag_v1.md` 第 38-43 行

**改进建议**：删除此条目。OOD 此处用法正确且与 Roadmap 约定一致，不构成问题。

---

### 1.2 [严重: 中] 2.2 "@phase0-mock-field 语义混淆" — 解读有偏差

**问题描述**：报告称 `@phase0-mock-field` 在"第902-906行描述为Phase 0子集冻结，在第628-632行又被当作Mock数据填充标记使用，二者混淆"。但经查阅 OOD 原文：

- 第 902-906 行定义的是"两层冻结策略"——`@phase0-mock-field` 属于"Mock 演示子集"而**非**"本阶段冻结"
- 第 627-632 行的 Mock 数据规则明确限定生效范围："Phase 0 仅对 8.2 节标记的 `@phase0-mock-field` 生效"
- 对于输入 DTO 字段（如 `TriageRequest.chiefComplaint`），标注 `@phase0-mock-field` 的作用是将其纳入 Phase 0 契约（第 906 行"该字段子集只服务于 Mock 联调"），而非要求 MockAiService 对该字段做数据填充

因此该注解不存在语义冲突，只是在输入/输出 DTO 上有不同的实际含义（输出端用作数据填充、输入端用作契约标记）。报告所称"二者混淆"是对 OOD 原文的过度解读。

**所在位置**：`a_v1_diag_v1.md` 第 45-51 行

**改进建议**：修订为此注解在输入/输出 DTO 上用途不同但不相矛盾；或降低严重度至"设计观察"而非"事实错误"。

### 1.3 [严重: 中] 2.3 分类错误 — springdoc-openapi 版本兼容性非"事实错误"

**问题描述**：报告将 springdoc-openapi v2.5.0 版本兼容性未验证归类为"事实错误"，但描述内容为"未经验证"而非"已知不兼容"。这不是一个已存在的事实错误，而是一个潜在的风险提示或兼容性观察。

**所在位置**：`a_v1_diag_v1.md` 第 52-57 行

**改进建议**：应将此项移至"风险提示"类别，或修改为"springdoc-openapi v2.5.0 兼容性未显式验证"并降为"低"严重度。

## 二、关键遗漏

### 2.1 [严重: 高] 遗漏了对 Roadmap Phase 0.4 "本阶段明确不包含"的对照检查

**问题描述**：`Docs/03_roadmap.md` Phase 0.4 声明"模块级接口契约冻结（在对应阶段启动前冻结）"——即 AI 模块的接口契约应在 Phase 2 启动前冻结，而非 Phase 0。但 OOD §8.2 在 Phase 0 冻结了全部 13 项 AI 能力的方法签名（AiService 接口的 13 个方法），这本质上是**模块级接口契约的预先冻结**。Roadmap 附录 A 虽列 Phase 0 为"全部（Mock 占位）"，但这与 Phase 0.4 的"不包含"之间存在张力。

报告对此未做任何分析，仅在 3.4 间接提及 ai-api 依赖问题。

**所在位置**：`a_v1_diag_v1.md` — 全文未覆盖

**改进建议**：在"偏离路线图"一节新增条目，分析 Phase 0.4 "模块级接口契约冻结"限制与 OOD §8.2 冻结 13 项 AiService 方法签名之间的矛盾：OOD 应明确论证为何在 Phase 0 冻结全部 13 个 AI 方法签名不违反 Phase 0.4 的限制，或推迟冻结非 Phase 2 首落地的 AI 能力方法签名。

---

### 2.2 [严重: 中] 未评估诊断建议的可操作性和副作用

**问题描述**：报告"高优先级修复建议"中第 4 条建议"将 FallbackAiService / DegradationStrategy 体系推迟至 Phase 2+"，但未评估此建议的副作用：
- `DegradationStrategy` 接口定义在 `ai-api` 子模块中，如果删除则 `ai-api` 的 `degradation/` 包路径将在 Phase 0 为空
- Route 表第 645 行（`ai.mock.enabled=false` 配置）的设计依赖于 FallbackAiService 的存在——删除后该配置行的行为会改变
- 将此部分推迟到 Phase 2+ 意味着后续 OOD 修订需要重新设计这部分结构，可能导致与 Phase 0 已冻结的 AiService 接口不兼容

报告给出了"改什么"但未分析"改了之后会带来什么新问题"。

**所在位置**：`a_v1_diag_v1.md` 第 196 行

**改进建议**：每条高优先级修复建议应附带副作用分析和前置条件说明。

## 三、分类与严重度不匹配

### 3.1 [严重: 低] 1.1 将不同维度的名称混为一谈

**问题描述**：报告将 Maven `<name>`（构建工具显示名，第 129 行）、`spring.application.name`（运行时标识，第 1006 行）、目录名（文件系统约定，第 54 行）归为"应用名不一致"，但三者用途各异：Maven name 仅用于构建报告显示，spring.application.name 用于运行期标识，目录名是磁盘路径约定。这不是"定义矛盾"而是对同一项目在不同上下文中的不同标识方式。报告将其标记为"定义矛盾"（低严重度）且归类可接受，但描述中将其称为"应用名不一致"有误导性。

**所在位置**：`a_v1_diag_v1.md` 第 12-24 行

**改进建议**：将描述改为"Maven 构建显示名与运行时应用名未对齐，可考虑统一以提高可识别性"。

## 四、深度与完整性评估

### 4.1 路线图偏离覆盖较全面，但分析方法可改进

报告对"偏离路线图"的检查覆盖了 4.1（协作规范）、4.2（入门引导）、4.3（推荐补齐项），定位准确。但缺少对 Roadmap Phase 0.4 "本阶段明确不包含"的反向验证（见2.1），也缺少对 Roadmap 附录 A 中"全部（Mock 占位）"与 OOD 实际设计范围的对比检查。

### 4.2 逻辑错误分析有深度，但部分判断偏主观

3.2（AI 降级框架过度设计）分析详尽，但评估尺度偏严——Roadmap 的"可跨阶段持续完善"不禁止结构设计提前就位，OOD 本身也声明 Phase 0 不激活这些组件。建议降为"设计争议"而非"逻辑错误"。

### 4.3 可操作性评估

- 4.1（协作规范）、4.2（入门引导）的建议具体可执行
- 3.1（GlobalExceptionHandler 位置调整）建议明确，未考虑副作用：如果移出 common，则 common 模块对 spring-boot-starter-web 的依赖可以去除，但 `Result<T>` 的 JSON 序列化仍然需要 Jackson（通过 spring-boot-starter-web 引入），这份依赖拆分需要在建议中澄清
- 4.3（推荐补齐项缺失）缺少"是否应补齐"的优先级判断——缺失的 5 项中哪些应优先纳入 Phase 0，哪些可顺延

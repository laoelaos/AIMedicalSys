# 质量审查报告：EntityMappingIT 实体映射诊断报告 v4

## 审查发现

### 问题 1 [严重] — @SQLRestriction 行为评判事实错误，导致 Issue 3 影响评估和优先级排序依据失准

**所在位置**：Issue 3「影响范围」第 247 行；「优先级排序」交叉对比分析第 351-354 行

**问题描述**：报告第 353 行声称 "`deleted IS NULL` 和 `deleted = 0` 在 `@SQLRestriction("deleted = false")` 过滤下均被视为'未删除'"。这是**事实错误**。

`BaseEntity.java:24` 声明 `@SQLRestriction("deleted = false")`，Hibernate 会追加 `WHERE deleted = false` (= `WHERE deleted = 0`) 到所有查询。在 SQL 三值逻辑中，`NULL = 0` 返回 **NULL（unknown）**，而非 TRUE。WHERE 子句将 NULL 视为 false，因此 `deleted IS NULL` 的记录会被过滤掉——**实际被当作"已删除"处理**。

这意味着：
- **Issue 3 运行时影响被严重低估**：报告称"运行时坏影响相对可控"（第 337 行）——实际影响比所述更严重，脏数据记录将从 JPA 查询中"消失"，可能导致权限未加载、软删除语义异常等业务问题。
- **优先级排序依据需要修正**：交叉对比分析（第 351-354 行）以"运行时 NULL 写入风险低"和"`deleted IS NULL` 和 `deleted = 0` 均被视为未删除"作为 Issue 3 评为 P2 的关键量化依据，该前提不成立。

**改进建议**：
1. 修正该错误陈述，明确 `deleted IS NULL` 在 `@SQLRestriction` 下的实际行为（被过滤）；
2. 重新评估 Issue 3 的业务影响等级——脏数据记录将被静默"隐藏"，影响范围可能超出当前描述；
3. 基于修正后的影响分析，重新审视交叉对比表格中的"业务影响"和"运行时异常风险"评分；
4. 若重新评估后 Issue 3 的优先级需调整，同步更新时序依赖图和修复顺序建议。

### 问题 2 [中等] — Issue 2 副作用分析未完成，v3 迭代已指出但 v4 未实质解决

**所在位置**：Issue 2「修复方案潜在副作用分析」第 72-73 行；v3「修订说明」第 2 条回应（第 391 行）

**问题描述**：v3 迭代要求"Issue 2 未排查创建 User 对象时未设置 password 的代码路径"（迭代历史第 44 行），v4 修订说明第 2 条声称已"接受"并补充了代码路径分析。但实际内容（第 72 行）仍然以"需确认那些模块中创建 User 后是否均设置了 password" 和"若存在未设置 password 就 persist 的路径，修改后将在运行时抛出异常"结尾——**排查本身并未执行，仅复述了风险**。

具体不足：
- 已知 admin 模块、注册流程等存在创建 User 的代码路径，但报告未实际阅读这些路径的代码；
- 风险提示未附带具体的排查结论或"高风险待确认"的标记；
- 后续优先级排序和修复计划未反映此不确定性。

**改进建议**：
1. 实际读取 admin 模块和注册流程中创建 User 的代码路径，确认是否均设置了 password；
2. 若无法在报告范围内完成排查，应在 Issue 2 的副作用分析中明确标注"高风险——未排查"的标记，而非以"需确认"模糊带过；
3. 在优先级排序的风险维度中反映此不确定性。

### 问题 3 [中等] — Issue 1 修复指引的可操作性显著低于其他 Issue，交付质量不一致

**所在位置**：Issue 1「修复指引」第 27-30 行；对比 Issue 2/3/4 的修复指引

**问题描述**：Issue 1 的修复指引仅给出"在 EntityMappingIT.java 中新增 User、Role、Post 三组集成测试方法，每组遵循「创建实体→persist→flush→find→断言」模式"，是**通用模式描述**，不包含：
- 测试方法签名（如 `void user_shouldMapUsernameUniqueConstraint()`）
- 具体断言逻辑（如何验证 unique 约束？需尝试两次 persist 同一 username 并捕获异常）
- `@ManyToMany` 关系的测试策略（需创建关联实体、持久化、查询验证关联表数据）
- Role 和 Post 的 enabled NULL 行为的测试条件指引

而 Issue 2/3/4 的修复指引均精确到**文件:行号级代码变更**（如 "将 `User.java:36` 的 `private Boolean enabled` 改为 `private Boolean enabled = true`"），Issue 3 甚至提供了 16 张表的逐条 ALTER TABLE 模板。Issue 1 的指引在精确度和可操作层面存在明显落差。

**改进建议**：
1. 为 Issue 1 提供至少一个实体的完整测试方法示例（含方法签名、断言逻辑、异常处理）；
2. 明确 `@ManyToMany`（User-Role、User-Post-Post-Function）关联映射测试的具体代码路径；
3. 补充 unique 约束、NOT NULL 约束的验证策略说明；
4. 在指引中引用"需验证的映射点"表中的行号信息，使映射点到测试代码之间具备可追溯性。

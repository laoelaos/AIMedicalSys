## 问题 1：[严重] Issue 3（deleted NOT NULL 不一致）缺少生产数据库迁移方案

**所在位置**：Issue 3 章节「修复指引」（第131-133行）

**问题描述**：Issue 2 提供了完整的生产脏数据行动指引（预检 SQL + 三种清理策略 + 操作顺序建议），但 Issue 3 的修复指引仅提示修改 `schema.sql` 中的 DDL 定义。报告自身在「影响范围」中已指出"手工 DDL 创建的数据库可能包含 `deleted IS NULL` 的脏数据"——但未提供任何脏数据预检、清理策略或 ALTER TABLE 迁移语句。对已存在的生产数据库，直接执行 `ALTER TABLE ... MODIFY COLUMN deleted TINYINT(1) NOT NULL DEFAULT 0` 会因现有 NULL 值而失败。执行者拿到报告后无法直接对生产库操作。

Issue 3 影响面覆盖报告中列举的 16 张表（已通过查阅 `schema.sql` 逐行核实，行号准确），但缺少迁移方案意味着执行者需要自行推导以下内容：脏数据预检方法、表级别逐张执行还是批量脚本、操作顺序、回滚方案。这与其他 Issue（特别是 Issue 2）的指引完整度形成显著不对称。

**严重程度**：严重

**改进建议**：为 Issue 3 补充生产库操作指引——(a) 脏数据预检 SQL（`SELECT table_name, COUNT(*) FROM 各表 WHERE deleted IS NULL`）；(b) 清理策略选项（批量更新为 0、标记为已删除后作归档处理等）；(c) ALTER TABLE 语句模板（可按表逐个生成或通过脚本批量生成）；(d) 操作顺序建议（先清理 → 后 DDL → 再验证约束生效）；(e) 变更后验证语句（尝试插入 NULL 应被拒绝）。

## 问题 2：[中等] 修复方案潜在副作用系统分析缺失

**所在位置**：全文各 Issue 章节

**问题描述**：报告未对各 Issue 修复方案的潜在副作用做系统性分析，导致执行者在实施时可能遇到未预期的后果。具体而言：

(a) **Issue 4 推荐方案 A+C**（Java 端设默认值 + 确认 DDL 已有 DEFAULT 1）——报告未提及以下副作用：现有数据库中 `enabled`/`visible` 为 NULL 的数据不会因 Java 默认值而自动修复。修复后，通过 Java 代码创建的新对象自动获得 `enabled = true`，但已有 NULL 数据继续保留 NULL 值，导致新旧数据在布尔语义上持续不一致。对于 query 层的 `WHERE enabled = 1` 过滤条件，NULL 记录不会被匹配到（SQL 三值逻辑），可能造成功能行为差异。

(b) **Issue 2 修复方案**（添加 `@Column(nullable = false)` + 修改 DDL）——报告的「生产脏数据行动指引」已覆盖主要风险，但未讨论以下问题：`User.java` 当前是否存在代码路径在创建 `User` 对象时未设置 `password` 字段？如果存在，添加 `nullable = false` 后 Hibernate 持久化时将抛出 `PropertyValueException`，需要在代码中同时排查并修复。

**严重程度**：中等

**改进建议**：在每个 Issue 的修复指引下增加「潜在副作用与风险」子节，至少涵盖：(a) 对现有数据的影响；(b) 对代码运行时行为的影响；(c) 需同步排查或修改的关联代码路径。Issue 4 需补充现有 NULL 数据的清理指引（与 Issue 3 类似）。

## 问题 3：[中等] 优先级排序缺乏系统论证

**所在位置**：「优先级排序」章节（第197-204行）

**问题描述**：报告以表格形式给出了 P0-P3 的优先级，并在排序理由栏中给出了每项的排序逻辑。但上一轮审查报告总结中称"优先级排序整体合理"仅为断言性结论，正文未从以下维度提供交叉对比分析：

- Issue 3（P2）影响 16 张表却排在 Issue 4（P1，影响 4 个实体）之后——排序理由是"运行时坏影响相对可控"，但未提供 `deleted IS NULL` 对查询语义的具体影响量化（SQL 三值逻辑下 `WHERE deleted = 0` 会漏掉 NULL 记录），也无从判断"相对可控"的基准是什么。
- Issue 1（P3）被报告自身定位为 Issues 2/3/4 修复后的验证手段——如果 Issue 1 是验证手段，理应在优先级排序中明确标注"与 P0/P1/P2 存在时序依赖：修复后须补充 Issue 1 以验证"，当前 P3 的理由虽提及了关联性但未形成清晰的时序约束说明。
- Issue 3 缺少生产迁移方案（问题 1）——在此缺失下将 Issue 3 评为 P2 是否合理？如果执行者发现无法直接对生产库操作，Issue 3 的实际可操作优先级可能高于 Issue 4。

**严重程度**：中等

**改进建议**：补充优先级交叉对比论证，至少包括：(a) 逐项说明每个 Issue 为什么高于/低于其他 Issue 的理由；(b) 明确标注时序依赖关系（哪些必须在哪些之前/之后修复）；(c) 对"运行时坏影响相对可控"这类主观判断提供依据或量化说明。

## 问题 4：[中等] Issue 1 交叉影响备注存在时序依赖矛盾

**所在位置**：Issue 1「交叉影响备注」（第25行）和 Issue 1「修复指引」（第29行）

**问题描述**：Issue 1 要求测试代码显式调用 `setEnabled(true/false)`，"因 Issue 4 的 enabled 无默认值问题"。但优先级排序中将 Issue 4 列为 P1（"应在新增测试前修复"），且 Issue 4 的推荐方案 A 正是为 enabled 字段添加 Java 默认值 `= true`。如果按优先级先修复 Issue 4（方案 A），Issue 1 的测试代码根本不需要 `setEnabled()` 调用。报告未对这两种可能路径做出条件化说明，导致执行者可能在不必要的情况下修改测试代码。

**严重程度**：中等

**改进建议**：在 Issue 1 的交叉影响备注中增加条件描述："如果 Issue 4 尚未修复（enabled 字段仍无 Java 默认值），测试代码需显式调用 `setEnabled()`；如果 Issue 4 的方案 A 已实施（即 enabled 已添加 `= true` 默认值），则无需显式设置。建议保持显式设置以提高测试可读性，但不应作为强制要求。"

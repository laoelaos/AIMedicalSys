## 逐维度审查

### 1. 类型系统可行性

**[通过]** 所有抽象的类型形态选择（interface / abstract class / class / enum）均与 Java 类型系统能力匹配。泛型 `CapabilityExecutor<T,R>`、`StructuredChatResult<T>`、`LocalRuleFallback<T,R>` 使用正确。单继承+多接口实现的约束被严格遵守。设计中的协作关系（依赖注入、Map 查找+委托模式）可在 Java 中实现。

- §3.1 预检降级路径调用点（第 1344 行）仅传入 13 参数 — 缺少 `modelId` 和 `sentinelReason`

## 修改要求（REJECTED）

### 问题 1：`doDegrade` 方法定义与 3 处调用点签名不一致

- **问题**：§2.3 类图和 §4.1 方法定义使用 15 参数签名（`modelId` 和 `sentinelReason` 作为最后两个参数），但：
  1. §3.1 预检降级路径调用点（第 1344 行）仅传入 13 参数 — 缺少 `modelId` 和 `sentinelReason`
  2. §3.1 薄适配器模板方法超时降级路径（第 1264 行）仅传入 14 参数 — 缺少 `sentinelReason`
  3. §3.1 薄适配器模板方法基础设施异常降级路径（第 1294 行）仅传入 14 参数 — 缺少 `sentinelReason`
- **原因**：三端不一致导致实现无法编译；修订说明声称已修复但实际未修复
- **建议方向**：统一三端为 15 参数签名。对于预检降级路径（第 1344 行），追加 `null, null`（对应 `modelId`, `sentinelReason`），附注释说明降级发生在模型路由和实验分流之前，两字段不可用。对于薄适配器模板方法（第 1264、1294 行），追加 `null`（对应 `sentinelReason`），附注释说明薄适配器不经过实验分流。完成修改后全文搜索 `doDegrade(` 确认所有 ~15 处调用点参数数量均与定义一致

### 问题 2：修订说明与正文内容脱节

- **问题**：v23 修订说明（第 4473-4480 行）宣称 4 项问题均已修复，但预检降级路径调用点（第 1344 行）和第 1264/1294 行的 `doDegrade` 调用参数数量与方法定义不匹配，与修订说明"已确认参数数量完全一致"的声称矛盾
- **原因**：修订说明的声声称未经过正文验证，降低了文档的可信度。此问题在第 15、21、22 轮迭代反复出现
- **建议方向**：v24 修订后，逐项对照修订说明和正文确认变更已实际落地；

## 审查概述

| 项目 | 内容 |
|------|------|
| 审查范围 | 需求响应充分度、整体深度与完整性、落地可执行性（侧重内部审议未充分覆盖维度） |

## 发现的问题

### 问题 1：【严重】修订说明与设计正文混合，产出不是一份可交付的 OOD 设计文档

**问题描述**：`a_v4_copy_from_v3.md` 中，「修订说明」章节从行 ~3443 延续至行 ~3833，长度约 400 行。这还仅是可见的尾部；实质上前序各轮的修订说明以相同模式堆叠——从 `修订说明（v2）` 到 `修订说明（v25）` 再到 `修订说明（v3 — 本轮修订）` 和 `修订说明（v4 — 本轮修订）`，累计约 **1500+ 行**。这些内容是设计过程的演变记录，不属于最终 OOD 设计交付物。将修订历史附着于产出正文中，导致：

- 实现者无法区分"当前设计"与"已废弃的旧提案"
- 文档定位混乱——无法确定这是一份设计规范还是设计日志
- 文档长度膨胀至 ~3800 行，核心设计内容被淹没在历史记录中

**所在位置**：§修订说明（v2）至 §修订说明（v4 — 本轮修订），约行 3443–3833 及所有历史迭代的修订说明段（散布在约 1500+ 行中）

**严重程度**：**严重**

**改进建议**：
1. 将全部修订说明剥离为独立归档文件（如 `design_evolution_log.md`），从产出正文中彻底删除
2. 产出正文只保留最终设计状态
3. 若需保留变更可追溯性，在文档开头用不超过 5 行的"变更摘要"表替代

### 问题 2：【重要】文档内部散布迭代标记，残留大量过程性注记

**问题描述**：正文中散布着大量版本标记，如 `(v4 新增)`、`(v5 新增)`、`(v8 修正)`、`(v11 修正)`、`(v14 修正)`、`(v20 新增)`、`(v23 补充)`、`(v24 补充)` 等，几乎遍布 §3 核心抽象各小节。这些标记是 Agent 跨轮次迭代的跟踪注记，对最终读者无信息价值，反而造成干扰：

- 实现者阅读时需要不断解码"这个 v8 修正和 v4 新增哪个是最终设计"
- 标记表明"此处曾存在问题"，但问题描述和解决逻辑已不可溯
- 文档的版本标识（标题写 v4，正文散布 v5/v8/v11/v14/v20/v23/v24/v25 等）自相矛盾

**所在位置**：散布于 §3.1–§3.11 全文，典型位置如行 734、817、834、856、861、867、900、904、1142、1148、1152、1309、1317、1385、1391、1424、1428、1475、1489、1527、1563、1721、1854、2048 等（不完全列举）

**严重程度**：**重要**

**改进建议**：
1. 全文搜索并清除所有 `(v{N} 新增/修正/补充/修订)` 类标记，替换为空白
2. 标题版本号改为最终稳定版本号（如移除 "v4" 后缀或固定为 "v1.0"）
3. 在文档元信息区（如文档头部）保留一行来源说明即可

### 问题 4：【重要】缺少实施者端的"首次编码引导"信息

**问题描述**：虽然 §1.6 API Surface 状态表按 P0–P3 标注了优先级，但该表仅列出 20+ 个需新建的 API 和组件，未提供：

- **第一行代码应从哪里开始**：无明确的"最小可工作原型"（Walking Skeleton）路径。实现者不清楚应先实现 `AiOrchestrator` 骨架还是 `SlidingWindowMetricsStore` 还是 `AbstractCapabilityExecutor`
- **组件间的编译依赖尚未标注**：状态表虽然有"依赖前提"列，但仅粗粒度标注（如 `依赖 SlidingWindowMetricsStore`），未给出拓扑排序或建议实施顺序
- **Mock/Stub 策略缺失**：哪些组件在初期可以使用简易实现替代以便快速验证管线，没有建议

**所在位置**：§1.6.2（行 121–140）状态表，及全文缺少编码实施引导

**严重程度**：**重要**

**改进建议**：
1. 在 §1.6 之后新增"实施拓扑顺序"小节，给出拓扑排序后的建议实施批次及每批次的验收标准
2. 对 P0 组件进一步细分首批（如先实现 `SlidingWindowMetricsStore` → 再实现 `DegradationStrategy` 骨架 → 再实现 `AbstractCapabilityExecutor`）
3. 补充每个批次的最小验证手段（如单元测试通过标准）

### 问题 5：【中等】需求响应中"风格一致性"参照缺乏具体对照

**问题描述**：需求要求"参考已有的 Phase0、Phase1ABD 的 OOD 设计成果，保持设计风格和结构一致性"。产出在 §1.2 末用一段话（行 40-41）声明"设计风格一致性"，声明已参照。但：

**所在位置**：§1.2（行 40–41）

**严重程度**：**中等**

**改进建议**：

2. 为每条规则标注来自哪个前序设计的哪一节

### 问题 6：【中等】doExecuteInternal() 伪代码中 structuredChat/chat 返回值的两层解包缺乏异常传播说明

**问题描述**：§4.1 中 `structuredChat()` 和 `chat()` 调用通过 `.join().getData()` 两层解包。`CompletableFuture.join()` 在异步任务异常时会包装抛出 `CompletionException`，当前伪代码缺少对该异常的捕获说明：

- `structuredChat()` 和 `chat()` 返回的是 `CompletableFuture<AiResult<...>>`，`.join()` 成功后可能拿到 `AiResult`（含错误码/失败状态），但伪代码直接调用 `getData()` 未检查 `AiResult` 的状态
- 若 `.join()` 抛出 `CompletionException`（LLM 实现层内部异常），伪代码中无对应 catch 块，异常将向上传播到 `supplyAsync` 的 lambda 边界外，最终被 `.exceptionally()` 捕获
- 但 `.exceptionally()` 仅检查 `TimeoutException`，将 `CompletionException` 重新包装为 `CompletionException` 抛出（行 2465–2466），这意味着 LLM 调用内部 Infrastructure 异常的实际分类信息丢失——未进入 `LlmInfrastructureException` catch 分支

**所在位置**：§4.1（行 2548–2582）

**严重程度**：**中等**

**改进建议**：
1. `.join()` 调用外层包裹 try-catch，区分 `CompletionException` 和非 `CompletionException`
2. `CompletionException` 中拆解出原始异常，按 `LlmInfrastructureException`/其他分类重新路由到对应 catch 分支
3. 补充 `AiResult` 状态检查：`.getData()` 前验证 `AiResult.isSuccess()`，失败时走对应降级路径而非静默获取 null data

---

### 问题 7：【中等】未定义 AiService 接口自身的线程安全契约

**问题描述**：`AiService` 接口被 13 个业务模块同时并发调用（Spring 默认单例），且其唯一正式实现 `AiOrchestrator` 的方法签名返回 `CompletableFuture`。虽然 §6.1 阐述了 `AiOrchestrator` 和 `CapabilityExecutor` 的线程安全模型，但接口层面上未定义：

- 调用方是否需要保证每次调用使用独立的 `request` 实例
- `AiService` 实现是否允许多线程并发调用同一方法传入共享的 `request` 对象
- 返回值 `CompletableFuture` 完成后 `AiResult` 的可变性契约

同类问题也存在于 `FallbackAiService`——它作为 `@Primary` 装饰器是真正注入到业务模块的入口点，其线程安全契约直接影响所有业务调用方。

**所在位置**：§1.3 核心抽象表（`AiService` 行）、§3.1（`FallbackAiService` Bean 装配段）、§6.1

**严重程度**：**中等**

**改进建议**：
1. 在 §1.3 `AiService` 职责描述中补充线程安全契约（建议：实现需线程安全；request 应为只读，调用方无需额外同步）
2. 在 `FallbackAiService` 描述中补充装饰器层面的线程安全保证

### 问题 8：【轻微】§3.11 共同约束中关于 DTO 改造的约定与 §3.5 过渡策略存在细微不一致

**问题描述**：§3.11 共同约束（行 3436）声明"所有 7 项能力的 DTO 均在 Phase 5 底座切流时完成 `AiRequestBase` 继承改造"。但 §3.5 过渡策略（行 1851）明确"Phase 5 底座切流时仅要求 7 项底座能力的 DTO 完成基类继承改造"——两处一致。然而 §3.5 关于 Jackson 兼容性测试的要求（行 1855–1859）定义了对 7 项底座能力 DTO 的测试，而 §3.11 的共同约束未引用这些测试要求，实现者可能完成 DTO 改造但遗漏配套的兼容性测试。

**所在位置**：§3.11 共同约束（行 3436–3439）与 §3.5（行 1851–1859）

**严重程度**：**轻微**

**改进建议**：
1. §3.11 共同约束中增加一行指向 §3.5 的 Jackson 兼容性测试要求
2. 或直接在 §3.11 末尾补充"各能力 DTO 改造完成后需执行 §3.5 定义的 3 个兼容性验证场景"

## 整体质量评价

设计的技术深度和细节覆盖度非常高（核心抽象定义完整、接口契约精确、错误分类表详尽、伪代码可执行可验证），在本项目 5 个 Phase 的 OOD 设计中属于质量较高的一批。上述 8 个问题主要集中在 **产出可交付性** 和 **结构规范性** 两个维度，而非设计本身的技术缺陷。核心建议为：(1) 将修订历史剥离为独立文件，产出回归"最终设计"定位；(2) 清除内部版本标记，实现者应阅读的是设计本身而非进化过程；(3) 增加实施引导信息，使产出从"是什么"延伸到"怎么做"。

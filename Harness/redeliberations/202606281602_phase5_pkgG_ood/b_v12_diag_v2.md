# 质量审查报告 — Phase 5 包G OOD 设计方案 v12（第2轮）

- **任务**：Phase 5 包G AI 进阶底座 OOD 设计方案质量审查
- **审查视角**：实际落地评估——设计是否可直接指导编码实现、接口定义是否足以支持下游消费者、异常场景和边界条件是否已考虑

## 问题列表

### 问题 1：[重要] 接口定义的消费者充分性——薄适配器能力在 AiCallRecord 中形成系统性分析盲区，下游消费者无法按关键维度聚合评估

**问题描述**：
§4.2 薄适配器成功路径（行 3116-3120）的指标记录中，`AiCallRecord.success()` 传入的 `modelId=null`、`retryCount=0`；薄适配器的所有降级路径和成功路径中 `promptVersion` 恒为 `null`。这导致 6 项薄适配器能力的 AiCallRecord 中 `modelId`、`promptVersion`、`retryCount` 三个字段**永远为空**，产出以下系统性分析缺口：

1. **模型维度完全不可见**：下游消费者（管理端统计报表、监控大盘）无法按模型维度聚合分析这近半数能力的调用分布和成本，这是"性能观测内建化"目标（§1.1）的关键缺口
2. **两种业务场景不可区分**：`modelId=null` 同时表示"薄适配器不感知模型路由"和"底座未配置模型路由"两种完全不同的运维状态，AiCallRecord 无法区分
3. **数据量级影响显著**：6 项薄适配器能力占总 AI 能力数的 46%，此维度的系统性缺失意味着近一半 AI 能力的性能分析无法跨模型聚合

设计在 §4.2 的注释中解释了 retryCount=0 的原因（"Phase 4 服务的内部重试行为对底座不可见"），但未评估此限制对下游分析的影响程度，也未为未来扩展预留接口契约。

**严重程度**：重要

**改进建议**：
1. 在 §3.1 Phase 4 薄适配器契约定义中，为 Phase 4 服务接口新增可选元数据暴露方法（如 `getUsedModelId(): String`、`getRetryCount(): int`），使可升级的 Phase 4 服务可填充这些字段；不可升级的保持 null
2. 在 §3.5 AiCallRecord 字段定义表的 modelId/promptVersion/retryCount 三行中补充"薄适配器场景下不可用的原因及未来扩展方向"说明
3. 在 §1.4「底座切流初期已知限制」中新增一条，集中说明 6 项薄适配器能力的指标分析维度缺口

### 问题 2：[重要] 需求响应充分度——实验分流异常与"无实验命中"两种场景的 assignment 返回值语义重叠，下游分析无法区分正常无实验与异常降级

**问题描述**：
§3.4 约定 `ExperimentManager.assign()` 在两种场景下返回不可区分的值：
- "无实验命中"（正常业务场景）：返回 `ExperimentAssignment.createDefault()`，`groupId="default"`, `targetPromptVersion=null`
- "实验分流异常"（§4.1 行 2800-2804 的 catch 块）：返回 `new ExperimentAssignment(null, "default", null, null)`，同样 `groupId="default"`

两种场景在 AiCallRecord 中均体现为 `promptVersion=null`、`groupId="default"`，下游消费者（实验效果分析报表）无法区分"该请求确实未参与任何实验"与"该请求的实验分流系统出现异常被迫回退"。这导致：

1. 实验效果分析中将异常降级的数据错误归入"无实验"基线，污染对比分析的基线条数据
2. 运维无法通过监控大盘感知实验分流异常的发生率和影响范围
3. 当 catch 块中 `experimentManager.assign()` 频繁抛出异常时，实验效果分析的失真量级不可追溯

此外，`promptVersion=null` 在三种降级路径（预检降级行 2772、超时降级行 2787、薄适配器降级行 3090）中承载多重语义——既是"实验分流前已降级，无实验上下文"，也可能是"实验分流后降级，但实验未命中"。三者无法从 AiCallRecord 的 `promptVersion` 字段值上区分。

**严重程度**：重要

**改进建议**：
1. 将"实验分流异常"场景改为返回 `ExperimentAssignment.createErrorFallback()`，使用 `groupId="assignment_error"` 或类似可区分的 Sentinal 值，与 `createDefault()` 的 `"default"` 明确区分
2. 在 §3.4「无实验命中的返回值语义」段落中补充两种场景的区分规则和 `createErrorFallback()` 的契约定义
3. 为 AiCallRecord.promptVersion 引入空值守卫标记方案，或在 §3.5 AiCallRecord 字段表中为 promptVersion 补充说明——区分"实验分流前降级"（promptVersion 恒未赋值）、"实验分流后未命中"（promptVersion=null 但有有效 groupId）

### 问题 3：[重要] 完整性——运行时配置热加载机制覆盖不完整，生产运维灵活性受限

**问题描述**：
文档为三个组件设计了运行时热加载机制：
- ModelRouter 路由配置 → `RouteConfigChangedEvent`（§3.2）
- PromptTemplate 模板配置 → `TemplateChangedEvent`（§3.3）
- ExperimentManager 实验配置 → `ExperimentChangedEvent`（§3.4）

但以下关键运行时配置**不具备热加载能力**，仅启动期通过 `@ConfigurationProperties` 一次性绑定：

1. **`execution.timeout.per-capability`**（§9.5 行 3591-3608）：各能力的端到端超时阈值。若运维发现某能力 P99 响应时间上涨需调整超时，必须重启底座
2. **`execution.timeout.thin-adapter.per-capability`**（§9.5 行 3613-3619）：薄适配器各能力的委托超时配置。同样仅启动期加载
3. **`degradation.strategies`**（§9.5 行 3621-3623）：各能力的降级策略白名单。若某能力因外部模型服务波动频繁熔断，运维无法在不重启的情况下替换为 noop 策略临时恢复服务
4. **`sliding-window.window-seconds`**（§9.5 行 3661）：滑动窗口时间宽度，影响降级判定灵敏度。启动期不可修改

虽然 §10.4 描述了分布式架构下的事件驱动重构方案，但**单体部署形态下**这些配置的热加载缺失直接影响日常运维效率。从"实际落地"视角评估，这些都是生产运行中需要运行时调整的参数，不支持热加载意味着任何调优都需要经历"修改配置 → 重启实例 → 验证效果"的重启周期，降低了运维响应速度和系统的可用性。

**严重程度**：重要

**改进建议**：
1. 为 `AiExecutionProperties`、`AiDegradationProperties`、`AiSlidingWindowProperties` 等配置类增加 Spring `@RefreshScope` 支持（需集成 Spring Cloud Context 或 Spring Cloud Config），使这些配置可通过 `/actuator/refresh` 端点热刷新
2. 若项目不使用 Spring Cloud，可自定义 `@Scheduled(fixedDelay = 60000)` 定时刷新机制——定期重读 `@ConfigurationProperties` 绑定的 PropertySource，检测到配置版本变更后通过 `ApplicationEvent` 通知各消费者
3. 两种方案均不适用时，至少在 §9.5 为每项非热加载配置添加显式注释"⚠️ 仅启动期加载，修改后需重启实例"，使运维明确配置管理边界
4. 在 §11 测试策略中新增配置热刷新的集成测试验证

### 问题 4：[一般] 逻辑一致性——Experiment 中流量分配百分比使用千分比（percentage 值域 0-1000）与业务语义不匹配，增加理解和配置出错风险

**问题描述**：
§3.4.4 ExperimentGroup 字段表中定义 `percentage: int`（0-1000，对应 0.0%-100.0%）。哈希分桶算法使用 `userId/sessionId 哈希值 % 1000` 映射到 [0, 1000) 区间后，进行累积百分比判断。

使用千分比而非百分比（0-100）的理由可能源于更精细的流量控制需求，但设计文档未提供"为什么千分比比百分比更合适"的决策理由。更重要的是：

1. 千分比与业界常见的百分比表述不一致——管理端配置页面需要将用户输入的百分比（如 "30%"）乘以 10 才能存入 `percentage` 字段，管理端开发者和运维人员需要额外的换算心智负担
2. 哈希模 1000 的边界检查（§3.4.4 行 1842-1845 的"所有分组 percentage 之和必须等于 1000"）增加了管理端的校验复杂度——若运维配置三个分组分别为 33.4%、33.3%、33.3%，千分比下的和为 1000 但业务语义上并不精确

**严重程度**：一般——此设计决策不产生运行时错误，但增加了管理端实现的复杂度和运维配置的使用门槛。

**改进建议**：
1. 在 §3.4.4 流量分配算法约束后补充"为何使用千分比（0-1000）而非百分比（0-100）"的设计决策理由
2. 或改为百分比（0-100）+ 内部浮点累加比较，消除千分比换算和累加和必须等于 1000 的强制约束

## 整体质量评价

本文档整体深度和完整度很高（3850 行），经过 12 轮迭代后，架构概述、模块划分、类图、核心抽象定义、关键行为伪代码、错误处理、并发设计、设计决策、Maven 依赖、迁移路径、YAML 配置、协作边界和测试策略等 OOD 核心要素均已覆盖到可直接指导编码的程度。经过 v1 审查发现的技术级指标双记问题、v11/v12 修正的 callerRole 不一致和 catch 死代码等已得到解决，设计在技术可行性维度已非常成熟。

剩余的主要质量风险集中在 **接口定义的消费者充分性**（问题 1：薄适配器指标分析盲区）、**业务场景区分度**（问题 2：实验分流异常不可追溯）和 **生产运维灵活性**（问题 3：配置热加载覆盖不完整）。这三个问题不会阻止首次编码实现，但将在底座投入生产后影响运维效率和数据分析能力，建议在 Phase 5 切流前优先解决。

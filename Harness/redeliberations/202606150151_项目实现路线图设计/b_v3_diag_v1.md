# 智慧云脑诊疗平台——项目实现路线图（v3）质量审查报告

> 审查模式：technical（技术方案设计）
> 审查对象：`a_v3_tech_v1.md`（v3 修订版）
> 历史迭代：`iteration_history.md`（第 1 轮 17 项 + 第 2 轮 3 项）
> 审查依据：`requirement.md`（再审议任务需求）、`Docs/01_requirement.md`（原始需求规格）
> 审查范围：未由组件 A 内部审议充分覆盖的维度（需求响应充分度、整体深度和完整性、时序一致性、关键遗漏等）

---

## 一、总体评价

v3 在 v2 基础上针对第 2 轮 3 项反馈（拓展 AI 耦合颗粒度、退药角色矩阵、健康档案基础/完整版边界）均做了实质性收紧与补全；并新增了 SQL-Toy 持久化双栈、AI 进阶五组件统一口径、§6 非功能需求承接一览、前向引用清理、协作硬约束表拆分等系统化改进。整体结构清晰、骨架阶段"最小化"判据充分、阶段 5~8 粒度适中、AI 分层策略收敛合理、退药/调拨/盘库的角色矩阵与状态机联动已细化到 OOD 可直接开工的层级。

但本轮仍存在若干**时序一致性**、**业务边界**与**少数关键功能承接**问题需进一步收口。详见下文。

## 二、问题清单（按严重程度排序）

### P1 问题（严重，须在交付前修复）

#### P1-1：3.1.3.2 病情咨询与 3.4.8 知识库问答存在显著时序矛盾

- **问题描述**：
  - `Phase 2.2` 阶段交付物明确列出 "3.1.3 AI 智能问诊（含 **3.1.3.1 导诊** / **3.1.3.2 病情咨询** / **3.1.3.3 智能挂号**）"作为 Phase 2 患者端交付内容。
  - 但需求 `§3.1.3.2` 病情咨询在输入/输出/降级上**强依赖 3.4.8 AI 知识库问答**（"本节调用 3.4.8 AI 知识库问答能力，输入为患者自然语言问题...输出为 AI 文本回复、追问建议与免责声明标志..."）。
  - 路线图 `Phase 5.2` 才将 3.4.8 列入交付（"3.4.8 AI 知识库问答（在 Phase 2 病情咨询基础上的知识库接入）"），`Phase 5.4` 才给出"患者端 3.1.3.2 病情咨询回答'应就医提醒'等"的可演示口径。
  - 这意味着 **Phase 2 期间 3.1.3.2 病情咨询在 UI 层面已存在，但其底层 AI 能力（3.4.8）要到 Phase 5 才落地**——Phase 2 完成后到 Phase 5 落地的中间若干阶段，3.1.3.2 处于"挂着 UI 入口但 AI 不可用"的状态；该状态的可演示性、降级路径（显示"暂不可用，请稍后重试"）、UI 提示（`disclaimer_required`）在 `Phase 2.4` 可演示能力中**完全未提及**。
- **所在位置**：
  - `a_v3_tech_v1.md` L170（Phase 2.2 交付物 3.1.3.2）
  - `a_v3_tech_v1.md` L359（Phase 5.4 可演示能力 3.1.3.2）
  - `a_v3_tech_v1.md` L339（Phase 5.2 3.4.8 落地）
  - `Docs/01_requirement.md` L293-302（3.1.3.2 对 3.4.8 的强依赖）
- **严重程度**：**严重**——这是 13 项 AI 能力中**唯一一项**在 Phase 2 阶段被 UI 消费但底层能力延后到 Phase 5 落地的能力，与"分层递进"和"可演示"两条硬约束都存在张力。
- **改进建议**：
  - **方案 A（推荐）**：明确将 3.1.3.2 拆为"UI 框架 + AI 占位"（Phase 2）与"接入 3.4.8 知识库"（Phase 5）两个增量。`Phase 2.4` 补充"3.1.3.2 病情咨询 UI 可达，AI 调用走 Mock 兜底文案（与 3.4.8 降级路径一致：显示'暂不可用，请稍后重试'），显著位置显示 AI 免责声明"的演示口径；`Phase 5.4` 补"3.1.3.2 升级为真实 3.4.8 调用"的可演示增量。
  - **方案 B**：将 3.4.8 知识库问答前移到 Phase 2，与 3.1.3.2 同阶段落地（这会使"分层递进"约束变重，需评估是否挤占 Phase 2 工作量上限）。

#### P1-2：Phase 2.4 取消挂号演示口径与"缴费记录未落地"事实存在时序冲突

- **问题描述**：
  - `Phase 2.4` 可演示能力写"取消挂号：距就诊 >2h 成功，**关联缴费记录由'已支付'变更为'待退费'**"。
  - 但实际 `Phase 2` 阶段：① 3.2.10 医嘱开立与记账属于 Phase 3 交付；② 3.2.9 线下窗口实际收银属于 Phase 4 交付。Phase 2 阶段**根本不存在任何"已支付"缴费记录**——挂号本身不直接产生缴费记录。
  - 需求 `§3.1.4.1` 也明确"若已生成缴费记录且为'已支付'状态..."——"若"条件意味着取消挂号可能不涉及缴费（"若挂号尚未绑定处方或已绑定处方但未发药，取消不触发库存动作"）。
  - 路线图 `Phase 2.3` 已正确写出"挂号状态机：待就诊 / 已就诊 / 已取消 / 待退费（中间态，与 Phase 4 缴费记录状态机的'待退费'态对齐）"——即"待退费"是**未来 Phase 4 的中间态占位**，但在 `Phase 2.4` 演示能力中**已演示该中间态的实际状态变更**，与 Phase 2 自身业务能力（无收费）不匹配。
- **所在位置**：
  - `a_v3_tech_v1.md` L187（Phase 2.4 取消挂号演示能力）
  - `a_v3_tech_v1.md` L178（Phase 2.3 挂号状态机"待退费"中间态）
  - `a_v3_tech_v1.md` L216（Phase 3.2 后端交付物"3.2.10 医嘱开立与记账"）
  - `Docs/01_requirement.md` L352（取消挂号"已支付 → 待退费"链路描述）
- **严重程度**：**严重**——会让 OOD 阶段误以为 Phase 2 就要落地"已支付→待退费"的完整事务链路，进而在 OOD 数据模型中过早引入缴费记录的完整状态机。
- **改进建议**：
  - `Phase 2.4` 修正为"取消挂号：距就诊 >2h 成功，挂号状态机由'待就诊'推进为'已取消'，并**预留'待退费'中间态字段**（实际值由 Phase 4 缴费记录联动写入，本阶段该字段恒为 null）"；或简化为"取消规则引擎的时间窗（>2h 允许 / ≤2h 拒绝）可演示"。
  - `Phase 2.6` 准出口径同步修订：将"待退费"相关项移到 Phase 4.6 准出。

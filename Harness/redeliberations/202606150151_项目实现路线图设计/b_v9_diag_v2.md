## 二、用户硬约束的深度验证（响应 v1 质询 §3 要求）

用户在 `requirement.md §2 与 §7` 明确"骨架完成后必须支持多人并行贡献"是 Phase 0 验收的**硬约束**，并要求组件 B 重点检查此项。v1 报告仅以"✓ Phase 0.2 协作规范四要素已可执行"一句带过，深度不足。v2 新增三项具体子验证。

### 2.1 接口契约冻结时机的并行起步影响

- **当前状态**：Phase 0.2 声明接口契约骨架仅含"框架级"（统一响应 / 分页 / 错误码命名空间 / 3.4.x AI 能力契约占位类型），"模块级"接口契约（7 个业务模块的端点清单与 schema）冻结时机为 **Phase 1 启动前**（L41 / L96）。
- **风险点**：Phase 0 阶段 7 个业务模块团队**无法在 Phase 0 内就各自模块的端点 schema 进入实质开发**——他们必须等待 Phase 1 启动前冻结才能并行；Phase 0 团队的工作被压缩到"搭骨架 + 等冻结"。
- **影响评估**：骨架阶段目标 L33 是"让多人可并行继续贡献"，但接口契约推迟冻结实际让"并行贡献"主要发生在 Phase 1 而非 Phase 0。**对骨架是否真的"能跑起来 + 能并行"的判定需细化为两层**：
  - **Layer 1（基础骨架层）**：框架级契约（4 类）+ 协作规范 4 要素 + 三端空壳 → 满足"能跑起来"——**达标**。
  - **Layer 2（业务模块层）**：模块级契约冻结推迟到 Phase 1 启动前 → Phase 0 期间各业务模块团队**只能"搭模块目录 + 写占位 Controller"**，实质业务开发在 Phase 1 启动后才开始。
- **判定**：骨架阶段对**基础架构层**的多人并行已具备充分条件；但对**业务模块层**的并行起步，文档需明示"Phase 0 内业务模块团队的实际贡献为模块目录 + README + 占位代码，实质端点开发需等待 Phase 1 启动前的接口契约冻结"。当前 L33 / L78 / L82-L89 的"并行贡献"措辞对此层区分模糊。
- **改进建议**：Phase 0.1 阶段目标增列一句"本阶段并行贡献的颗粒度为基础骨架层（CI / 协作规范 / 三端空壳），业务模块层（端点 schema）需等待 Phase 1 启动前接口契约冻结"；L82-L89 任务分配维度也据此区分"基础骨架层任务"与"业务模块目录占位任务"。

### 2.2 CI 软预备对并行 PR 的质量门控影响

- **当前状态**：Phase 0.2 L45 将"持续集成"行"层级"列由"硬必需"降为"软预备"——CI 占位流水线仅固化"CI 配置文件存在且能跑构建+单测占位"；CI 必过项清单（构建+单测+接口契约 OpenAPI diff+代码风格+单测覆盖率阈值）作为 **Phase 1 首期交付**。
- **风险点**：Phase 0 期间若有 PR 提交到主分支，CI 流水线**仅校验"能跑构建+单测占位"**，缺少 OpenAPI diff、代码风格、单测覆盖率三类质量门控——**Phase 0 阶段产生的 PR 质量门控比 Phase 1 启动后更松**。
- **影响评估**：与 v7 P1-1（协作规范可执行性）的初衷一致——Phase 0 阶段不强行引入 4 类质量门控（避免"骨架最小化被误读"）；但**风险表"骨架阶段'最小化'被误读为'省略关键基础设施'"行（L545）未提及"CI 推迟导致 PR 质量门控短期薄弱"这一具体子风险**。
- **改进建议**：风险表 L545 行增列"CI 软预备导致 Phase 0 期间 PR 质量门控短期薄弱"子项，缓解策略"Phase 0 期间所有 PR 必须由模块 owner 人工复核四要素（接口契约 / 迁移 SQL / AGENTS.md / 单测），与 CI 必过项完全等价——Phase 1 启动后由 CI 自动接管"；或将该子风险下沉到附录 D「下沉的实施级风险」。

### 2.3 跨模块接口变更的同步流程覆盖

- **当前状态**：Phase 0.2 L44 协作规范"Code Review 必查项"已包含"接口契约是否变更并同步 OpenAPI / 是否有跨模块调用并经被调用方 owner 评审"两条；Phase 0.3 L64 明确"OpenAPI / TypeScript 类型生成（后端导出 → 前端 `@types` 包）"作为前后端契约决策。
- **判定**：跨模块接口变更的同步流程（PR 模板字段 + CR 必查项 + OpenAPI 双向同步 + 跨模块 owner 评审）**已在协作规范四要素中闭环**。
- **影响评估**：**达标**。新加入的开发者基于 Phase 0.2 协作规范即可独立完成跨模块接口变更的提交、评审、同步三步流程，无需 OOD 阶段介入。

### 2.4 综合判定

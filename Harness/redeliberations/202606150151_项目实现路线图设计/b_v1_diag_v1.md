# 质量审查报告：项目实现路线图设计（a_v1_tech_v1.md）

> 审查对象：`a_v1_tech_v1.md`（智慧云脑诊疗平台——项目实现路线图）
> 审查模式：technical（技术方案设计文档质量审查）
> 审查重点：需求响应充分度、整体深度和完整性、内部审议未覆盖的维度
> 审查时间：第 1 轮再审议
> 审查员：组件 B（diagnose 子 Agent）

---

## 0. 总体评价

产出文档在**结构化、可执行性、协作维度**三个维度上完成度较高：阶段划分（7 阶段）满足 5~8 的要求，骨架阶段（Phase 0）的"协作地形"清单、并行贡献模块切分（模块 A~F）描述清晰，每个阶段都包含"阶段目标/交付物/技术决策/可演示能力/可分配子任务/准入准出"六要素，整体优于同类路线图文档的常见水准。

但经过对需求文档与产出文档的逐项对照，**存在若干中等及严重级别的质量问题**，主要集中在以下几类：
1. **多处事实性错误与逻辑自相矛盾**（核心三件套编号顺序与需求不一致；Phase 5.1 自相矛盾的 AI 清单；准出口径与实际交付物不匹配）；
2. **关键需求遗漏**（拓展任务中的"微服务拆分"与"硬件采集"完全缺失；药房药库库存调拨的过程描述弱化；医嘱开立与记账边界不清）；
3. **AI 能力分层递进策略的执行出现"越位"**（Phase 3 提前落地 3.4.4 / 3.4.9 / 3.4.10 三项"拓展 AI"，破坏了"核心三件套→拓展"分层收口约束）；
4. **若干含混表述**（"仅记账"、"○² 权限"、">2h"符号、AI 3.4.7 在两阶段的重叠职责）影响下游可执行性。

下文按严重程度逐项展开。

## 1. 严重问题（必须修复）

### 1.1 【严重】核心三件套编号顺序与需求定义相反

- **问题描述**：需求文档 §4 AI 能力维度明确"核心三件套（分诊/审核/病历生成）"。按 3.4.x 编号对应关系，应为：3.4.1=分诊（核心 1）、3.4.2=审核（核心 2）、3.4.3=病历生成（核心 3）。但产出文档在两处将 3.4.2 与 3.4.3 调换：
  - 文档 L177（Phase 3.1）："落地核心三件套中第二件'AI 病历生成'（3.4.3）与第三件'AI 处方审核'（3.4.2）"
  - 文档 L185（Phase 3.2）："**核心三件套之 [AI 病历生成]** ……**核心三件套之 [AI 处方审核]**"
  - 文档 L365（阶段间总览对照表 Phase 3 行）："3.4.3（核心 2）、3.4.2（核心 3）"
- **所在位置**：L177、L185、L365
- **严重程度**：严重（事实错误，破坏阶段交付物的"核心→拓展"编号体系，下游 OOD 文档将基于此映射类/接口，影响全局）
- **改进建议**：
  - 将 L177、L185、L365 中所有"3.4.2 = 核心 3 / 3.4.3 = 核心 2"的表述统一修正为"3.4.2 = 核心 2（AI 处方审核）、3.4.3 = 核心 3（AI 病历生成）"；
  - 同时核查全文（含 3.6 准出口径"任务四/任务五"）中与该编号相关的描述是否一致。

### 1.2 【严重】Phase 5.1 出现"剩余 7 项 AI"与同节备注自相矛盾

- **问题描述**：Phase 5.1（L268-L272）正文列出"补齐剩余 7 项拓展 AI（3.4.7 影像 / 3.4.8 知识库 / 3.4.12 排班 / 3.4.13 综合讨论结论 / 3.4.5 检查报告 / 3.4.6 检验报告 / 3.4.11 执行顺序推荐）"——这 7 项被定性为"剩余"。但同节备注立刻说明"3.4.5 / 3.4.6 / 3.4.11 已在 Phase 4 落地于医生端工作台"。两句话在同一段落内直接矛盾。
- **所在位置**：L268-L272
- **严重程度**：严重（路线图最核心的"阶段准出口径"出现自反逻辑错误，审阅者无法判断哪一句为准）
- **改进建议**：
  - 修正为"补齐剩余 4 项拓展 AI（3.4.7 影像 / 3.4.8 知识库 / 3.4.12 排班 / 3.4.13 综合讨论结论）+ 3.4.7 的模型管理面 + 3.3.3 管理员分诊台规则"；
  - 与 5.2、5.4、5.5 中的 AI 清单保持完全一致。

### 1.3 【严重】Phase 5.6 准出口径"除 3.4.5/3.4.6/3.4.11 外其余 10 项"漏算 3.4.7

- **问题描述**：Phase 5.6（L308）写"13 项 AI 能力中除 3.4.5 / 3.4.6 / 3.4.11 已在 Phase 4 落地外，其余 10 项均可演示"。但 13 - 3 = 10 看似自洽，实际漏算了 3.4.7：Phase 4.2（L228）已明确将 3.4.7（AI 影像分析：头部 CT / 肺部 CT / 肿瘤分割）作为交付物。Phase 4 实际落地 4 项 AI 能力（3.4.5、3.4.6、3.4.7、3.4.11），Phase 5 才需要落地 9 项。准出口径与实际交付物矛盾。
- **所在位置**：L308
- **严重程度**：严重（准出条件是验收硬约束，口径错误会导致 Phase 5 提前收口、Phase 6 联调时才发现能力缺失）
- **改进建议**：将 5.6 修正为"13 项 AI 能力中除 3.4.5 / 3.4.6 / 3.4.7 / 3.4.11 已在 Phase 4 落地外，其余 9 项均可演示"。

### 1.4 【严重】需求中"微服务"与"硬件采集"两个拓展任务完全缺失

- **问题描述**：需求文档 §4 与 §6 关键参考维度明确"任务分层维度：核心任务一～六 → 拓展任务（实时通信/微服务/AI 进阶/部署/硬件采集/影像 AI）"。产出路线图覆盖了实时通信（Phase 6）、AI 进阶（Phase 5）、部署（Phase 6）、影像 AI（Phase 4-5），但 **"微服务"与"硬件采集"两个拓展任务没有任何阶段/任务包承接**：
  - 微服务：Phase 0 仅以"Spring Boot 3 多模块工程"占位，未说明单仓多模块→微服务的演进路径，Phase 6 部署章节也未提及微服务化（如服务注册发现、配置中心、网关拆分）。
  - 硬件采集：需求 §1.4 检查检验域中硬件检查设备的数据采集是真实业务环节，路线图完全未提及（既未列入任何阶段的交付物，也未列入风险/依赖章节）。
- **所在位置**：全文（无对应章节）
- **严重程度**：严重（需求明确列举的 6 个拓展任务缺失 1/3，与需求响应充分度直接冲突）
- **改进建议**：
  - 在 Phase 0.3 关键技术决策"工程组织"决策列中追加"演进路径：单仓多模块（Maven 聚合）→ Phase 5/6 视负载热点服务按 `services/` 切分为独立 Jar 部署形态（服务注册发现 Nacos/Eureka + Spring Cloud Gateway + 配置中心 Nacos Config）"，并说明"分阶段微服务化"原则；
  - 新增或在 Phase 4 增补"硬件接入"任务包（如"医疗设备数据采集适配层：HL7 / DICOM / 串口 / 网关协议，预留 `device-gateway` 后端模块占位；Phase 4 仅做接口契约与 Mock 数据"），并明确"骨架阶段（Phase 0）即在 `services/` 下预留 `device-gateway` 目录与 README 占位，避免后续大改"。

### 1.5 【严重】药房药库库存调拨的过程与子任务描述不足

- **问题描述**：需求 §1.4 药房药库域明确强调"药房药库的库存调拨"是核心业务场景。Phase 4.2（L230-L231）的"药库域"交付物仅一行"3.2.6 药库管理；含库存调拨记录、盘库后差额回退药库流程"，未描述：
  - 调拨的发起方、审批方、执行方（管理员？药房医生？药库管理员？）；
  - 调拨单的状态机（草稿/审批/在途/已接收/已拒绝）；
  - 调拨与日配额/子库存的勾稽关系；
  - 调拨失败的回退路径。
  - Phase 4.5 包 D"药库域（3.2.6 + 调拨 + 盘库）"也未将"调拨"作为独立子任务列出。Phase 4.3 仅说"药房库存扣减/回补"而未说"药库→药房调拨"。
- **所在位置**：L230-L231、L240-L241、L256-L257
- **严重程度**：严重（关键业务场景的"过程"未在路线图层级收敛，落到 OOD 阶段会因缺乏边界而出现接口反复）
- **改进建议**：
  - 4.2 药库域交付物表格展开"调拨单"与"盘库单"为独立行；
  - 4.3 关键技术决策追加"调拨单状态机（草稿/审批/在途/已接收/已拒绝/已撤销）+ 调拨过账（药库出库 → 药房入库双写对账）+ 调拨差异告警"；
  - 4.5 拆为"包 D1：药库基础管理（3.2.6 + 盘库）"、"包 D2：调拨管理（调拨单生命周期 + 药库→药房过账）"，并将可分配粒度提升到 OOD 阶段可直接开工的层级。

### 1.6 【严重】"医嘱开立与记账"边界不清，可能被误读为记账只做半截

- **问题描述**：Phase 3.2（L184）"医嘱开立与记账（3.2.10，仅记账）"与 Phase 3.5 包 F（L210）"医嘱开立与记账（3.2.10，本阶段仅记账动作）"两处都把"医嘱开立"和"记账"并列后又用"仅记账"修饰。"仅记账动作"既可被理解为"开立部分已做、本阶段只补充记账"，也可被理解为"开立部分不做、本阶段只做记账"——两种解读会导致 OOD 阶段对 3.2.10 的实现范围产生分歧。
- **所在位置**：L184、L210
- **严重程度**：严重（影响 OOD 对 3.2.10 范围的判定，进而影响 Phase 3 与 Phase 4 的接口契约）
- **改进建议**：
  - 明确表述为"3.2.10 医嘱开立（医生端 UI + 处方/检验/检查单据生成）+ 记账（收费前置单生成）在本阶段一并完成；记账仅指生成收费前置单据（带'待缴费'状态），实际收银动作在 Phase 4 线下窗口（3.2.9）落"；
  - 或拆为两行交付物：3.2.10-a 医嘱开立、3.2.10-b 记账前置单。

## 2. 中等问题（应修复）

### 2.1 【中等】AI 能力"核心→拓展"分层在 Phase 3 出现越位

- **问题描述**：需求文档 §4 与 §7 隐含标准明确"AI 能力按'核心三件套→拓展'分层递进，避免一上来就铺 13 项 AI 能力导致失控"。Phase 3 在落地核心三件套的 3.4.2 / 3.4.3 同时，把三项拓展 AI（3.4.4 智能诊断 / 3.4.9 开立检查 / 3.4.10 辅助开方）一并提前落地（L186-L187）。这一做法在工程上合理（诊疗闭环确实需要这些能力），但与"分层递进"的收口约束在字面上冲突，且未给出合理解释。
- **所在位置**：L186-L187
- **严重程度**：中等（不影响交付，但破坏与需求的对齐口径）
- **改进建议**：
  - 在 Phase 3.1 阶段目标中明确"为支撑'AI 病历生成'与'AI 处方审核'的诊疗闭环，本阶段同步落地 3.4.4 / 3.4.9 / 3.4.10 三项与三件套强耦合的拓展 AI 能力（不视为'AI 失控铺开'）"；
  - 或将 3.4.4 / 3.4.9 / 3.4.10 后移到 Phase 4，Phase 3 用"占位接口 + 本地规则降级"先保证病历/审核可演示。

### 2.2 【中等】AI 能力 3.4.7（影像分析）在 Phase 4 与 Phase 5 的职责边界不清晰

- **问题描述**：
  - Phase 4.2（L228）："3.4.7：头部 CT / 肺部 CT / 肿瘤分割，岗位决定可用模型"——读作"分析能力在 Phase 4 落地"；
  - Phase 5.2（L280）："3.4.7 AI 影像分析（在 Phase 4 检查域基础上的模型管理面，按岗位开放）"——读作"分析能力在 Phase 4，Phase 5 仅做模型管理"；
  - 阶段间总览对照表（L366 与 L367）同时把 3.4.7 列入 Phase 4 与 Phase 5 两行。
- **所在位置**：L228、L280、L366-L367
- **严重程度**：中等（产生任务包重复分配的可能）
- **改进建议**：
  - 4.2 中明确"3.4.7 仅在 Phase 4 落地'分析推理'能力"；
  - 5.2 明确"3.4.7 在 Phase 5 仅扩展'模型管理面'（注册/授权/审计），不再涉及推理逻辑"；
  - 阶段间总览表中将 Phase 5 行的 3.4.7 拆为"3.4.7-管理"或备注"同 3.4.7，仅管理面"。

### 2.3 【中等】Phase 4.2 "○² 权限"符号含义不明

- **问题描述**：Phase 4.2（L230）描述"药房医生经手处方的检查/检验结果调取路径（○² 权限，入口收敛在处方详情卡片）"。"○²" 符号既非标准 RBAC 标记也非需求文档的引用规范，下游审阅者无法判断该权限的级别、归属、作用范围。
- **所在位置**：L230
- **严重程度**：中等（影响 OOD 阶段对药房医生权限模型的实现）
- **改进建议**：
  - 改为自然语言："药房医生通过处方详情卡片**内嵌入口**查看该处方所关联的检查/检验结果（权限：药房医生岗位 + 处方关联校验，不开放独立列表查询能力）"；
  - 删除 "○²" 符号，或在文档开头"阅读指引"中给出符号说明。

### 2.4 【中等】"取消挂号（>2h）"符号含混，未覆盖边界场景

- **问题描述**：Phase 2.4（L157）"取消挂号（>2h）成功"，" >2h" 既可读作"距离就诊时间大于 2 小时"（业务可取消的合法时间窗），也可读作"取消时间大于 2 小时"（动作时长，与业务无关）。同时未演示"≤2h 不可取消"的反例，路线图在准入/准出口径上缺少边界冒烟场景。
- **所在位置**：L157、Phase 2.3 的"取消规则引擎（2 小时窗 + 退费状态机）"（L142）
- **严重程度**：中等（影响 Phase 2 准出冒烟用例的完备性）
- **改进建议**：
  - 改为"取消挂号（距就诊 >2h 成功，≤2h 拒绝并提示'请到线下窗口办理'）"；
  - 准出口径（Phase 2.6）补一句"取消规则引擎的时间窗、状态机、关联缴费记录状态机冒烟通过"。

### 2.5 【中等】"AI 进阶"拓展任务缺少集中承接

- **问题描述**：需求 §4 与 §6 拓展任务包含"AI 进阶"（即在 13 项基础能力之上做能力深化/调优/模型管理）。产出路线图把"模型管理面"散落在 Phase 5 多个子任务中，但未将"AI 进阶"作为独立的拓展任务包（如 LLM 路由、Prompt 工程、模型 A/B、向量库管理、推理性能优化）。
- **所在位置**：Phase 5（缺独立任务包）
- **严重程度**：中等（与需求"AI 进阶"拓展任务逐项缺失对应）
- **改进建议**：
  - 在 Phase 5.5 增加"包 G：AI 进阶底座（LLM 凭证集中管理 + 多模型路由 + Prompt 模板版本管理 + 推理性能埋点）"；
  - 或在 Phase 6.5 增补"包 F：AI 能力持续优化（向量库管理 + 模型 A/B 框架 + 推理性能基线）"，明确 Phase 6 是上线与运维收口期，AI 进阶适合放在此处以避免与 Phase 5 主线争夺资源。

### 2.6 【中等】微服务化的演进路径在风险/依赖章节未提及

- **问题描述**：与 1.4 关联：风险章节（L387-L395）未列出"微服务化演进过程中的拆分成本/接口兼容/数据一致性"风险，缺少对"单仓多模块→多 Jar 部署"这一路径的过渡期风险预案。
- **所在位置**：L387-L395
- **严重程度**：中等
- **改进建议**：
  - 在风险表中新增"行：'单仓多模块→微服务化'拆分时跨模块调用可能产生分布式事务/链路追踪复杂度上升"；
  - 缓解策略："Phase 0 即在 common 模块统一引入 Spring Cloud 兼容依赖（OpenFeign、Sleuth、Seata 占位）；拆分时通过配置开关在'同进程调用'与'远程调用'间切换"。

## 3. 轻微问题（建议优化）

### 3.1 【轻微】Phase 0 最小化清单的边界可更克制

- **问题描述**：Phase 0.2（L34-L43）共 10 个交付类别、约 20+ 个具体交付项，相对于"最小化骨架"略显饱满（如"前端 UI 库"中具体到 Vant/自研 Mobile 壳的二选一决策，"包管理"具体到 pnpm/Maven，"代码风格"具体到 Checkstyle/EditorConfig/ESLint/Prettier 四个工具）。这些决策在 OOD 阶段做也来得及。
- **所在位置**：L34-L43
- **严重程度**：轻微（不破坏骨架验收，但增加 Phase 0 的工作量）
- **改进建议**：
  - 把"前端 UI 库"和"包管理"两类并入"前端骨架"和"后端骨架"内不再单列；

## 5. 深度与完整性评估

### 5.1 优点

- **结构高度规范**：每个阶段统一 6 要素（目标/交付物/技术决策/可演示/可分配/准入准出），便于横向对比与下游 OOD 切入；
- **可分配性较好**：每阶段列出 4~6 个可分配子任务包，并标注模块前缀（包 A~F），可指导 Sprint 划分；
- **协作约束落地充分**：Phase 0 给出 6 个并行贡献模块的接口前缀、分支策略、共享类型包等"地形"要素；
- **风险与依赖章节质量较高**：识别了 AI 不可用、13 项 AI 失控铺开、接口契约不同步等关键风险并给出缓解策略；
- **跨阶段连续性较好**：缴费记录状态机、健康档案→AI 输入契约、检查预约分支（`registration_type=EXAMINATION`）等跨阶段依赖在多个阶段持续被引用。

### 5.2 不足

- **关键拓展任务（微服务、硬件采集）完全缺失**，与需求 §4 任务分层维度直接冲突；
- **关键业务场景（药房药库调拨、医嘱开立与记账边界）的过程描述颗粒度不足**，下游 OOD 阶段需要再次厘清范围；
- **AI 能力 3.4.7 的双阶段职责**未在文档中显式约定管理面与分析面的边界；
- **3 处编号/清单自相矛盾**（核心 2/3 编号、Phase 5.1 自相矛盾、Phase 5.6 漏算 3.4.7）属于路线图文档的硬伤，必须在定稿前修正。

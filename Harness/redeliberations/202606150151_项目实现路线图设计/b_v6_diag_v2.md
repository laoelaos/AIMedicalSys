## 一、骨架最小化对照核查（v2 新增，响应质询严重问题 2）

> 依据 `requirement.md` §2 第 3 点"骨架阶段是整个流程的第一步，且应**最小化**"硬约束。本节逐节审查 Phase 0 的硬必需条目是否真正不可下沉至 Phase 1。

### 1.1 Phase 0.2 硬必需类目逐条最小化质询

Phase 0.2 表共列 7 项硬必需交付类目（"层级"列 = **硬必需**）+ 3 项软预备。本节对硬必需 7 项逐条质询：

| 硬必需类目 | 最小化质询 | 判定 | 理由 |
|-----------|------------|------|------|
| 工程结构（Monorepo + 接口契约骨架） | 能否下沉到 Phase 1？ | **不可下沉** | Monorepo 目录是所有后续工作的物理依赖，必须 Phase 0 落地 |
| 后端骨架（Spring Boot 3 多模块 + 数据契约 + 权限模型） | 能否下沉到 Phase 1？ | **不可下沉** | 多模块占位是 Phase 1+ 多人并行的物理基础 |
| 前端骨架（三套 Vue 3 + TS + Vite） | 能否下沉到 Phase 1？ | **不可下沉** | 三端骨架是 Phase 1+ 三端并行的物理基础 |
| 协作规范（README/CONTRIBUTING/AGENTS.md/分支模型/Commit 规范/PR 模板/Code Review 清单） | 能否下沉到 Phase 1？ | **部分可下沉**——见 S-2 | "README/CONTRIBUTING/AGENTS.md" 是文档级元信息，理论上"工程结构"占位后即可最小化（一句话指针）；但"分支模型/Commit 规范/PR 模板/Code Review 清单"是 §2 第 4 点"支持多人并行"硬约束的硬必需（见第二节） |
| 持续集成（CI 占位流水线） | 能否下沉到 Phase 1？ | **可下沉**——见 S-3-a | CI 占位仅是"构建+单测占位"骨架，在 Phase 1 第一个 PR 出现前不必强求（Phase 0 阶段不会产生真实业务代码） |
| 基础组件（共享包：request/Result/GlobalExceptionHandler/BaseEntity/统一日志/审计/鉴权拦截器） | 能否下沉到 Phase 1？ | **不可下沉** | 这些是 Phase 1 鉴权中心与各模块复用的基础，下沉会形成 Phase 1 阻塞 |
| 部署骨架（本地一键启动脚本 + QUICKSTART.md） | 能否下沉到 Phase 1？ | **不可下沉** | requirement §7 第 2 条明确"新加入的开发者能基于骨架文档独立开始工作"——一键启动脚本是这一硬约束的最小化兜底 |

**质询结论**：

- 7 项硬必需中，5 项**确属不可下沉**（工程结构 / 后端骨架 / 前端骨架 / 基础组件 / 部署骨架）
- 1 项**部分可下沉**（协作规范——见 S-2）
- 1 项**可下沉**（CI 占位——见 S-3-a）

Phase 0.2 硬必需条目数为 7，可下沉数为 1-2 项；"最小化"程度**基本达标**但仍有 1-2 项优化空间（详见 S-3）。

### 1.2 Phase 0 子节数量本身是否过度

Phase 0 包含 0.1~0.7 共 7 个子节。`requirement.md` §7 阶段粒度要求"建议在 5~8 个阶段之间"——这是**阶段**粒度约束，非子节粒度。但若 Phase 0 单阶段就占 7 个子节，则下游"骨架阶段"可能不再"最小化"。

| 子节 | 用途 | 必要性 |
|------|------|--------|
| 0.1 阶段目标 | 描述骨架阶段目标 | 必要 |
| 0.2 阶段交付物 | 列出最小化清单 | 必要 |
| 0.3 关键技术决策 | 列出 9 项关键技术决策 | 必要 |
| 0.4 可演示能力 | 描述"能跑起来"的可视化证据 | 必要 |
| 0.5 可并行贡献的地形 | 列出 7 个并行模块 | 必要 |
| 0.6 阶段准入/准出 | 列出 3 条准出 + 软预备达成标志 | 必要 |
| 0.7 预备性占位 | 软预备条目总览 | **可考虑下沉**——见 S-3-b |

**质询结论**：Phase 0.7 与 Phase 0.2 / 0.3 / 0.5 在"层级"列与软预备条目上存在结构性冗余（已在 v1 报告 P3-3 识别为"轻微"——本节将其升格为"骨架最小化"维度的优化项 S-3-b）。

### 1.3 软预备条目是否"预备"过度

Phase 0.7 软预备条目 4 项：device-gateway 占位 / SQL-Toy 依赖 / Spring Cloud 兼容依赖 / 演进路径声明。

**质询**：

- 软预备条目是 v5 收口"硬必需 vs 软预备"二分层级的产物，目的是"避免 v1/v2 阶段因未在 Phase 0 启动而被迫后期大改"。**这与"骨架最小化"约束的张力是什么？**
- v6 Phase 0.7 软预备的"达成标志"全部为"已固化位置 / 已引入依赖"，不要求 Phase 0 写真实代码。**这种"声明式预备"是否真的增加骨架复杂度？**

**判定**：软预备的"声明式预备"形式（仅占位 / 仅声明）不增加骨架代码量，但增加了路线图文档的 4 行表格篇幅。这与"骨架最小化"约束是文档粒度的张力，**轻微违反**而非严重违反。

## 二、多人并行硬约束三要素核查（v2 新增，响应质询严重问题 2）

> 依据 `requirement.md` §2 第 4 点"骨架阶段完成标准：完成后**该骨架必须足够支撑多人同时继续在其上并行贡献**。这是骨架阶段验收的硬约束" 与 §7 第 2 条"新加入的开发者能基于骨架文档独立开始工作（而不必从零理解整体）" 验收隐含标准。本节按三要素（接口契约冻结粒度 / 提交规范与 CI 门禁 / 新人独立产出 PR 验证路径）逐项核查。

### 2.1 要素 1：接口契约冻结粒度

**Phase 0.2 / 0.3 中"接口契约"的实际定义**：

- Phase 0.2 工程结构行：接口契约骨架 = "API 接口清单、请求/响应通用规范：`Result<T>` 统一响应、分页结构、错误码命名空间；AI 能力契约骨架：3.4.x 的输入/输出占位类型与错误码"
- Phase 0.3 前后端契约决策行：OpenAPI / TypeScript 类型生成（后端导出 → 前端 `@types` 包）

**核查结论**：

- 接口契约在 Phase 0 仅冻结了**框架级规范**（统一响应、分页、错误码命名空间、OpenAPI 工具链），**未冻结 7 个并行模块的端点清单与请求/响应 schema**。
- 7 个模块 A ~ G 的"独立接口前缀"在 Phase 0.5 提了一句"每个业务模块有独立的目录、独立的 README、**独立的接口前缀**、独立的开发分支建议"——**但具体前缀是什么、每个模块开放哪些端点、各端点的请求/响应 schema 是什么、错误码如何分配**——这些均未在 Phase 0 给出。
- 这意味着：7 位开发者分别接走 A ~ G 模块后，**实际开工时仍需在 PR 评审中协商具体接口契约**。这与 requirement §7 第 2 条"独立开始工作（而不必从零理解整体）"的隐含标准有差距。

**严重度判定**：

- 接口契约框架已冻结（统一响应、错误码命名空间、OpenAPI 工具链）——这是 Phase 0 的合理交付
- 7 个模块的具体端点 schema 未冻结——这是 Phase 0 → Phase 1 衔接的合理状态（OOD 阶段产出），但需在 Phase 0 末或 Phase 1 初补充"接口契约冻结时机表"
- 综合判定：**一般**（S-1），非"严重"——因为：
  - 接口契约冻结时机在路线图层面是 OOD 阶段产出，过度细化反而违反 requirement §5"不写 OOD 类图、不定义具体类、不写方法签名、不写接口字段表"
  - 但当前 Phase 0.2/0.5 的措辞强度（"接口契约骨架"+"独立的接口前缀"）未明示冻结时机，下游可能误读为"Phase 0 已冻结"

**改进建议（S-1）**：

- Phase 0.2 / 0.5 增列"接口契约冻结时机表"——框架级（统一响应/分页/错误码）Phase 0 冻结，模块级（端点清单 / schema）Phase 1 启动前冻结
- 避免下游误读为"Phase 0 接口契约已冻结到可分配粒度"

### 2.2 要素 2：提交规范与 CI 门禁

**Phase 0.2 协作规范行**：仓库 README、CONTRIBUTING、AGENTS.md（含构建/启动/联调指令）、分支模型、Commit 规范、PR 模板、Code Review 清单；建立代码风格统一约束并接入 CI 校验（**具体工具选型下沉到 OOD 阶段**）

**核查结论**：

- 提交规范提了"分支模型、Commit 规范、PR 模板、Code Review 清单"四个文件名/类目，但**未给出任何具体内容或示例**——分支命名格式、Commit message 模板、PR 模板字段、CR 检查项均未在 Phase 0 显式声明
- CI 门禁提了"建立代码风格统一约束并接入 CI 校验"——但**未声明 CI 必须通过的检查项清单**（构建？单测？Lint？接口契约校验？）
- 工具选型（Checkstyle/EditorConfig/ESLint/Prettier）下沉到 OOD 阶段——这是合理的，但**具体的"分支命名规则"等非工具性规范不应下沉 OOD**——分支命名是流程级规范，与工具无关

**严重度判定**：

- 提交规范的"四要素"（分支/Commit/PR/CR 清单）只有文件名占位，没有**实质性流程规范**——新人读完 Phase 0 文档**仍不知道"我应该怎样命名一个分支"**。这是 requirement §7 第 2 条"独立开始工作"的硬约束违反。
- **判定为"严重"**（S-2）——理由：流程级规范与工具选型不同，**不应下沉 OOD**；当前 Phase 0 的"提交规范"实质上是"占位文件名"，不是"可执行规范"

**改进建议（S-2）**：

- Phase 0.2 协作规范行**追加最小化流程规范**：
  - 分支命名：`feature/<TICKET-ID>-<short-desc>` / `fix/<TICKET-ID>-<short-desc>` / `hotfix/<desc>`
  - Commit message：`feat|fix|docs|refactor|test: <scope>: <subject>`（参考 Conventional Commits，路线图仅固化"约定式"格式不绑定工具）
  - PR 模板：包含"变更范围/影响模块/接口契约变更/测试覆盖/截图"等字段
  - CR 清单：包含"接口契约是否变更/是否有迁移 SQL/是否更新文档"等检查项
- 这些流程级规范**与工具选型无关**，应在 Phase 0 落地以满足 §2 多人并行硬约束

### 2.3 要素 3：新人独立产出 PR 验证路径

**Phase 0.4 可演示能力**：

- 三端工程均可一键启动到"占位首页 + 登录/注册空页 + 一个 ping 接口可达"
- 通过 `AGENTS.md` 与 `QUICKSTART.md`，新加入的开发者可在 1 小时内本地拉起全部工程
- 后端 API 文档（Knife4j）可访问，模块列表清晰
- CI 流水线跑通构建+单测占位

**核查结论**：

- Phase 0.4 描述的"1 小时内拉起工程"是**起步门槛**——满足"工程能跑起来"
- 但"独立产出第一个 PR"的验证路径**未显式给出**——新人从"工程跑起来"到"提交第一个 PR"中间需经过：创建分支 → 编写代码 → 提交 commit → 创建 PR → CI 校验通过 → CR 通过 → 合并。这些步骤在 Phase 0.4 中**无任何一句话兜底**
- requirement §7 第 2 条"独立开始工作"隐含的"独立"含两层：(a) 起步独立（已满足）；(b) 产出独立（未明示路径）

**严重度判定**：

- 当前 Phase 0.4 仅覆盖(a)起步独立，未覆盖(b)产出独立
- 这是 requirement §2/§7 的硬约束违反——但**严重度判定为"一般"**（S-1 弱化版），理由：产出独立涉及工具链细节（IDE 配置、Git hooks、pre-commit 等），路线图层面只需承诺"提交规范 + CI 门禁 + CR 清单在 Phase 0 落地"即可——**这一承诺本身已部分满足产出独立**，但**承诺内容的实质深度不足**（见 2.2 S-2）

**与 S-2 的关系**：S-1 与 S-2 实际是同一硬约束的"骨架侧"与"流程侧"两面。建议 v7 在修复 S-2 时一并强化 S-1。

### 2.4 三要素核查总评

| 三要素 | 当前 Phase 0 满足程度 | 与 requirement §2/§7 硬约束距离 |
|--------|----------------------|-------------------------------|
| 1. 接口契约冻结粒度 | 框架级已冻结，模块级未冻结时机 | 一般距离（OOD 阶段产出路线图无需提前冻结） |
| 2. 提交规范与 CI 门禁 | 仅有"文件名占位"无"实质内容" | **严重距离**（流程级规范应 Phase 0 落地） |
| 3. 新人独立产出 PR 验证路径 | 起步独立满足，产出独立未明示路径 | 一般距离（与 S-2 强相关） |

**结论**：Phase 0 在 requirement §2/§7 硬约束上**存在 1 项严重违反（S-2 提交规范） + 2 项一般违反（S-1 接口契约冻结时机 / S-1 弱化版新人独立产出）**。

## 三、识别的实质性问题（按 requirement §2/§5/§7 硬约束严重度校准）

> 严重度校准原则（采纳质询第 4 点建议）：
> - **严重** = 直接违反 requirement.md §2/§5/§7 硬约束/边界/验收隐含标准
> - **一般** = 违反 §4 关键参考维度但有间接补救路径
> - **轻微** = 对用户验收无影响的可选优化

### S-1 [一般] Phase 0 接口契约"已冻结"措辞可能误导下游对冻结粒度的判断

- **问题描述**：Phase 0.2 工程结构行写"接口契约骨架（API 接口清单、请求/响应通用规范：Result<T> 统一响应、分页结构、错误码命名空间；AI 能力契约骨架：3.4.x 的输入/输出占位类型与错误码）作为工程结构层面交付的一部分"——其中"作为工程结构层面交付的一部分"措辞与"接口契约骨架"并列出现，可能让下游误读为"接口契约已在 Phase 0 冻结到可分配粒度"。但实际上 Phase 0 仅冻结了框架级规范（统一响应格式、分页、错误码命名空间），7 个模块的端点 schema 未冻结。
- **所在位置**：`a_v6_tech_v1.md` L41（Phase 0.2 工程结构行）

- **问题描述**：用户在 `requirement.md` §5 明确边界"不写 OOD 类图、不定义具体类、不写方法签名、不写接口字段表"，并在 §2 明确要求"比 OOD（面向对象设计）文档**更上层、更笼统**"。但 v6 中出现了大量 OOD 级别命名：

### P1 一般问题

#### 问题 1：Phase 3 拓展 AI "强耦合" 论据的颗粒度不均衡
- **问题描述**：路线图将 3.4.4 智能诊断、3.4.9 开立检查/检验、3.4.10 辅助开方均归为"与核心 2/3 强耦合同步落地"，但三者耦合度并不对等。**3.4.10 → 3.4.2**（审核处方）确实是直接上下游耦合；**3.4.9 → 3.4.2/3.4.3**（开立检查/检验）实际可独立于审核/病历存在（医生可手动开单后再走审核）；**3.4.4 → 3.4.3**（诊断结论→病历生成）属于"上游数据输入"关系，可被"占位接口 + 本地规则降级"替代而不破坏诊疗闭环。将这 3 项一并提前落地于 Phase 3 实质上扩大了 Phase 3 的 AI 工作量（3 项拓展 AI + 2 项核心 AI 共 5 项），与"避免 AI 失控铺开"的约束存在张力。
- **所在位置**：Phase 3.1 阶段目标后的"关于 AI 能力'核心→拓展'分层的说明"段落（L181）；Phase 3.5 包 E（L213）；风险表"13 项 AI 同时铺开导致失控"行（L423）
- **严重程度**：一般
- **改进建议**（二选一）：
  - **方案 A（推荐）**：将 3.4.4 / 3.4.9 后移到 Phase 4（与 3.4.5 检查报告、3.4.6 检验报告、3.4.7 影像、3.4.11 执行顺序推荐同阶段落地），Phase 3 仅保留 3.4.10 辅助开方（因为与 3.4.2 审核强耦合）。3.4.4 / 3.4.9 在 Phase 3 用"占位接口 + 本地规则降级"保证诊疗闭环可演示。Phase 3 的 AI 工作量收敛为 3 项（核心 2/3 + 拓展 1），更符合分层策略。
  - **方案 B**：保留现状但将"强耦合"细分为"3.4.10 与 3.4.2 直接耦合（审核输入）"与"3.4.4 / 3.4.9 为病历生成提供输入数据（数据耦合，可降级）"两层论证，避免论据被简化为单一标签。同时在 Phase 3.4 可演示能力中明示"AI 诊断/开立检查失败时降级到本地规则"路径已通过冒烟。

#### 问题 2：退药流程的过程角色与审批机制描述不足
- **问题描述**：Phase 4.2 药房域给出了"待发药/已发药（待取）/已发药（已取）/已退药"状态机；Phase 4.3 关键技术决策给出了"已发药（已取）取药 24 小时内允许退药"的窗口规则；Phase 4.4 可演示能力有"处方提交 → 药房发药 → 患者取药 → 退药 → 联动'待退费'中间态 → 线下窗口实际退费"端到端描述。但**未明确**：①退药申请由谁发起（患者 / 医生 / 药房医生）；②是否需要医生审批（如抗生素、特殊管制药品）；③执行退药的角色与库存回补的事务边界；④退药被拒的兜底路径。这与同节中"调拨单"详尽到"发起方=药库管理员 / 审批方=药库主管 / 执行方=药库出库与药房入库"的颗粒度形成明显不对称。
- **所在位置**：Phase 4.2 药房域行（L234）；Phase 4.3 关键技术决策"药房库存扣减/回补"段（L245-L246）；Phase 4.5 包 C（L268）
- **严重程度**：一般
- **改进建议**：参照调拨单的"发起方/审批方/执行方"格式，补全退药流程的角色矩阵与状态机联动描述，例如：
  > "退药申请（发起方=患者或开单医生）→ 药房医生审核（特殊管制药品需原开方医生复核，普通药品药房医生可直接批准）→ 药房执行退药（库存回补）→ 关联缴费记录由'已支付'变更为'待退费'（中间态）→ 线下窗口实际退费；退药被拒走原渠道告知。"

#### 问题 3：健康档案"基础版"与"完整版"边界模糊
- **问题描述**：Phase 1.2 交付物为"3.1.6 健康档案基础版（增删改条目）"；Phase 1.3 关键技术决策提到"健康档案结构化存储（含过敏史/慢病/家族史/手术史/用药史子字段）"；Phase 2.3 提到"保留 `allergy_details` 可选扩展容器"。但**未明确**：①"基础版"与"完整版"的功能分界点（如：是否含跨机构历史查询、长期健康趋势合并展示、过敏史扩展字段的实际启用）；②哪些"完整版"特性需在 Phase 1 后续阶段增量交付；③健康档案的"档案查询"（患者端按时间/机构/类别查看历史档案列表）能力在哪个阶段交付。健康档案作为 AI 输入契约的关键数据源（见 3.3"健康档案 → AI 输入契约字段映射"），其完整度直接决定 AI 能力的实际可用性。
- **所在位置**：Phase 1.2（L99）、Phase 1.3（L109）、Phase 2.3（L153）
- **严重程度**：一般
- **改进建议**：
  - (a) 在 Phase 1.2 明确"基础版"范围（基础字段 CRUD + 过敏史/慢病/家族史/手术史/用药史结构化子字段录入）；
  - (b) 在 Phase 4.2 真实数据接入行或新增"健康档案增强"行，明确患者端查看历史档案列表（按时间/机构/类别）与长期健康趋势合并展示的落地阶段（建议 Phase 4 末或 Phase 6）；
  - (c) 标注"过敏史扩展容器 `allergy_details`"在 Phase 1 即开放 schema，使用场景（如：AI 输入映射）在 Phase 2 启用。

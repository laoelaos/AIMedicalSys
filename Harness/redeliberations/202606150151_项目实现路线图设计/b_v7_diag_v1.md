## 一、整体评价

v7 在协作规范可执行性、接口契约冻结时机表、任务分层映射表、OOD 命名约定等方面均有针对性修补，**核心三件套编号、AI 落地阶段定义、13 项 AI 能力归口、5~8 阶段划分、骨架最小化判据、退药流程角色矩阵、跨阶段衔接规则等关键问题与历史反馈保持闭环**。文档主干逻辑严密，已能直接支撑 OOD 阶段启动。

但**第 7 轮迭代后，文档体量从 v6 报道的 ~99KB 进一步增长到 101,545 字节（732 行），并未因多轮"去膨胀"动作而显著缩减**。内容膨胀的风险点已从"协作硬约束表 / 风险表"转移到"Phase 0 子表分层"、"阅读指引 / 任务分层映射表 / 接口契约冻结时机表 / 修订说明（v6/v7）"等多个新增长点；OOD 边界渗透通过"约定段落"做了语义封口，但 30+ 处具体 OOD 级命名（如 `TriageRuleProvider`、`ExamRecommendRuleProvider`、`PENDING_PHASE4_DISPATCH`、`exam_application`、`api/triage/consult`、`knowledge/qa`、`Result<T>`、`BaseEntity`）在正文中依然散布。**v6 P2-4 的"约定代替下沉"方案治标不治本，路线图层面与 OOD 文档的可分性下降**。

下文按严重程度排列本次审查发现的具体问题。

## 二、P1 级问题（严重 / 必须在下一轮修复）

### P1-1：v6 的 P2-4 "OOD 边界渗透"未真正解决，"约定段落"封口失败

- **问题描述**：v6 P2-4 要求将散布在正文中的 OOD 级命名（TriageRuleProvider / ExamRecommendRuleProvider / Result<T> / GlobalExceptionHandler / BaseEntity / PENDING_PHASE4_DISPATCH / exam_application / api/triage/consult / knowledge/qa / allergy_history / allergy_details / reaction_type 等约 30+ 处）下沉或抽象化。v7 的处理方案是**仅在 Phase 0.3 前增加一段"路线图层面 OOD 级别命名约定"（L65）的免责说明**，未对具体命名做下沉。该方案存在三个根本问题：
  - (1) **约定不强制**：声明"行为契约的语义优先于具体命名"等同于承认"具体命名不重要"，但下游 OOD 工程师在阅读时仍会被这些命名误导，把"占位 + Mock"理解为"接口定型"。
  - (2) **示例性命名引发固化认知**：Phase 0.3 决策表与 Phase 2.3 / Phase 4.3 / Phase 5.3 多处使用具体类名与字段名（`TriageRuleProvider` / `ExamRecommendRuleProvider` / `PENDING_PHASE4_DISPATCH` / `allergy_details` / `reaction_type`），OOD 阶段一旦直接采用，就抹消了重新命名的机会。
  - (3) **约定段落本身是"路线图层面 OOD 边界"内容**——一段解释"为什么这些命名不该被理解为 OOD"的元说明，反而把 OOD 边界的讨论内化进了路线图，违背 requirement.md §5 "不写 OOD 类图、不定义具体类、不写方法签名、不写接口字段表"的硬约束。
- **所在位置**：v6 → v7 修改说明 L728 承认"未做 30+ 处命名的逐一下沉"；具体散布位置至少包括 L42（模块命名 `pharmacy` / `drug-storage`）、L61（`Result<T>` / `GlobalExceptionHandler` / `BaseEntity`）、L77（`AIService`）、L184（`api/triage/consult`）、L193（`PENDING_PHASE4_DISPATCH` / `exam_application`）、L242（`TriageRuleProvider` / `ExamRecommendRuleProvider`）、L326（同 L242）、L386（`triage/consult` / `knowledge/qa`）、L388（`disclaimer_required`）、L241-242（`allergy_history` / `allergy_details`）、L139（`reaction_type` / `severity`）等。
- **严重程度**：严重
- **改进建议**：
  - 方案 A（推荐）：**真实下沉**——将所有具体类名替换为"行为契约描述"（如 `TriageRuleProvider` → "分诊规则降级提供器"；`PENDING_PHASE4_DISPATCH` → "Phase 4 上线前的占位状态"；`api/triage/consult` → "分诊咨询接口"；`Result<T>` → "统一响应包装"；`BaseEntity` → "实体基类"）；将具体字段名替换为语义描述（`allergy_details` → "过敏详情扩展容器"）；将具体状态枚举值替换为语义占位（`PENDING_PHASE4_DISPATCH` → "Phase 4 派单上线前的占位状态"）。这是 v6 P2-4 的本意。
  - 方案 B（兜底）：如认为方案 A 工作量过大（v7 已声明），应**明确**这一选择的代价——删除约定段落，但在文末附"v7 收口说明"声明：路线图层面保留具体 OOD 命名为"协作便利"，要求 OOD 阶段在第一次评审时重审全部命名。这一兜底虽不彻底，但避免"约定段落"成为新的元信息负担。
  - 顺带：**L65 整段（路线图层面 OOD 级别命名约定）应删除**——该段落违反路线图边界且无约束力。

### P1-3：核心任务六的 Phase 归口有歧义，任务分层映射表与正文存在张力

- **问题描述**：v7 新增"任务分层 ↔ Phase 映射表"（L511-L535）将核心任务六（联调）归口为 **Phase 3 + Phase 6**（L524："Phase 3（医生端单端诊疗闭环冒烟）+ Phase 6（全链路联调收口）"）。但同时：
  - Phase 3.6 准出（L270）声明"任务六（联调）的核心诊疗场景冒烟通过（区别于 Phase 6 全链路联调……本阶段仅验证医生端单端诊疗闭环的可走通性……后者属 Phase 6 收口目标）"；
  - Phase 6.6 准出（L491）声明"六大业务域、三类角色、13 项 AI 能力……全部可演示且通过冒烟"；
  - 历史 iteration_history L122-L124（v3 P2-9）已专门修订过此问题。
  - **当前 v7 把"任务六"拆分为"任务六（单端诊疗闭环）"与"任务六（全链路联调）"两个变体**，但**两个变体都被映射表简称为"核心任务六（联调）"**——一个任务被映射到两个阶段，但文档从未定义"任务六"是单一可交付物还是可分阶段累积的可交付物。如果视作单一物，Phase 3 准出不应称"任务六（联调）冒烟通过"；如果视作可累积物，则 L524 表格列应写"任务六分两个增量"。v7 取了第三种含糊方案：**用一个阶段列写两个阶段、未显式拆分子任务**。
- **所在位置**：L524（任务分层映射表核心任务六行）、L270（Phase 3.6 准出）、L491（Phase 6.6 准出）、L504（阶段间总览对照表 Phase 3 行）。
- **严重程度**：严重
- **改进建议**：
  - 方案 A：**显式拆分**——将"核心任务六"改为"核心任务六-a（单端诊疗闭环联调）"和"核心任务六-b（全链路联调）"两个子任务；映射表与各阶段准出同步更新。
  - 方案 B：**显式说明累积关系**——保持单一"核心任务六（联调）"但显式声明"联调的可演示范围在 Phase 3 是医生端单端、在 Phase 6 是全链路，视为同一任务的可演示范围扩展"；映射表注明"Phase 3 = 子集冒烟 / Phase 6 = 全部可演示"。
  - 当前 v7 既未拆分也未说明，是历史 v3 P2-9 的反向回潮。

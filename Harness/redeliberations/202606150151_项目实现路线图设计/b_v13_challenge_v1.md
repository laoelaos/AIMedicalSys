## 质询要点

### 严重问题 1：核心严重问题清单严重偏离任务描述焦点

- **问题**：审查报告 10 个具体问题中有 8 个属于 quality-challenger.md 明确禁止的"文档校对、统计行数、数行号、文档版本号等与任务描述关系不大的细节"，4 个"严重"问题中 3 个均围绕字节数 / 行数统计。
- **原因**：
  - quality-challenger.md 顶部明确写："关注任务，不要陷入不必要的细节。审查报告应该关注任务描述，不要关注文档校对、统计行数、数行号、文档版本号等与任务描述关系不大的细节。"
  - requirement.md 第 12 节明确指示"组件 B 的质量审查应重点检查：阶段划分是否合理、骨架是否最小化、骨架完成后是否真能支持多人并行贡献、是否遗漏了需求中的关键功能/AI 能力"——4 项核心审查点中，仅"骨架最小化"被部分回应（且以行数为判据），其余 3 项均仅在通过项中草草带过。
  - 当前审查报告若被组件 A 接收，可能误导其下一轮（v14）大量投入"删除附录 A.1-A.3 / 元信息下沉 / 字节数核测脚本化"等非任务核心改进，反而牺牲实质性内容。
- **建议方向**：
  - 重写"具体问题清单"，剔除字节数 / 行数 / 版本号 / 排版细节类问题，保留与"实现路径 / 阶段划分 / 协作地形 / 需求覆盖"直接相关的问题。
  - 对"骨架最小化"改以质性指标审查："Phase 0 段是否存在与'支撑多人并行贡献'目标无关的冗余内容？若有，请逐条列举其与协作目标的非关联性。"

### 严重问题 2：把历史轮次内部派生目标包装为"用户硬约束"

- **问题**：审查报告反复将"v8 P1-2 设定 Phase 0 ≤ 60-70 行""v11 子 agent 设定 ≤ 95,000 字节"援引为"硬约束""三次重演""连续 5 轮未达成"——但 requirement.md 全文未出现行数 / 字节数指标。
- **原因**：
  - 这种包装会让组件 A 误以为存在"用户三令五申必须达成的量化指标"，从而把改进重心错放在压缩字节数上。
  - 用户真实硬约束是"骨架最小化 = 完成后能支撑多人并行贡献"（功能性指标），与"字节数 / 行数"无直接关系——骨架可以同时是"100 行且能支撑并行"或"60 行但不能支撑并行"。
  - quality-challenger.md 第 30 行（报告必要性维度）明确禁止追究"与任务描述关系不大的细节"，"内部派生量化指标"恰属此类。
- **建议方向**：
  - 删除"v11 子 agent 设定 ≤ 95,000 字节"作为评估基线的相关表述，或明确标注为"内部工艺改进项，非用户硬约束"。
  - "Phase 0 最小化"评估改以"任务必要性扫描"代替行数计数：逐条审视 Phase 0 段每一表 / 每一行，判定其是否对"支撑多人并行贡献"具直接贡献；无直接贡献的内容即为可压缩对象。

### 严重问题 3："多人并行贡献"硬约束的深度审查缺位

- **问题**：requirement.md 第 12 节明确要求"重点检查骨架完成后是否真能支持多人并行贡献"——这是 Phase 0 验收的**硬约束**。但审查报告对此仅以"Phase 0.5 模块 A-G + Phase 0.2 协作规范行存在"作为通过依据，未做实质性深度审查。
- **原因**：
  - 状态机契约 6 大（缴费 / 调拨 / 盘库 / 处方 / 退药 / 病历）被建议下沉到附录后，并行开发的 5+ 团队在 Phase 1-Phase 4 实施时如何保持状态机一致性，未被审视。
  - "模块级端点字段细节随各业务模块首次提交时冻结"在多团队并发时是否真能避免跨团队接口冲突，未被审视。
  - 7 模块（patient / doctor / exam / lab / pharmacy / admin / ai-ability）的接口依赖图是否会在 Phase 2 / Phase 3 形成关键路径瓶颈，未被审视。
- **建议方向**：
  - 增设"骨架对多人并行贡献的支撑度"专项审查段，覆盖：①模块边界是否清晰且无重叠 / 无遗漏；②接口契约冻结时点是否能避免跨团队等待；③状态机契约位置（Phase 0 主体 vs 附录）对并发开发的影响；④AI 能力降级路径是否能让前端团队在 AI 团队未就绪时独立推进。

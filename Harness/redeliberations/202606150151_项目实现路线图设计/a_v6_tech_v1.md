## Phase 0 — 最小化骨架（项目初始化阶段）

### 0.1 阶段目标

搭出**让多人可并行继续贡献**的最小工程骨架。骨架阶段不实现任何业务功能，只交付协作所需的"地形"。

> 关于"硬必需 / 软预备"二分层级（v5 收口）：Phase 0 的最小化交付需区分"硬必需（直接支撑 Phase 1 起步）"与"软预备（仅固化位置/依赖，可在后续阶段首期补齐）"两类条目。硬必需缺一不可，软预备如未在 Phase 0 启动，可在对应后续阶段首期补齐。本节各表新增"层级"列；Phase 0.7 集中陈列软预备条目并说明触发补齐时机。

### 0.3 关键技术决策

| 决策项 | 决策 | 层级 | 决策理由 |
|--------|------|------|----------|
| 工程组织 | Monorepo（单仓多模块） | **硬必需** | 三端 + 多后端模块 + 共享类型，单仓可原子化重构与跨端接口同步 |
| 工程组织演进路径 | 单仓多模块（Maven 聚合）→ Phase 5/6 视负载热点服务按 `services/` 切分为独立 Jar 部署形态（服务注册发现 Nacos/Eureka + Spring Cloud Gateway + 配置中心 Nacos Config） | **软预备** | 在骨架阶段固化"同进程调用"接口风格，演进时通过配置开关在"同进程"与"远程调用"间切换，避免后期重构；Phase 0 仅声明路径，真实拆分在 Phase 5/6 启动 |
| 后端模块切分 | 按"业务域"垂直切模块，模块间通过内部 API 调用而非共享 DB | **硬必需** | 保证 Phase 1+ 多人可在不同模块内并行开发不相互阻塞 |
| 后端模块命名约定 | `pharmacy` 模块当前以"药房 + 药库子包"形态承载；后续视 OOD 阶段规模评估（药库域实体数量、子模块数）决定是否拆为 `pharmacy` 与 `drug-storage` 两个独立 Maven 模块 | **硬必需**（命名约定本身） | 在 Phase 0 即固化药库与药房在工程结构上的归属位置，避免 Phase 4 业务落地时出现跨模块越界 |
| 前后端契约 | OpenAPI / TypeScript 类型生成（后端导出 → 前端 `@types` 包） | **硬必需** | 解决"前后端改一处要改多处"的协作痛点 |
| 数据库选型 | MySQL（与原始文档一致），编码 utf8mb4 | **硬必需** | 实践路径最成熟、文档支持最全 |
| 持久化方案 | Spring Data JPA（CRUD 主路径）+ SQL-Toy（复杂报表/统计查询） | **硬必需（JPA）/ 软预备（SQL-Toy 仅引入依赖）** | JPA 用于日常 CRUD 主路径，SQL-Toy 用于复杂报表聚合（Phase 4 检验趋势图、Phase 5 排班统计、Phase 6 性能指标），与原始技术栈文档 §2 一致；骨架阶段 JPA 即用、SQL-Toy 仅引入依赖，复杂查询在后续阶段按需启用 |
| 鉴权落地 | Spring Security + JWT，骨架阶段只走通"登录 → 拿 Token → 受保护接口" 三个端点的空实现 | **硬必需** | 后续业务模块只需复用鉴权拦截器即可 |
| AI 接入 | 后端独立 `ai-ability` 模块，对外暴露 `AIService` 内部接口；骨架阶段以"可注入的 Mock 实现"占位 | **硬必需** | 业务模块可在无 LLM 凭证情况下基于 Mock 实现并行开发 |
| 硬件采集预留 | `services/device-gateway` 目录 + README 占位 | **软预备** | 提前固化"硬件采集"作为独立业务模块的位置，避免后期跨模块越界；真实代码在 Phase 4 首期补齐 |

## Phase 1 — 基础设施 + 统一认证与权限（地基阶段）

### 1.1 阶段目标

打通"账号 / 角色 / 岗位 / 菜单 / 接口"五件套，让三大终端能基于统一的身份进入平台骨架。

### 1.2 阶段交付物

| 类别 | 交付项 |
|------|--------|
| 数据基础 | 用户、角色、岗位、用户-角色-岗位关联、菜单、按钮权限、字典等基础表与初始 SQL |
| 鉴权中心 | 登录（密码 + JWT）、刷新 Token、登出、当前用户信息接口；统一鉴权拦截器；CORS 配置 |
| 患者端基础 | 3.1.1 注册/登录、3.1.2 个人中心（仅查看）、**3.1.6 健康档案基础版**（增删改条目） |
| 医生端基础 | 医生账号登录、岗位标识加载、个人资料维护（3.2.7） |
| 管理员端基础 | 管理员登录、用户/角色/岗位管理、菜单管理、字典管理（3.3.4 基础数据） |
| 权限模型 | 角色-岗位-功能三级权限模型按 2.6 矩阵落地；前后端按权限动态渲染菜单与按钮 |
| 公共能力 | 统一响应 `Result<T>`、全局异常处理、参数校验（Validation）、审计日志、操作日志 |

> **健康档案基础版边界声明**：
> - **基础版范围** = 基础字段（姓名/性别/年龄/联系方式/紧急联系人）CRUD + 5 类结构化子字段录入（过敏史 / 慢病 / 家族史 / 手术史 / 用药史，每类子字段支持新增/编辑/删除条目）。
> - **`allergy_details` 子字段的录入能力**：**前端在 Phase 1 即提供录入 UI**（含 `reaction_type` / `severity` 等细粒度字段），即使后端 Phase 1 仅存字符串 `allergy_history` 兼容；Phase 2 起后端将已录入的 `reaction_type` / `severity` 拼接进 `allergy_details` 容器供 AI 消费——即"前端先提供录入能力、后端逐步消费"，避免 Phase 2 AI 接入时回退修改前端 UI。
> - **基础版明确不含**（避免与完整版混淆）：
>   - ① "按时间/机构/类别查看历史档案列表"——属 Phase 4 增强项（4.2"健康档案增强"行）
>   - ② "档案历史版本对比"——属 Phase 4 增强项
>   - ③ "长期健康趋势合并展示"——属 Phase 4 增强项
> - 基础版与完整版的分界点统一为：**基础版 = 当前可录入与可编辑的字段集合**；**完整版 = 在基础版之上叠加"按维度查询/对比/趋势"等阅读侧能力**。

### 1.3 关键技术决策

- 密码使用 BCrypt 强哈希存储；JWT 携带 userId/role/position 关键 claims
- 前端按角色动态加载路由表；菜单权限由后端下发
- 健康档案结构化存储（含过敏史/慢病/家族史/手术史/用药史子字段），过敏史条目支持 `allergy_details` 可选扩展容器（schema 在本阶段开放；前端录入 UI 在本阶段即提供，后端 Phase 1 仅持久化字符串 `allergy_history`，Phase 2 起消费 `reaction_type` / `severity`）
- 后端日志统一接入 Logback + MDC 追踪号；前端 Axios 拦截器注入 `X-Request-Id`

### 1.4 可演示能力

- 患者注册/登录、查看个人中心
- 医生按岗位登录看到对应菜单（5 个子岗位）
- 管理员可创建用户、分配角色与岗位
- 任何受保护接口未带 Token 都被拦截；权限不足返回 403

### 1.5 可分配子任务包

- 包 A：用户/角色/岗位/菜单/字典数据建模与初始 SQL
- 包 B：Spring Security + JWT 实现
- 包 C：患者端注册/登录/个人中心/健康档案 UI + API
- 包 D：医生端与管理端登录/菜单动态化

### 1.6 阶段准入/准出

- 准出：三大终端能完成"登录 → 看到与自己角色匹配的菜单"全链路，权限矩阵 2.6 全部通过冒烟

## Phase 2 — 患者服务域 + 核心三件套之"智能分诊"（患者端主流程）

### 2.1 阶段目标

### 2.3 关键技术决策

- 分诊调用方式：单次（主诉 + Q&A 拼接）与多轮（`session_id` + 增量 `additional_responses`）并存，由前端按 `session_id` 是否存在自动选择
- 分诊降级：AI 超时（3 次失败/单次超时）→ 前端兜底文案"建议直接联系线下接诊窗口"
- 挂号状态机：待就诊 / 已就诊 / 已取消 / 待退费（中间态字段由 Phase 2 预留占位，Phase 4 缴费记录联动写入；与 Phase 4 缴费记录状态机的'待退费'态对齐）
- 取消规则：取消操作以"距就诊时间"为判定基准；>2h 允许取消并将挂号状态置为"已取消"并预留"待退费"中间态字段（实际值由 Phase 4 缴费记录联动写入），≤2h 拒绝并提示走线下窗口
- 检查预约分支（`registration_type=EXAMINATION`）的检查申请生成时机与流转，与 Phase 4 检查管理协调。本阶段规则统一：**Phase 2 阶段生成的检查预约占位申请在写入 `exam_application` 实体时 `assignment_status=PENDING_PHASE4_DISPATCH`**——仅占位、不入派单队列；Phase 4 自动派单上线时**首次回填**已存在的占位申请（按创建时间顺序入队），后续新申请按 Phase 4 派单逻辑即时分配。此规则避免 Phase 2 阶段产生的"占位申请"在 Phase 4 派单时落入歧义
- 健康档案 → AI 输入契约字段映射：按 3.1.6 末段过渡方案，保留 `allergy_details` 可选扩展容器（Phase 1 schema 已开放，前端录入 UI 已提供，本阶段启用为 AI 输入映射的细粒度过敏数据源）

## Phase 3 — 医生端核心 + 核心三件套之"AI 处方审核 / AI 病历生成"（诊疗主战场）

### 3.1 阶段目标

跑通门诊医生"叫号 → 接诊 → AI 辅助开方 → AI 处方审核 → AI 病历生成 → 保存"的诊疗闭环；落地核心三件套中**第二件"AI 处方审核"（3.4.2，对应核心 2，**首次落地阶段 = Phase 3**）**与**第三件"AI 病历生成"（3.4.3，对应核心 3，**首次落地阶段 = Phase 3**）**。

> 关于 AI 能力"核心→拓展"分层的说明：3.4.10 辅助开方与 3.4.2 AI 处方审核构成"开方→审核"直接上下游耦合，必须同步落地。3.4.4 智能诊断、3.4.9 开立检查/检验与 3.4.2 / 3.4.3 仅为"数据输入关系"（3.4.4 的诊断结论与 3.4.9 推荐的检查项目最终都进入 3.4.3 病历生成的输入）——可被"占位接口 + 本地规则降级"替代而不破坏诊疗闭环，故后移到 Phase 4，与 3.4.5 / 3.4.6 / 3.4.7 / 3.4.11 同阶段落地。Phase 3 仅保留 3.4.10 一项拓展 AI，工作量收敛为"核心 2 项 + 拓展 1 项"，更贴合"核心→拓展"分层策略。Phase 3 期间，3.4.4 / 3.4.9 由 `ai-ability` 模块的 Mock 占位接口 + 本地规则降级保证诊疗闭环可演示，UI 显著标识"由本地规则生成"，AI 不可用时降级路径同样冒烟通过。

### 3.3 关键技术决策

- 病历生成：流式输出（SSE）放在 Phase 6 拓展；本阶段先做一次性 Structured Output 同步返回
- 处方审核：审核结果独立持久化（与处方解耦），便于事后回溯
- AI 审核 6 秒硬超时 + 降级标识：前端须在审核弹窗显著位置标识
- 健康档案 → AI 输入契约字段映射：实现统一提取器（`allergy_history` 字符串 + 可选 `allergy_details` 容器（Phase 1 schema 已开放，前端录入 UI 已提供，本阶段启用）、`comorbidities` 数组），由 `ai-ability` 模块复用
- 3.4.4 / 3.4.9 占位接口：与 3.4.2 / 3.4.10 共用 `ai-ability` 模块的 Mock 实现入口；本地规则降级表配置在 `common` 模块，由 `TriageRuleProvider` / `ExamRecommendRuleProvider` 暴露，Phase 4 升级为真实 AI 推理时仅替换实现类
- 检查/检验结果：本阶段数据先由后端占位 API 模拟，Phase 4 接入真实检查/检验流程
- 病历/处方数据对患者端 3.1.5 的可见性：本阶段病历生成 / 处方审核落库后，患者端 3.1.5"报告/病历/处方查询"即可显示对应**草稿版本**（病历草稿、AI 审核后的处方记录），完整报告（检查/检验）与趋势图在 Phase 4 接入
- **3.4.10 降级路径（v5 收口）**：3.4.10 AI 辅助开方的降级路径复用 3.4.2 处方审核的降级链路——AI 不可用时回退到本地剂量规则表（`common` 模块统一维护），UI 显著标识"由本地剂量规则生成，未经过 AI 审核"，原始处方提交不受阻断；降级路径冒烟通过作为本阶段准出条件之一

## Phase 4 — 检查 / 检验 / 药房 / 药库 / 线下窗口 / 硬件接入（业务域拓展）

### 4.1 阶段目标

把医生端的 5 个子岗位全部落地：检查医生、检验医生、药房医生、线下接诊医生；同时打通"检查预约 → 检查执行 → 报告 → 反馈门诊"与"处方 → 发药 → 退药 → 联动缴费"两条主流程；落地药房药库库存调拨的完整业务闭环；落地硬件接入适配层骨架（接口契约 + Mock 数据）；承接 Phase 3 占位的 3.4.4 / 3.4.9 真实 AI 推理；承接患者端"健康档案增强"功能（档案查询 + 长期健康趋势合并展示）；承接患者端 3.1.8"药房取药"入口与 3.1.7"缴费记录查询"入口的真实数据接入。

> **关键路径与里程碑（v5 收口，回应 P2-1 工作量集中风险）**：本阶段承接 4 个业务域 + 硬件接入 + 真实数据接入 + 健康档案增强 + 4 项 AI 真实落地，工作量高于其他阶段。9 个任务包建议按以下依赖顺序推进——

### 4.3 关键技术决策

- 检查预约任务分配：系统自动按最短队列或轮转规则分配至检查医生，无需手动认领。**Phase 2 历史占位申请衔接**：Phase 2 生成的 EXAMINATION 占位申请（`assignment_status=PENDING_PHASE4_DISPATCH`）在 Phase 4 派单上线时按**创建时间顺序首次回填入队**；后续新申请按 Phase 4 派单逻辑即时分配
- 趋势图聚合：按 `item_id`（内部编码）匹配；同报告期取最近一次；单位不一致时按 SI 标准单位换算（架构设计阶段固化换算规则），无法换算的不参与绘制并标注；**聚合查询使用 SQL-Toy（与 Phase 0.3 持久化双栈一致）**
- 药房库存扣减/回补：发药扣减药房库存、退药回补药房库存；盘库差额回退药库
- **退药角色矩阵与状态机联动**（颗粒度与 4.2 调拨单对齐；**v5 收口：业务规则边界保留在路线图，事务边界与时间窗下沉 OOD**）：
  - **发起方** = 患者（通过患者端发起）或开单医生（通过医生端处方详情发起）——**保留在路线图**（业务规则边界）
  - **审批方** = 药房医生，按药品分类规则决定是否需双签（具体规则下沉产品配置）——**v6 简化**：移除具体药品分类（管制/抗生素）表述，颗粒度向 OOD 收敛
  - **执行方** = 药房医生：执行药房库存回补 + 缴费记录联动变更——**保留在路线图**（业务规则边界）
  - **退药被拒兜底** = 走原渠道（患者端 / 医生端）告知，退药申请状态变更为"已拒绝"，库存与缴费记录均不变更——**保留在路线图**（业务规则边界）
  - **事务边界**（库存回补与缴费记录状态变更的原子性保证）——**下沉到 OOD 阶段**，路线图仅声明"退药库存回补与缴费记录状态变更需保证事务原子性，任一失败触发回滚"
  - **退药时间窗**（已发药后允许退药的时间范围）——**下沉到产品配置层**，路线图仅声明"退药有可配置时间窗（管理员可配置）"
- 缴费记录状态机落地（详见 4.2 通用行）：原缴费记录变更为"待退费"中间态而非新建记录（避免双记录歧义）
- **调拨单状态机**：草稿 → 审批中 → 在途 → 已接收 / 已拒绝 / 已撤销；调拨过账采用"药库出库 + 药房入库"双写对账（出库与入库任一失败触发事务回滚 + 差异告警）；调拨差异按差异类型分级告警（警告/阻塞；具体字段下沉 OOD 阶段）
- **盘库单状态机**：草稿 → 盘点中 → 已复盘 → 已归档；盘盈自动入库，盘亏自动出库，误差超阈值触发管理员复核
- 药房医生查看检查/检验结果：药房医生通过处方详情卡片内嵌入口查看该处方所关联的检查/检验结果（权限：药房医生岗位 + 处方关联校验，不开放独立列表查询能力）
- 硬件接入：`device-gateway` 模块独立于 `exam` / `lab` 模块；通过统一的"设备事件 → 业务事件"映射层将原始设备数据（HL7 消息、DICOM 文件、串口数据、网关推送）转为 `exam` / `lab` 模块可直接消费的业务事件；Phase 4 仅落接口契约与 Mock 数据，真实设备对接在后续阶段
- 3.4.4 / 3.4.9 真实 AI 推理承接：复用 Phase 3 暴露的 `TriageRuleProvider` / `ExamRecommendRuleProvider` 接口；本阶段新增真实 AI 实现类并通过配置开关切换；本地规则降级实现保留为兜底（真实 AI 不可用时回退）
- 3.4.5 / 3.4.6 AI 失败回退：调用失败 / 超时时回退到医生手动判读模式（具体 UI 规范下沉 OOD 阶段）；**v6 简化**：移除"医生可手工录入…与初步结论"与"AI 标签栏显著标识"等 UI/UX 细节
- **3.4.7 / 3.4.11 降级路径（v5 收口）**：3.4.7 影像分析推理与 3.4.11 执行顺序推荐在 AI 调用失败 / 超时时，回退到"医生手动操作"路径——3.4.7 退回到检查医生手工录入影像所见与初步结论、3.4.11 退回到检查医生按经验/默认规则手工排序；UI 显著标识"由医生手动判读 / 排序，AI 不可用"，原始申请单数据仍可正常流转，不阻断业务流程

## 整体评价与质量评级

### 综合判断

v4 相比 v3 在跨阶段依赖与时序一致性方面取得了**实质性进展**：3.1.3.2 病情咨询的 Phase 2↔Phase 5 跨阶段拆分、EXAMINATION 占位申请的 `PENDING_PHASE4_DISPATCH` 衔接规则、退药角色矩阵的"发起方/审批方/执行方/事务边界/退费收口/被拒兜底"颗粒度承接，均显著降低了下游 OOD 阶段的二义性风险。13 项 AI 能力、六大业务域、三大终端、5 子岗位的覆盖在阶段间总览表中均有显式承接。

但本报告通过反向覆盖率矩阵的独立核查，识别出 **1 项严重事实错误**、**1 项关键功能隐性遗漏**、**1 项并行可分配性硬约束未满足**——三者均影响下游使用，必须在 v5 收口前修订。其余问题为一般或轻微。

### 质量评级

**有条件通过**——v4 在任务核心四维度的整体表现达到"可作为路线图正式基线"的下限要求，但下述 **3 项 P1 问题必须在 v5 收口前修订**，否则下游 OOD/实施工程师将基于错误信息开展工作：
1. P1-A：Phase 5.6 准出 "共 7 项在本阶段落地" 与各阶段实际落地阶段相互矛盾（事实错误）
2. P1-B：3.1.7 缴费记录查询的"占位 → 真实数据接入"时点遗漏（关键功能未承接）
3. P1-C：Phase 3.5 任务包 D 将 AI 处方审核（核心 2）与 AI 病历生成（核心 3）打包于同一子包，违反"可分配性"硬约束

## 一、阶段划分合理性审查（任务核心维度①）

### 1.1 7 阶段划分的合理性

需求 §7 要求"建议在 5~8 个阶段之间"。v4 采用 7 阶段（Phase 0~Phase 6），处于建议区间中段。从需求 §1.4 六大业务域 + §2 三终端 + §3 功能层级倒推，7 阶段切分点的合理性如下：

| 阶段 | 业务覆盖 | 切分合理性论证 |
|------|----------|----------------|
| Phase 0 骨架 | 全部（占位） | **必要且唯一**：用户硬约束 §2 要求最小化骨架单独切出 |
| Phase 1 鉴权地基 | 全部（地基） | **合理**：地基横跨三端、纵贯全部业务域；与业务域 Phase 解耦，避免 Phase 2+ 重复造轮子 |
| Phase 2 患者主流程 + 核心 1 | 患者服务 | **合理**：核心三件套之核心 1（智能分诊）本就是患者域入口，业务与 AI 同步收敛 |
| Phase 3 医生端诊疗 + 核心 2/3 | 门诊诊疗 | **合理**：核心 2/3（审核/病历）的演示场景就是门诊诊疗，业务与 AI 自然耦合 |
| Phase 4 业务域拓展 | 检查/检验/药房/药库 + 硬件 | **存在隐忧**（见下条）：本阶段承接 4 个业务域 + 硬件接入 + 真实数据接入 + 健康档案增强 + 3.4.4/3.4.9 真实 AI + 3.1.8 真实接入 + 3.4.5/3.4.6 降级，9 个并行任务包，**工作量明显高于其他阶段** |
| Phase 5 管理员端 + 拓展 AI | 医院管理 + AI | **合理**：管理员端的主要价值就是配置/管理"前面阶段已落地的能力"，含 AI 进阶底座五组件 |
| Phase 6 收口 | 全部 | **合理**：联调/部署/性能/可观测性属"横切性能力"，本就应单独阶段收口 |

### 1.2 【P2-1】Phase 4 工作量集中度过高的潜在风险

- **问题描述**：Phase 4 是 7 个阶段中工作量最重的阶段，承接交付物包括：检查域（含 4 项 AI）+ 检验域（含 1 项 AI + 趋势图）+ 药房域（含退药角色矩阵 + 状态机）+ 药库域（含调拨 + 盘库 2 套独立状态机）+ 线下窗口 + 缴费记录状态机 + 健康档案增强 + 真实数据接入（5 类）+ 3.1.8 药房取药真实接入 + 硬件接入（4 类协议）+ 3.4.4/3.4.9 真实 AI 推理 + 3.4.5/3.4.6 降级路径。其 4.5 节列出 9 个任务包（A/B/C/D/D1/D2/E/F/G），是全部阶段中并行任务包数量最多的。
- **原文证据**："准出：六大业务域中'门诊诊疗 / 检查检验 / 药房药库 / 患者服务'四条主流程全部可演示；调拨单六态、盘库单四态、缴费记录五态、退药流程（发起/审批/执行/被拒兜底）全部可被冒烟……3.4.5 / 3.4.6 降级路径（医生手动判读）冒烟通过"（Phase 4.6 准出）
- **影响**：在 Phase 4 任一子任务包延期都会拖累后续阶段；新加入开发者面对 9 个并行任务包难以判断关键路径
- **严重程度**：一般（P2）——属阶段划分粒度问题，但 v4 已通过子任务包 A~G 的拆分形式做了内部缓解；如要进一步降低风险，可考虑将 Phase 4 拆为"4a 检查检验 + 患者真实数据"与"4b 药房药库 + 硬件接入"两个子阶段（总阶段数将达 8 个，仍在 5~8 区间内上限）
- **改进建议**：（任选其一）
  1. 在 Phase 4 阶段目标后追加"关键路径与里程碑"小段，显式标注 9 个任务包的依赖顺序与并发可能性（哪几个可并发、哪几个有上下游依赖）
  2. 或在风险表追加"Phase 4 工作量集中导致拖延"行 + 缓解策略
  3. 或考虑 v5 阶段拆分（仅在 v4 通过验收后再行规划）

### 1.3 阶段间过渡是否清晰

v4 在每个 Phase 的"X.6 阶段准入/准出"小节均给出准出条件；阶段间总览对照表（L489-L501）以 7 列结构呈现"阶段—关键交付—业务域—角色/终端—AI 能力—演示主线"对应关系。**未发现阶段顺序错位或过渡断裂问题**。

## 二、骨架最小化与"支持多人并行贡献"硬约束审查（任务核心维度②③）

### 2.1 Phase 0 是否真正"最小化"

需求 §2 明确"骨架阶段应**最小化**"，且 §7 验收隐含标准"骨架阶段交付后，新加入的开发者能基于骨架文档独立开始工作"。"最小化"的判据应当是"足够支撑并行贡献的最小集合"，而非"少即是好"。

#### 2.1.1 Phase 0 交付物按"硬必需 vs 软预备"切分

通过对 Phase 0.2（7 类别交付物）与 Phase 0.3（10 行关键技术决策）的逐项审视，得到下述切分：

| 类别 | Phase 0 当前条目 | 硬必需（支撑并行起步） | 软预备（为后续阶段铺路） |
|------|------------------|------------------------|--------------------------|
| 工程结构 | Monorepo + 接口契约骨架 | ✓ 全部硬必需 | — |
| 后端骨架 | 10 模块占位 + 数据契约 + 权限模型 | ✓ 模块占位与权限模型硬必需 | `device-gateway` 占位仅为硬件采集铺路（拓展任务，Phase 4 才用） |
| 前端骨架 | 三套 Vue 3 工程 | ✓ 全部硬必需 | — |
| 协作规范 | AGENTS.md / CONTRIBUTING / PR 模板 | ✓ 全部硬必需（这正是并行的前提） | — |
| 持续集成 | CI 占位流水线 | ✓ 全部硬必需 | — |
| 基础组件 | Result / 全局异常 / Axios 拦截器 | ✓ 全部硬必需 | — |
| 部署骨架 | dev-up 脚本 + QUICKSTART.md | ✓ 全部硬必需 | — |
| 技术决策 | 工程组织 / 模块切分 / 命名约定 | ✓ 全部硬必需 | — |
| 技术决策 | 工程组织演进路径（→ 微服务） | — | ✓ 软预备：Phase 5/6 才需 |
| 技术决策 | 持久化方案（JPA + SQL-Toy） | ✓ JPA 硬必需 | ✓ SQL-Toy 软预备：Phase 4 才需 |

| 技术决策 | AI 接入（ai-ability + Mock） | ✓ 硬必需 | — |
| 技术决策 | 硬件采集预留 | — | ✓ 软预备：Phase 4 才需 |
| 技术决策 | 数据库选型 MySQL/utf8mb4 | — | △ 已在 02_tech.md 中规定（路线图不必重复） |

### 2.2 【P2-2】骨架最小化的"硬必需 vs 软预备"未做显式分层

- **问题描述**：Phase 0.2 / 0.3 / 0.5 中"硬必需"与"软预备"条目混排，未做显式分层。新加入的并行贡献者无法第一眼判断"哪些必须在 Phase 0 全部到位、哪些可以延后到首期补齐"。例如 `device-gateway` 目录占位、SQL-Toy 依赖引入、Spring Cloud 兼容依赖占位（OpenFeign/Sleuth/Seata），三者均非"支撑 Phase 1 起步"的硬必需项，但与硬必需项平级列出，会让骨架阶段的实际工作量被高估。
- **原文证据**：
  - Phase 0.2 "后端骨架"行写："`common` / `gateway` / `auth` / `patient` / `doctor` / `exam` / `lab` / `pharmacy` / `admin` / `ai-ability` 等模块占位"——`device-gateway` 在同一行作为"硬件采集预留目录"加注，与其他业务模块占位平级
  - Phase 0.3 "硬件采集预留"决策行写："`services/device-gateway` 目录 + README 占位"——独立成行
  - Phase 0.5 任务分配 7 个模块平级（A~G），"模块 G：搭 `device-gateway` 占位（硬件采集模块骨架）"与其他 6 个支撑并行起步的核心模块同级列出
- **严重程度**：一般（P2）——不影响实施正确性，但影响"骨架最小化"的可读性，且可能造成新人对 Phase 0 工作量的误判
- **改进建议**：
  1. Phase 0.2 / 0.3 / 0.5 各自在表格末段或正文中显式标注"硬必需 / 软预备"二分（可在表格新增"层级"列）
  2. 软预备条目独立成"Phase 0 预备性占位"小段（例如 0.7 节），强调"如未在 Phase 0 启动，可在对应后续阶段首期补齐"
  3. 软预备条目至少应明确包括：`device-gateway` 目录占位、SQL-Toy 依赖、Spring Cloud 兼容依赖占位

### 2.3 多人并行贡献的接口/数据/权限契约是否真已就绪

需求 §7 验收隐含标准的核心是"骨架完成后**该骨架必须足够支撑多人同时继续在其上并行贡献**"。这一硬约束需要逐项核查：

| 并行起步硬必需项 | v4 是否提供 | 充分性评估 |
|--------------------|-------------|-----------|
| 接口契约骨架 | ✓ Phase 0.2 工程结构行声明"API 接口清单、请求/响应通用规范：`Result<T>`、分页结构、错误码命名空间；AI 能力契约骨架：3.4.x 输入/输出占位类型与错误码" | **充分**：含三端共用规范 + AI 能力契约骨架 |
| 数据契约骨架 | ✓ Phase 0.2 后端骨架行声明"数据库命名规范、ER 图文档骨架、主键/时间戳/审计字段约定、Flyway/Liquibase 初始空迁移脚本" | **充分**：含命名约定 + 初始迁移脚本 |
| 权限契约骨架 | ✓ Phase 0.2 后端骨架行声明"角色—岗位—功能三级矩阵、菜单与前端路由占位" | **充分**：含角色-岗位-功能三级矩阵骨架 |
| CI 占位 | ✓ Phase 0.2 持续集成行 | **充分**：含构建+单测占位+lint+接口契约校验 |
| 共享类型包 | ✓ Phase 0.2 工程结构行"`packages/shared-types`" + Phase 0.5 任务分配段"共享类型在 `packages/shared-types` 中维护；新增字段在此一处定义" | **充分** |
| AGENTS.md | ✓ Phase 0.2 协作规范行 | **充分** |
| QUICKSTART.md | ✓ Phase 0.2 部署骨架行 | **充分** |
| 模块独立 README | ✓ Phase 0.5 段"每个业务模块有独立的目录、独立的 README、独立的接口前缀" | **充分** |

**结论**：v4 在"支持多人并行贡献"的 8 项硬必需基础设施上**均已就绪**，骨架质量满足需求 §7 验收隐含标准。

### 2.4 【P1-C】Phase 3.5 任务包 D 违反"可分配性"硬约束

- **问题描述**：Phase 3.5 任务包 D 描述为 **"包 D：AI 病历生成（3.4.3，核心 3）+ AI 处方审核（3.4.2，核心 2）（含风险等级与降级）"**——即把核心三件套之核心 2 与核心 3 两项独立的 AI 能力打包到**同一子任务包**。这违反需求 §6 "每个阶段应能拆解为可分配给不同开发者的子任务包"的硬约束。
  - 若将本包分配给单一开发者：工作量过大（核心 2 含三档风险等级 + 6 秒硬超时 + 降级标识 + 审核记录持久化；核心 3 含 Structured Output + 按科室 Prompt 模板 + 关键字段缺失提示）
  - 若让两个开发者并行：包内边界不清，难以避免 PR 冲突
  - 对比 Phase 2.5 包 C（仅 3.4.1）/ Phase 3.5 包 E（仅 3.4.10）的颗粒度，包 D 异常臃肿
- **原文证据**：
  - Phase 3.5 包 D 原文（L288）：`包 D：AI 病历生成（3.4.3，核心 3）+ AI 处方审核（3.4.2，核心 2）（含风险等级与降级）`
  - 对比包 C（L287）：`包 C：处方中心 + 病历中心后端` — 后端先行
  - 对比包 E（L289）：`包 E：AI 辅助开方（3.4.10，与核心 3.4.2 强耦合同步落地的拓展 AI）` — 单一 AI 能力即一个独立包
- **严重程度**：**严重（P1）**——直接违反需求 §6 "可分配性" 硬约束，导致 Phase 3 核心三件套的并行实施受阻
- **改进建议**：
  1. Phase 3.5 包 D 拆分为：
     - **包 D1：AI 处方审核（3.4.2，核心 2）**——含三档风险等级 + 6 秒硬超时 + 降级标识 + 审核记录持久化
     - **包 D2：AI 病历生成（3.4.3，核心 3）**——含 Structured Output + 按科室 Prompt 模板 + 关键字段缺失回提示
  2. Phase 3.5 子任务包重新编号为 A/B/C/D1/D2/E/F/G（共 8 个，与 Phase 4.5 的 D1/D2 命名风格一致）
  3. 协作硬约束表（L514）"核心任务四 = Phase 3 / 核心任务五 = Phase 3" 行的颗粒度自然与"包 D1/包 D2"一一映射

## 逐维度审查

### 1. 证据充分性

**[问题-严重]** 多处关键判断仅依靠"行号 + 简略概括"展示，未附原文片段供复核者验证。报告在 P1-2 中声明"v4 在 Phase 4.3 中给出了退药角色矩阵 7 要素、调拨单状态机 6 态、盘库单状态机 4 态、缴费记录状态机 5 态"——这四项核心断言关系到"是否真的逼近 OOD 边界"的判定依据，但读者无法在不打开 a_v4_tech_v1.md 的情况下确认这 4 套状态枚举的字面内容。P1-1 同样如此：报告称"meta 信息约 85 行 / 14.5%"，但 85 行的具体拆分（L28-84 的 57 行 + L527-553 的 8 行 + L566-583 的 18 行）是否准确相加得到 83 行而非 85 行，存在计算与口径上的不确定性。

**[问题-一般]** P2-2 中"性能埋点 vs 性能基线是否同一组件的两个演进阶段"属于推测性表述，缺乏原文支撑。报告在 §三 未发现的问题中宣称"v4 自检的 12 项反馈全部回应"——但报告未附原文证据链，仅以"自检报告"为引用源。

### 2. 逻辑完整性

**[问题-严重]** 报告存在**任务焦点偏离**的系统性问题。需求文档 §7 明确审查重点为：(a) 阶段划分是否合理；(b) 骨架是否最小化；(c) 骨架完成后是否真能支持多人并行贡献；(d) 是否遗漏了需求中的关键功能/AI 能力。但本报告在四个 P1 问题中，仅 P1-3 涉及(b)，P1-1 / P1-2 / P2-1 主要落在"文档形态 vs OOD 边界"这一**次级维度**，而 (a) 阶段划分合理性、(c) 多人对并行贡献硬约束、(d) 关键功能/AI 能力是否遗漏这三大核心维度的审查深度明显不足。例如：
- 阶段划分（7 阶段）是否合理：报告仅在整体评价中用"已显著改善"带过，未对"为什么 7 阶段、而不是 6 或 8 阶段"做实质论证
- 多人并行贡献硬约束：报告未逐项检查"接口契约、数据契约、权限契约、CI 占位、共享类型包、AGENTS.md、QUICKSTART.md"七项是否真的已定义清晰且互不冲突
- 关键功能遗漏：报告 §三 仅列"已闭环"清单，但**未采用"反向检查法"**——即未从需求 §3 / §5 倒推每条功能是否都有阶段承接

**[问题-一般]** P1-1 与 P2-4 存在论证冗余：P1-1 指出"meta 信息 14.5% 体量过大"，P2-4 指出"'v4 已收口'类注释放错位置"——两者实为同一现象（meta/历史叙事污染正文）的两个切面，但报告未将它们合并或显式标注关联，导致"meta 瘦身"的整改建议在两处分别给出，可能造成 v5 收口时的重复处理。

**[问题-一般]** 报告整体语调偏"建议性"而非"判定性"。综合判断部分使用"v4 同时已经触及'路线图'形态的体量上限"、"v5 收口时需要做一次瘦身 + OOD 边界回拉"等弱化表述，对当前 v4 版本的最终质量未给出明确评级（通过 / 有条件通过 / 不通过），与"质量审查报告"应有的判定功能不匹配。

### 3. 覆盖完备性

**[问题-严重]** 报告对任务描述明确点名的审查维度**系统覆盖不足**。具体缺失如下：
- **阶段划分合理性**：报告未分析"为什么是 7 个阶段"——例如为何不将 Phase 5 的"AI 进阶底座"前移至 Phase 4，为何不将 Phase 6 的"实时通信"作为 Phase 1 的可选组件
- **多人并行贡献硬约束**：报告仅在 P1-1 的严重程度判定中提及"新加入的开发者第一眼会被历史修订细节干扰"，但**未对协作硬约束表 L511 行的 13 项 AI 能力归属做并行可行性检查**——例如 Phase 3 同时承接 3.4.2 / 3.4.3 / 3.4.10 三项 AI 能力，是否真的能让三个不同开发者并行实现（这正是需求 §6"可分配性"的硬约束）
- **关键功能反向检查**：报告未对照需求文档 §3 / §5 列举"6 大业务域 + 5 个医生子岗位 + 3 大终端 + 13 项 AI 能力"做覆盖率矩阵核验——§三 的"已闭环"清单是组件 A 自报，缺少审查者的独立倒推
- **降级路径完整性**：报告在 §三 第 9 项列出"3.4.2/3.4.5/3.4.6/3.4.7/3.4.8/3.4.4/3.4.9 降级路径已显式声明"——但**未对 13 项 AI 能力做逐项降级路径核查表**，仅以"显式声明"为结论

**[问题-一般]** P1-3 关于"Phase 0 硬必需 vs 软预备"二分法，提出的硬必需项与软预备项的具体清单（"三端空壳、接口契约骨架、数据契约骨架、权限契约骨架、CI 占位、共享类型包、AGENTS.md、QUICKSTART.md" vs "SQL-Toy 依赖引入、device-gateway 目录与 README 占位、ai-ability 模块 Mock 入口"）是审查者的**新增建议**而非 a_v4 原文分析，但报告未明确标注"以下为审查者建议"——这模糊了"审查"与"重设计"的边界。

### 4. 报告必要性

**[问题-严重]** 报告存在**过度关注"细节问题"的明显倾向**，违背质询指令"关注任务，不要陷入不必要的细节"原则。下列问题均属于指令明确点名的应避免的"细节问题"类型：

1. **P1-1 主体** —— 围绕"0.1 节文档变更记录 57 行 + 末尾修订说明 18 行 + 风险表 8 行 ≈ 85 行 / 14.5%"展开，本质是**统计行数 + 计算占比**的细节审查。指令明确"统计行数、数行号"应避免。
2. **P2-4 整体** —— "'v4 已收口 / v3 已统一' 类注释放错位置"是**文档版本号污染**的典型问题，指令明确"文档版本号等与任务描述关系不大的细节"应避免。
3. **P3-3 整体** —— "v3 → v4 修订摘要 L69-84 与修订说明(v4) L566-583 存在内容重复"是**文档校对/排版**问题，指令明确"文档校对"应避免。
4. **多处行号引用** —— 报告全文大量使用"L28-84"、"L99"、"L131-139"、"L325-332"、"L511"等精确行号引用，指令明确"数行号"应避免。即使行号引用本身必要，也应转化为内容引述或节标题引用。

**[问题-一般]** 报告篇幅 202 行中，至少 60 行（约 30%）用于"细节问题"论述（P1-1 / P2-4 / P3-3 / 行号统计），而用于"任务核心维度"（阶段划分合理性、骨架最小化实质证据、多人并行贡献约束、关键功能反向核查）的深度论述明显不足。**资源错配**导致核心审查维度的论证被挤压。

## 质询要点

### 要点 1：报告任务焦点偏离，核心审查维度深度不足

- **问题**：报告将大量审查资源投入到"meta 信息占比 / 版本号污染 / 文档重复"等次级维度，未对任务描述 §7 明确点名的四大核心审查维度（阶段划分合理性、骨架最小化、多人并行贡献硬约束、关键功能/AI 能力是否遗漏）做深度审查。
- **原因**：这导致审查结论的**信度受损**——读者无法从本报告中确认 v4 路线图在"阶段为何是 7 个"、"多人并行是否真可行"、"13 项 AI 能力是否全部承接且无遗漏"等关键问题上是否合格。即便 P1-2（OOD 边界）与 P1-3（Phase 0 最小化）的论证较为扎实，也不足以替代核心维度的审查。
- **建议方向**：
  1. 重新组织报告结构，将"任务核心维度"作为一级章节，每个维度独立成节并展开实质论证
  2. 对"阶段划分（7 个）合理性"给出独立判断（不仅说"已显著改善"，还需说"为什么 7 个是合适的 / 风险点是什么"）
  3. 对"多人并行贡献硬约束"做逐项可行性核验（接口契约、数据契约、权限契约、CI 占位、共享类型包、AGENTS.md、QUICKSTART.md 是否真能让 N 个开发者并行起步）
  4. 对"13 项 AI 能力 + 6 大业务域 + 3 大终端 + 5 个医生子岗位"做反向覆盖率矩阵，**从需求文档倒推**而非从组件 A 自报确认

### 要点 2：细节问题过度，违反"关注任务"原则

- **问题**：报告 P1-1 主体、P2-4 整体、P3-3 整体均属"统计行数 / 文档版本号 / 文档校对"类细节问题，质询指令明确要求避免。
- **原因**：这些细节问题在最终读者（下游 OOD 工程师、新加入开发者）的视角中权重很低；过度关注细节会**稀释核心质量问题的信号强度**，让读者对"路线图本身质量是否合格"产生误判。
- **建议方向**：
  1. **删除** P1-1 主体中"85 行 / 14.5%"的计算段落，改为质性描述（如"v4 文档中变更记录与修订说明在正文章节中占据显著篇幅，对最终读者构成干扰"）
  2. **删除** P2-4 整体（"v4 已收口 / v3 已统一"类注释放错位置）—— 这完全是版本号污染问题
  3. **删除或合并** P3-3 —— 重复段在再审议迭代过程中本就合理，不构成质量缺陷
  4. **大幅减少** 全文中"L28-84"、"L99"、"L131-139"等行号引用，改为节标题引用或内容引述

### 要点 3：证据展示不充分，关键判断缺乏可复核支撑

- **问题**：报告多处关键判断（如 P1-2 的 4 套状态枚举、P2-2 的"埋点 vs 基线"区分）仅给出"行号 + 简略概括"，未附原文片段；某些量化数据（如 P1-1 的 14.5%）存在计算口径模糊。
- **原因**：质量审查报告应**自证**而非**自报**——读者在没有 a_v4_tech_v1.md 的情况下，应能仅凭审查报告验证 70%+ 的关键判断。当前报告的证据密度不足。
- **建议方向**：
  1. P1-2 中每项状态枚举附 5-10 字原文片段（如"调拨单：草稿/审批中/在途/已接收/已拒绝/已撤销"原文已含此枚举）
  2. P1-1 中"85 行 / 14.5%"改为引用结构（"变更记录节 + 修订说明节 + 风险表已闭环行"），不重复精确行数
  3. §三 未发现的问题中"v4 自检的 12 项反馈"应附原文证据（v4 自检报告中的对应段落或编号），或明确声明"本审查未独立验证 v4 自检结论，仅承接其结论"

### 要点 4：报告语调偏弱，缺乏明确质量评级

- **问题**：报告综合判断部分使用"v4 同时已经触及路线图形态的体量上限"、"v5 收口时需要做一次瘦身 + OOD 边界回拉"等弱化表述，对当前 v4 版本的最终质量未给出明确评级。
- **原因**：质量审查报告的核心价值之一是**判定功能**——读者需要知道当前版本是"通过"还是"不通过"。当前报告更像"改进建议清单"而非"质量审查报告"。
- **建议方向**：
  1. 在"综合判断"段落末尾增加明确的质量评级（如"v4 在任务核心维度上通过，但在 OOD 边界与文档体量上需 v5 收口"或"v4 有条件通过，需 v5 收口后才可作为正式路线图基线"）
  2. P1-1 / P1-2 / P1-3 三项的严重程度应统一使用任务描述中的"阻断性"标准（影响多人并行贡献的为 P1-阻断；影响 OOD 设计空间的为 P1-阻断；仅影响可读性的为 P3）

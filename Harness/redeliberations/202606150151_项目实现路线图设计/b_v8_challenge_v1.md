**[建议]** 报告应：(1) 撤回"兜底声明仅在文末附录"的结论，重新核查 L56 引用块；(2) 将"Phase 0 行数压缩未达标"与"全文字节数 +1.5%"作为两个独立观察分别陈述，不再合并定性为"严重 / 反向恶化"。

## 质询要点（CHALLENGED 触发条目）

### 要点 1：「严重 2」核心事实错误——兜底声明并非仅在附录

- **问题**：报告认定"v8 删除命名约定段落后，正文再无任何'该命名仅作锚点'的就近提示；兜底声明仅出现在文末'修订说明（v8）'附录（v8 L727）"，并据此评级"严重"、定性"比 v7 状态更危险"、推动 v9「必做」改进列表。但 v8 L56（Phase 0.3 节首引用块）已显式含同一兜底声明，与 v7「命名约定段落」原位置完全相同。
- **原因**：这是报告全篇最高严重度结论之一，直接驱动 v9 必做改进的方向。若该结论事实有误，将误导组件 A 增加冗余声明而非真正下沉 OOD 命名；同时也会让组件 A 误判 v8 整体状态"比 v7 更危险"——但客观上 v8 至少删除了一段违反 §5 的"命名约定段落"，相对 v7 是局部改进，并非"实质后退"。
- **建议方向**：重新阅读 v8 L56 并确认兜底声明位置；将「严重 2」修订为"30+ 处 OOD 命名未下沉（与 v7 同等问题，未恶化）"，重新评估严重度；改进建议从"补充兜底声明"改为"是否需要按方案 B 真实下沉 30+ 处命名"的方向论证。

### 要点 2：「严重 1」与「一般 4」对同一根因的定级不自洽

- **问题**：v8 整体字节数 +1.5% 同时被「严重 1」（嵌套在 P1-2 反向恶化论证中）和「一般 4」（独立条目）讨论，但定级分别为"严重"和"一般"，且「一般 4」自述"不直接破坏可演示性"。
- **原因**：同一现象同时被定为「严重」和「一般」削弱报告自洽性，给组件 A 造成"该问题到底有多严重"的判断混乱。
- **建议方向**：合并「严重 1」与「一般 4」为单一条目；同时将"Phase 0 行数压缩 75 vs 目标 60-70"（确有局部未达标）与"全文字节数 +1.5%"（v8 未做整体压缩承诺）拆分为两个独立观察，避免混合归罪。

### 要点 3：「严重 2」推荐方案 A 与 v8 已存在内容重复

- **问题**：推荐方案 A 要求"在阅读指引 §0 增加一句话兜底"，但 v8 L56 已含语义等同的声明；方案 A 实质是位置微调，无法支撑"严重"级评级的改进强度。
- **原因**：方案 A 若被组件 A 采纳，会增加文档冗余而非解决根本问题；若组件 A 识别出冗余而拒绝采纳，又会陷入"严重问题无可行方案"的僵局。
- **建议方向**：明确推荐方案 B（真实下沉 30+ 处 OOD 命名）的优先级；或论证为何 §0 位置比 L56 现位置更具兜底效力。

# 再审议任务：智慧云脑诊疗平台——项目实现路线图设计

## 1. 任务来源

用户原始指令：依据需求 `Docs/01_requirement.md` 与技术栈 `Docs/02_tech.md`，设计一套完整的从零到完整项目的实现流程。

## 2. 用户对产出的明确要求

- **定位**：比 OOD（面向对象设计）文档**更上层、更笼统**。它不应当是 OOD 文档，不应深入到类、方法、接口字段级别。
- **核心目标**：告诉观看者"**下一步可以实现哪些功能，要做什么**"。即：实现路径、阶段划分、阶段内任务清单。
- **必须包含骨架阶段**：此阶段是整个流程的第一步，且应**最小化**。
- **骨架阶段完成标准**：完成后**该骨架必须足够支撑多人同时继续在其上并行贡献**。这是骨架阶段验收的硬约束。
- **覆盖范围**：从"零"（项目尚未初始化）到"完整项目"（需求文档 §1.4 所列六大业务域、§2 所列三大终端角色、§3 所列功能需求、§5 所列数据需求、§6 所列非功能要求均已实现并可演示）。

## 3. 输入资料

- **需求文档**：`Docs/01_requirement.md`（智慧云脑诊疗平台需求规格说明书）
- **技术栈文档**：`Docs/02_tech.md`（智慧云脑诊疗平台技术栈文档）

## 4. 关键参考维度

实现路线图设计时需统筹以下维度（来自需求与技术栈）：

1. **业务域维度**：六大业务域（患者服务、门诊诊疗、检查检验、药房药库、医院管理、AI 能力）的功能分布。
2. **角色维度**：患者、医生（含 5 个子岗位：门诊/检查/检验/药房/线下接诊）、管理员。
3. **终端维度**：患者端（手机形态）、医生端（PC）、管理员端（PC）。
4. **技术栈维度**：前端 Vue 3 + TS + Vite + Element Plus + Pinia；后端 Spring Boot 3 + Spring Security + JWT + JPA/SQL-Toy + MySQL/PostgreSQL；AI 能力层 Spring AI + LLM；中间件 REST/WebSocket/SSE；部署 Nginx + Jar。
5. **AI 能力维度**：3.4.1~3.4.13 共 13 项 AI 能力，按"核心三件套（分诊/审核/病历生成）→ 拓展能力（诊断/检查/检验报告/排班/分诊台/讨论结论等）"分层实现。
6. **任务分层维度**：核心任务一～六（基础设施 + 患者端 + 分诊 + 审核 + 病历 + 联调）→ 拓展任务（实时通信/微服务/AI 进阶/部署/硬件采集/影像 AI）。
7. **协作维度**：骨架完成后必须支持多人并行贡献——意味着模块边界、接口契约、提交规范必须在骨架阶段就定义清晰。

## 5. 不应做的事（边界）

- 不写 OOD 类图、不定义具体类、不写方法签名、不写接口字段表。
- 不替代需求文档（不重复 §3 各功能模块的细节描述）。
- 不替代技术栈文档（不重复各技术点的能力描述）。
- 不写代码片段、不写具体配置文件内容。
- 不写测试用例的细节。
- 不做"已完成"层面的描述——这是路线图，是规划而非现状。

## 6. 产出形态建议（仅作提示，不强制）

- 一份 Markdown 文档，文件名建议 `Docs/03_implementation_roadmap.md` 或由主Agent 协调确定。
- 文档应当具备**可执行性**：每个阶段结束时应能明确回答"该阶段可以演示什么、哪些功能已可用"。
- 文档应当具备**可分配性**：每个阶段应能拆解为可分配给不同开发者的子任务包。

## 7. 验收隐含标准

- 观看者阅读后能立即回答："现在/下一步我该做什么？"
- 骨架阶段交付后，新加入的开发者能基于骨架文档独立开始工作（而不必从零理解整体）。
- 阶段划分粒度合理：阶段过多会拖延、阶段过粗会失控。建议在 5～8 个阶段之间。
- AI 能力按"核心→拓展"分层递进，避免一上来就铺 13 项 AI 能力导致失控。

---

**注意事项**：本次任务的"再审议"采用 generic-redeliberation-harness + technical 模式（技术方案设计）。组件 A 的 technical 子Agent 应当把这份路线图作为"技术方案"来设计——确保每个阶段有明确的"技术决策"和"可执行的实现路径"，但不深入到 OOD 类的粒度。组件 B 的质量审查应重点检查：阶段划分是否合理、骨架是否最小化、骨架完成后是否真能支持多人并行贡献、是否遗漏了需求中的关键功能/AI 能力。

# 智慧云脑诊疗平台 —— 项目实现路线图

> 文档定位：实现路线图（Implementation Roadmap）。比 OOD 类图更上层、更笼统。
> 文档职责：告诉阅读者"现在/下一步可以实现哪些功能、要做什么"，回答实现路径、阶段划分、阶段内任务清单。
> 文档边界：不写 OOD 类图、不写方法签名、不写接口字段表；不复述需求/技术栈细节；不写代码片段；不写测试用例细节。
> 阶段粒度：7 个阶段（满足 5~8 阶段要求）。Phase 0 为最小化骨架，是本路线图的硬性起点。
> 编号约定：核心三件套严格按需求 §4 与 §0 阅读指引定义 — 3.4.1 = 核心 1（智能分诊）、3.4.2 = 核心 2（AI 处方审核）、3.4.3 = 核心 3（AI 病历生成）；全文（含总览表、任务四/任务五相关描述）保持一致。
> 历次修订摘要：见本文 0.1 节「文档变更记录」；本轮（v3 → v4）定向修订清单见文末「修订说明（v4）」。

### 0.3 关键技术决策

| 决策项 | 决策 | 决策理由 |
|--------|------|----------|
| 工程组织 | Monorepo（单仓多模块） | 三端 + 多后端模块 + 共享类型，单仓可原子化重构与跨端接口同步 |

| 硬件采集预留 | `services/device-gateway` 目录 + README 占位 | 提前固化"硬件采集"作为独立业务模块的位置，避免后期跨模块越界 |

## Phase 2 — 患者服务域 + 核心三件套之"智能分诊"（患者端主流程）

### 2.1 阶段目标

跑通患者从"出现症状 → AI 分诊 → 挂号 → 查看记录"的患者端主流程，并接入核心三件套中第一件"AI 智能分诊"（3.4.1，对应核心 1）。

### 2.2 阶段交付物

| 类别 | 交付项 |
|------|--------|
| 患者端 | 3.1.3 AI 智能问诊（含 **3.1.3.1 导诊 / 3.1.3.2 病情咨询（UI 可达，AI 调用走 Mock 兜底文案——显示"暂不可用，请稍后重试"，显著位置显示 AI 免责声明 `disclaimer_required=true`；真实 3.4.8 知识库接入延后到 Phase 5.4） / 3.1.3.3 智能挂号**）、3.1.4 线上挂号（含 `registration_type=OUTPATIENT/EXAMINATION` 双分支）、3.1.4.1 取消挂号、3.1.5 报告/病历/处方查询（占位可空；病历/处方草稿数据从 Phase 3 起即可见，真实数据接入详见 Phase 4.2 说明）、3.1.7 缴费记录查询（占位可空）、3.1.8 手机首页 + 便民服务子功能（3.1.8.1）（**其中"药房取药"入口在本阶段为占位卡片——点击提示"暂无可取药品"或空状态；真实数据接入延后到 Phase 4 药房发药后**） |
| 后端 | 挂号中心（按 `registration_type` 双分支）、**取消规则引擎（>2h 允许取消并将挂号状态置为"已取消"并预留"待退费"中间态字段，实际值由 Phase 4 缴费记录联动写入；≤2h 拒绝并提示"请到线下窗口办理"）**、分诊记录表、AI 智能分诊接口 `api/triage/consult`、AI 调用结果落库（主诉、推荐科室/医生、患者 ID、时间戳） |
| AI 能力 | **核心三件套之核心 1 [智能分诊]**（3.4.1）：Spring AI ChatModel 接入（HTTP API 调用大模型），按 3.1.3.1 单次/多轮两种调用方式；Prompt 模板可配置；超时与降级（回退按科室列表选择）；高风险分诊台规则管理接口（3.3.3，由 Phase 5 落地，此处占位） |

### 2.3 关键技术决策

- 分诊调用方式：单次（主诉 + Q&A 拼接）与多轮（`session_id` + 增量 `additional_responses`）并存，由前端按 `session_id` 是否存在自动选择
- 分诊降级：AI 超时（3 次失败/单次超时）→ 前端兜底文案"建议直接联系线下接诊窗口"
- 挂号状态机：待就诊 / 已就诊 / 已取消 / 待退费（中间态字段由 Phase 2 预留占位，Phase 4 缴费记录联动写入；与 Phase 4 缴费记录状态机的'待退费'态对齐）
- 取消规则：取消操作以"距就诊时间"为判定基准；>2h 允许取消并将挂号状态置为"已取消"并预留"待退费"中间态字段（实际值由 Phase 4 缴费记录联动写入），≤2h 拒绝并提示走线下窗口
- 检查预约分支（`registration_type=EXAMINATION`）的检查申请生成时机与流转，与 Phase 4 检查管理协调。本阶段规则统一：**Phase 2 阶段生成的检查预约占位申请在写入 `exam_application` 实体时 `assignment_status=PENDING_PHASE4_DISPATCH`**——仅占位、不入派单队列；Phase 4 自动派单上线时**首次回填**已存在的占位申请（按创建时间顺序入队），后续新申请按 Phase 4 派单逻辑即时分配。此规则避免 Phase 2 阶段产生的"占位申请"在 Phase 4 派单时落入歧义

### 4.3 关键技术决策

- 检查预约任务分配：系统自动按最短队列或轮转规则分配至检查医生，无需手动认领。**Phase 2 历史占位申请衔接**：Phase 2 生成的 EXAMINATION 占位申请（`assignment_status=PENDING_PHASE4_DISPATCH`）在 Phase 4 派单上线时按**创建时间顺序首次回填入队**；后续新申请按 Phase 4 派单逻辑即时分配
- 趋势图聚合：按 `item_id`（内部编码）匹配；同报告期取最近一次；单位不一致时按 SI 标准单位换算（架构设计阶段固化换算规则），无法换算的不参与绘制并标注；**聚合查询使用 SQL-Toy（与 Phase 0.3 持久化双栈一致）**
- 药房库存扣减/回补：发药扣减药房库存、退药回补药房库存；盘库差额回退药库
- **退药角色矩阵与状态机联动**（颗粒度与 4.2 调拨单对齐）：
  - **发起方** = 患者（通过患者端发起）或开单医生（通过医生端处方详情发起）
  - **审批方** = 药房医生：普通药品药房医生可直接批准；**特殊管制药品 / 抗生素需原开方医生复核后再由药房医生批准**（双签）
  - **执行方** = 药房医生：执行药房库存回补 + 缴费记录联动变更
  - **事务边界** = 药房库存回补 + 关联缴费记录由"已支付"变更为"待退费"（中间态）在同一事务内完成；任一失败触发事务回滚
  - **退费收口** = 线下窗口（3.2.9）按"待退费"中间态执行线下实际退费
  - **退药被拒** = 走原渠道（患者端 / 医生端）告知，退药申请状态变更为"已拒绝"，库存与缴费记录均不变更
  - 退药窗口：已发药（已取）取药 24 小时内允许退药（管理员可配置）；超时不退或走特殊流程
- 缴费记录状态机落地（详见 4.2 通用行）：原缴费记录变更为"待退费"中间态而非新建记录（避免双记录歧义）
- **调拨单状态机**：草稿 → 审批中 → 在途 → 已接收 / 已拒绝 / 已撤销；调拨过账采用"药库出库 + 药房入库"双写对账（出库与入库任一失败触发事务回滚 + 差异告警）；调拨差异（数量、批号、效期）按差异类型分级告警（警告/阻塞）
- **盘库单状态机**：草稿 → 盘点中 → 已复盘 → 已归档；盘盈自动入库，盘亏自动出库，误差超阈值触发管理员复核
- 药房医生查看检查/检验结果：药房医生通过处方详情卡片内嵌入口查看该处方所关联的检查/检验结果（权限：药房医生岗位 + 处方关联校验，不开放独立列表查询能力）
- 硬件接入：`device-gateway` 模块独立于 `exam` / `lab` 模块；通过统一的"设备事件 → 业务事件"映射层将原始设备数据（HL7 消息、DICOM 文件、串口数据、网关推送）转为 `exam` / `lab` 模块可直接消费的业务事件；Phase 4 仅落接口契约与 Mock 数据，真实设备对接在后续阶段
- 3.4.4 / 3.4.9 真实 AI 推理承接：复用 Phase 3 暴露的 `TriageRuleProvider` / `ExamRecommendRuleProvider` 接口；本阶段新增真实 AI 实现类并通过配置开关切换；本地规则降级实现保留为兜底（真实 AI 不可用时回退）
- 3.4.5 / 3.4.6 AI 失败回退：调用失败 / 超时时 UI 降级为"医生手动判读"模式（医生可手工录入检查/检验结果描述与初步结论），AI 标签栏显著标识"AI 不可用"，原始设备数据 / 申请单数据仍可正常录入并落地，不阻断业务流程

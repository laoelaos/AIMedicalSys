## Phase 0 — 最小化骨架（项目初始化阶段）

### 0.1 阶段目标

搭出**让多人可并行继续贡献**的最小工程骨架。骨架阶段不实现任何业务功能，只交付协作所需的"地形"。

### 0.2 阶段交付物（最小化清单）

| 类别 | 交付项 | 说明 |
|------|--------|------|
| 工程结构 | Monorepo 目录骨架 | `apps/`（三端）、`services/`（后端模块）、`packages/`（共享类型/工具）、`docs/`、`scripts/` |
| 后端骨架 | Spring Boot 3 多模块工程 | `common` / `gateway` / `auth` / `patient` / `doctor` / `exam` / `lab` / `pharmacy` / `admin` / `ai-ability` 等模块占位，每个模块具备三层分层结构与包名约定 |
| 前端骨架 | 三套 Vue 3 + TS + Vite 工程 | `patient-portal`（手机壳形态）、`doctor-station`（PC）、`admin-console`（PC）；统一 Pinia 状态管理骨架、Axios 实例（拦截器、JWT 注入、统一错误处理）、Vue Router 占位路由 |
| 接口契约 | API 接口契约骨架 | 接口清单、请求/响应通用规范（`Result<T>` 统一响应、分页结构、错误码命名空间）；AI 能力契约骨架（3.4.x 的输入/输出占位类型与错误码） |
| 数据契约 | 数据库骨架 | 数据库命名规范、ER 图文档骨架、主键/时间戳/审计字段约定、Flyway/Liquibase 初始空迁移脚本 |
| 权限契约 | 角色—岗位—功能矩阵骨架 | 2.6 矩阵编码进后端权限模型占位；菜单与前端路由占位 |
| 协作规范 | 协作与提交规范 | 仓库 `README`、`CONTRIBUTING`、`AGENTS.md`（含构建/启动/联调指令）、分支模型、Commit 规范、PR 模板、Code Review 清单 |
| 持续集成 | CI 占位流水线 | 至少含构建+单测占位；前端 lint/format；后端编译/单测；接口契约校验 |
| 基础组件 | 共享包 | 前端：`request` 封装、统一登录态 store、通用表单/列表组件占位；后端：`Result`、`GlobalExceptionHandler`、`BaseEntity`、统一日志/审计/鉴权拦截器 |
| 部署骨架 | 本地一键启动脚本 | `scripts/dev-up.sh` 拉起数据库、后端、前端三端；`docs/QUICKSTART.md` 5 分钟跑起来 |

### 0.3 关键技术决策

| 决策项 | 决策 | 决策理由 |
|--------|------|----------|
| 工程组织 | Monorepo | 三端 + 多后端模块 + 共享类型，单仓可原子化重构与跨端接口同步 |
| 后端模块切分 | 按"业务域"垂直切模块，模块间通过内部 API 调用而非共享 DB | 保证 Phase 1+ 多人可在不同模块内并行开发不相互阻塞 |
| 前后端契约 | OpenAPI / TypeScript 类型生成（后端导出 → 前端 `@types` 包） | 解决"前后端改一处要改多处"的协作痛点 |
| 数据库选型 | MySQL（与原始文档一致），编码 utf8mb4 | 实践路径最成熟、文档支持最全 |
| 持久化方案 | Spring Data JPA | 易于在骨架阶段快速搭出可运行的 Repository |
| 鉴权落地 | Spring Security + JWT，骨架阶段只走通"登录 → 拿 Token → 受保护接口" 三个端点的空实现 | 后续业务模块只需复用鉴权拦截器即可 |
| AI 接入 | 后端独立 `ai-ability` 模块，对外暴露 `AIService` 内部接口；骨架阶段以"可注入的 Mock 实现"占位 | 业务模块可在无 LLM 凭证情况下基于 Mock 实现并行开发 |
| 前端 UI 库 | Element Plus（PC 端）/ Vant 或自研 Mobile 壳（患者端）| 与技术栈文档一致 |
| 包管理 | 后端 Maven / 前端 pnpm | 与既有约定一致 |
| 代码风格 | 后端 Checkstyle/EditorConfig；前端 ESLint + Prettier | 避免合并冲突与风格争议 |

### 0.4 可演示能力

- 三端工程均可一键启动到"占位首页 + 登录/注册空页 + 一个 ping 接口可达"
- 通过 `AGENTS.md` 与 `QUICKSTART.md`，新加入的开发者可在 1 小时内本地拉起全部工程
- 后端 API 文档（Knife4j）可访问，模块列表清晰
- CI 流水线跑通构建+单测占位

### 0.5 可并行贡献的"地形"已具备

- 每个业务模块有独立的目录、独立的 README、独立的接口前缀、独立的开发分支建议
- 共享类型在 `packages/shared-types` 中维护；新增字段在此一处定义
- 任务分配维度（与下游阶段对应）：
  - 模块 A：搭 `patient-portal` 与 `patient` 后端模块
  - 模块 B：搭 `doctor-station` 与 `doctor` 后端模块
  - 模块 C：搭 `exam` / `lab` / `pharmacy` 后端模块
  - 模块 D：搭 `admin-console` 与 `admin` 后端模块
  - 模块 E：搭 `ai-ability` 模块的 Mock 实现
  - 模块 F：搭基础设施（DB / CI / 部署脚本）

### 0.6 阶段准入/准出

- 准入：仓库已初始化、技术栈基线已对齐 Docs/02_tech.md
- 准出（三条硬性）：①三端能跑通空壳 + 一个 `ping` 接口；②`AGENTS.md` 与模块 README 完备；③共享类型与接口契约占位定义完成

## Phase 1 — 基础设施 + 统一认证与权限（地基阶段）

### 1.1 阶段目标

打通"账号 / 角色 / 岗位 / 菜单 / 接口"五件套，让三大终端能基于统一的身份进入平台骨架。

### 1.2 阶段交付物

| 类别 | 交付项 |
|------|--------|
| 数据基础 | 用户、角色、岗位、用户-角色-岗位关联、菜单、按钮权限、字典等基础表与初始 SQL |
| 鉴权中心 | 登录（密码 + JWT）、刷新 Token、登出、当前用户信息接口；统一鉴权拦截器；CORS 配置 |
| 患者端基础 | 3.1.1 注册/登录、3.1.2 个人中心（仅查看）、3.1.6 健康档案基础版（增删改条目） |
| 医生端基础 | 医生账号登录、岗位标识加载、个人资料维护（3.2.7） |
| 管理员端基础 | 管理员登录、用户/角色/岗位管理、菜单管理、字典管理（3.3.4 基础数据） |
| 权限模型 | 角色-岗位-功能三级权限模型按 2.6 矩阵落地；前后端按权限动态渲染菜单与按钮 |
| 公共能力 | 统一响应 `Result<T>`、全局异常处理、参数校验（Validation）、审计日志、操作日志 |

### 1.3 关键技术决策

- 密码使用 BCrypt 强哈希存储；JWT 携带 userId/role/position 关键 claims
- 前端按角色动态加载路由表；菜单权限由后端下发
- 健康档案结构化存储（含过敏史/慢病/家族史/手术史/用药史子字段）
- 后端日志统一接入 Logback + MDC 追踪号；前端 Axios 拦截器注入 `X-Request-Id`

### 1.4 可演示能力

- 患者注册/登录、查看个人中心
- 医生按岗位登录看到对应菜单（5 个子岗位）
- 管理员可创建用户、分配角色与岗位
- 任何受保护接口未带 Token 都被拦截；权限不足返回 403

### 1.5 可分配子任务包

- 包 A：用户/角色/岗位/菜单/字典数据建模与初始 SQL
- 包 B：Spring Security + JWT 实现
- 包 C：患者端注册/登录/个人中心/健康档案 UI + API
- 包 D：医生端与管理端登录/菜单动态化

### 1.6 阶段准入/准出

- 准出：三大终端能完成"登录 → 看到与自己角色匹配的菜单"全链路，权限矩阵 2.6 全部通过冒烟

## Phase 2 — 患者服务域 + 核心三件套之"智能分诊"（患者端主流程）

### 2.1 阶段目标

跑通患者从"出现症状 → AI 分诊 → 挂号 → 查看记录"的患者端主流程，并接入核心三件套中第一件"AI 智能分诊"（3.4.1）。

### 2.2 阶段交付物

| 类别 | 交付项 |
|------|--------|
| 患者端 | 3.1.3 AI 智能问诊（含 3.1.3.1 导诊 / 3.1.3.2 病情咨询 / 3.1.3.3 智能挂号）、3.1.4 线上挂号（含 `registration_type=OUTPATIENT/EXAMINATION` 双分支）、3.1.4.1 取消挂号、3.1.5 报告/病历/处方查询（占位可空）、3.1.7 缴费记录查询（占位可空）、3.1.8 手机首页 + 便民服务子功能（3.1.8.1） |
| 后端 | 挂号中心（按 `registration_type` 双分支）、取消规则引擎（2 小时窗 + 退费状态机）、分诊记录表、AI 智能分诊接口 `api/triage/consult`、AI 调用结果落库（主诉、推荐科室/医生、患者 ID、时间戳） |
| AI 能力 | **核心三件套之 [智能分诊]**：Spring AI ChatModel 接入（HTTP API 调用大模型），按 3.1.3.1 单次/多轮两种调用方式；Prompt 模板可配置；超时与降级（回退按科室列表选择）；高风险分诊台规则管理接口（3.3.3，由 Phase 5 落地，此处占位） |

### 2.3 关键技术决策

- 分诊调用方式：单次（主诉 + Q&A 拼接）与多轮（`session_id` + 增量 `additional_responses`）并存，由前端按 `session_id` 是否存在自动选择
- 分诊降级：AI 超时（3 次失败/单次超时）→ 前端兜底文案"建议直接联系线下接诊窗口"
- 挂号状态机：待就诊 / 已就诊 / 已取消 / 待退费（中间态，与 5.4.6 一致）
- 检查预约分支（`registration_type=EXAMINATION`）的检查申请生成时机与流转，与 Phase 4 检查管理协调，本阶段只生成申请记录占位
- 健康档案结构化字段 → AI 输入契约字段映射：按 3.1.6 末段过渡方案，保留 `allergy_details` 可选扩展容器

### 2.4 可演示能力

- 患者以"胸痛伴气短"主诉进行 AI 分诊，得到推荐科室与医生
- 一键挂号成功，在"我的挂号"中可见
- 取消挂号（>2h）成功，关联缴费记录由"已支付"变更为"待退费"
- 患者端首页所有范围内入口可达（除"住院服务"明确占位）

### 2.5 可分配子任务包

- 包 A：挂号中心后端（含双 `registration_type` 分支与取消规则）
- 包 B：患者端 UI（智能问诊 + 挂号 + 取消 + 首页 + 便民服务）
- 包 C：AI 智能分诊（后端 `ai-ability/triage` 模块 + Prompt 模板 + Mock 兜底）
- 包 D：分诊记录表与历史查询 API

### 2.6 阶段准入/准出

- 准出：任务三（智能分诊）+ 任务二（挂号流程）演示链路通畅；AI 不可用时降级路径可演示

## Phase 3 — 医生端核心 + 核心三件套之"AI 病历生成 / AI 处方审核"（诊疗主战场）

### 3.1 阶段目标

跑通门诊医生"叫号 → 接诊 → AI 诊断/开立检查/开方/病历生成/审核 → 保存"的诊疗闭环；落地核心三件套中第二件"AI 病历生成"（3.4.3）与第三件"AI 处方审核"（3.4.2）。

### 3.2 阶段交付物

| 类别 | 交付项 |
|------|--------|
| 医生端 | 3.2.1 挂号管理、3.2.2.1 病人基本信息、3.2.2.2 病情信息录入、3.2.2.3 AI 智能诊断助手（3.4.4）、3.2.2.4 AI 开立检查/检验（3.4.9）、3.2.2.5 查看检查/检验结果（数据来源先为检查/检验完成后的占位记录，渲染/趋势图实现本阶段先做骨架）、3.2.2.6 AI 辅助开方（3.4.10）、3.2.2.7 AI 处方审核、3.2.2.8 AI 病历生成、3.2.2.9 病历保存/列表/详情 |
| 后端 | 处方中心、医嘱开立与记账（3.2.10，仅记账）、病历中心、审核结果持久化（审核记录表）、本地规则校验（AI 不可用降级） |
| AI 能力 | **核心三件套之 [AI 病历生成]**：Spring AI Structured Output 将对话文本转为结构化病历字段；按科室可配置 Prompt 模板；输入 < 50 字符不允许触发；关键字段缺失时回提示补全。**核心三件套之 [AI 处方审核]**：风险等级（高/中/低）三档；高风险阻断、中风险三种可选操作（强制提交/修改重审/撤销审核）、低风险建议不阻断；6 秒硬超时回退到本地规则校验并打标"未经过 AI 审核"；MEDIUM 强制提交留痕 |
| 拓展 AI（提前落地部分） | **3.4.4 AI 智能诊断**（置信度 < 60% 醒目标识、12 秒硬超时降级手动诊断）、**3.4.9 AI 开立检查/检验**、**3.4.10 AI 辅助开方**（剂量阈值告警） |

### 3.3 关键技术决策

- 病历生成：流式输出（SSE）放在 Phase 6 拓展；本阶段先做一次性 Structured Output 同步返回
- 处方审核：审核结果独立持久化（与处方解耦），便于事后回溯
- AI 审核 6 秒硬超时 + 降级标识：前端须在审核弹窗显著位置标识
- 健康档案 → AI 输入契约字段映射：实现统一提取器（`allergy_history` 字符串 + 可选 `allergy_details` 容器、`comorbidities` 数组），由 `ai-ability` 模块复用
- 检查/检验结果：本阶段数据先由后端占位 API 模拟，Phase 4 接入真实检查/检验流程

### 3.4 可演示能力

- 门诊医生"叫号" → 接诊 → 录入病情 → AI 诊断 → AI 开立检查 → AI 辅助开方 → AI 处方审核 → AI 生成病历 → 保存全链路
- AI 不可用时降级到本地规则校验、UI 显示"未经过 AI 审核"标签
- 病历生成后患者端 3.1.5 可看到
- 处方提交后自动创建"待缴费"单据（仅记账）

### 3.5 可分配子任务包

- 包 A：医生端 3.2.1 / 3.2.2.1 / 3.2.2.2 / 3.2.2.9 UI
- 包 B：医生端 3.2.2.3 / 3.2.2.4 / 3.2.2.6 / 3.2.2.7 / 3.2.2.8 UI
- 包 C：处方中心 + 病历中心后端
- 包 D：AI 病历生成 + AI 处方审核（含风险等级与降级）
- 包 E：AI 诊断 / 开立检查 / 辅助开方（提前落地的拓展 AI）
- 包 F：医嘱开立与记账（3.2.10，本阶段仅记账动作）

### 3.6 阶段准入/准出

- 准出：任务四（AI 处方审核）+ 任务五（AI 病历生成）演示链路通畅；任务六（联调）核心场景可走通

## Phase 4 — 检查 / 检验 / 药房 / 药库 / 线下窗口（业务域拓展）

### 4.1 阶段目标

把医生端的 5 个子岗位全部落地：检查医生、检验医生、药房医生、线下接诊医生；同时打通"检查预约 → 检查执行 → 报告 → 反馈门诊"与"处方 → 发药 → 退药 → 联动缴费"两条主流程。

### 4.2 阶段交付物

| 类别 | 交付项 |
|------|--------|
| 检查域 | 3.2.3 检查管理；含 AI 分诊台（3.4.11 执行顺序推荐）、AI 智能检查报告（3.4.5 一次性输出）、AI 影像分析（3.4.7：头部 CT / 肺部 CT / 肿瘤分割，岗位决定可用模型） |
| 检验域 | 3.2.4 检验管理；含 AI 智能检验报告（3.4.6）、趋势图渲染（按 `item_id` 聚合、SI 单位统一换算；管理员在 3.3.4 配置最近 N 次默认 5） |
| 药房域 | 3.2.5 药房管理；含"待发药/已发药（待取）/已发药（已取）/已退药"状态机；药房库存 = 药库日配额下的子库存；药房医生经手处方的检查/检验结果调取路径（○² 权限，入口收敛在处方详情卡片） |
| 药库域 | 3.2.6 药库管理；含库存调拨记录、盘库后差额回退药库流程 |
| 线下窗口 | 3.2.9 线下窗口；含线下挂号/退号、收费/退费、费用记录查询 |
| 通用 | 缴费记录状态机（待缴费/已支付/待退费/已退费/已撤销）按 5.4.6 落地；"医嘱开立与记账"行子动作（开单医生撤销/管理员代撤销）落地 |
| 拓展 AI | 3.4.11 AI 执行顺序推荐（分诊台工作台调用，概念上独立于 3.3.3 管理端分诊规则） |

### 4.3 关键技术决策

- 检查预约任务分配：系统自动按最短队列或轮转规则分配至检查医生，无需手动认领
- 趋势图聚合：按 `item_id`（内部编码）匹配；同报告期取最近一次；单位不一致时按 SI 标准单位换算（架构设计阶段固化换算规则），无法换算的不参与绘制并标注
- 药房库存扣减/回补：发药扣减药房库存、退药回补药房库存；盘库差额回退药库
- 退药窗口：已发药（已取）取药 24 小时内允许退药（管理员可配置）；超时不退或走特殊流程
- 缴费记录状态机：原缴费记录变更为"待退费"中间态而非新建记录（避免双记录歧义）
- 药房医生查看检查/检验结果：仅在处方详情卡片内嵌入口，不开放独立列表查询

### 4.4 可演示能力

- 患者"预约检查"分支（`registration_type=EXAMINATION`）→ 系统自动派单 → 检查医生执行 → AI 报告 → 反馈门诊
- 检验结果趋势图（按 `item_id` 聚合 + SI 单位换算）
- 处方提交 → 药房发药 → 患者取药 → 退药 → 联动"待退费"中间态 → 线下窗口实际退费
- 线下窗口：挂号/退号/收费/退费/费用记录查询

### 4.5 可分配子任务包

- 包 A：检查域（3.2.3 + 3.4.5 + 3.4.7 + 3.4.11）
- 包 B：检验域（3.2.4 + 3.4.6 + 趋势图聚合）
- 包 C：药房域（3.2.5 + 状态机 + 库存）
- 包 D：药库域（3.2.6 + 调拨 + 盘库）
- 包 E：线下窗口（3.2.9 + 缴费记录状态机 + 3.2.10 撤销子动作）

### 4.6 阶段准入/准出

- 准出：六大业务域中"门诊诊疗 / 检查检验 / 药房药库 / 患者服务"四条主流程全部可演示；缴费记录状态机所有状态可被冒烟

## Phase 6 — 全链路联调 + 实时通信 + 部署与运维（收口与上线）

### 6.1 阶段目标

把任务六（联调）的"典型演示场景"扩展为覆盖三大终端的端到端冒烟；落地实时通信拓展；打包部署跑通；性能/可观测性/可演示性收口。

### 6.2 阶段交付物

| 类别 | 交付项 |
|------|--------|
| 联调 | 任务六典型场景贯通：注册登录 → AI 分诊 → 挂号 → 接诊 → AI 诊断 → AI 开立检查 → 检查执行 → AI 报告 → AI 辅助开方 → AI 审核 → AI 病历生成 → 缴费 → 发药 → 退药 → 退费 |
| 拓展 | 实时通信：WebSocket（高风险用药实时告警推送）、SSE（病历生成流式输出）、Pinia 模块化状态机（复杂流程状态机） |
| 部署 | Nginx 静态资源部署前端；Spring Boot Jar 独立运行后端；一键 `dev-up` / `dev-down` 脚本；生产环境部署文档；健康检查接口与就绪探针 |
| 可观测性 | 日志聚合（MDC + 追踪号）、指标埋点、错误监控、慢调用追踪 |
| 性能 | 按 6.7 性能指标验收（推断阈值，详见 OOD/性能阶段文档） |
| 文档 | `QUICKSTART.md`、`DEPLOY.md`、`ARCHITECTURE.md`（含架构图/数据流图/AI 能力全景）、`API.md`（Knife4j）、演示 PPT 大纲 |

### 6.3 关键技术决策

- 流式输出：病历生成从一次性返回升级到 SSE 流式，前端逐字渲染
- 高风险告警：WebSocket 通道从服务端主动推送，门诊医生端实时显示
- 可观测性：后端 Logback + MDC；前端 Sentry/自建埋点；统一追踪号贯穿
- 部署：Nginx 反向代理 + 静态资源 + 后端 Jar；HTTPS 由 Nginx 终结
- AI 凭证与模型路由：LLM 凭证集中管理；多模型/多租户路由由 `ai-ability` 抽象层屏蔽

### 6.4 可演示能力

- 一条"患者 → 医生 → 检查 → 检验 → 药房 → 退费"全链路贯通；高风险告警实时弹出
- 病历生成逐字流式渲染
- 演示视频脚本可逐镜头拍摄
- 本地 `dev-up` 一键拉起整套系统；生产环境按 `DEPLOY.md` 5 步部署

### 6.5 可分配子任务包

- 包 A：联调（接口契约、跨模块数据流、Postman 测试集）
- 包 B：实时通信（WebSocket 高风险告警 + SSE 病历流式）
- 包 C：部署（Nginx + Jar + 启动脚本 + 部署文档）
- 包 D：可观测性（日志、指标、追踪）
- 包 E：演示材料（架构图、PPT、演示脚本）

### 6.6 阶段准入/准出

- 准出：六大业务域、三类角色、13 项 AI 能力、§5 数据需求、§6 非功能要求全部可演示且通过冒烟

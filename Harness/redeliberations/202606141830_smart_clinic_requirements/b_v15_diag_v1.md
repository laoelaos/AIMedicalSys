审查视角：需求响应充分度、整体深度与完整性、实现者可行动性

## 总体评价

该文档经过 15 轮迭代审议，成熟度很高，在业务覆盖、字段级契约、状态机定义等方面已达到较高的需求基线水平。多数之前在内部审议中发现的问题（角色边界、权限矩阵、AI 契约、状态机完整性等）已得到有效修正。以下列出在本次审查中发现的 6 项问题，侧重内部审议尚未充分覆盖的维度——即可实现性隐含模糊地带、验收判定的完整性，以及跨章节的结构性缺失。

## 问题 1：缴费记录状态机图示与文本描述不一致（§5.4.6）

- **问题描述**：§5.4.6 的状态图绘有两条并行箭头——"已支付 → 待退费"（触发条件：退费申请）与"已支付 → 待退费（部分）"（触发条件：部分退费申请）。但同一章节的文本语义段（"待退费（部分）状态语义（v11 修订）"）已明确说明部分退费与全额退费统一使用单一"待退费"状态，通过"未退金额"字段区分，不应存在两条独立的状态路由。图示与文本描述的不一致会导致实现方在数据库设计层误建两个独立状态枚举值，增加实现复杂性并引入不必要的前端分支判断。
- **所在位置**：§5.4.6 状态图（约 line 1501–1507）及下方的"待退费（部分）状态语义"段（约 line 1514）
- **严重程度**：一般
- **改进建议**：将状态图统一为单条"已支付 → 待退费"箭头，在箭头旁加注"全额/部分退费由'未退金额'字段区分"，删除"已支付 → 待退费（部分）"的独立箭头分支。同时确保文字描述中的"部分退费"在状态图中不再独立呈现为状态路由。

## 问题 2：§3.4.13 超时与最低质量门槛错误码优先级未定义（§3.4.13）

- **问题描述**：§3.4.13 定义了两种可能返回错误的场景——超时（返回 `DISCUSS_AI_TIMEOUT`）和 `discussion_text` 字符数 < 50（返回 `DISCUSS_AI_OUTPUT_INCOMPLETE`）。当 AI 在达到 50 字符之前发生超时，两个条件同时满足，但文档没有定义哪个错误码优先返回。实现方不确定是返回超时错误（保留已生成的部分内容）还是返回质量门槛错误（附带"质量未达标"醒目标识）。两种处理路径在前端行为上差异明显（前者显示超时提示+重试按钮，后者显示质量未达标提示+重新生成入口），错误码选择直接影响用户体验。
- **所在位置**：§3.4.13 "超时阈值"段（约 line 1063）与"50 字符最低质量门槛的处理规则"段（约 line 1068）
- **严重程度**：一般
- **改进建议**：在"50 字符最低质量门槛的处理规则"段末尾追加错误码优先级规则，建议：若因超时导致 `discussion_text` < 50 字符，优先返回 `DISCUSS_AI_TIMEOUT`（与超时行为一致，`partial_content` 携带已生成的 < 50 字符片段），超时不触发质量门槛校验；仅当 AI 正常返回（无超时）但 `discussion_text` < 50 字符时返回 `DISCUSS_AI_OUTPUT_INCOMPLETE`。若需要超时场景下仍校验质量门槛，则需明确定义两错误码的级联关系。

## 问题 3：§7.1 部分验收项缺少可操作的判定边界（§7.1）

- **问题描述**：§7.1 的 13 项功能验收中，部分项（#1 注册登录、#4 叫号、#11 患者端查看）行为简单、判定边界明确；少数项（#6 检查管理、#7 检验管理、#9 线下窗口）以场景描述替代通过标准，缺少"在此条件下执行动作 A，系统应返回可观察的结果 B"的统一判定模板。具体而言：#6 描述了一整段完整业务流程（接收→执行→AI 报告→修订→提交→权限验证），但未定义通过判定的最低标准——是"任意一次成功的调用链"即可，还是必须同时验证 authorized 医生可见识别结果、unauthorized 医生不可见识别结果（正反测试）？验收方与实现方对"通过"的预期可能不一致。
- **所在位置**：§7.1 #6（约 line 1638）、#7（约 line 1639）、#9（约 line 1641）
- **严重程度**：一般
- **改进建议**：对 #6 和 #7，建议追加最小判定条件，例如格式为"满足以下条件视为通过：(a) 医生端完成从任务接收到提报的完整 UI 操作闭环；(b) AI 返回结果在 UI 上可观察（含降级场景）；(c) 岗位权限校验至少覆盖一个正向用例（授权岗位可见）和一个负向用例（非授权岗位不可见）"。#9 可参照#3 的格式追加明确的判定条件。

## 问题 4：趋势分析图中"同一检验项目"的匹配规则未定义（§3.2.2.5 / §3.4.6）

- **问题描述**：§3.2.2.5 定义了趋势分析图显示"最近 N 次结果"，数据来源为"从检验报告历史记录中聚合"。但"同一检验项目"的匹配规则未定义——是按 `item_name` 精确字符串匹配（"白细胞计数"与"WBC"被视为不同项目），还是按项目 ID 或标准化编码匹配？如果同一患者在一个报告周期内同一项目有多次测量，是否去重？如果同一项目的 `unit` 在不同报告间不同（如 mmol/L vs mg/dL），是否做单位换算？缺少这些定义，实现方需要在开发阶段自行假设，且不同实现者的假设可能不同，导致验收时行为不一致。
- **所在位置**：§3.2.2.5 关键交互段趋势图子段（约 line 538–539）；§3.4.6 输出契约 `abnormal_items` 的 `delta` 排序规则段（line 936）仅定义了"上一次"的排序，未定义"同项目"的匹配标准
- **严重程度**：轻微
- **改进建议**：在 §3.2.2.5 或 §3.4.6 服务质量要求段追加"同项目判定规则"，建议：(a) 按检验项目内部编码（`item_id` 或等效标识）匹配，而非 `item_name` 字符串；(b) 同一报告中的重复测量按提交时间取平均值或最近值，需明确定义；(c) 若涉及单位换算，标注"由架构设计阶段确定"或给出常见单位的换算约束。

## 问题 5：§3.4.5 内部调用 §3.4.7 失败时 `image_recognition` 缺省行为存在两个差异表述（§3.4.5）

- **问题描述**：§3.4.5 在"与 3.4.7 的服务组合关系"段（line 906）写明：若因 `raw_data_ref` 指向非影像数据格式导致 3.4.7 调用失败，返回 `EXAM_AI_INTERNAL_IMG_FAIL` 并**将 `image_recognition` 缺省**。但同一章节"服务质量要求"段的降级路径（line 919）写明：3.4.5 内部调用 3.4.7 失败时**回退到无 `image_recognition` 的报告草稿**（`EXAM_AI_INTERNAL_IMG_FAIL`）。两处表述一处在输入校验场景中写"缺省"、一处在降级路径中写"回退到无 image_recognition"，虽然实际结果同为"不返回 image_recognition"，但前者的触发条件是"输入数据不匹配"而后者是更广泛的"调用失败"，触发条件范围不同。实现方无法判断：若 3.4.7 因网络超时而失败（非输入数据问题），是否同样适用"`image_recognition` 缺省"？是否也返回 `EXAM_AI_INTERNAL_IMG_FAIL`？还是返回 `IMG_AI_TIMEOUT`？
- **所在位置**：§3.4.5 "与 3.4.7 的服务组合关系"段（约 line 906）与"服务质量要求"段降级路径（约 line 919）
- **严重程度**：轻微
- **改进建议**：将两处合并在"服务质量要求"段中统一表述为："3.4.5 内部调用 3.4.7 失败时（包括超时、无权限、输入数据不匹配等任何原因），`image_recognition` 缺省（不返回），同时返回 `EXAM_AI_INTERNAL_IMG_FAIL` 错误码（透传 3.4.7 的具体错误细节）。报告草稿与判读仍正常返回。若 `raw_data_ref` 指向非影像数据格式，同样按此规则处理。" 删除组合关系段中的"返回 EXAM_AI_INTERNAL_IMG_FAIL 并将 image_recognition 缺省"的独立表述，避免触发条件歧义。

## 问题 6：§3.1.6 健康档案→AI 映射约定中过敏史 `allergy_history` 字段对 `reaction_type` 和 `severity` 的遗漏可能影响临床风险判断，但文档仅以"业务方评审提示"推后处置，未提供过渡期默认行为（§3.1.6）

- **问题描述**：§3.1.6 末段"健康档案数据→AI 契约字段的映射约定"明确 `allergy_history` 仅传递 `allergen` 名称列表，不传递 `reaction_type`（过敏反应类型）与 `severity`（严重程度），并在末尾追加了"业务方评审提示"建议业务方确认是否纳入。从实现者视角看：若业务方在评审会议中确认纳入，则 3.4.2/3.4.4/3.4.9/3.4.10/3.4.13 的输入契约字段集均需同步更新、前端健康档案编辑界面需补充反应类型与严重程度字段、映射逻辑需重写——这是一个波及多个模块的变更。但文档未定义"过渡期默认行为"：在业务方确认之前，实现方是否应预留扩展字段？AI 契约的输入是否应设计为可扩展结构？如果后续变更为新增字段而非替换现有字段，现有字段（仅 allergen）的兼容性如何保证？目前仅以"由需求变更流程追加"一句带过，缺少架构层面的兼容性指引。
- **所在位置**：§3.1.6 末段"健康档案数据→AI 契约字段的映射约定"及"业务方评审提示"段（约 line 390–395）；波及 §3.4.2/3.4.4/3.4.9/3.4.10/3.4.13 的 `allergy_history` 输入字段定义
- **严重程度**：轻微（由于文档已主动标注为待确认项，不作为当前版本缺陷；但从"可直接作为开发依据"视角判断，缺少过渡期设计指引）
- **改进建议**：在 §3.1.6 映射约定段末尾追加强制性建议（而非可选提示）："在业务方确认前，实现方应按以下过渡方案实施——(a) AI 契约的 `allergy_history` 字段继续保持 string 类型（以逗号拼接过敏原名称），但建议在契约定义中预留 `allergy_details`（array of object，可选）作为后续扩展字段的容器，当前暂不传递；架构设计阶段应确保输入契约向后兼容，后续追加 `reaction_type`/`severity` 时无需修改现有字段；(b) 健康档案编辑界面本节字段（过敏原、反应类型、严重程度、发生时间）前端建议**直接实现完整字段**，通过 UI 展示但不强制填写，为后续 AI 消费做好数据储备。若业务方在评审会议中确认不纳入，前端可隐藏反应类型与严重程度字段，不影响已存储数据。"

## 总结

以上 6 项问题中：
- 问题 1（状态图不一致）属于**逻辑矛盾**——同一章节内图示与文字描述冲突，实现方按任一方案实现均会与另一方案冲突。
- 问题 2（错误码优先级未定义）属于**隐含模糊地带**——两个正交规则在边界条件重叠时无优先级，属于典型的"两个正确规则同时适用时无规则"的缺口。
- 问题 3（验收判定边界）属于**需求深度不足**——部分验收项缺少可操作的最小通过标准，实现方无法自行判定是否"完成"。
- 问题 4（趋势图匹配规则）属于**实现细节缺失**——需要在当前粒度上追加一层匹配规则以消除歧义。
- 问题 5（降级触发条件歧义）属于**表述冲突**——同一行为在文档不同位置给出不同触发条件的描述。
- 问题 6（过渡期设计指引缺失）属于**完整性不足**——文档主动识别了潜在变更但未为过渡期提供设计指引。

该文档整体已达到高度成熟的需求基线水平，上述问题不影响文档作为开发依据的基本可用性，但建议在下一轮迭代中修正。

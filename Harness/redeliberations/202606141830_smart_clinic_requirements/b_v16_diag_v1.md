## 发现的问题

### 问题 1：检查预约（EXAMINATION）从患者预约到检查医生执行的交接机制未闭合

- **问题描述**：§3.1.4 线上挂号描述了患者端完成检查预约的动作（选择检查项目→选择时间段→提交），§5.4.1 EXAMINATION 路径状态机将"检查科室接收该预约"定义为"待就诊→检查执行中"的触发条件，但整个文档未定义"检查科室接收"的具体机制——是系统自动分配到某个检查医生、由检查医生从队列中手动认领、还是按科室排队后由首名空闲医生自动接单？§3.2.3 检查管理的核心交互起点是"查看待执行检查任务"，但"待执行检查任务"的来源渠道和分配规则未明确。实现者无法据此确定后端应实现"推送式分配"还是"拉取式认领"机制，架构设计阶段可能因缺乏业务约束而出现多种实现方案，导致后续联调不一致。

- **所在位置**：§3.1.4 线上挂号→关键交互（检查预约分支）、§3.2.3 检查管理→关键交互、§5.4.1 挂号记录状态机→EXAMINATION 路径触发条件

- **严重程度**：一般

- **改进建议**：在 §5.4.1 EXAMINATION 路径触发条件段补充"待就诊→检查执行中"的具体触发机制说明，至少明确是"系统自动按检查项目归属科室分配至对应检查医生的待执行队列（推送式）"还是"检查医生从共享待执行列表中手动认领（拉取式）"，或者在 §3.2.3 关键交互中补充"接收检查预约"动作的描述。如当前需求阶段无法确定分配策略，应标注"由架构设计阶段确定分配策略"以消除模糊地带。

### 问题 2：检查/检验报告状态机缺少已提交报告的修订回退路径

- **问题描述**：§5.4.4 检查报告状态机和 §5.4.5 检验报告状态机均定义为严格的线性流转（待执行→执行中→AI 报告生成中→AI 报告就绪→医生终稿→已提交→已归档），每个终态（已提交/已归档）均不允许回退到任何前置状态。但在实际医疗业务中，已提交的报告可能因录入误差、补充检查结果或 AI 解读偏差需要修订或重新出具。与此形成对比的是，§5.4.3 病历状态机已通过版本号（`version`）和父病历标识（`parent_record_id`）机制支持"已保存"病历的修订（创建新版本而非修改原记录）。检查/检验报告状态机与病历状态机在支持修订的能力上存在不对称，实现者如果仅按状态机字面实现，将无法满足医疗业务中"报告更正"的真实需求。

- **所在位置**：§5.4.4 检查报告状态机→不允许的流转、§5.4.5 检验报告状态机→不允许的流转

- **严重程度**：一般

- **改进建议**：方案 A——在 §5.4.4 和 §5.4.5 中补充已提交报告的修订回退机制（参考 §5.4.3 病历状态机的版本化修订模式，追加"已提交→修订中"分支，新版本以独立报告记录存在，原报告标记为"已作废"或"已取代"）。方案 B——如本次需求范围确定不需要支持报告修订后重新提交，在状态机中显式声明"已提交报告不可修订，如需更正通过新建检查/检验申请实现"，以消除实现者的猜测。

### 问题 3：部分验收项仍以一句话场景描述替代可操作判定条件

- **问题描述**：§7.1 中 #6 检查管理、#7 检验管理、#9 线下窗口已在 v16 追加了精细的判定条件（a/b/c/d 条款），但 #1（患者注册、登录、查看个人中心）、#4（医生叫号）、#8（发药/退药联动退费）、#11（患者端查看各记录）仍以一句话场景描述替代"通过/不通过"判定标准。例如 #11："患者在患者端查看挂号记录、处方记录、电子病历、健康档案与缴费记录"——没有定义：数据为空时是否要求显示空状态、数据加载失败时是否要求显示错误提示、各项数据的展示字段最小集合是什么、是否要求覆盖"有数据"和"无数据"两种状态。实现者无法将此验收项直接转化为测试用例，演示方也难以判断是否"通过"。

- **所在位置**：§7.1 功能验收 #1、#4、#8、#11

- **严重程度**：一般

- **改进建议**：参照 #6/#7/#9 的判定条件格式，为 #1/#4/#8/#11 逐项追加可操作的通过条件，至少应明确：(a) 正常数据场景下的 UI 可见元素（如挂号记录至少展示科室/医生/时间/状态四个字段）；(b) 空数据场景的 UI 表现（空状态提示）；(c) 异常场景的 UI 表现（加载失败时的错误提示）。#8 退药联动退费还需补充验证退费金额是否正确的判定条件（如退药金额与待退费金额是否一致）。

### 问题 4：缴费记录状态机"待退费（部分）"触发条件与"单一待退费状态"注记不一致

- **问题描述**：§5.4.6 状态机图附注明确声明"全额/部分退费统一使用单一'待退费'状态，由'未退金额'字段区分"，但在同一节的触发条件段中（v7 修订段）仍将"已支付→待退费（部分）"列为一条独立的触发条件分支。虽然括号中的"（部分）"意在描述退费类型而非状态名称，但状态机的标准阅读习惯会将"待退费（部分）"理解为一个独立的状态变体。这与"单一待退费状态"原则在字面上存在矛盾——实现者阅读时会困惑：到底系统中是只有一个"待退费"状态而用金额字段区分，还是有两种不同的"待退费"（全额 vs 部分）状态？这可能导致后端状态字段设计与金额校验逻辑出现分歧。

- **所在位置**：§5.4.6 缴费记录状态机→触发条件段"已支付→待退费（部分）：部分退费申请..."

- **严重程度**：轻微

- **改进建议**：将触发条件中的"已支付→待退费（部分）"修订为"已支付→待退费（部分退费申请金额 < 原始金额）"，删除"待退费（部分）"中的括号后缀以避免被误解为独立状态名。同时在附注中更明确地表述："部分退费与全额退费在状态机中共用同一个'待退费'状态，由缴费记录实体的'未退金额'字段区分退费程度（部分退费时 0 < 未退金额 < 原始金额，全额退费完成后未退金额 = 0）"。

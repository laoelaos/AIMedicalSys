## 当前审查结果

**问题 1：缴费记录状态机图示与文本描述不一致（§5.4.6）**
严重程度：一般。状态图绘有两条并行箭头——"已支付 → 待退费"与"已支付 → 待退费（部分）"，但同节文本已说明部分退费与全额退费统一使用单一"待退费"状态、通过"未退金额"字段区分。图示与文本冲突会导致实现方误建两个独立状态枚举值。建议将状态图统一为单条"已支付 → 待退费"箭头，加注"全额/部分退费由'未退金额'字段区分"，删除独立箭头分支。

**问题 2：§3.4.13 超时与最低质量门槛错误码优先级未定义（§3.4.13）**
严重程度：一般。超时返回 `DISCUSS_AI_TIMEOUT`，`discussion_text` < 50 字符返回 `DISCUSS_AI_OUTPUT_INCOMPLETE`，但 AI 在达到 50 字符前发生超时时两个条件同时满足，未定义优先级。建议优先返回 `DISCUSS_AI_TIMEOUT`（`partial_content` 携带已生成片段），超时不触发质量门槛校验；仅当 AI 正常返回但文本 < 50 字符时返回 `DISCUSS_AI_OUTPUT_INCOMPLETE`。

**问题 3：§7.1 部分验收项缺少可操作的判定边界（§7.1）**
严重程度：一般。#6 检查管理、#7 检验管理、#9 线下窗口以场景描述替代通过标准，缺少"在此条件下执行动作 A，系统应返回可观察的结果 B"的统一判定模板。建议对 #6/#7 追加最小判定条件，如"(a) 医生端完成从任务接收到提报的完整 UI 操作闭环；(b) AI 返回结果在 UI 上可观察（含降级场景）；(c) 岗位权限校验至少覆盖一个正向用例和一个负向用例"。#9 参照 #3 格式追加判定条件。

**问题 4：趋势分析图中"同一检验项目"的匹配规则未定义（§3.2.2.5 / §3.4.6）**
严重程度：轻微。趋势图显示"最近 N 次结果"需从检验报告历史记录中聚合，但"同一检验项目"的匹配规则未定义——按 `item_name` 字符串、项目 ID 还是标准化编码？同一报告周期内重复测量是否去重？`unit` 不同时是否换算？建议按检验项目内部编码（`item_id`）匹配，重复测量取均值或最近值，单位换算由架构设计阶段确定。

**问题 5：§3.4.5 内部调用 §3.4.7 失败时 `image_recognition` 缺省行为存在两个差异表述（§3.4.5）**
严重程度：轻微。§3.4.5 "与 3.4.7 的服务组合关系"段写明调用失败返回 `EXAM_AI_INTERNAL_IMG_FAIL` 并将 `image_recognition` 缺省（触发条件为输入数据不匹配）；"服务质量要求"段降级路径写回退到无 `image_recognition` 的报告草稿（触发条件为更广泛的"调用失败"）。两处触发条件范围不同。建议合并统一表述：任何原因（超时、无权限、输入数据不匹配等）导致的失败均缺省 `image_recognition` 并返回 `EXAM_AI_INTERNAL_IMG_FAIL`，报告草稿与判读正常返回。

**问题 6：§3.1.6 健康档案→AI 映射约定中过敏史 `allergy_history` 字段对 `reaction_type` 和 `severity` 的遗漏可能影响临床风险判断，但文档仅以"业务方评审提示"推后处置，未提供过渡期默认行为（§3.1.6）**
严重程度：轻微。`allergy_history` 仅传递 `allergen` 名称列表，不传递 `reaction_type` 与 `severity`，文档仅以"业务方评审提示"建议业务方确认，但未定义过渡期默认行为——实现方是否预留扩展字段？AI 契约是否设计为可扩展结构？建议追加强制性过渡方案：(a) AI 契约保持 string 类型但预留 `allergy_details`（array of object，可选）作为扩展容器；(b) 健康档案编辑界面前端直接实现完整字段（反应类型、严重程度、发生时间），UI 展示但不强制填写。

## 问题1：缺少对原始 todo.md 的显式覆盖确认

### 位置
整篇报告（标题、问题清单、汇总表均未提及）。

### 严重程度
中等。

### 描述
`requirement.md` 明确要求"对代码评审 todo.md 中的问题进行定位分析"。报告列举的 10 个问题与 `todo.md` 的 10 个待办项一一对应（映射关系可推断），但报告通篇**没有任何一处明确提及 todo.md 或声明"所有 todo 项已覆盖"**。

具体缺失：
- 未说明各诊断问题对应哪一条 todo 项（如 "Problem 1 对应 todo 第 9 行 BaseEnum 项"）
- 未汇总声明 10 个 todo 项的处理状态（哪些确认为代码缺陷、哪些为 OOD 问题、哪些为其他类型）
- 未说明是否有的 todo 项被认定为误报（虽然每个问题都标注了"是否为误报:否"，但没有从 todo 项视角给出总体回答）

### 改进建议
在标题下方或汇总表前增加一段覆盖声明，例如：`todo.md 共 10 项待办已全部覆盖分析，对应关系如下：问题1→第1项、问题2→第2项……`。同时可在汇总表中增加"对应 todo 项"列，便于追溯。

## 问题2：问题10 的 P0 优先级在 Phase 0 语境下可能高估

### 位置
问题10，「优先级」字段。

### 严重程度
低~中等。

### 描述
问题10（ApiClient 错误拦截器未实现 NETWORK_ERROR）被归类为 P0，理由是"直接影响运行时行为：前端无法通过统一 code 判断网络错误，未捕获的 Promise 异常可能产生运行期报错"。但：

1. **Phase 0 前端骨架的实际影响范围**：Phase 0 的前端仅包含占位页面和健康检查（`/api/ping`），无实际 API 调用链路会触发网络错误。`Promise.reject(error)` 在 Phase 0 的正常联调流程中几乎不会被执行到。
2. **Phase 0 验收标准**中前端仅需展示占位页面、代理 ping 请求到后端，不涉及 `code === "NETWORK_ERROR"` 模式。
3. **问题10 的 P0 影响主体**是 Phase 1+ 引入认证跳转和真实 API 调用后的场景——报告自身已在影响范围第 5 项承认"Phase 1+ 如果引入 AuthStore 的 401 跳转逻辑，错误拦截器的统一处理能力将更加必要"。

代码确实偏离了 OOD §3.5 的设计要求，需要修复。但将 Phase 0 具身运行时影响判定为 P0 与实际场景不完全吻合。

### 改进建议
将优先级调整为 P1（当前 Phase 0 无实质运行时影响，但属设计偏离需修复），或保留 P0 但修正理由措辞——明确区分"Phase 1+ 场景下的 P0"与"Phase 0 场景下的实际影响"，让执行者可据此判断是否在 Phase 0 立即修复。

## 问题3：问题2/3 的 P1 优先级与推荐方案（方案B：低风险文档更新）存在表述张力

### 位置
问题2、问题3 的「优先级」字段（P1）及「修复者指引」中「推荐选择：方案B」。

### 严重程度
低。

### 描述
问题2/3 被标记为 P1（中优先级），理由为"涉及全仓库最大结构性偏离"和"需先决策再行动"。但「修复者指引」最终推荐的方案 B（保留扁平布局，更新 OOD）属于**无代码修改、低风险、纯文档更新**——与 P1 所隐含的"较高修复成本或较大影响"语义不完全匹配。

造成这种张力的原因是 P1 实际指**决策过程优先级**（方向选择影响后续所有 POM 相关修复），而非**修复执行优先级**或**运行时影响程度**。但报告未在优先级字段或修复指引中明确区分"决策优先级"与"修复优先级"这两个维度，可能导致执行者混淆——看到 P1 后会以为需要投入较大修复工作量，但阅读修复指引后才发现推荐的是纯文档更新。

### 改进建议
在问题2/3 的优先级字段中明确标注"P1（决策优先级 — 方向选择影响后续修复路径，但推荐的方案 B 为低风险文档更新）"，或在修复指引中增加"优先级说明"脚注。

## 问题4：未从"其他类型问题"角度对 todo 项做总体分析

### 位置
汇总表及报告整体结构。

### 严重程度
低。

### 描述
`requirement.md` 要求判断四种情况：真实代码缺陷、误报、OOD 文档问题、其他类型问题。报告通过每个问题的「分类」和「是否为误报」字段间接回答了前三类，但没有从第四类"其他类型问题"的视角给出总体分析。

具体来说，`todo.md` 中10项问题的分布是：
- **OOD文档问题**：问题1（BaseEnum 定义遗漏）
- **代码与设计偏离**：问题2~7（POM 相关配置偏离）
- **代码缺陷**：问题8、10（GlobalExceptionHandler 遗漏 handler、ApiClient 未实现 NETWORK_ERROR）
- **测试覆盖不足**：问题9（FallbackAiServiceTest 未验证日志）

这些分类覆盖了"其他类型问题"的范畴，但报告没有从 todo 项视角聚合回答"哪些项属于代码缺陷、哪些属于文档问题、哪些属于其他类型"——这个视角的聚合有助于执行者快速理解 todo 项的本质分布。

### 改进建议
在汇总表之前或之后增加一段总体分析，按 requirement.md 的四类标准对全部 10 项进行归类统计，例如：
- 真实代码缺陷：2 项（问题8、10）
- 误报：0 项
- OOD 文档问题：1 项（问题1）
- 其他类型（代码与设计偏离、测试覆盖不足等）：7 项（问题2~7、9）

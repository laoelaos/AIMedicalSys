## 发现的问题

### 问题1：问题5/6 缺少独立修复方向，降低可操作性

- **位置**：问题5（dependencyManagement 缺少外部 starter 声明）、问题6（Common POM 多引入 validation optional）以及「修复者指引」节
- **严重程度**：高
- **问题描述**：报告为问题2/3、问题4、问题7 分别提供了明确的修复方向（"推荐方案B"、"降 OOD 到 3.2.5"、"备选方案：将三个 ignore 条目从父 POM 移入 application/pom.xml"），但问题5 和问题6 缺少独立的修复方向说明。

  - 问题5：未指明是 **"代码向设计对齐"**（向 `pom.xml` `<dependencyManagement>` 补充 5 个 starter 条目）还是 **"设计向代码对齐"**（从 OOD §2.2 删除这 5 个条目），或维持现状（利用 Spring Boot parent BOM 兜底）。跨问题根因整合中仅模糊建议"在同一个 POM 编辑会话中依次处理"，修复者无法做出有依据的选择。
  - 问题6：报告已分析出 validation 依赖"既不必要（`MethodArgumentNotValidException` 来自 spring-web），也不符合 OOD 原则"，但未将该分析转化为可执行的结论——是从 `common/pom.xml` 中移除该依赖，还是保留并在 OOD 中补充说明？

  **可操作性缺口**：5 个问题（2/3/4/5/6/7）中，4 个有明确修复方向，2 个没有。执行者在无指引时可能做出与报告其他部分矛盾的决策（例如：在问题7 中严格对齐 OOD，在问题5 中却保留实际代码现状），造成修复方向不一致。

- **改进建议**：
  1. 问题5：推荐**代码向设计对齐**——向 `<dependencyManagement>` 补充 5 个 starter 条目（OOD 意图是将它们作为统一管理清单和门禁可视化依据）。版本无需显式指定，由 Spring Boot parent BOM 管理即可。如果选择保留现状，应在 known_issues.md 中记录该偏离及其理由（parent POM BOM 已兜底）。
  2. 问题6：推荐**从 common/pom.xml 移除 validation starter**——理由已由报告自身提供（异常类来自 spring-web，非 validation starter 专有；子模块已独立声明 validation）。移除前验证 common 模块是否有其他类型依赖于 validation starter 即可。

### 问题2：todo 覆盖声明未对 `[严重]` 断言给出核查结论

- **位置**：「todo.md 覆盖声明」节
- **严重程度**：低
- **问题描述**：todo.md 首项为 `[严重] *(无严重问题)*`。报告声明覆盖了"10 项待办事项"，但未对"无严重问题"这一断言本身给出明确的核查或确认。虽然该条目不是需要"诊断"的问题（属于声明性断言），但 requirement.md 要求"对 todo.md 中的问题清单进行定位分析"，完整覆盖应包含对该断言真实性的一句话确认。
- **改进建议**：在覆盖声明中补充："`[严重]` 项（无严重问题）：经核查，10 项诊断结论中无任何问题达到严重级别，确认该断言成立。"

### 问题3：修复顺序编排中问题5/6 的并行机会标注与问题7 的步骤归属可优化

- **位置**：「修复者指引」→「可执行修复顺序编排」节的第3步 + 并行机会说明
- **严重程度**：低
- **问题描述**：当前编排将问题5/6/7 合并为"POM 配置一轮对齐"，但问题7 的修复方案（将 ignore 条目从父 POM 移入 `application/pom.xml`）与问题5（向 dependencyManagement 补充 starter）的修改位置和风险等级不同。问题7 涉及 POM 继承行为改变和门禁效果验证，实际执行复杂度高于问题5/6，绑定在同一"一轮对齐"中可能导致修复者低估问题7 的验证工作量。

  另外，并行机会说明标注问题5/6/7 "可合并为『POM 配置一轮对齐』任务"，但问题7 的推荐备选方案（移入 application/pom.xml）是否能在不先完成问题2/3 决策的前提下安全执行？若目录结构决策后 POM 内容发生变化，则问题7 的代码修改可能被覆盖。

- **改进建议**：
  1. 在第3步中将问题7 与问题5/6 拆分标注（例如标记为"问题7（P1，需验证 CI 门禁，见 CI 门禁影响分析）vs 问题5/6（P2，纯补充/清理，无需额外验证）"），避免执行者低估问题7 的验证成本。
  2. 在问题7 的依赖说明中标注其与问题2/3 决策的关系：若选择重构为分层布局（方案A），POM 结构将大幅变动，问题7 的修改可能被覆盖，应先决策后执行。

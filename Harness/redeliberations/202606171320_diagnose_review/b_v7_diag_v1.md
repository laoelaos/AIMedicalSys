## 发现问题

### 问题1：P0 级别真实代码缺陷缺失修复方向，可操作性与优先级倒挂

**所在位置**：问题8（GlobalExceptionHandler）、问题10（ApiClient 错误拦截器）

**问题描述**：报告中唯二归类为"代码缺陷"（类型①）的问题8和问题10，均缺少「修复方向」小节——这是 10 个问题中仅有的两个无独立修复方向的问题。对比之下，P2 级别的 POM 配置偏离问题（问题5/6/7）均有详细的修复方向（包括具体要添加/移除的条目名称、备选方案、联合验证命令）。

具体缺失内容：
- 问题8（P0(Phase 1+)）：未给出 `@ExceptionHandler` 方法的代码骨架，未说明 `HttpMessageNotReadableException` → 400 时应返回哪个 `ErrorCode` 枚举值（PARAM_INVALID 还是自定义值），也未说明 `HttpMessageNotWritableException` → 500 时的响应格式是否沿用现有 `SYSTEM_ERROR`
- 问题10（P0(Phase 1+)）：未给出错误拦截器的代码骨架，未说明格式转换后应 `return Promise.resolve()` 还是 `return Promise.reject()`（问题10 证据链第3项提到"返回统一格式的 `Promise.resolve()`"但文本本身有歧义——若 `Promise.resolve` 则上游 `.then()` 分支会收到错误对象，与正常响应难以区分）。另外，未讨论 `error.response` 存在但 `error.response.status` 为特定值的情形的处理策略（如 401 是否应区分于 NETWORK_ERROR）

修复方向缺失的直接后果：这恰好是优先级最高的两个代码缺陷（P0），执行者在 Phase 1+ 场景激活时需要自行推断修复方案，而 P2 的 POM 配置问题却有完整的操作指引——形成"重要性越高，指引越少"的倒挂。

**严重程度**：高（直接影响执行者的修复效率和质量，P0 级别问题缺少可执行修复指引）

**改进建议**：
- 为问题8补充修复方向：给出 `HttpMessageNotReadableException` 和 `HttpMessageNotWritableException` 的 `@ExceptionHandler` 方法骨架，明确每个 handler 的 HTTP 状态码、ErrorCode 和响应消息
- 为问题10补充修复方向：给出错误拦截器代码骨架，明确 `error.response === undefined` 时的返回值格式和 Promise 控制流（resolve vs reject），至少讨论 401/403 等 HTTP 错误码是否应在此层统一处理

### 问题2：问题9 的日志测试方案与 known_issues.md K3 存在隐蔽冲突

**所在位置**：问题9「修复方向」中的 ListAppender 代码骨架（line 401-418）及注意事项（line 421）

**问题描述**：报告推荐使用 Logback `ListAppender` 为 `FallbackAiService.handleEmptyDelegates()` 的日志行为追加测试，并在代码骨架中示范了首次调用验证 ERROR 级别、第二次调用验证 WARN 级别的断言逻辑。然而，报告自身引用 known_issues.md K3（line 381）指出："known_issues.md K3 记录的 ERROR 日志触发时机偏差（首次调用 vs 启动期）"。

这意味着：当前代码的 ERROR 日志触发时机（首次调用时）已被确认为已知偏差，正确的语义应当是"启动期输出 ERROR 日志"。如果按照当前代码行为编写测试断言（首次调用 → ERROR），待 K3 偏差被修复后（改为启动期触发 ERROR），该测试将失效——要么测试失败（ERROR 日志不见于首次调用），要么测试通过但不再覆盖实际的修复后行为。

报告虽然提醒了"注意首次调用触发 ERROR、后续调用触发 WARN 的语义差别"，但未提醒"当 K3 被修复后，此测试需要同步更新断言逻辑"这一关键风险。执行者可能直接复用代码骨架中的断言模式，在测试中固化一个已知错误行为。

**严重程度**：中（测试方案未考虑已知偏差的未来修复，可能导致测试用例在偏差修复后失效）

**改进建议**：
- 在 ListAppender 代码骨架后追加注释或警告：⚠️ 当前代码的 ERROR 日志触发时机（首次调用）已被 known_issues.md K3 记录为已知偏差。推荐的测试策略是对 K3 修复后的期望行为（启动期输出 ERROR）编写断言，而非对当前代码行为编写断言，以避免未来失效
- 或进一步建议：在测试前先触发启动期事件（如通过构造函数注入或反射调用 `@PostConstruct` 方法），使日志行为对齐设计语义，再基于对齐后的行为编写断言

### 问题3：问题5 证据链计数不精确，影响可信度

**所在位置**：问题5「证据链」第1项和第2项（line 203-206）

**问题描述**：报告称"OOD 骨架代码中 `<dependencyManagement>` 包含 11 个条目（6 内部 + 5 starter + springdoc + h2 + security + validation + test）"，但括号内枚举项之和为 16（6+5+1+1+1+1+1），且 "5 starter" 与后续单独列出的 "security + validation + test" 存在重复计数——OOD 的 5 个 starter 正是 web、data-jpa、security、validation、test。同样，报告称"实际 `<dependencyManagement>` 仅含 8 个条目（6 内部 + springdoc + h2）"，但实际 POM 的 dependencyManagement 还包含 3 个业务模块（patient、doctor、admin），总计应为 11 项而非 8 项。

A 方内部审议（a_v7_challenge_v1.md）已识别此计数不精确问题但标注为"轻微 —— 不影响核心结论（5 个 starter 条目确实缺失）"。该判断在技术层面成立——XML 内容对比已独立完成证明。但从审查报告的质量标准看，证据链是支撑报告可信度的核心部分，可验证的计数错误（即使不影响结论）会削弱读者对分析严谨性的信心。修复者可能质疑"计数都不准，其他数据是否也经得起验证？"

**严重程度**：低（事实性错误但核心结论不受影响，与 A 方审议结论一致）

**改进建议**：
- 将 OOD 骨架条目数更正为 13（6 内部 + 5 starter + springdoc + h2）
- 将实际 `pom.xml` 条目数更正为 11（6 内部 + 3 业务模块 + springdoc + h2）
- 修正后需确认枚举数与计数一致

**所在位置**：未发现（全报告缺漏）

## 总体评价

该报告经过 8 轮内部修订，在证据链完整性、根因分析深度、跨问题整合和决策引导方面已达到良好质量水平。修复顺序编排考虑了依赖关系和并行机会，偏离容忍度标注提供了执行者判断依据。

主要发现的 4 个质量问题中：
- **问题1**（P0 修复方向缺失）属于严重问题，是 AI 内部审议（关注证据充分性、逻辑完整性、覆盖完备性）未覆盖的可操作性维度——内部审议验证了"what is wrong"的准确性，但未评估"how to fix"的充分性
- **问题2**（测试方案与已知偏差冲突）属于中等问题，是内部审议未深入涉及的修复建议副作用分析
- **问题3**（计数不精确）属于低等问题，已由 A 方内部审议识别但未修复
- **问题4**（排除项未显式确认）属于低等问题，是需求响应完备性的补充建议

核心改进建议：为 P0 代码缺陷补充修复方向，使优先级与可操作性对齐；解决测试方案与 known_issues 偏差的冲突风险；修正已知的计数错误。

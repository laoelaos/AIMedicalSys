## 问题1：优先级标签与执行顺序建议之间存在误导性张力

**所在位置**：汇总表（第594-595行，Priority列）与「可执行修复顺序编排」（第616-624行）

**问题描述**：
问题8和问题10的优先级标注为 `P0(Phase 1+)/P2(Phase 0)`，其Phase 0等级为**P2**（低优先级）。但「可执行修复顺序编排」明确推荐这两个问题**最先执行**（"完全独立，可第一时间并行分配"）。与此同时，问题7标注为**P1**（中优先级）却被安排在更后的步骤。

这种错位导致：
1. 扫描阅读者先看汇总表形成"P2=不重要"的初步印象，与后续"先执行"的指引矛盾
2. P1项被排在P2项之后执行，削弱了优先级体系的基本语义（高优先级反而后做）
3. 执行者可能绕过优先级栏直接跳到执行顺序图，形成"只看图不看表"的信息损耗

**严重程度**：中

**改进建议**：
选项A（推荐）：将问题8/10的Priority列改为 `P2[Phase 0] → P0(Phase 1+)`（优先标注当前Phase的有效等级，未来升档条件次之），同时在汇总表底部或执行顺序图中增加注解："P2标注仅反映Phase 0无运行时影响，不表示不推荐优先执行——因其修复独立无阻塞，列为优先并行项。"

选项B：在汇总表优先级列保留现有双值格式，但增加一列「执行优先级」直接标注修复顺序中的实际优先度（如"优先""决策后""随时"），消除语义混淆。

## 问题2：问题10修复方案缺乏TypeScript类型兼容性分析

**所在位置**：问题10「修复方向」（第510-548行），代码骨架

**问题描述**：
修复方案将错误拦截器从 `return Promise.reject(error)` 改为 `return Promise.resolve({ code, message })`，同时保持成功拦截器返回 `response.data.data`。方案未分析以下类型的兼容性影响：

- 如果 `api` 实例的 TypeScript 返回类型泛型定义为 `Promise<T>`（成功时T = `response.data.data` 的类型），那么错误拦截器返回 `{ code: string, message: string }` 会导致 TypeScript 推断的联合类型为 `T | { code: string, message: string }`。当前项目前端的消费方代码是否兼容这一联合类型？是否所有调用方都需要额外的类型断言？
- 其他HTTP错误分支（401/403/其他）同样返回 `Promise.resolve({ code, message })`，进一步扩大了联合类型的成员数。
- 报告建议"所有分支统一使用 Promise.resolve() 保持与成功拦截器一致的返回格式（成功分支 `return response.data.data`）"——但成功分支返回的是 `data.data`（嵌套解包后的业务数据），而错误分支返回的是 `{ code, message }`（模拟 Result 结构）。两者格式并不一致，报告却声称"一致"。

**严重程度**：中

**改进建议**：
在问题10修复方向中补充TypeScript类型兼容性分析小节，至少说明：
1. 当前前端代码中所有 `api.get/post` 调用的消费方（现仅有健康检查和占位页面）是否受联合类型影响
2. 是否需要对 Axios 实例的响应类型泛型做约束（如 `AxiosInstance` 的 `get<T>()` 签名是否需调整）
3. 明确 `Promise.resolve({ code, message })` 与 `Promise.resolve(response.data.data)` 的格式差异，以及如何保证前端调用方感知的错误类型与业务返回类型之间不会产生类型污染（如使用 discriminated union 或类型守卫）

## 问题3：问题9的测试备选方案（优先方案）缺乏可操作性的代码指引

**所在位置**：问题9「修复方向」→ K3冲突警告→「优先方案」（第458-461行）

**问题描述**：
报告为known_issues.md K3（日志触发时机偏差）提供了两种测试策略：
- **优先方案**（推荐）："在测试中先触发 `@PostConstruct` 启动期逻辑（如通过反射调用或 Spring Test 上下文刷新），使日志行为对齐设计语义"

优先方案虽标为推荐，但仅提供了"通过反射调用或Spring Test上下文刷新"的方向性描述，没有任何代码示例或实现步骤。这与ListAppender方案（完整18行代码骨架 + 清理逻辑 + 断言模式）形成鲜明反差。执行者无法直接从报告中获取"怎么做"的信息，需要自行调研反射调用 `@PostConstruct` 的方法或Spring Test上下文刷新的配置方式。

**严重程度**：低

**改进建议**：
为优先方案补充具体实现指引，至少包括：
1. 如果选择反射方式：给出 `MethodHandle` 或 `ReflectionTestUtils.invokeMethod()` 的调用骨架
2. 如果选择Spring Test上下文刷新：说明如何通过 `@SpringBootTest` 配合 `@DirtiesContext` 或在 `@BeforeEach` 中重新初始化Bean来触发 `@PostConstruct`
3. 或者如果确认优先方案在当前Spring Boot 3.2.5 + JUnit 5环境中实现复杂度较高，可降级优先方案为备选方案，避免为执行者提供"推荐但无法直接使用"的指引

## 当前审查结果

### 问题1（中严重度）：问题9的测试推荐方案基于不存在的 `@PostConstruct` 方法
- **位置**：`a_v10_diag_v1.md` 问题9章节，"优先方案"方式A（Line 461-504）
- **问题描述**：报告推荐通过 `ReflectionTestUtils.invokeMethod(service, "init")` 反射调用 `@PostConstruct` 方法触发启动期日志，但经代码验证 `FallbackAiService.java` 中不存在任何 `@PostConstruct` 注解的方法。代码示例中的 `// 假设 @PostConstruct 方法名为 init` 属于虚构前提，直接执行将抛出 `NoSuchMethodException`。
- **改进建议**：方式A需先声明"当前代码无 `@PostConstruct` 方法"的前提条件，说明需先修复 K3 偏差再按此方案测试；或改为更务实的方案——直接对当前代码行为（首次调用→ERROR、后续→WARN）编写日志断言。

### 问题2（中严重度）：执行顺序编排存在内部矛盾
- **位置**：`a_v10_diag_v1.md` Line 732-733（"完全独立于第1步"）与 Line 749（"依赖第1步决策"）
- **问题描述**：第3步标题标注"完全独立于第1步"，但第3步包含的问题7条目注明"应先决策后执行"。标题级的矛盾降低读者对编排可靠性的信任，扫读者可能遗漏依赖关系。
- **改进建议**：将第3步标题改为"除标注依赖关系外，其余大部分任务可与第1步并行"，并在问题7条目后标注"[依赖第1步决策，决策后方可执行]"。

### 问题3（低严重度）：问题5的证据链计数分类不一致
- **位置**：`a_v10_diag_v1.md` Line 205-207
- **问题描述**：OOD 骨架条目计数用"6 内部"，实际 POM 条目计数用"6 内部 + 3 业务模块"，两种计数使用不同分类粒度，语义精确性受损。
- **改进建议**：按统一口径重新表述，如"OOD 骨架示例代码列出 6 个内部模块 + 5 starter + springdoc + h2 共 13 项；实际 POM 列出 9 个内部模块 + springdoc + h2 共 11 项"。

### 问题4（低严重度）：问题8修复方案中 `PARAM_INVALID` 的选择缺少论证
- **位置**：`a_v10_diag_v1.md` Line 358-378
- **问题描述**：将 `HttpMessageNotReadableException`（请求体 JSON 格式错误）映射到 `GlobalErrorCode.PARAM_INVALID`（"参数校验失败"），但 OOD 仅指定 HTTP 400 状态码，未指定具体 ErrorCode。"请求体格式错误"与"参数值非法"语义不完全一致，前端可能需区分两条文案。
- **改进建议**：在"关键决策点"中补充 ErrorCode 选择的语义匹配度讨论，或标注"待前端确认是否需要区分两类 400 错误"。

### 问题5（低严重度）：问题10未识别成功拦截器本身的类型不一致
- **位置**：`a_v10_diag_v1.md` Line 661 及"TypeScript 类型兼容性分析"章节（Line 629-661）
- **问题描述**：成功拦截器已在返回两种不同类型（`T` vs `ApiResult<T>`），这是修复前的既有缺陷。报告将其标注为"连带变更"而非"需要独立评估的影响点"，可能低估修复工作量。
- **改进建议**：明确将成功拦截器的类型不一致列为"既有设计问题"，要求修复者在实施问题10修复前先完成对现有消费方代码的扫描，再统一选择类型方案。

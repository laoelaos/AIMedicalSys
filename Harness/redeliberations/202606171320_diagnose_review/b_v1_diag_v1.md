### 问题2：缺少明确的修复优先级排序

| 项目 | 内容 |
|------|------|
| **所在位置** | 「汇总 — 修复者指引」及整份报告 |
| **问题描述** | 报告按修复目标文件类型（代码/OOD文档/测试）对 10 个问题分组，但未给出明确的优先级排序。对于诊断报告的使用者（执行者），需要知道"先修哪个、后修哪个"。从可操作性视角看，10 个问题并列呈现增加了执行者的决策成本。例如：问题8（序列化异常处理器缺失）和问题10（前端错误拦截器）直接影响运行时行为，优先级应高于问题5（dependencyManagement 列表不完整，功能无影响）；问题2/3（目录结构偏差）涉及重大重构决策，可能需要先决策再行动，与简单配置修复不在同一优先级。 |
| **严重程度** | **一般** — 降低可操作性，执行者需要自行判断处理顺序 |
| **改进建议** | 增加明确的优先级排序（如 P0/P1/P2 或 高/中/低），考虑因素包括：(1) 运行时影响范围（是否改变 API 行为），(2) 修复代价与风险（目录重构 vs 简单配置修改），(3) 对后续阶段的阻塞程度。 |

### 问题3：问题4 的影响范围遗漏 OOD §1.4 的版本引用

| 项目 | 内容 |
|------|------|
| **所在位置** | 问题4「影响范围」 |
| **问题描述** | 报告正确指出了 OOD §2.1 父 POM 骨架中 Spring Boot 版本（3.3.0）与代码（3.2.5）的不一致，并给出"升代码到 3.3.0 或降 OOD 到 3.2.5"的建议。但 OOD §1.4（line 41）运行时环境要求表中同样引用了"Spring Boot 3.3.0 最低要求"。若选择"降 OOD 到 3.2.5"，§1.4 也需要同步更新（否则会出现「3.3.0 最低要求」与「实际使用 3.2.5」的内部矛盾）。当前影响范围分析未覆盖此关联。 |
| **严重程度** | **轻微** — 影响范围不完整，但不会导致错误的修复方向 |
| **改进建议** | 在影响范围中补充 §1.4 的同步更新需求，或明确指出该处标注的是 JDK 17+ 与 3.3.0 的关联（实际 3.2.5 同样兼容 JDK 17+），以便决策者全面评估。 |

### 问题4：问题7 "代码缺陷"分类可商榷

| 项目 | 内容 |
|------|------|
| **所在位置** | 问题7「分类」 |
| **问题描述** | 报告将问题7（maven-dependency-plugin 的 ignoredUnusedDeclaredDependencies 额外包含 business 模块）分类为「代码缺陷」。但该问题的本质是**配置范围过宽**（extends OOD 设计约定的豁免范围），与问题5（dependencyManagement 缺少 starter 声明）、问题6（Common POM 多引入 validation optional）同为「代码与 OOD 的设计偏离」，而非典型的会导致运行时失败或错误行为的代码缺陷。原文影响范围分析也指出"当前无运行时影响"——与"代码缺陷"的分类存在微妙矛盾。 |
| **严重程度** | **轻微** — 不影响修复方向，但可能误导执行者对问题紧迫性的判断 |
| **改进建议** | 建议将分类调整为「代码与 OOD 的设计偏离」或「配置偏差」，或保留「代码缺陷」但在影响范围中更明确地说明这是一个"潜在风险"（可能掩盖未来依赖问题）而非"当前缺陷"。 |

| 项目 | 内容 |
|------|------|
| **所在位置** | 整份报告 |

### 问题6：known_issues.md 引用不充分

| 项目 | 内容 |
|------|------|
| **所在位置** | 问题9「证据链」 |
| **问题描述** | 报告在数据来源中列出了 known_issues.md，但在问题9（FallbackAiServiceTest 未验证日志输出）的分析中未引用 known_issues.md 中的 K3 条目（"FallbackAiService 空委托 ERROR 日志触发时机为首次调用而非启动期"）。K3 与问题9直接相关——它记录了错误日志的具体触发行为与文档描述之间已存在的细微偏差，这个已知偏差是测试需要额外关注日志输出的另一个理由。引用 K3 可以使证据链更完整，也可帮助执行者理解该问题存在两层含义（测试覆盖不足 + 已知的文档描述偏差）。 |

| **改进建议** | 在问题9的证据链或根因分析中引用 known_issues.md K3，说明该已知偏差与日志测试覆盖之间的关系。 |

### 问题1（中严重度）：问题9的测试推荐方案基于不存在的 `@PostConstruct` 方法

**位置**：`a_v10_diag_v1.md` 问题9章节，"优先方案"方式A（Line 461-504）

**问题描述**：
报告在问题9的"优先方案"中提供了完整的代码示例，通过 `ReflectionTestUtils.invokeMethod(service, "init")` 反射调用 `@PostConstruct` 方法来触发启动期日志。代码示例明确写有 `// 假设 @PostConstruct 方法名为 init`。

**经实际代码验证**：`FallbackAiService.java` 中**不存在**任何 `@PostConstruct` 注解的方法。`AtomicBoolean firstEmptyDelegateCall = new AtomicBoolean(true)` 在字段声明处直接初始化，**没有任何 `init()` 或 `@PostConstruct` 启动期回调**。当前代码行为是首次 `handleEmptyDelegates()` 调用时输出 ERROR——也就是运行期调用时触发，而非启动期。

**问题本质**：
- 报告虽然正确识别了 K3 已知偏差（"代码行为与设计语义偏离——首次调用而非启动期"），但在测试方案推荐中**未披露"添加 `@PostConstruct` 方法是使用该测试方案的前提条件"**。
- 代码示例中 `ReflectionTestUtils.invokeMethod(service, "init")` 在现有代码上直接执行会抛出 `NoSuchMethodException`。
- 修复者若按此方案编码，要么因方法不存在而卡住，要么需要自行推断要先修改源代码——但报告未做此说明。

**严重程度**：中。因为这不是纯理论讨论——报告给出了可直接"复制"的代码骨架，但该骨架引用的方法不存在。对于需要按报告行动的执行者，这是一个实际障碍。

**改进建议**：
1. 在方式A的"前提条件"中明确声明：当前代码无 `@PostConstruct` 方法，需先修复 K3 偏差（在 `FallbackAiService` 中补充 `@PostConstruct` 初始化方法，将 ERROR 日志逻辑移到启动期），然后才能按此方式编写测试。
2. 或者，将方式A改为更务实的方案：直接对当前代码行为编写日志断言（首次调用→ERROR、后续→WARN），并在断言旁注释 K3 依赖关系。

### 问题4（低严重度）：问题8修复方案中 `PARAM_INVALID` 的选择缺少论证

**位置**：`a_v10_diag_v1.md` Line 358-378

**问题描述**：
报告将 `HttpMessageNotReadableException`（请求体 JSON 格式错误）映射到 `GlobalErrorCode.PARAM_INVALID`（code="PARAM_INVALID", message="参数校验失败"），但未提供 OOD 文档依据——OOD 仅指定 HTTP 400 状态码（§5.1），未指定具体 ErrorCode。

"请求体 JSON 格式错误"与"参数校验失败"在语义上不完全一致：前者是反序列化失败（JSON 语法错误、字段类型不匹配），后者是业务参数校验未通过（如必填字段为空）。将两者映射到同一个 `PARAM_INVALID` 意味着前端无法区分"请求格式错误"与"参数值非法"——这可能与前端错误处理策略相关（需在前段错误提示中分别展示"请求格式错误"和"参数校验失败"两条不同文案）。

**严重程度**：低。当前 Phase 0 无触发路径，此决策在 Phase 1+ 前纠正也来得及。但报告将 `PARAM_INVALID` 作为确定方案提出，建议标注为"初步建议，需前端团队确认"。

**改进建议**：在"关键决策点"中补充讨论此 ErrorCode 选择的语义匹配度，或标注"待前端确认是否需要区分两类 400 错误"。

**问题描述**：

```typescript

    if (response.data?.code === 'SUCCESS') {
      return response.data.data   // 类型: T

    return response.data           // 类型: ApiResult<T>
```

成功拦截器已经在返回 **两种不同类型**（`T` vs `ApiResult<T>`），这是修复前的**既有缺陷**。报告的修复方案中，方案B 要求将成功拦截器统一改为 `return response.data`，这本身就是一个独立的设计决策——需要对所有现有消费方进行扫描以确保兼容。

报告将此标注为"连带变更"而不是"需要独立评估的影响点"，可能低估了修复的工作量。

**严重程度**：低。不改变问题10本身的诊断结论（错误拦截器确实未实现 NETWORK_ERROR），但影响修复方案的实施范围评估。

**改进建议**：明确将成功拦截器的类型不一致列为"既有设计问题"，并要求修复者在实施问题10修复前先完成对现有消费方代码的扫描（确认 `response.data.data` 的用法分布），再统一选择类型方案。

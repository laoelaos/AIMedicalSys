## 问题列表

### 问题 1（重要）："连续"语义定义与实现机制不匹配

- **所在位置**: 4.1 节「第二层：登录失败计数（LoginAttemptTracker）」实现方式描述及「"连续"语义定义」
- **问题描述**: 文档为 LoginAttemptTracker 定义了严格的"连续失败"语义——"中间不得插入任何非登录请求或登录成功，非登录请求均视为中断连续序列"。但描述的实现仅为 `ConcurrentHashMap<String, AttemptRecord>`，其中 AttemptRecord 仅记录失败次数和首次失败时间戳。这套数据结构和机制**无法检测**"非登录请求插入"这一中断条件，因为 LoginAttemptTracker 仅在 `AuthServiceImpl.login()` 中被调用，对非登录请求完全无感知。实际可实现的机制仅为"基于时间窗口的累计失败计数"，而非文档定义的"严格连续"语义。
- **迭代溯源**: 第 8 轮审查（iteration_history.md 问题 9）要求补充语义定义，产出方已在 v10 修订中补充了定义文本，但该定义与实现机制仍然不匹配——解决了"未定义"的表象，未解决"定义与实现矛盾"的实质。
- **改进建议**: 二选一：
  (a) 简化"连续"语义定义为"时间窗口内累计失败次数"，移除"中间不得插入非登录请求"的表述，使定义与 ConcurrentHashMap + count/timestamp 的实现一致；
  (b) 若坚持严格"连续"语义，需设计额外检测机制（如 LoginAttemptTracker 需感知非登录请求、或在每个 Filter 中插入心跳重置逻辑），并在文档中描述具体的实现方案和数据结构变更。

### 问题 2（一般）：刷新端点未检查账户锁定状态，形成绕过路径

- **所在位置**: 3.1.3 节 Token 刷新流程、4.1 节第三层「临时账户锁定」
- **问题描述**: 当账户因连续登录失败触发锁定后（ACCOUNT_LOCKED），`AuthServiceImpl.login()` 的第一步即拒绝登录。但 `POST /api/auth/refresh` 刷新流程中（3.1.3 节步骤 5-7）仅检查了用户 enabled/deleted 状态和 tokenVersion，**未检查账户锁定状态**。这意味着攻击者即使触发了账户锁定，只要持有有效的 Refresh Token，仍可通过刷新端点持续获取新的 Access Token，使账户锁定保护形同虚设。
- **改进建议**: 在 3.1.3 节步骤 5（从 DB 加载用户最新状态）之后增加账户锁定检查。若用户被锁定，应拒绝刷新并返回 ErrorCode.ACCOUNT_LOCKED。如果此为有意设计决策（锁定仅限制登录、不限制已签发的令牌使用），应在 11 节「设计决策」表中明确记录此选择及其安全理由。

### 问题 3（一般）：版本号不一致可能导致文档追溯混淆

- **所在位置**: 文档标题（第 1 行 "Phase 1 包 A/B/D 统一修复与包 B OOD 设计方案（v9）"）vs 修订说明最后一节（Section 20 "修订说明（v10）"）vs 文件名（a_v9_copy_from_v8.md）
- **问题描述**: 文档标题声明为 v9，但修订说明从 v2 递增到 v10（共 10 轮修订记录），与标题的 v9 不匹配。文件名暗示当前文档为 v9 且从 v8 复制而来，但修订说明 v10 的内容对应的是第 8 轮审查的修复（iteration_history.md 中第 8 轮即为最后轮次），版本号对应关系混乱。在多人协作或后续迭代追溯时，此不一致可能导致"当前版本究竟是 v9 还是 v10"的混淆。
- **改进建议**: 统一版本号体系。建议将修订说明中最后一节编号与文档版本号对齐，或明确说明修订说明编号与文档版本号之间的对应关系。

### 问题 4（一般）："清除 SecurityContext 作为安全最佳实践"的说法不准确

- **所在位置**: 3.1.6 节步骤 10：「清除当前请求 SecurityContext（作为安全最佳实践），客户端可继续使用旧 Access Token 至其自然过期」
- **问题描述**: SecurityContext 是请求作用域的（request-scoped），当前请求处理完毕后 Spring Security 的 `SecurityContextHolderFilter` 或 `SecurityContextPersistenceFilter` 会自动清除。在 `AuthServiceImpl.changePassword()`（Service 层，位于 Filter 链下游）手动调用 `SecurityContextHolder.clearContext()`，发生在当前请求即将完成之际，实际上不产生额外的安全效果——当前请求后续已无核心业务代码需要访问 SecurityContext。将其描述为"安全最佳实践"夸大了此操作的实际价值，可能导致读者误认为这是密码变更后必要的安全步骤。
- **改进建议**: 移除"作为安全最佳实践"的表述，改为准确描述此步骤的真实目的，例如"清除当前请求 SecurityContext 确保后续同一请求中的其他 Filter 不会误用变更前用户状态（若存在），但当前请求的 Access Token 仍有效至自然过期"；或者直接删除此步骤，因为其效果在实践层面可忽略。

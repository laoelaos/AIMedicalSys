## 发现的问题

### 🔴 问题 1：登录流程存在时序侧信道攻击风险（安全遗漏）

**问题描述**：登录流程步骤 5（用户名不存在）不执行 BCrypt 密码比对，而步骤 7（密码不匹配）执行 `passwordEncoder.matches()`。BCrypt（Strength 10）比对耗时通常在 100ms 以上，而步骤 5 仅查询 DB 后直接返回。攻击者可通过响应时间差异枚举有效用户名，部分抵消统一错误消息的防御效果。当前 IP 限流（5 次/10 秒）仅降低攻击速率，不能消除时序差异。

**所在位置**：3.1.1 节步骤 5 vs 步骤 7

**严重程度**：重要

**改进建议**：在步骤 5 中用户名不存在时，对虚拟哈希值执行 dummy BCrypt 比对以消除时序差异（固定延迟方案），使步骤 5 和步骤 7 的执行时间无法区分。或在设计决策中明确承认此风险并推迟到 Phase 2。

---

### 🔴 问题 2：密码变更后旧 Refresh Token 撤销机制未设计（关键遗漏）

**问题描述**：4.2 节"用户可主动撤销"声称"用户修改密码后，所有已签发的 Refresh Token 在下次使用时因密码变更而被拒绝"，但立即补充"（需设计密码变更后标记机制）"。此机制在 Phase 1 无 Refresh Token 黑名单的情况下是唯一的撤销路径，但文档仅将其作为 TODO 占位，未给出任何具体设计（如 `tokenVersion` 字段、claims 校验逻辑等）。这给读者以虚假的安全感。

**所在位置**：4.2 节「Refresh Token 的安全补偿策略」末条

**严重程度**：重要

**改进建议**：二选一——(a) 设计具体机制：User 实体新增 `tokenVersion`（Integer），Refresh Token claims 携带此版本号，刷新时比对，密码变更时递增版本号使旧 Refresh Token 失效；或 (b) 删除该条声明，明确说明 Phase 1 中密码变更后旧 Refresh Token 仍可被使用，承认此风险。

---

### 🔴 问题 3：密码变更成功后前端恢复流程未定义（关键遗漏）

**问题描述**：3.4 节定义密码变更成功后清除 `passwordChangeRequired` 标记，但未定义前端的后置流程：前端应在密码变更成功后调用哪些 API（`GET /api/auth/me`? `GET /api/menu/tree`?），以及如何跳转到正常页面。7.4 节的前端补偿机制也未覆盖此场景。

**所在位置**：3.4 节「密码变更强制策略」、7.4 节「包 D 前端对包 B 的补偿机制」

**严重程度**：重要

**改进建议**：在 3.4 节补充密码变更后的完整前后端交互流程：PUT /api/auth/password 成功后 → 前端清除 passwordChangeRequired 标记 → 调用 GET /api/auth/me 刷新用户信息 → 调用 GET /api/menu/tree 获取菜单 → 跳转到首页。

---

### 🔴 问题 4：菜单递归展平方案与路由注册策略存在冲突（可落地性缺陷）

**问题描述**：8.2 节 H5 提出 `convertMenusToRoutes` 改为递归展平版本，生成展平路由表。但现有代码（`menu.ts:99`）将所有路由通过 `routerInstance.addRoute('Layout', route)` 注册为 `Layout` 路由的子路由。对于递归展平生成的完整路径（如 `/parent/child/grandchild`），此注册方式在 Vue Router 中的实际行为未明确：是拼接为 `/Layout/parent/child/grandchild` 还是保持绝对路径？递归展平产生的 path 是相对路径还是绝对路径？现有 `path === '/'` 等跳过滤逻辑如何与递归兼容？路由 name 唯一性如何保证？

**所在位置**：8.2 节 H5、「修订说明（v5）」P10

**严重程度**：重要

**改进建议**：明确以下三点：(a) 路由注册策略：展平路由是以根路由（`router.addRoute`）还是以 Layout 子路由（`router.addRoute('Layout', ...)`）注册，path 格式是绝对路径还是相对路径；(b) 与现有跳过滤条件的兼容关系；(c) name 唯一性保证策略（permission code 已基本唯一，但需要确认跨数据库无重复）。

---

### 🔴 问题 5：前端缺少 PASSWORD_CHANGE_REQUIRED 错误码处理（关键遗漏）

**问题描述**：7.4 节（前端补偿机制）覆盖了 401 自动刷新、登出清理、多 Tab 并发互斥，但未覆盖 HTTP 403 + ErrorCode `PASSWORD_CHANGE_REQUIRED` 的处理。如果用户在 `passwordChangeRequired=true` 状态下手动导航到 URL（绕过登录后的前端跳转），PasswordChangeCheckFilter 返回 403，前端 axios 拦截器缺少对应的处理规则——既不会弹出修改密码提示，也不会跳转到修改密码页。

**所在位置**：7.4 节「包 D 前端对包 B 的补偿机制」

**严重程度**：重要

**改进建议**：在 7.4 节新增一条：axios 响应拦截器识别 HTTP 403 + ErrorCode `PASSWORD_CHANGE_REQUIRED` 时，重定向到密码修改页面并终止原始请求，不展示通用 403 错误提示。

---

### 🟡 问题 6：登录流程步骤 7 的失败计数维度未指定（可落地性缺陷）

**问题描述**：3.1.1 节步骤 5 明确指定"IP 维度"的失败计数，步骤 6 明确指定"用户名和 IP 双维度"，但步骤 7（密码不匹配）仅说"递增 LoginAttemptTracker 失败计数"，未指定维度。由于步骤 7 执行时用户名已确定（User 已加载），应至少递增用户名维度的计数。

**所在位置**：3.1.1 节步骤 7

**严重程度**：一般

**改进建议**：将步骤 7 的措辞补充为"递增 LoginAttemptTracker **用户名维度**的失败计数"。

---

### 🟡 问题 7：`JwtTokenProvider` 的 `@PostConstruct` 启动验证依赖未显式声明的 Bean 类型（可落地性缺陷）

**问题描述**：4.7 节要求 `JwtTokenProvider` 在 `@PostConstruct` 中执行 JWT_SECRET 启动验证。但 2.1 节目录结构将 `JwtTokenProvider` 列在 `auth/jwt/` 下，文档未说明其 Spring Bean 注册方式（`@Component`/`@Service`/`@Bean in AuthModuleConfig`）。若以非 Spring-managed 方式实例化（如手动 `new`），`@PostConstruct` 不会生效，密钥验证逻辑将被绕过。

**所在位置**：4.7 节「启动验证逻辑」、2.1 节目录结构

**严重程度**：一般

**改进建议**：在 1.3 节核心抽象表或 2.1 节目录结构中对 `JwtTokenProvider` 标注其 Spring stereotype（如 `@Component`），或将启动验证逻辑移至 `AuthModuleConfig.@PostConstruct` 中。

## 整体质量评价

该文档经过 4 轮迭代修正，已在内部审议（设计-验证循环）中修复了大量事实性错误、内部矛盾和技术可行性问题。需求覆盖完整、模块划分清晰、核心流程定义详尽。仍存在的质量问题集中在三个维度：

1. **端到端完整性**：登录时序攻击防护、密码变更后前端恢复流程、PASSWORD_CHANGE_REQUIRED 前端处理等"边界到边界"的完整流转路径未定义
2. **安全深度**：时序侧信道攻击未被纳入安全设计考虑
3. **可落地细节**：菜单递归化与路由注册的兼容性、Bean 生命周期假设等实现细节需进一步澄清

以上 7 个问题均不影响架构正确性，但若不解决将影响最终的集成质量和安全性。

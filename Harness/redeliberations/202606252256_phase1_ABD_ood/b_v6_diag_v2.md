## 二、深度与完整性评估（本次新增）

### 2.1 可直接编码性

**优点**：
- 核心抽象一览（1.3 节）明确了每个接口/类的类型形态和职责，编码者可直接映射到 Java 代码
- 目录结构（2.1 节）精确到文件名，无歧义
- Filter 行为契约（3.3 节）以伪代码步骤描述，可逐行翻译为 Java 实现
- DTO 定义（5.2 节）直接使用 Java record 语法，无歧义
- 并发设计（第 9 节）给出了具体的数据结构和锁策略

**缺口**（部分已在 v1 报告中检出）：
- 密码变更流程（3.1.6 缺失）无独立流程描述，实现者需跨节拼接 3.4 节策略 + 7.4 节恢复流程 + 10.2 节 ErrorCode → 降低编码效率
- AuthenticationEntryPoint 的 ACCOUNT_DISABLED 区分逻辑未定义 → 实现者需自行推断
- CRUD Controller 层面缺少数个典型异常场景的处理说明（如更新不存在的菜单 ID、删除仍有子菜单的父菜单等）

### 2.2 异常场景与边界条件

**已覆盖**：
- 用户名不存在（步骤 5 → dummy BCrypt）
- 用户被禁用（步骤 6、Filter 层）
- 用户被删除（Filter 层静默跳过）
- 密码不匹配（步骤 7）
- 密码变更后旧 Refresh Token（tokenVersion 机制）
- 登出清理兜底（前端 finally 块）
- 多 Tab 并发刷新互斥（Promise 单例模式）
- PASSWORD_CHANGE_REQUIRED 阻断与前端重组

**未覆盖**：
- 同时修改密码的场景：管理员在用户正在操作时标记密码过期 → 用户下一个请求被阻断，但当前页面操作未保存 → 缺少用户体验说明
- 删除仍有子菜单的父菜单时应返回的错误码和校验逻辑
- 菜单 CRUD 中，当 `parentId` 指向不存在的菜单时的错误处理
- 登录锁定状态（ACCOUNT_LOCKED）在锁定时间到期后的自动解锁机制仅有"惰性清除"描述，未说明首次解锁尝试时检查锁定的具体逻辑

### 2.3 接口定义完整性（下游消费者视角）

接口清单（6.1 节）和响应格式（6.2-6.3 节）完整定义了对前端的所有请求/响应契约。主要缺口：
- `/api/auth/logout` 的请求体可选性对前端代码产生分支（携带 vs 不携带 refreshToken），未给出推荐策略
- 刷新端点失败时的具体错误码链未定义（如 Refresh Token 过期返回何种 ErrorCode，是 TOKEN_REFRESH_FAILED 还是 UNAUTHORIZED）
- `GET /api/menu/tree` 无响应格式定义（虽 MenuResponse 在 5.2 有定义，但缺少 JSON 示例）

### 🟡 重要

#### 10. 登录流程步骤 6 存在时序侧信道缺口

**问题描述**：v6 修订说明第 1 条为步骤 5（用户名不存在）补充了 dummy BCrypt 比对，消除了与步骤 7（密码错误，真实 BCrypt）的时序差异。但步骤 6（用户已禁用或已删除）未执行任何 BCrypt 比对，仅递增计数器后直接返回。导致三种场景的响应时间：
- 步骤 5：dummy BCrypt → 中耗时（~50ms）
- 步骤 6：无 BCrypt → 低耗时（<5ms）
- 步骤 7：真实 BCrypt → 高耗时（~50ms）

攻击者可通过响应时间差异区分"用户存在但已禁用"（快）与"用户不存在"（中）与"密码错误"（中，与 step 5 接近但可统计区分）。虽比 v5 的原始缺陷（仅区分"存在"与"不存在"）有所改善，但步骤 6 的时序差异仍然存在。

**所在位置**：3.1.1 节步骤 6

**严重程度**：重要

**改进建议**：在步骤 6 中增加 dummy BCrypt 比对，消除与步骤 5/7 的时序差异。或在设计决策中明确承认此残余风险（已禁用账号枚举攻击面较小），并记录计划在 Phase 2 解决。

---

#### 11. `LoginResponse` 中 `tokenType` 与 `expiresIn` 的定位不明确

**问题描述**：5.2 节 `LoginResponse` 包含 `String tokenType`（固定 "Bearer"）和 `long expiresIn`（access token 过期秒数），但在 3.1.1 节步骤 10-11 的流程描述以及 6.2 节 JSON 响应示例中均为涉及这两个字段的业务使用说明。实现者不知 `expiresIn` 的语义是"从签发时间起的绝对秒数"还是"剩余有效秒数"。当刷新 token 时，`TokenRefreshResponse` 同样包含 `expiresIn`，但新 token 签发后立即返回，其理论值与登录响应中的同名字段语义一致。

**所在位置**：5.2 节 LoginResponse、TokenRefreshResponse；6.2 节 JSON 示例

**严重程度**：重要

**改进建议**：在 5.2 节 LoginResponse 定义下方补充注释，明确 `expiresIn` 为"access token 剩余有效秒数（从签发时计算）"，供前端用于预估下一次刷新时机。同步更新 6.2 节 JSON 示例字段注释。

---

### 🔵 一般

#### 12. `PUT /api/auth/password` 成功后 `tokenVersion` 递增对已登录 Access Token 的影响未说明

**问题描述**：3.4 节密码变更流程说明 `tokenVersion` 递增使旧 Refresh Token 失效。但已签发的 Access Token（15 分钟有效期）仍在 Filter 层每次查库验证 enabled/deleted，却不验证 tokenVersion。这意味着密码变更后，旧 Access Token 在剩余有效期内仍可使用，这是否符合预期未说明。

**所在位置**：3.4 节「密码变更强制策略」、4.2 节 tokenVersion 描述

**严重程度**：一般

**改进建议**：在 3.4 节或 4.2 节补充说明：Access Token 不检查 tokenVersion 是已知设计决策（短有效期 + 查库验证 enabled 已提供基本保护），非疏忽。

---

#### 13. `<script setup>` 或 Options API 使用约定未说明

**问题描述**：前端修复方案（8.2 节）提到了 Pinia store、`getCurrentInstance()` 等 Vue 3 Composition API 特性，但未说明前端代码的统一风格约定（是否使用 `<script setup>`、是否使用 Composition API vs Options API）。`getCurrentInstance()` 的使用方式对上下文敏感，若项目统一使用 `<script setup>` 则 `getCurrentInstance()` 在 setup 外部不可用。

**所在位置**：8.2 节 H6 修复方案

**严重程度**：一般

**改进建议**：在 8.2 节开头补充前端代码风格约定说明，或在 H6 修复方案中将 `getCurrentInstance()` 示例替换为更适合项目的注入模式。

v6 文档在结构完整性和关键安全设计深化方面已达到较高水平，多数要求的子项均有对应章节覆盖。主要改进空间集中在三个维度：

## 问题 1：登录流程未定义账户锁定时的返回行为

**问题描述**：Section 3.1.1 登录流程步骤 2 执行了 LoginAttemptTracker 的锁定状态检查，但未定义检查命中时（即账户已被锁定）应返回什么 ErrorCode 和 HTTP 状态码。步骤 3 直接跳转到用户加载，暗示即使锁定也继续执行。对比 Section 3.1.3 Token 刷新流程步骤 6 明确描述了"若已锁定，返回 TOKEN_REFRESH_FAILED 错误并强制前端清除本地 token"，登录流程的对应处理完全缺失。

**所在位置**：3.1.1 节步骤 2-3；4.1 节第三层描述"AuthServiceImpl.login() 在第一步即检查锁定状态"

**严重程度**：重要

**改进建议**：在步骤 2 后补充锁定命中分支：若用户名已锁定，返回 ErrorCode.ACCOUNT_LOCKED（HTTP 429），消息"账户已锁定，请 15 分钟后重试"，无需继续步骤 3-11。同时澄清是否需同步检查 IP 维度锁定（见问题 2）。

## 问题 2：IP 维度锁定检查未在登录流程中显式体现

**问题描述**：Section 4.1 第二层登录失败计数表定义了 IP 维度的锁定条件（同一 IP 20 次失败后锁定 30 分钟），第三层描述"AuthServiceImpl.login() 在第一步即检查锁定状态"暗示应在登录入口处检查双维度锁定。但 Section 3.1.1 登录流程步骤 2 仅检查"该用户名是否处于临时锁定状态"，未提及 IP 维度的锁定检查。这意味着 IP 维度锁定虽然在设计中定义，但在实际登录流程中可能被绕过——IP 被锁定后，攻击者可切换用户名继续尝试。

**所在位置**：3.1.1 节步骤 2；4.1 节第二层锁定表 + 第三层描述

**严重程度**：重要

**改进建议**：在 3.1.1 节步骤 1（速率限制）和步骤 2（用户名锁定检查）之间或在步骤 2 中补充 IP 维度的锁定检查，明确 LoginAttemptTracker 在登录入口处同时校验用户名和 IP 双维度的锁定状态。对于步骤 5（用户名不存在）的场景，IP 维度锁定仍应被检查，说明此时仅检查 IP 维度（因无有效用户名）的逻辑。

## 问题 3：M17/M18/M19 未出现在缺陷追踪表中

**问题描述**：原始需求的问题清单包含 19 个一般问题（M1-M19），Section 8.1（后端）和 8.2（前端）的追踪表覆盖了 M1-M16，但以下 3 个条目不在此追踪表中：

- M17 Actuator 端点暴露（设计方案在 4.4 节以 denyAll 处理，但未追踪）
- M18 401 响应体结构不一致（设计方案在 6.3 节和 10.2 节统一了格式，但未追踪）  
- M19 用户不存在时菜单返回空数组（设计方案通过 JwtAuthenticationFilter 静默跳过+ExceptionTranslationFilter 间接修复为返回 401，但未追踪且未显式说明行为变更）

虽然三个问题的设计方案已分散体现，但追踪表的遗漏意味着实施阶段可能遗漏回归验证。

**所在位置**：8.1 节（后端追踪表）、8.2 节（前端追踪表）、8.3 节（包 A 追踪表）

**严重程度**：一般

**改进建议**：在 8.1 节补充三条目的追踪行（M17/M18/M19），包含当前状态、修复方案、潜在副作用、影响范围四列。对于 M19 尤其需要显式说明行为变更（从"返回空数组"变为"返回 401"）以消除实施者的困惑。

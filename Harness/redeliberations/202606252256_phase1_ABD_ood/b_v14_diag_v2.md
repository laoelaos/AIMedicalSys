## 二、事实错误与逻辑矛盾

### 2.1 ACCOUNT_LOCKED 错误消息与 ErrorCode 表不一致（重要）

- **位置**：3.1.1 节步骤 3 vs 10.2 节 ErrorCode 表
- **问题描述**：3.1.1 节步骤 3 为 IP 维度锁定和用户名维度锁定分别定义了不同的错误消息（"请 30 分钟后重试" vs "请 15 分钟后重试"），但 10.2 节 ErrorCode 表中 `ACCOUNT_LOCKED` 仅有一条固定消息"请 15 分钟后重试"。两处描述在语义上矛盾——ErrorCode 表暗示消息是固定的枚举属性，而流程描述暗示消息动态依赖于锁定维度。
- **严重程度**：重要 — 实现者若以 ErrorCode 表为唯一参考，IP 锁定场景将返回错误的锁定时长消息
- **改进建议**：二选一统一：(a) 在 10.2 节为 ACCOUNT_LOCKED 新增"账户已锁定，请 30 分钟后重试"消息行对应 IP 维度，并在消息字段标注"根据锁定维度返回对应消息"；(b) 在 10.2 节补充注释说明"ACCOUNT_LOCKED 消息根据锁定维度动态生成"

### 2.2 tokenVersion 比对步骤存在设计歧义（重要）

- **位置**：3.1.3 节步骤 5 vs 步骤 9
- **问题描述**：步骤 5 从 DB 加载用户（含 tokenVersion），步骤 9 要求"验证 Refresh Token claims 中的 tokenVersion 与 DB 中当前 tokenVersion 一致"。表述未明确步骤 9 是从 DB 重新查询 tokenVersion，还是复用步骤 5 已加载的实体字段值。若复用步骤 5 的值，则存在细竞态窗口——步骤 5 至步骤 9 之间若其他线程/实例完成密码变更，旧 Refresh Token 仍可能通过验证；若重新查询，则步骤 5 的 tokenVersion 加载存在冗余。
- **严重程度**：重要 — 设计歧义直接影响实现决策，不同实现行为差异影响密码变更即时生效的安全保证
- **改进建议**：明确步骤 9 的实现决策：(a) 若复用步骤 5 的实体字段值（允许微竞态），需在设计决策表记录此权衡及理由（并发窗口极窄、影响仅限于真正同时发生密码变更的极端场景）；(b) 若重新查询以保证强一致性，则步骤 9 改为"重新从 DB 加载用户当前 tokenVersion 并比对"，同时移除步骤 5 中 tokenVersion 的预加载以消除冗余

## 三、深度与完整性评估

### 3.1 `UserFacade` 门面接口缺少完整定义（严重）— 同 v1 诊断

- **位置**：2.2 节依赖规则（用户数据访问途径段）、1.3 节核心抽象一览、2.1 节目录结构
- **问题描述**：2.2 节引入了 `UserFacade` 作为业务模块访问用户数据的门面接口，但存在以下完整性问题：
  - 未出现在 1.3 节「核心抽象一览」表中，与同属 `common-module-api` 的 `CurrentUser` 形成对比（后者已收录）
  - 未出现在 2.1 节目录结构中
  - 未定义任何方法签名、返回类型、参数列表
  - 未说明与 `CurrentUser` 的职责分工（都是业务模块获取用户数据的途径）
- **严重程度**：严重 — 实施者无法据此编码门面接口
- **改进建议**：
  - 在 1.3 节补充 `UserFacade` 条目
  - 在 2.1 节 `common-module-api` 目录树中增加 `UserFacade.java`
  - 定义至少 2-3 个方法签名
  - 补充 `UserFacade` 与 `CurrentUser` 的分工说明

### 3.2 `PasswordChangeCheckFilter` 白名单路径匹配策略未定义（重要）— 同 v1 诊断

- **位置**：3.3 节 PasswordChangeCheckFilter 行为契约
- **严重程度**：重要
- **改进建议**：明确指定使用 `AntPathRequestMatcher` 进行路径匹配，并在 SecurityConfig 中给出 Filter 注册的示例代码段

### 3.3 `MenuUpdateRequest` 的 PATCH 语义与 Java record 反序列化存在歧义（重要）— 同 v1 诊断

- **位置**：5.2 节 MenuUpdateRequest
- **严重程度**：重要 — 见 v1 诊断详细分析
- **改进建议**：二选一：(a) 改用传统 POJO + 字段标记组；(b) 正式声明降级为全量替换语义（PUT）

### 3.4 Refresh 端点在禁用用户场景不递增失败计数（重要）— 同 v1 诊断

- **位置**：3.1.3 节步骤 7
- **严重程度**：重要 — 见 v1 诊断详细分析
- **改进建议**：在步骤 7 中增加 IP 维度的失败计数递增

### 3.5 登录成功后 IP 维度计数器清空前置条件未定义（一般）

- **位置**：3.1.1 节步骤 8
- **问题描述**：步骤 8 要求"清除该请求来源 IP 的失败计数"，但未说明清除操作的前置假设。在 NAT/代理共享 IP 场景下，用户 A 登录成功将清除同一公网 IP 下所有用户的 IP 维度失败计数，可导致用户 B 的 IP 锁定被意外绕过。此场景属于 IP 维度防暴力破解的已知局限性，应在设计决策表中记录。
- **严重程度**：一般 — 在私有网络/Direct IP 场景下影响有限；NAT 场景下需要在前置负载均衡或反向代理层面区分真实客户端 IP
- **改进建议**：在 11 节设计决策表增加一行记录此局限性，或补充说明"IP 维度的失败计数以请求来源 IP 为准（若经过反向代理，应取 `X-Forwarded-For` 或 `X-Real-IP` 头，由 `GlobalRateLimitFilter` 和 `LoginAttemptTracker` 统一从 `RequestHelper.getClientIp()` 获取）"

### 3.6 `ProfileUpdateRequest.phone` 可选性未声明（一般）— 同 v1 诊断

- **位置**：5.2 节 ProfileUpdateRequest
- **严重程度**：一般
- **改进建议**：在 `phone` 和 `email` 字段补充注释说明"可选字段"

### 3.7 `PasswordChangeRequest.oldPassword` 缺少约束（一般）— 同 v1 诊断

- **位置**：5.2 节 PasswordChangeRequest
- **严重程度**：一般
- **改进建议**：为 `oldPassword` 补充 `@Size(max = 128)` 约束

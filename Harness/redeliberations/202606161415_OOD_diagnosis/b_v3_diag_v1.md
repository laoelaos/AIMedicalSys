## 问题 1："偏离路线图"维度仅覆盖内容溢出，未覆盖内容遗漏

- **所在位置**：第四章"偏离路线图 Phase 0 阶段"（问题 5、6）
- **严重程度**：中等
- **问题描述**：
  用户需求中"偏离路线图 Phase 0 阶段"维度的检查范围明确包含两方面："超出 Phase 0 范围的内容"与"遗漏 Phase 0 应包含的内容"。但报告仅识别了超出范围的问题（问题 5：完整 DTO 契约属于超前冻结；问题 6：DataScopeType 枚举值属于 Phase 1 预设计），未对"遗漏"方向做任何检查说明。

  例如，路线图第 38 行要求"接口契约框架"（统一响应包装、分页、错误码命名空间、AI 能力占位类型等），第 42 行要求"持续集成占位"（CI 流水线配置文件就位），第 39 行要求"数据与权限模型骨架"——报告应说明 OOD 文档是否覆盖了全部 Phase 0 必备项，或至少声明已检查且无遗漏。当前报告没有任何关于"遗漏检查"的结论，使得该维度的覆盖不完整。

- **改进建议**：
  在第四章末尾增加一段说明，明确：(1) 已完成 OOD 文档与路线图 Phase 0 任务清单（路线图第 57–64 行）的逐一对照检查；(2) 检查结论——是否有遗漏，或确认覆盖完整。具体格式可为：
  > **遗漏检查**：已对照路线图第 57–64 行逐一核实，OOD 文档覆盖了搭建任务 A–H 对应的所有设计要素（多模块骨架→第 2 章、数据库骨架→第 7 章、AI Mock→第 3.4 节/第 8 章、CI 占位→第 9 章等），未发现遗漏。

## 问题 2：问题 5 推荐方案缺少字段选择标准，执行者无法独立决策

- **所在位置**：问题 5 结论部分（第 136 行）
- **严重程度**：中等
- **问题描述**：
  报告推荐方案 A + 子方案 A2，要求"保留空壳 DTO 类声明"并为"Phase 0 保留 `TriageRequest.patientId`、`TriageResponse.level`、`TriageRequest.symptoms` 等核心字段作为 Mock 占位字段子集"。但报告未定义"核心字段"的选择标准或判定方法。

  执行者面对 13 组 DTO（每组可能有 5–20 个字段），需要自行判断哪些字段属于"Phase 0 核心"。不同执行者可能做出不同选择，导致 Phase 0 实现中出现多种版本的字段保留策略，影响后续阶段的字段重构。

- **改进建议**：
  补充字段选择的判定标准，例如：
  - 对 Phase 0 前端数据绑定验证必需的字段（如 ID 字段、名称字段、状态枚举字段）
  - 对后续阶段不会发生结构性变化的字段（如基础 ID、创建时间）
  - 对后续阶段可能新增但不会改变现有含义的字段
  或更具体地给出"保留规则"（如：仅保留 String/枚举/数值等基础 JSON 类型字段，排除嵌套 DTO 引用，因为嵌套对象在空壳 DTO 下无字段可展示）。

## 问题 3：优先级依赖关系描述内部不一致

- **所在位置**：第六章"汇总与修复优先级"（第 169 行、第 178 行）
- **严重程度**：中等
- **问题描述**：
  主表中 P0 条目（第 169 行）明确标注"无前置依赖，不依赖其他问题修复"。但依赖关系说明段落（第 178 行）指出"需在子方案 A2 确认后再实施 P1 的文档文本修订"。这意味着 P1 中的文档文本修订部分依赖 P0 决策。

  这两个陈述存在方向性矛盾：如果 P0"无前置依赖"但 P1 需要等待 P0 输出，则 P1 与 P0 之间存在一条单向依赖链，主表应标注"涉及文档文本修订的 P1 项有前置依赖"而非笼统说"无前置依赖"。当前表述会误导执行者认为 P0/P1 可完全并行推进，直到实施 P1 文本修订时才发现需要等待 P0。

- **改进建议**：
  将 P0 条目的"无前置依赖"修正为更精确的描述，如：
  > 方案决策无前置依赖，可立即启动。但 P1 中涉及第 304–307 行依赖描述文本修订的问题（1、3）需待 P0 子方案 A2 确认后方可实施，以避免文档文本在不同版本间被重复修改。

## 问题 5：问题 4 归类为"逻辑错误"不够准确

- **所在位置**：第三章"逻辑错误"（问题 4，第 64–76 行）
- **严重程度**：轻微
- **问题描述**：
  问题 4 描述的是 FallbackAiService 在配置异常（`ai.mock.enabled=false` 且无可用的非 Fallback 实现）时的兜底行为缺乏日志告警。OOD 文档在第 658 行明确描述了这一兜底行为（"不抛出异常，直接返回 AiResult 实例"），文档自身不存在逻辑错误。报告中指出的"没有任何日志或启动警告提示运维人员"属于**改进建议**层面，而非**文档逻辑错误**。

  按照任务维度的定义，逻辑错误应指"推理链条断裂、前提与结论不匹配、遗漏关键因果关系"。这里的核心问题是"功能实现缺少告警机制"，属于设计完善度问题，更适合放在改进建议或非功能性缺陷类别中。

- **改进建议**：
  将问题 4 的类别调整为"改进建议"或"设计完善度"，而非"逻辑错误"。如果坚持归类为逻辑错误，需明确说明 OOD 文档的哪部分逻辑链条存在断裂——例如可论证"文档在描述兜底行为时遗漏了可观测性的关键环节"。或者，将其归入"逻辑错误"时，与问题描述的对应关系应在报告中显式说明。

## 问题 6：报告中对照文档包含 `Docs/01_requirement.md`，超出任务指定的参考范围

- **所在位置**：报告文件头（第 4 行）
- **严重程度**：轻微
- **问题描述**：
  用户需求明确指定了两份参考文档：`Docs/04_ood_phase0.md`（被诊断文档）和 `Docs/03_roadmap.md`（对照文档）。报告文件头将对照文档扩展为 `Docs/03_roadmap.md` 和 `Docs/01_requirement.md` 两份。

- **改进建议**：
  (1) 在报告文件头的对照说明中明确标注 `Docs/01_requirement.md` 是附加参考而非任务指定文档；或 (2) 将 `Docs/01_requirement.md` 的使用限定在"补充说明"而非正式对照依据；或 (3) 如果报告中的所有判断均不依赖于 `Docs/01_requirement.md`，则将其从文件头中移除。

建议在下一迭代中优先修复问题 1（补充遗漏检查）和问题 2（补充字段选择标准），这两个问题直接影响执行者的决策质量。

## 发现的问题

### 问题 1：问题 5 方案 A 遗漏对 MockAiService 占位数据约定冲突的分析（关键遗漏）

- **所在位置**：第 99–115 行（问题 5，方案 A 描述及结论）
- **严重程度**：中等
- **问题描述**：
  方案 A 建议将 Phase 2~5 的 DTO 缩减为仅含类声明的空壳，声称"MockAiService 在 Phase 0 只需用 `new XXXDto()` 返回占位数据即可"（第 115 行）。但被诊断文档 `Docs/04_ood_phase0.md` 第 623–629 行定义了详细的 Mock 数据占位约定——包括字符串字段填 `"mock_" + 字段名`、数值字段填 0/1、枚举填中性值、嵌套 DTO 递归填充等——这些约定全部依赖字段级元信息（字段名、类型、可空标注）。若 DTO 被缩减为空壳（零字段），MockAiService 将丧失填充任何具含义占位数据的能力，只能返回所有字段为 null 的 DTO 实例。这与 Phase 0 路线图的"搭建任务 F：AI 能力模块 Mock 占位实现"的预期产出存在显著偏差——前端开发团队将收到无任何字段结构的 Mock 响应，无法验证数据绑定和 UI 展示逻辑。

  诊断报告未识别这一冲突，也未评估方案 A 对 MockAiService 设计的具体影响，导致执行者拿到方案 A 后需自行发现并解决该矛盾，降低了建议的可操作性。
- **改进建议**：在问题 5 的方案 A 中补充以下分析：
  1. 明确指出空壳 DTO 与第 623–629 行 Mock 占位约定的冲突
  2. 给出具体应对选项，例如：
     - **子方案 A1**：同步缩减 Mock 数据占位约定（将第 623–629 行改为"Phase 0 Mock 返回所有字段为 null/default 的空 DTO"），明示前端团队在 Phase 0 不依赖字段结构进行 UI 验证
     - **子方案 A2**：为 Phase 0 保留少量关键字段（如 `TriageRequest.patientId`、`TriageResponse.level` 等真正需要 Mock 填充的字段）作为"Phase 0 Mock 字段子集"，其余字段保持空壳，在 DTO 类中以注释标记哪些属于 Phase 0 占位字段
     - **子方案 A3**：维持当前空壳原则，在 MockAiService 中通过 `org.springframework.beans.BeanUtils.instantiateClass()` + `@MockBean` 级别的数据回填机制独立维护 Mock 数据生成，与 DTO 字段结构解耦
  3. 评估每个子方案对前端开发和验收的影响

### 问题 2：问题 4 改进建议仅提供被动检测方案，范围偏窄

- **所在位置**：第 71–72 行（问题 4 的建议）
- **严重程度**：轻微
- **问题描述**：
  对 FallbackAiService 自排除后委托为空的兜底场景，仅建议"输出 WARN 级别日志"。这是运行时被动检测——仅在 AI 调用发生时才会触发日志输出，调用前无人知晓系统处于全降级状态。未讨论启动期主动验证方案（如 `@PostConstruct` 中检查委托列表是否为空并立即输出 ERROR 日志或抛 `IllegalStateException` 快速失败）作为替代或补充。在 Phase 0 骨架阶段虽可接受，但建议的覆盖面不足以让执行者了解可选方案空间。
- **改进建议**：
  在建议中增加启动期验证方案作为可选增强：
  - FallbackAiService 增加 `@PostConstruct` 方法，在应用启动时检查排除自身后的委托列表 `delegateAiServiceList` 是否为空，若为空则输出 `ERROR` 级别日志（明确提示"未找到可用的 AiService 实现，本实例将始终返回降级结果"）
  - 评估是否需要在 Phase 0 采用 fail-fast（启动抛异常）还是 degrade-gracefully（WARN + 运行时降级）策略，给出推荐

### 问题 3：问题 1 的修复建议未考虑 security 依赖在 common 中的实际用途说明的兼容更新

- **所在位置**：第 18–19 行（问题 1 的建议）
- **严重程度**：轻微
- **问题描述**：
  建议仅提及"移除 common 模块对 spring-boot-starter-security 的依赖声明"，但被诊断文档第 292 行同时描述了该依赖的用途说明（"用于可复用的通用安全类型"）。若仅移除 POM 声明而未同步删除或修正用途说明，文档会出现"声明了用途但不依赖对应库"的新矛盾。该问题虽小，但足以影响一次修复的完整性。
- **改进建议**：在建议中补充："同步删除第 292 行中 `spring-boot-starter-security（用于可复用的通用安全类型）`的用途说明，或将其改为注释指出该依赖已移除、各模块按需声明"。在汇总表中与问题 3（依赖传播描述修正）共同标注涉及第 304–307 行的文本修订，统一排入 P1 一并处理。

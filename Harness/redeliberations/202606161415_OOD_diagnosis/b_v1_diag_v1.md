# 质量审查报告：OOD Phase 0 文档诊断报告（a_v1_diag_v2.md）

> 被审查产出：`a_v1_diag_v2.md`
> 审查视角：需求响应充分度、事实准确度、深度与完整性、可操作性

## 一、需求响应充分度

### 问题 1：缺失"事实错误"检查维度（严重）

**问题描述**：任务需求明确要求从 4 个维度诊断 OOD 文档——(1) 定义矛盾、(2) 事实错误、(3) 逻辑错误、(4) 偏离路线图 Phase 0 阶段。当前报告覆盖了维度 1、3、4，但完全缺失维度 2"事实错误"。

被诊断的 OOD 文档（`Docs/04_ood_phase0.md`）包含大量可核查的技术事实陈述，例如：
- 第 292 行声明 common 模块依赖 `spring-boot-starter-security` 的原因描述
- 第 304–307 行关于 common 模块的三个 Starter 依赖的传播决策
- 第 780–785 行关于安全类型放置位置的断言
- 第 1414 行关于 H2 依赖 scope 切换策略的说明

这些陈述均属于事实性声明，应被逐一核查。当前诊断未涉及任何事实错误检查，构成对任务要求的实质性遗漏。

**所在位置**：报告中无对应章节
**严重程度**：严重
**改进建议**：补充"事实错误"检查章节，逐项验证 OOD 文档中的技术事实陈述是否准确（尤其是 Maven 依赖传播、Spring Boot 配置机制、JPA 行为等层面的陈述）。

## 二、诊断深度与完整性

### 问题 2：问题 5 的改进建议忽略编译期依赖约束（中等）

**问题描述**：问题 5 将 Section 8.2 的完整 DTO 定义判定为"偏离路线图 Phase 0 阶段"，严重程度标注为"严重"。其改进建议为："Section 8.2 缩减为 Phase 0 实际需要的 AI 能力占位部分（仅包括 `AiService` 接口方法签名骨架和 `MockAiService` 涉及的少数核心 DTO）"。

然而，OOD 文档第 882–888 行明确将 `AiService` 接口设计为**类型化方法签名**模式：
```
CompletableFuture<AiResult<TriageResponse>> triage(TriageRequest request);
```

这意味着：
- 所有 13 个方法的输入/输出 DTO 类型（如 `TriageRequest`、`TriageResponse`）必须在编译期存在
- 审计中未说明"仅保留少数核心 DTO"后，哪些能力的 DTO 被保留、哪些被移除、被移除能力的 `AiService` 方法签名如何调整
- 如果仅移除字段级细节而保留 DTO 类壳，实质与移除外层壳的差异不大，仍需回答到底哪些字段留在 Phase 0 哪些字段可以延迟
- 如果改用泛型/Object 签名以消除 DTO 类型依赖，则改变了 `AiService` 接口形态，这与报告认定"接口方法签名骨架"应保留的立场矛盾

该改进建议未考虑上述编译期依赖约束，导致修复者无法依据该描述直接采取行动。

**所在位置**：报告第 63–71 行（问题 5 改进建议）
**严重程度**：中等
**改进建议**：补充编译期依赖约束分析，给出至少两种可行方案并评估其权衡：(a) 保持类型化签名但仅定义字段壳（占位 DTO 无业务约束），明确告知后期有重构成本；(b) 使用通用类型签名（如 `CompletableFuture<AiResult<?>>`），但需说明对前端类型安全的影响。

### 问题 3：问题 6 的改进建议缺少粒度区分（中等）

**问题描述**：问题 6 将 DataPermissionEvaluator 接口声明和 DataScopeType 枚举值定义一同归类为"偏离路线图 Phase 0 阶段"，改进建议要求全部移除，仅留一句话占位。

然而以下两个要素的性质不同：
- `DataPermissionEvaluator` **接口声明**：属于扩展点预留（evolvable skeleton），这与路线图 Phase 0 0.2 节"可演进"目标一致——"骨架预留 AI 能力抽象层、权限模型扩展点、微服务化拆分演进路径"
- `DataScopeType` **枚举值**（`SELF_OWNED`、`SELF_HANDLED`）：属于接口契约冻结，与路线图 §0.4 "模块级接口契约冻结（在对应阶段启动前冻结）"冲突

将二者不加区分地全部移除是过度修正，等同于否定了 Phase 0 "可演进"的设计目标。此外，需求文档 §2.6 明确将 ○¹ 和 ○² 定义为数据权限语义，完全移除 DataPermissionEvaluator 接口会导致需求到实现的可追溯链条断裂。

**所在位置**：报告第 75–81 行（问题 6 改进建议）
**严重程度**：中等
**改进建议**：区分对待——保留 `DataPermissionEvaluator` 接口声明（作为扩展点预留），仅移除 `DataScopeType` 枚举值和接口方法体中的具体约定。同时补充说明该扩展点与需求 §2.6 ○¹/○² 的追溯关系。

---

### 问题 4：问题 2 的改进建议缺乏可行性与副作用评估（中等）

**问题描述**：问题 2 指出纯接口模块（common-module-api、ai-api）通过 transitive 依赖获得 `spring-boot-starter-data-jpa` 全量依赖，与"轻量接口契约"定位矛盾。改进建议给出两个选项：将 data-jpa 标注为 `<optional>true</optional>`，或新建独立 `common-jpa` 子模块。

报告未评估两个选项的副作用：
1. **标注 optional**：所有业务模块（patient/doctor/admin）必须在各自 POM 中显式声明 data-jpa 依赖，增加 POM 维护负担，且存在模块遗漏声明导致编译错误的风险
2. **新建 common-jpa 子模块**：新增一个 Maven 子模块，影响父 POM 聚合、CI 流水线分阶段构建（需调整 5 个构建阶段中的基础层模块列表）、IDE 模块加载等
3. 报告未指明在 OOD 文档所述的所有业务模块"均包含 JPA 实体和 Repository"（第 306 行）的前提下，选项 2 的额外模块隔离是否真正必要

**所在位置**：报告第 27–29 行（问题 2 改进建议）
**严重程度**：中等
**改进建议**：补充每个选项的副作用评估和变更影响范围说明，或基于 OOD 文档上下文给出明确建议（而非并列选项），使修复者能直接决策。

## 三、分类准确性

### 问题 5：问题 3 的分类不匹配（轻微）

**问题描述**：问题 3（依赖关系图符号歧义）被归类为"定义矛盾"，但实际描述的是 ASCII 图的可读性问题——ai-api 在图中出现两次，阅读者易误判为存在循环依赖。这属于**文档表达清晰度**问题，与"定义矛盾"（定义不一致或自相矛盾）无关。

强行归入"定义矛盾"类别可能导致后续梳理问题时，修复者误以为该问题与模块定义冲突同类，影响问题排序和修复优先级判断。

**所在位置**：报告第 33–38 行（问题 3）
**严重程度**：轻微
**改进建议**：重新归类为"逻辑错误"（图形表达不清晰导致读者产生逻辑误判）或新增"文档质量"类别。如认为 ASCII 图表达本身的歧义属逻辑问题，应在问题描述中明确定义为什么归入该类别。

## 四、可操作性评估

### 问题 6：缺乏明确的修复优先级排序（轻微）

**问题描述**：报告末尾的汇总表按严重程度排序（严重→中等→轻微），但未给出**实施顺序建议**或**依赖关系说明**。从修复者视角看，以下信息缺失：
- 哪些问题修复存在先后依赖（如问题 5 的 Section 8.2 改完后再修复问题 1、2 的 POM 依赖矛盾？还是独立并行修复？）
- 哪些问题存在自愈可能（如问题 3 的 ASCII 图优化，可以在解决其他问题时顺便调整）
- 大多数问题修复需要接触 POM 或代码文件，报告中未给出具体的文件路径

虽然单个问题已有足够描述，但缺乏整体执行指引，会降低修复者在动手前的决策效率。

**所在位置**：报告第 87–95 行（汇总表）
**严重程度**：轻微
**改进建议**：在汇总表基础上补充"修复依赖"和"建议顺序"维度，或增加一段文字说明高优修复事项及并行关系。

## 五、综合质量评价

本诊断报告对 OOD 文档的定义矛盾和路线图偏离两个维度的分析较为深入，各问题均提供了位置标注和改进建议。但存在以下结构性不足：

| 维度 | 评价 |
|------|------|
| 需求响应完整性 | **缺失事实错误维度**，覆盖 3/4 |
| 建议可操作性 | **中等**，3 处建议未考虑编译期约束或副作用 |
| 分类准确性 | **中等**，1 处明显分类不匹配 |
| 优先级指导 | **欠缺**，仅有严重程度分级，无实施排序 |

主要风险在于：修复者拿到报告后，对问题 2、5、6 的改进建议难以直接行动，需自行补充分析。建议在下一轮迭代中优先补充上述缺失的深度分析。

### 问题 1（中等）：问题 5 方案 A 遗漏对 MockAiService 占位数据约定冲突的分析

- **所在位置**：a_v2_diag_v1.md 第 99–115 行（问题 5，方案 A 描述及结论）
- **问题描述**：方案 A 建议将 Phase 2~5 的 DTO 缩减为仅含类声明的空壳，声称"MockAiService 在 Phase 0 只需用 `new XXXDto()` 返回占位数据即可"。但被诊断文档 `Docs/04_ood_phase0.md` 第 623–629 行定义了详细的 Mock 数据占位约定——包括字符串字段填 `"mock_" + 字段名`、数值字段填 0/1、枚举填中性值、嵌套 DTO 递归填充等——这些约定全部依赖字段级元信息。若 DTO 被缩减为空壳（零字段），MockAiService 将丧失填充任何具含义占位数据的能力，只能返回所有字段为 null 的 DTO 实例。这与 Phase 0 路线图的"搭建任务 F：AI 能力模块 Mock 占位实现"的预期产出存在显著偏差——前端开发团队将收到无任何字段结构的 Mock 响应，无法验证数据绑定和 UI 展示逻辑。诊断报告未识别这一冲突，也未评估方案 A 对 MockAiService 设计的具体影响，降低了建议的可操作性。
- **改进建议**：在问题 5 的方案 A 中补充以下分析：
  1. 明确指出空壳 DTO 与第 623–629 行 Mock 占位约定的冲突
  2. 给出具体应对选项，例如：
     - **子方案 A1**：同步缩减 Mock 数据占位约定（将第 623–629 行改为"Phase 0 Mock 返回所有字段为 null/default 的空 DTO"），明示前端团队在 Phase 0 不依赖字段结构进行 UI 验证
     - **子方案 A2**：为 Phase 0 保留少量关键字段（如 `TriageRequest.patientId`、`TriageResponse.level` 等真正需要 Mock 填充的字段）作为"Phase 0 Mock 字段子集"，其余字段保持空壳，在 DTO 类中以注释标记哪些属于 Phase 0 占位字段
     - **子方案 A3**：维持当前空壳原则，在 MockAiService 中通过 `BeanUtils.instantiateClass()` + `@MockBean` 级别的数据回填机制独立维护 Mock 数据生成，与 DTO 字段结构解耦
  3. 评估每个子方案对前端开发和验收的影响

### 问题 2（轻微）：问题 4 改进建议仅提供被动检测方案，范围偏窄
